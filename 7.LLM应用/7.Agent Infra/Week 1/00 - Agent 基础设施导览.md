## 知识点速查表

| 概念                  | 核心要点                                    | 对 infra 工程师的意义                                |
| ------------------- | --------------------------------------- | --------------------------------------------- |
| Token 生成            | 每个 token 严格串行，一次前向传播                    | 延迟 = TTFT + (token 数 × 每 token 毫秒数)。解码阶段无法并行。 |
| Agent 循环            | 观察 → 思考 → 执行，直到收到停止信号                   | 这个循环是服务层最核心的抽象，其他一切都围绕它展开。                    |
| 工具调用协议              | `tool_use` 输出，`tool_result` 输入，通过 ID 关联 | 你所写的框架，正是负责拦截并执行这些消息的那一层。                     |
| 上下文窗口即内存            | 每次调用大模型时，完整对话记录都会被重新读取                  | Agent 跑得越久，上下文越大，KV Cache 压力也随之升高。            |
| System prompt 即基础设施 | 定义工具清单、人设、停止条件、安全规则                     | 用版本控制管理它，用 eval 验证它，把它当作一次部署来对待。              |
| 推理服务栈               | HTTP → 调度器 → KV Cache → GPU 权重          | 显存往往是瓶颈，而不是算力（FLOPS）。                         |
| KV Cache            | 为每个活跃上下文中的每个 token 保存注意力状态              | 高并发或长上下文下容易被填满，设计时需要提前考虑。                     |
| 连续批处理               | 在 token 边界处把新请求填充进 GPU 的空闲槽位            | 这正是现代推理服务器在解码串行的前提下，仍能保持高吞吐的秘诀。               |
|                     |                                         |                                               |
