# 工具调用的四层保险

## 理想路径只是个幻觉

当你通过大模型调用工具时，模型并没有真正"调用函数"。它只是生成*看起来像*函数调用的文本——一个带名称和参数的 JSON 对象。你的基础设施解析这段文本，然后分派真正的调用。整个系统的可靠性，完全取决于生成的文本是否有效、类型正确、语义通顺。

**完整的失效分类：**

![[gemini-svg (5).svg|814]]

注意，**语义错误**与其他三类是分开的。验证和约束解码可以捕获结构问题——但模型传对了参数、值却是*错误*的情况，对 schema 来说是不可见的。在整篇文章中，请牢记这种不对称性。

## 第一层：JSON Schema 验证 —— 它到底检查什么

JSON Schema 是一种声明式词汇，用于描述 JSON 值的形态。当你定义工具时，你就是在为其输入编写 JSON Schema。你的基础设施会在分派前，用 schema 验证模型的输出。

```json
{
  "name": "search_documents",
  "description": "Search the knowledge base for relevant documents",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "minLength": 1,
        "maxLength": 500,
        "description": "The search query string"
      },
      "top_k": {
        "type": "integer",
        "minimum": 1,
        "maximum": 20,
        "default": 5,
        "description": "Number of results to return"
      },
      "filters": {
        "type": "object",
        "properties": {
          "date_after": { "type": "string", "format": "date" },
          "source": { "type": "string", "enum": ["wiki", "docs", "tickets"] }
        },
        "additionalProperties": false
      }
    },
    "required": ["query"],
    "additionalProperties": false
  }
}
```

每个约束都在做实际工作：
- `additionalProperties: false` —— 捕获幻觉出来的参数名
- `source` 上的 `enum` —— 模型只能输出你指定的确切字符串，不能是听起来像的变体
- `top_k` 上的 `minimum`/`maximum` —— 捕获模型传 9999 或 -1 的情况
- `required: ["query"]` —— 捕获模型遗漏唯一必填字段的情况

**JSON Schema 验证无法捕获的：** 模型传了 `query: "the"` 而本应有实质内容的查询，或者传了 `top_k: 1` 而任务显然需要更多结果。这些是语义失效——结构上合法，逻辑上错误。

---

## 第二层：约束解码 —— 推理服务器如何保证输出合法

这是大多数人不知道存在的机制，也是推理层最强大的可靠性杠杆。

**标准生成：** 模型在每一步输出整个词表上的概率分布，然后从中采样。任何 token 都可能被采样，包括会破坏 JSON 结构的那些。

**约束解码：** 在每一步生成时，推理服务器拦截 logit 分布，把在当前解析状态下会产生非法延续的 token *置零掩码*。模型从根本上就不可能输出非法 token——不是事后抑制，而是在采样前就把它从候选中剔除。

机制通常是一个**有限状态机（FSM）**或语法，用于追踪当前解析状态：

![[Pasted image 20260402174727.png]]

实现这一点的推理服务器：
- **vLLM** — `guided_json` / `guided_grammar`
- **Outlines** —  pioneered 这一方法，与大多数后端集成
- **llama.cpp** — `--grammar`
- **Anthropic/OpenAI APIs** — 在服务端实现基于 schema 的约束工具调用

权衡：约束解码会增加少量每 token 开销（FSM 状态转移），非常复杂的 schema（大量嵌套对象）编译语法可能较慢。但对于你编写的大多数工具 schema，这个开销可以忽略不计。

**约束解码仍然无法修复的：** 模型选择了 `source: "wiki"` 而本应该选择 `"docs"`。两者都是合法的 enum 值。约束只消除了物理上不可能的输出——它并不能让模型在合法选项中做出明智的选择。

---

## 第三层：重试逻辑设计 —— 大多数工程师错在哪里

天真的实现就是一个裸 `try/except`，用同样的调用重试。这在三个方面都是错的：它没告诉模型错在哪里，没调整任何调用内容，还可能无限循环。

![[gemini-svg (6).svg|819]]

**把错误以 `tool_result` 的形式反馈，而不是新的 `user` 消息。** 当错误落在对话结构中正确的位置时，模型对自己正在尝试做什么有更多上下文：

```python
# 工具调用的四层保险
"Your last response was invalid JSON. Please try again."

# 工具调用的四层保险
{
  "role": "tool",
  "tool_use_id": "call_abc123",
  "content": (
    "Tool call failed schema validation. "
    "Error: 'top_k' must be an integer, got string '\"five\"'. "
    "The valid range is 1-20. "
    "Your output was: {\"query\": \"...\", \"top_k\": \"five\"}"
  )
}
```

糟糕的版本什么都没告诉模型。良好的版本给出了确切的字段名、确切的错误值、以及违反的约束。当错误信息精确时，模型通常能在第一次重试时纠正。

**关键设计决策：**

- **最多重试 2–3 次，不要无限。** 如果模型对同一个工具调用失败三次，再重试几乎不会有帮助。结构性问题：工具描述模糊、schema 令人困惑、或者任务需要别的工具。此时应上报。
- **指数退避仅用于瞬态错误。** 区分模型错误（立即重试，反馈错误）和基础设施错误如限流（退避后重试，不添加到对话中）。
- **不要重试语义错误。** 如果模型传了合法参数但工具返回"未找到结果"或"权限不足"，这不是重试场景——而是模型需要推理并选择不同行动的结果。

## 第四层：提示词层面 —— 你的第一道防线

在上述所有机制运行之前，模型的生成已经受到你描述工具方式的影响。

```python
# 工具调用的四层保险
{
  "name": "create_ticket",
  "description": "Create a ticket",
  "input_schema": {
    "type": "object",
    "properties": {
      "priority": {"type": "string"},
      "assignee": {"type": "string"}
    }
  }
}

# 工具调用的四层保险
{
  "name": "create_ticket",
  "description": (
    "Create a support ticket. Use this when the user needs something tracked "
    "or escalated. Do NOT use for questions you can answer directly. "
    "Priority must be exactly one of: 'low', 'medium', 'high', 'critical'. "
    "Assignee must be a team name like 'backend' or 'infra', not a person's name."
  ),
  "input_schema": {
    "type": "object",
    "properties": {
      "priority": {
        "type": "string",
        "enum": ["low", "medium", "high", "critical"],
        "description": "Urgency level. Use 'critical' only for production outages."
      },
      "assignee": {
        "type": "string",
        "enum": ["backend", "infra", "frontend", "data"],
        "description": "Team responsible. Do not use individual names."
      },
      "title": {
        "type": "string",
        "maxLength": 100,
        "description": "One-line summary of the issue."
      }
    },
    "required": ["priority", "assignee", "title"],
    "additionalProperties": false
  }
}
```

`enum` 约束身兼两职：它通过描述语义上引导模型选择合法值，又通过 schema 验证/约束解码机械地强制执行。描述中明确的"do NOT use for"可以减少当简单动作就能解决问题时，模型幻觉出工具调用的情况。

## 完整的可靠性防护栈

| 层级 | 能捕获什么 | 会遗漏什么 | 何时运行 |
|---|---|---|---|
| 工具描述 + 字段描述 | 引导模型选择合法值 | 什么都不捕获——它只是概率性的 | 生成前 |
| 约束解码 | 格式错误的 JSON、错误类型、非法 enum | 语义错误、合法范围内的错误值 | 生成中 |
| JSON Schema 验证 | 所有结构错误（事后） | 语义错误 | 生成后 |
| 错误反馈重试 | 可恢复的结构错误 | 持续的语义失败 | 失败时 |
| 最大重试 + 硬失败 | 不可恢复的失败 | N/A —— 上报 | N 次失败后 |

这些层级是**互补而非冗余的。** 你需要全部。只有 schema 验证的系统会比重试更多。有约束解码但无 schema 验证的系统是在信任推理服务器没有 bug。没有重试逻辑的系统会把每个瞬态模型抖动都变成用户可见的失败。

---

## 语义错误的缺口

语义错误对所有这些层级都是不可见的。一个模型调用 `delete_file` 传 `path: "/tmp/cache"`，而它本应该用 `path: "/tmp/cache/old"`，会通过所有检查并造成实际问题。

对语义错误的防御不是机械性的——而是：
1. **谨慎的工具设计** —— 缩小范围、尽可能使用幂等操作、对破坏性动作设置显式确认步骤
2. **幂等性** —— 设计工具使得重试产生相同结果；这让意外重复调用变得安全
3. **日志记录** —— 在生产中记录实际传递的参数，以便事后检测和调查语义失败
