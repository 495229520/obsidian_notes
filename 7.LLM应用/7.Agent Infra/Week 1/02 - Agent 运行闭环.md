# Agent 运行闭环

普通的大模型只是接收输入、产生输出——一次性的。而 **Agent** 把大模型包装在一个循环里，让它能够执行操作、观察结果，并在任务完成前持续运行。

---

## 核心循环：观察 → 思考 → 执行

Agent 循环有三个环节在不断重复，直到任务完成：

**思考。** 用完整上下文调用大模型，它要么输出一个工具调用（结构化的 JSON，告诉框架该做什么），要么给出最终答案。本质上仍是 token 生成——"智能"完全体现在输出格式上。

**执行。** 实际执行工具的**是框架，而不是大模型**。大模型本身无法直接访问互联网、文件或 API，它只能*描述*自己想做什么，由周围的系统来执行。

**观察。** 工具结果被追加到上下文窗口，作为新的消息。此时大模型拥有了更长的对话记录：原始任务 → 第一次推理 → 工具调用 → 结果 → …… 每轮循环都会让这段记录越来越长。

## 关键特性

**大模型在调用之间没有记忆。** Agent 唯一的状态就是能装进上下文窗口的内容。每次调用大模型，都会从头重新读取整个对话记录。这就是超长 Agent 运行容易崩溃的原因：上下文被填满，早期信息被截断，Agent 逐渐失去对任务的整体把握。

**停止条件必须明确。** 设计良好的 Agent 在完成时会发出特定信号（比如调用 `final_answer` 工具，或干脆停止使用工具）。如果没有这个机制，Agent 可能无限循环，或者幻觉自己已经完成了实际上并未完成的任务。

**这种设计的优雅与脆弱** 在于，一切都只是文本在对话记录中流动。没有 RPC、没有 socket、没有特殊协议——只有精心编排的 JSON 消息序列，而大模型经过训练，能够正确生成和消费这些消息。

## 常见的失效模式

| 失效类型 | 原因 | 缓解措施 |
|---|---|---|
| **上下文溢出** | 长时间运行的 Agent 在任务中途填满了上下文窗口 | 保持上下文"整洁"：把已完成的子任务卸载到外部存储，只保留当前需要的内容 |
| **无限循环** | 没有定义停止条件 | 始终定义明确的停止信号（如 `final_answer` 工具），并设置最大迭代次数保护 |
| **幻觉已完成** | Agent 输出最终答案，声称任务已完成，实际上并没有 | 在调用 `final_answer` 前要求通过工具调用来验证；把这个要求写进 system prompt |
| **忽略工具结果** | 模型基于过时的上下文推理，而非最新的工具结果 | 确保工具结果被正确追加，并在模型下一次提示中强制它确认这些结果 |
