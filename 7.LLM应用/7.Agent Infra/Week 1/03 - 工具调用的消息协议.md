# 工具调用的消息协议

工具调用并非魔法——它是大模型与框架事先约定好的**结构化对话格式**。

核心洞察：**大模型从未真正"调用"任何东西。** 它只是按照特殊格式生成文本。框架读取这段文本，运行真正的代码，然后把结果以更多文本的形式反馈回去。整个过程就是一个结构化的对话协议。

---

## 底层消息流转

![[Pasted image 20260402165405.png]]

逐轮的实际消息载荷：

**第 0 轮 —— 工具定义注入上下文**
```json
{
  "tools": [
    {
      "name": "web_search",
      "description": "Search the web for current information",
      "input_schema": {
        "type": "object",
        "properties": {
          "query": { "type": "string" }
        },
        "required": ["query"]
      }
    }
  ]
}
```

**第 1 轮 —— 大模型发出 `tool_use` 块**
```json
{
  "role": "assistant",
  "content": [
    {
      "type": "tool_use",
      "id": "call_abc123",
      "name": "web_search",
      "input": { "query": "current LLM inference benchmarks 2025" }
    }
  ]
}
```

**第 2 轮 —— 框架执行并返回 `tool_result` 块**
```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "call_abc123",
      "content": "Search results: ..."
    }
  ]
}
```

---

## 三个需要牢记的要点

**1. 工具定义只是 prompt 中的文本。**
`tools` 数组在对话开始前就被注入模型的上下文。大模型在训练中学会了如何阅读 JSON Schema，以及如何生成符合规范的 `tool_use` 块。并不存在特殊的"工具调用模式"——这是训练时学会的，是一种模式匹配。

**2. 大模型的"调用"只是结构化输出。**
当模型输出 `tool_use` 块时，它只不过是生成了一个描述其意图的 JSON 对象。框架解析这个对象，运行真正的函数（`web_search`、`execute_python`、`read_file` 等），然后获得真实结果。模型本身对网络、文件系统或任何外部系统都毫无访问权限。

**3. `tool_use_id` 是握手机制。**
每次调用都会分配一个唯一 ID（如 `call_abc123`）。`tool_result` 消息必须回显同样的 ID。这让模型可以在一轮中进行多次工具调用——每个结果都能对应回正确的调用，模型也能分别对它们进行推理。

---

## 并行工具调用

模型可以在**单轮中输出多个 `tool_use` 块**。框架会把它们全部派发（可能并行执行），收集所有 `tool_result` 块，追加到上下文后，才再次调用大模型。对于需要同时从多个来源收集信息的 Agent 来说，这是显著的吞吐优化。

```json
{
  "role": "assistant",
  "content": [
    { "type": "tool_use", "id": "call_001", "name": "search_docs", "input": { "query": "auth flow" } },
    { "type": "tool_use", "id": "call_002", "name": "read_file", "input": { "path": "auth.py" } }
  ]
}
```

两个结果都在下一次大模型调用前返回——模型可以同时看到它们。
