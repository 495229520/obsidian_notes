# 推理服务怎么运转

作为编写调用模型 API 的 Agent 代码的人，你需要对 HTTP 调用另一端发生的事情建立心智模型。

---

## 服务栈

你的 Agent 代码发送 HTTP POST。在另一端，推理服务器（vLLM、TGI、Triton）接收它，在单个 GPU 内核运行之前就立即做出调度决策。

```
Agent 代码
    ↓  HTTP POST
推理服务器
    ↓  调度
KV Cache 管理器    ← 内存瓶颈
    ↓
GPU 权重         ← 启动时加载一次，7B 模型约 14GB，70B 模型约 140GB
```

## GPU 显存是瓶颈 —— 而非算力

对于从 CPU 背景过来的工程师，这是最反直觉的一点。

7B 参数模型在 fp16 下的模型权重大约需要 **14 GB**。70B 模型需要 **约 140 GB**。这是固定的——启动时加载一次。剩下的一切都用于 KV Cache，它存储每个活跃上下文中每个 token 的注意力 Key 和 Value。

KV Cache 会随着并发请求增多或上下文变长而收缩。当它满了，服务器就会排队新来的请求。

**为什么这对 Agent 设计很重要：** 长 Agent 循环——在上下文中累积多轮工具调用——之所以昂贵，是因为内存压力，而非算力。处于第 50 轮循环的 Agent 消耗的显存，比第 1 轮时多得多。

---

## 静态批处理 vs 连续批处理

GPU 在做很多事并行时效率最高。

**静态批处理：** 把固定的一组请求分组，一起运行，然后等待批处理中*最慢*的请求完成后才接受新的。如果一个请求生成 1000 个 token，其他只生成 10 个，快的那些就空闲等待。

**连续批处理**（现代方法，vLLM 使用）：在 token 级别操作。当批处理中任意请求完成一个 token 的生成并结束时，它的 GPU 槽位立即被等待的请求填充。批处理的组成在每个 token 边界动态变化。

这是让大规模 LLM 服务在经济上可行的关键创新——生产推理服务器的吞吐远高于朴素基准测试，因为 GPU 利用率始终保持高位。

## Prefill/Decode 解耦（进阶）

由于 prefill 是计算密集型而 decode 是内存带宽密集型（见 [[04 - 上下文就是工作内存]]），一些高级服务系统在物理上将它们分离到不同硬件：

- **Prefill 工作节点** —— 高算力机器，并行处理大提示
- **Decode 工作节点** —— 为内存带宽优化的机器，服务流式 decode 阶段

实现这一点的系统：**Splitwise**、**Mooncake**。对于长上下文输入的异步 Agent 流水线（如加载大文档），这意味着 prefill 阶段的路由可能与随后的 decode 流不同。

## 对 Agent 构建者的实际启示

| 设计选择 | 为何重要 |
|---|---|
| 保持 system prompt 简短且可缓存 | 前缀缓存让后续请求完全跳过 prefill 成本 |
| 避免不必要的长工具结果 | 它们膨胀每个后续 decode 步骤的 KV Cache 压力 |
| 对长回复使用异步模式 | 500 token 的 decode 在 20 tok/s 下 = 25 秒；不要同步阻塞 |
| 在一轮中批量处理独立的工具调用 | 让服务器在工具执行期间服务其他请求 |
