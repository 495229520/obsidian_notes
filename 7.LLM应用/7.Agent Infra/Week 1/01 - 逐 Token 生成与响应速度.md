# 逐 Token 生成与响应速度

> **为什么这对 Agent 很重要：** 每个 Agent 步骤的耗时都是 TTFT + N×解码延迟。一个包含 5 个步骤的 Agent，每次输出 200 个 token，模型速度为 20 tok/s，光是解码阶段就要花掉约 50 秒——这还没算上工具执行的时间。所以，token 预算要精打细算。

---

## Token 是什么？

Token 是大模型处理的基本单位——它既不是单词，也不是字符。像 "running" 这样的词可能只占 1 个 token，而 "uncharacteristically" 可能会被切成 3 到 4 个。标点、空格、以及子词片段都是独立的 token。平均来说，**1 个 token ≈ 0.75 个英文单词**。

词表是通过 **BPE（Byte-Pair Encoding，字节对编码）** 构建的——本质上是一张查找表，把常见的子词序列映射成 token ID。比如 "unbelievable" → `["un", "believ", "able"]`。

---

## 生成循环详解

整个过程是**逐个 token 严格串行**的：

**1. 分词。** 输入文本通过 BPE 查找表被切分成 token。

**2. Transformer 层的前向传播。** 每个 token 都通过自注意力机制与其他所有 token 交互，结果再流经前馈层。这是计算量最重的步骤——对于大模型来说，涉及数十亿次浮点运算。

**3. Logits → 概率。** 最后一层会为词表中的每个条目（约 3.2 万到 10 万个 token）输出一个原始分数（logit）。Softmax 把这些分数转换成总和为 1 的合法概率分布。

**4. 采样。** 从分布中选取一个 token。
- `temperature` 控制分布的尖锐程度（低 = 贪婪/确定性，高 = 更随机）
- `top-p`（核采样）把采样范围限制在最可能的那部分 token，直到它们的累积概率超过阈值

**5. 追加并重复。** 新 token 被追加到上下文，然后重新运行完整的前向传播，预测下一个。

## 为什么延迟用「token/秒」来衡量

因为生成循环是**严格串行**的——每个 token 都依赖于之前所有的 token，所以无法并行生成。这是根本性的约束：

- **首 token 时间（TTFT）** —— 从请求到第一个输出现身的耗时。这是预填充阶段的延迟（一次性并行处理你的整个输入提示）。
- **每秒 token 数（TPS）** —— 首 token 之后，生成循环本身的吞吐量。这就是你感受到的模型"打字速度"。

如果一个模型每秒生成 40 个 token，那么 200 个 token 的回复需要约 5 秒。成本纯粹取决于输出 token 的数量——而不是答案的复杂程度。一句话的简单回复和一句话的花哨回复，只要 token 数相同，耗时就一样。

**流式输出**（边生成边展示）的体验会比等待完整回复好得多——挂钟时间其实一样，但你在 TTFT 之后就能立即获得反馈。
