# 上下文就是工作内存

> **核心心智模型：** 上下文窗口是 **RAM，而非磁盘。** 它快速、昂贵、容量有限且易失。放进去的每样东西都要消耗显存和解码延迟。

在推理阶段，大模型没有持久状态。它对当前任务的"记忆"完全存在于送入前向传播的 token 序列中。这就是上下文窗口——而在这个前向传播的架构内部发生的事情，才真正构成了工程约束。

---

## KV Cache：它究竟是什么

在每个注意力层中，针对每个 token，模型会计算三个向量：**Query（Q）**、**Key（K）** 和 **Value（V）**。注意力操作的本质是：*当前 token 的 Q 与每个历史 token 的 K 有多匹配？* —— 这个分数决定了每个 token 的 V 有多少会被混合进输出。

关键洞察：**一旦计算完成，某个 token 的 K 和 V 就永远不会改变。** 模型不需要在后续前向传播中重新计算它们。所以它们被缓存起来——这就是 KV Cache。

对于一个 32 层、32 头、10 万 token 上下文的模型，KV Cache 很容易达到**数 GB 的 GPU 显存**。

---

## Prefill 快，Decode 慢——原因各不相同

![[gemini-svg (7).svg|827]]

**Prefill** 一次性处理你的整个提示——比如 8000 个 token。这是一个巨大的矩阵乘法：`tokens × d_model`。GPU 正是为此优化的。算术强度很高，每个计算核心都在忙碌。GPU 处于*计算密集型*状态。结果：所有 K 和 V 向量在一轮内完成计算并缓存。

**Decode** 则完全不同。你刚生成了一个新 token。你需要把它送过全部 32 层，在每一层，注意力机制必须*读取整个 KV Cache* 来计算新 token 的注意力分数。但你只计算一个 token 的输出。这是极小的计算量对应巨大的内存读取。GPU 变成*内存带宽密集型*——它大部分时间都在把数据从 HBM 搬到计算核心，而不是真正计算。

这就是为什么你看到"首 token 时间"很快（那是 prefill 完成），之后的生成却缓慢而稳定（那是 decode，每 N 毫秒一个 token）。

---

## 四项真正的工程约束

**约束 1 —— 显存是硬天花板。**
KV Cache 随上下文长度线性增长。对于 Llama 3 70B 这样的模型，80 层注意力、64 头，每个 token 都会在每层增加 K 和 V 向量——在 FP16 下，12.8 万 token 的上下文仅缓存就可能消耗 20–30 GB 显存，这还没算模型权重本身。

**约束 2 —— 上下文越长，decode 越慢。**
在 Agent 任务的第 100 步，模型每次生成单个 token 都要读取 10 万 token 的 KV Cache。内存总线是瓶颈。生成吞吐（tokens/秒）会随着在上下文中越深而下降。运行了很多轮的 Agent 的响应速度，会比刚开始时明显更慢。

**约束 3 —— Prefill 成本在注意力中是平方级的。**
标准注意力要计算每对 token 之间的相似度——序列长度的 O(n²)。10 倍长的提示可能意味着 100 倍的注意力计算。这就是为什么 TTFT 是与 token 间延迟不同的独立指标。对于异步 Agent 流水线，缓慢的 TTFT 意味着调度器被阻塞，等待模型"醒来"。

**约束 4 —— 输入占用了输出空间。**
如果你的模型有 12.8 万 token 的窗口，而提示 + 工具结果 + 对话历史已经达到 12 万 token，那你只剩下 8 千 token 的输出预算。对于需要大量推理才能行动的 Agent，这可能构成实际约束。

---

## 分块 Prefill 调度

对于超长的提示，一些推理服务器会把 prefill **分块**处理，而不是一次性处理整个提示。这让 GPU 可以把 prefill 块与其他请求的 decode 工作交错执行——即使单个长提示本会独占计算核心，也能保持 GPU 利用率高位运行。

权衡：TTFT 略微增加（prefill 被分散到多轮执行），但整体系统吞吐提升了，因为其他请求不会被饿死。

---

## 对 Agent 基础设施的启示

**Prefill/decode 解耦。**
由于 prefill 是计算密集型而 decode 是内存密集型，一些服务系统（如 Splitwise、Mooncake）在物理上将它们分离到不同机器上。Prefill 工作节点重计算；decode 工作节点优先保证内存带宽。

**为什么长上下文下异步更重要。**
如果 decode 以 20 tokens/秒 生成 token，而你的上下文很长，500 token 的回复需要 25 秒。同步系统会阻塞在这上面。异步设计让调度器可以排队请求、去做别的工作，然后通过回调或轮询来处理响应。

**上下文作为 Agent 唯一的草稿纸，存在逐出风险。**
如果一个长时间运行的 Agent 在任务中途填满了上下文窗口，无论你用什么策略恢复（截断早期轮次、总结、重启）都是有损的。良好的 Agent 架构会把已完成的子任务卸载到外部存储，只拉入当前需要的内容。

**KV Cache 复用（前缀缓存）。**
如果多个请求共享相同的 system prompt 或文档，该前缀的 KV Cache 可以跨请求复用。对于总是以相同长 system prompt 开头的 Agent，这意味着第一个请求承担完整的 prefill 成本；后续请求使用缓存的 KV，完全跳过这一步。Anthropic 的 API 明确以"prompt caching"的形式暴露了这一点。
