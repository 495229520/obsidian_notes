# System Prompt 不是文案，而是配置层

在简单的聊天机器人中，system prompt 只是人格描述。但在 Agent 中，它是**承重的基础设施**——如果它有问题，Agent 会以很难调试的方式崩溃，因为失败看起来像是"模型行为怪异"，而不是"有 bug"。

## 四个层次

这四个层次各自独立失效，且方式不同。

**第一层 —— 人格与行为边界。**
这告诉模型它是谁、以及绝对不能做什么。对于基础设施 Agent，可能是"你是部署助手；永远不要直接修改生产数据库"。如果这一层缺失或模糊，模型会用训练先验来填补空白，而这可能不符合系统的要求。

**第二层 —— 工具清单。**
这是最具基础设施关键性的层次。每个工具描述实际上是函数签名的注释，模型用它来判定*何时*以及*如何*调用工具。描述的质量直接决定调用准确度。像 `"search the web"` 这样的描述，产生的工具调用行为会比 `"search the web for current, time-sensitive information that is unlikely to be in training data"` 更差——后者还告诉了模型*何时不应该*调用它。

**第三层 —— 循环终止条件。**
没有明确的停止规则，Agent 会无限循环或过早停止。你需要编码类似这样的规则："一旦用至少一个工具验证了结果，就调用 `final_answer`"，或者"每个用户请求不要超过 5 次工具调用"。这是纯粹的控制流逻辑，但它以自然语言的形式存在于 prompt 中。

**第四层 —— 安全与信任边界。**
哪些工具在执行前需要人工确认？Agent 被允许读取什么数据？这是你的访问控制层——不是由模型本身强制执行，而是由框架读取 prompt 的意图，并在循环周围用代码实现。

---

## 关键的工程洞察

对 system prompt 的改动是**部署**，而非配置微调。你需要对它做版本控制、做 eval、在它出问题时回滚——就像对待代码一样。

---

## 上线前检查清单

在发布或改动 system prompt 前，请验证：

1. **工具清单是否完整？** 每个工具的描述是否清楚说明了何时使用它*以及*何时不使用？
2. **停止条件是否明确？** prompt 是否定义了"完成"是什么样的，以及 Agent 完成时发出什么信号？
3. **破坏性动作的信任边界是否已定义？** 哪些操作需要确认？哪些可以直接安全执行？
4. **是否已版本控制？** 当前 prompt 是否已提交、打标签，并与验证它的 eval 结果关联？
