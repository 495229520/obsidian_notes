# 🔧 1-Year Agent Infrastructure Learning Plan
### For Systems Engineers (C++/Java background)

---

## Phase 1 — Bridge to the AI Layer (Weeks 1–8)
*Goal: Learn just enough about how agents work so you can build the right infrastructure for them*

### Week 1 — How LLMs & Agents Actually Work (Concepts Only)
- How LLMs generate tokens, why latency is per-token
- What an agent loop looks like: observe → think → act → repeat
- What "tool calling" means at the protocol level (just JSON over HTTP)
- **No coding this week** — read, watch, understand the workload you'll be serving

### Week 2 — Python for Systems Engineers
- Python from a C++ engineer's perspective: GIL, memory model, ctypes
- `asyncio` internals: how the event loop works (compare to your epoll/kqueue knowledge)
- `subprocess`, `os`, `signal` — the UNIX stuff you already know, in Python
- **Project:** Rewrite one of your C++ IPC demos in Python — feel the differences

### Week 3 — The LLM API as a Network Service
- Anthropic/OpenAI API: just HTTP/SSE under the hood — read the raw protocol
- Streaming responses: SSE framing, chunked transfer encoding
- Token counting, timeouts, backpressure problems
- **Project:** Call the LLM API using raw `httpx` with streaming — no SDKs

### Week 4 — Tool Call Protocol Deep Dive
- How tool calling works at the wire level (JSON schema negotiation)
- MCP (Model Context Protocol): read the full spec — this is your home turf (it's basically RPC over stdio/SSE)
- Compare MCP to gRPC, JSON-RPC, and D-Bus
- **Project:** Read the MCP spec and diagram its transport and message format

### Week 5 — Agent Execution Model
- What an agent runtime needs to track: current step, tool results, message history
- Where things go wrong: loops, hangs, runaway tool calls, OOM
- Resource consumption patterns of agent workloads (CPU, memory, network)
- **Project:** Instrument a simple Python agent loop — measure latency per step, token throughput

### Week 6 — Observability Foundations
- OpenTelemetry: traces, spans, metrics — the standard you'll instrument everything with
- Structured logging for long-running agent runs
- What to measure: tool call latency, queue depth, memory per agent, crash rates
- **Project:** Add OTel tracing to your Week 5 agent loop

### Week 7 — Survey the Landscape
- Read the source code of one agent framework (LangGraph or AutoGen) — focus on the runtime, not the prompt logic
- Identify the infrastructure weak points: where does it block? where does it leak? where does it crash?
- **Deliverable:** Write a 1-page technical critique of the framework's runtime

### Week 8 — Consolidation
- **Milestone:** You understand the workload. You know what agents need from infrastructure. Now you build it.

## Phase 2 — Core Infrastructure Primitives (Weeks 9–24)

### Week 9 — Agent Sandboxing I — Concepts & Linux Primitives
- namespaces: `pid`, `net`, `mnt`, `user`, `ipc` — you likely know some of these
- cgroups v2: CPU, memory, IO limits per agent process
- `seccomp` syscall filtering for untrusted agent-generated code
- Read: how Docker and gVisor use these primitives

### Week 10 — Agent Sandboxing II — Implementation
- Build a minimal process sandbox in C++ using `clone()` + namespaces + cgroups
- Add `seccomp-bpf` filter to whitelist only safe syscalls
- **Project:** A sandbox that runs arbitrary Python code with hard CPU/memory limits and no network access

### Week 11 — Agent Sandboxing III — Escape & Hardening
- Common sandbox escapes: `/proc` leaks, shared memory, socket hijacking
- Filesystem isolation: `pivot_root`, read-only overlayfs, tmpfs for scratch
- **Project:** Try to escape your own sandbox — then fix the holes

### Week 12 — High-Throughput Tool Call Dispatching
- Design a tool dispatcher: agents submit tool calls → dispatcher routes → result returned
- Thread pool vs. async I/O vs. process pool — tradeoffs for tool execution
- Backpressure: what happens when tools are slow or unavailable
- **Project:** Build a tool dispatcher in C++ that handles 1000 concurrent tool calls

### Week 13 — State Persistence I — Design
- What agent state needs to be persisted: message history, tool results, memory, current step
- Serialization formats: protobuf vs. MessagePack vs. JSON — pick for agent workloads
- Write-ahead logging (WAL) for resumable agent runs
- Schema design for agent runs, steps, tool calls

### Week 14 — State Persistence II — Implementation
- Implement a lightweight agent state store backed by SQLite (with WAL mode)
- Checkpoint and restore an agent mid-run
- **Project:** Agent that can be killed at any step and resume exactly where it left off

### Week 15 — Transport Layer I — MCP Server from Scratch
- Implement an MCP server in C++ over stdio transport (the simplest profile)
- Handle `initialize`, `tools/list`, `tools/call` message types
- Error handling, unknown method responses, malformed JSON
- **Project:** A working MCP server that exposes 3 real tools (file read, shell exec, HTTP fetch)

### Week 16 — Transport Layer II — MCP over SSE/HTTP
- Implement the HTTP+SSE transport profile of MCP
- Session management, reconnection, event ordering guarantees
- Compare to your TCP/UDP programming experience — what's new?
- **Project:** Upgrade your Week 15 MCP server to support SSE transport

### Week 17 — Agent-to-Agent Communication
- How multi-agent systems pass messages: shared queue, direct RPC, pub/sub
- Design a message bus for agents: topics, routing, delivery guarantees
- Your IPC background (pipes, sockets, shared memory) applies directly here
- **Project:** Message bus in C++ that routes tool calls between a planner agent and executor agents

### Week 18 — Scheduling & Resource Management
- Scheduling multiple concurrent agent runs: priority queues, fair share
- Resource accounting: tokens budgeted per agent, cost caps
- Preemption: pausing a low-priority agent to free resources
- **Project:** A simple agent scheduler that manages 10 concurrent agents under a shared token budget

### Week 19 — Reliable Tool Execution
- Idempotency for tool calls (what if the agent retries?)
- Timeouts and deadlines propagated through the call chain
- Circuit breakers for flaky tools
- **Project:** Add retry logic + idempotency keys + circuit breakers to your tool dispatcher

### Week 20 — Memory Architecture
- Short-term (in-context), working (external store), long-term (vector DB)
- Designing a memory service: read/write API, eviction policy, TTL
- Your background: think of it as a cache hierarchy (L1/L2/L3 but for agent context)
- **Project:** Build a memory service with Redis (working memory) + pgvector (long-term)

### Week 21 — Streaming & Backpressure
- How to stream LLM tokens through your infrastructure to a client
- Backpressure: slow consumers, fast producers — apply your IO multiplexing knowledge
- SSE vs. WebSocket for streaming agent output
- **Project:** Build a streaming proxy that sits between the LLM API and client, with flow control

### Week 22 — Crash Recovery & Fault Isolation
- Detecting stuck/looping agents: heartbeat, step counter, timeout watchdog
- Isolating a crashing agent from affecting others (your process/thread knowledge applies)
- Automatic restart with state recovery from checkpoint
- **Project:** Watchdog process in C++ that monitors agent workers and restarts them on failure

### Week 23 — Security: Prompt Injection at the Infrastructure Layer
- Prompt injection: an attacker manipulates tool output to hijack agent behavior
- Infrastructure-level defenses: output sanitization before re-injection, content policies
- Audit logging of all tool inputs/outputs for forensics
- **Project:** Add an output sanitization layer to your tool dispatcher

### Week 24 — Phase 2 Review
- **Milestone:** You've built: a sandbox, a tool dispatcher, a state store, an MCP server, a message bus, a scheduler, and a streaming layer. Integrate these into one cohesive agent runtime.

## Phase 3 — Production Systems (Weeks 25–40)

### Week 25 — Containerizing Your Runtime
- Package your C++ agent runtime into a minimal Docker image
- Multi-stage builds, distroless base images
- Resource limits in container runtime vs. your cgroup implementation — understand the relationship

### Week 26 — Kubernetes for Agent Workloads
- Pods, Deployments, StatefulSets — when to use each for agents
- Scheduling agent jobs with `Jobs` and `CronJobs`
- Resource requests/limits for LLM-heavy workloads

### Week 27 — Distributed State
- Why SQLite isn't enough at scale
- PostgreSQL with connection pooling (PgBouncer)
- Distributed locking with Redis — compare to your mutex/lock experience
- **Project:** Migrate your state store to PostgreSQL with proper locking

### Week 28 — Queue-Based Agent Orchestration
- Kafka or RabbitMQ as the backbone for agent task distribution
- Your DPDK/IO multiplexing background: understand the throughput requirements
- Dead letter queues for failed agent runs
- **Project:** Replace your message bus with Kafka-backed orchestration

### Week 29 — High-Performance Tool Serving
- gRPC for internal tool services (compare to your TCP/IP programming)
- Protocol Buffers for tool input/output serialization
- Load balancing tool servers: round-robin, least connections, consistent hashing
- **Project:** Rewrite your tool dispatcher client/server as a gRPC service

### Week 30 — Observability at Scale
- Distributed tracing across: client → agent runtime → tool servers → LLM API
- Metrics pipeline: Prometheus + Grafana for agent fleet monitoring
- Dashboards: agent throughput, p50/p99 tool latency, error rates, cost per run
- **Project:** Full observability stack for your agent runtime

### Week 31 — Cost Control Infrastructure
- Token metering: count every token in/out per agent run
- Cost attribution: per user, per agent type, per tool
- Semantic caching: cache LLM responses for repeated queries (your caching knowledge applies)
- **Project:** Token metering service with per-user budget enforcement

### Week 32 — Multi-Tenancy & Isolation
- Isolating agent runs per tenant: separate cgroups, namespaces, DB schemas
- Noisy neighbor problem: one tenant's agents shouldn't starve others
- **Project:** Add tenant isolation to your agent scheduler

### Week 33 — Human-in-the-Loop Infrastructure
- Pausing an agent mid-run and waiting for human input (think: `SIGSTOP` but for agent state)
- Durable execution: agent sleeps, system restarts, agent wakes up
- **Project:** Agent pause/resume mechanism with a simple web UI to approve pending actions

### Week 34 — Agent Fleet Management
- Running hundreds of concurrent agents: operational challenges
- Health checks, readiness probes, graceful shutdown
- Rolling deploys of agent runtime without dropping in-flight runs

### Week 35 — Performance Profiling & Optimization
- Profile your C++ runtime: `perf`, `flamegraph`, `valgrind`
- Identify bottlenecks: is it the sandbox? the dispatcher? the state store? the network?
- **Project:** Reduce p99 tool call latency by 50% through targeted optimization

### Week 36 — Load Testing Agent Infrastructure
- `k6` or `locust` for simulating concurrent agent workloads
- Find your breaking points: where does throughput collapse?
- Capacity planning: how many agents can one node handle?

### Week 37 — Chaos Engineering
- Intentionally kill processes, inject latency, corrupt messages
- Verify your fault isolation and recovery mechanisms hold
- **Tools:** `chaos-mesh` or manual fault injection scripts

### Week 38 — Security Hardening
- Threat model for your agent infrastructure
- Network policies: agents should not talk to each other directly
- Secrets management: Vault or cloud KMS for API keys used by tools
- Penetration test your sandbox

### Week 39 — Benchmarking Against Existing Solutions
- Compare your runtime to E2B, Modal, Fly.io Machines on: latency, isolation, cost
- Identify where yours is better (likely: lower latency, better isolation control)
- **Deliverable:** Benchmark report with graphs

### Week 40 — Phase 3 Review
- **Milestone:** Production-grade agent infrastructure: observable, secure, multi-tenant, fault-tolerant, and benchmarked.
