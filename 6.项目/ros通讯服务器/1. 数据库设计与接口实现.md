# 数据库设计与接口实现

> MFMS (Manufacturing Flexible Manufacturing System) 是一个混合机器人制造控制系统，集成 ROS 2、Qt GUI、MySQL 数据库，用于管理异构设备（机器人、AGV、相机、控制器）。本文档分析其数据库设计与核心接口实现。

---

## 1. 系统架构概述

### 1.1 整体架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                           Qt GUI 上位机                              │
├─────────────────────────────────────────────────────────────────────┤
│  MfmsCommandService  │  MfmsDbService  │    MfmsRosBridge          │
│  (命令下发服务)        │  (数据库事件服务)  │    (ROS话题桥接)          │
├─────────────────────────────────────────────────────────────────────┤
│                         MySQL 数据库                                 │
│              (状态表 + 事件表 + 触发器)                               │
├─────────────────────────────────────────────────────────────────────┤
│                    ROS 2 通信层                                      │
│           (Topics / Services / Actions)                             │
├─────────────────────────────────────────────────────────────────────┤
│   FR机器人   │   华数机器人   │   Seer AGV   │   相机   │   控制器    │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 设计思路

系统采用**事件驱动架构**，通过数据库作为上下位机之间的状态同步中心：

| 层级 | 组件 | 职责 |
|------|------|------|
| 上位机 | Qt GUI | 用户交互、状态展示、命令发起 |
| 服务层 | 三大服务 | 数据库操作、ROS通信、状态同步 |
| 数据层 | MySQL | 状态持久化、事件队列、触发器自动通知 |
| 通信层 | ROS 2 | 设备间标准化通信协议 |
| 设备层 | 异构设备 | 实际执行、状态上报 |

**核心优势**：
- 上下位机解耦：通过数据库事件表实现异步通信
- 状态可追溯：所有状态变更都有事件记录
- 扩展性强：新增设备只需添加对应的 Topic/Service

### 2.2 设备管理表

#### device 表（设备基础信息）

存储设备的静态配置信息。

```sql
CREATE TABLE device (
    id VARCHAR(10) PRIMARY KEY,    -- 设备ID，格式：type(3) + module(3) + id(4)
    group_id INT,                  -- 设备分组ID，用于逻辑分组
    address VARCHAR(50),           -- 设备网络地址（IP或其他标识）
    create_ts BIGINT               -- 创建时间戳（毫秒）
);
```

**设备ID编码规则**：

| 位置 | 长度 | 含义 | 示例 |
|------|------|------|------|
| 1-3 | 3位 | 设备类型 | `rba`=机器人适配器, `agv`=AGV, `ctr`=控制器 |
| 4-6 | 3位 | 模块型号 | `fra`=法奥, `hsu`=华数, `ser`=Seer |
| 7-10 | 4位 | 序列号 | `0001`, `0002` |

**示例**：
```
rbafra0001  →  机器人(rba) + 法奥(fra) + 编号0001
agvser0001  →  AGV(agv) + Seer(ser) + 编号0001
ctrkos0001  →  控制器(ctr) + Kossi(kos) + 编号0001
```

#### device_state 表（设备实时状态）

存储设备的动态运行状态，这是事件驱动架构的核心表。

```sql
CREATE TABLE device_state (
    id VARCHAR(10) PRIMARY KEY,
    state ENUM('offline', 'online', 'load', 'unload', 'connected'),
    info JSON,          -- 设备详细信息（IP、版本、位姿等）
    err_code INT        -- 错误码，0表示正常
);
```

**状态枚举说明**：

| 状态 | 含义 | 触发方 |
|------|------|--------|
| `offline` | 离线/未连接 | 系统/下位机 |
| `load` | 正在加载 | 上位机请求 |
| `online` | 在线就绪 | 下位机确认 |
| `connected` | 已建立连接 | 下位机确认 |
| `unload` | 正在卸载 | 上位机请求 |

**info JSON 结构示例**（以法奥机器人为例）：

```json
{
    "ip": "192.168.1.100",
    "soft_version": "v3.8.1",
    "robot_type": "法奥机器人",
    "joint_pos": [0.0, -90.0, 90.0, 0.0, 90.0, 0.0],
    "cart_pos": [500.0, 0.0, 400.0, 180.0, 0.0, 0.0]
}
```

#### device_state_event 表（下位机消费）

记录设备状态变更事件，供下位机轮询消费。

```sql
CREATE TABLE device_state_event (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    device_id VARCHAR(10),
    from_state ENUM('offline', 'online', 'load', 'unload', 'connected'),
    to_state ENUM('offline', 'online', 'load', 'unload', 'connected'),
    reason VARCHAR(255),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**设计思路**：下位机（如 ROS 节点）定期查询此表，获取需要执行的状态变更请求。当上位机调用 `loadDevice()` 时，状态变为 `load`，触发器自动生成事件，下位机读取后开始加载设备。

#### device_ui_event 表（上位机消费）

记录需要通知上位机的状态变更事件。

```sql
CREATE TABLE device_ui_event (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    device_id VARCHAR(10),
    from_state ENUM('offline', 'online', 'load', 'unload', 'connected'),
    to_state ENUM('offline', 'online', 'load', 'unload', 'connected'),
    reason VARCHAR(255),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_device_id (device_id),
    INDEX idx_to_state (to_state),
    INDEX idx_created_at (created_at)
);
```

**索引设计说明**：
- `idx_device_id`：按设备过滤事件
- `idx_to_state`：按目标状态过滤（如只关心 `online` 事件）
- `idx_created_at`：按时间范围查询

### 2.3 脚本管理表

#### lua_script 表（脚本内容）

```sql
CREATE TABLE lua_script (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,

    script_content TEXT,          -- Lua 脚本源代码
```

#### lua_state 表（脚本执行状态）

```sql
CREATE TABLE lua_state (
    script_id BIGINT,
    group_id INT,                 -- 执行该脚本的设备组
```

**状态机设计**：

```
                    ┌──────────────────────────────────────┐
                    │                                      │
                    ▼                                      │
    ┌───────┐  startScript()  ┌───────┐  下位机确认  ┌─────────┐
    │ wait  │ ──────────────► │ ready │ ──────────► │ running │
    └───────┘                 └───────┘             └─────────┘
        ▲                         │                      │
        │                         │ pauseScript()        │
        │                         ▼                      │
        │                    ┌────────┐                  │
        │                    │ pause  │◄─────────────────┘
        │                    └────────┘   pauseScript()
        │                         │
        │     abortScript()       │ resumeScript()
        │◄────────────────────────┘
        │
        │     abortScript()
    ┌───────┐◄────────────────────────────────────────────
    │ abort │
    └───────┘
```

### 2.4 触发器设计

触发器是事件驱动架构的关键，实现状态变更的自动通知。

#### trg_device_state_event 触发器

```sql
DELIMITER //
CREATE TRIGGER trg_device_state_event
AFTER UPDATE ON device_state
FOR EACH ROW
BEGIN
    -- 当状态发生变化时
    IF OLD.state != NEW.state THEN
        -- load/unload 状态：生成下位机消费事件
        IF NEW.state IN ('load', 'unload') THEN
            INSERT INTO device_state_event (device_id, from_state, to_state, reason)
            VALUES (NEW.id, OLD.state, NEW.state, '上位机请求');
        END IF;

        -- online/connected/offline 状态：生成上位机消费事件
        IF NEW.state IN ('online', 'connected', 'offline') THEN
            INSERT INTO device_ui_event (device_id, from_state, to_state, reason)
            VALUES (NEW.id, OLD.state, NEW.state, '下位机确认');
        END IF;
    END IF;
END //
DELIMITER ;
```

**触发逻辑**：

| 新状态 | 事件目标表 | 消费方 | 场景 |
|--------|-----------|--------|------|
| `load` | device_state_event | 下位机 | 上位机请求加载设备 |
| `unload` | device_state_event | 下位机 | 上位机请求卸载设备 |
| `online` | device_ui_event | 上位机 | 下位机确认设备就绪 |
| `connected` | device_ui_event | 上位机 | 下位机确认连接建立 |
| `offline` | device_ui_event | 上位机 | 设备离线通知 |

#### trg_lua_state_event 触发器

```sql
DELIMITER //
CREATE TRIGGER trg_lua_state_event
AFTER UPDATE ON lua_state
FOR EACH ROW
BEGIN
    IF OLD.state != NEW.state THEN
        -- 所有状态变更都生成双向事件
        INSERT INTO lua_state_event (script_id, group_id, from_state, to_state, reason)
        VALUES (NEW.script_id, NEW.group_id, OLD.state, NEW.state, '状态变更');

        INSERT INTO lua_ui_event (script_id, group_id, from_state, to_state, reason)
        VALUES (NEW.script_id, NEW.group_id, OLD.state, NEW.state, '状态变更');
    END IF;
END //
DELIMITER ;
```

**`loadDevice` 实现逻辑**：

```cpp
void MfmsDbService::loadDevice(const QString& deviceId) {
    if (!m_connected) {
        emit errorOccurred(DbErrorCode::NOT_CONNECTED, "数据库未连接");
        return;
    }

    QSqlQuery query(m_db);
    query.prepare("UPDATE device_state SET state = :state WHERE id = :id");
    query.bindValue(":state", DeviceState::LOAD);
    query.bindValue(":id", deviceId);

    if (!query.exec()) {
        emit errorOccurred(DbErrorCode::UPDATE_FAILED, query.lastError().text());
        return;
    }

    // 注意：不直接 emit deviceStateChanged
    // 状态变更通过触发器 → ui_event 表 → 轮询线程 → 信号 的路径传递
    // 保证上下位机看到的状态变更顺序一致
}
```

**关键设计点**：
- 使用参数绑定防止 SQL 注入
- 不直接发出状态变更信号，而是依赖事件表，保证一致性
- 错误处理通过信号传递，不抛异常

**`sendRobotMotion` 实现逻辑**：

```cpp
void MfmsCommandService::sendRobotMotion(const RobotMotionCommand& cmd) {
    // 1. 参数校验
    if (cmd.speed < 0 || cmd.speed > 100) {
        emit errorOccurred(CmdErrorCode::INVALID_PARAMETER, "速度超出范围");
        return;
    }

    // 2. 查找设备
    auto it = std::find_if(m_devices.begin(), m_devices.end(),
        [&](const OnlineDeviceInfo& d) { return d.id == cmd.deviceId; });

    if (it == m_devices.end()) {
        emit robotMotionExecuted(cmd.deviceId, false,
            CmdErrorCode::DEVICE_NOT_FOUND, "设备不存在");
        return;
    }

    // 3. 构造 ROS Service 请求
    auto request = std::make_shared<com_interfaces::srv::RobotCmd::Request>();
    request->mode = cmd.mode;
    request->speed = cmd.speed;
    request->use_cartesian = cmd.useCartesian;

    for (int i = 0; i < 13; ++i) {
        request->params[i] = cmd.params[i].value;
    }

    // 4. 异步调用 Service
    auto client = m_node->create_client<com_interfaces::srv::RobotCmd>(
        it->serviceName().toStdString());

    auto future = client->async_send_request(request,
        [this, deviceId = cmd.deviceId](auto response) {
            emit robotMotionExecuted(deviceId, response->success,
                response->error_code, QString::fromStdString(response->message));
        });
}
```

**关键设计点**：
- 异步 Service 调用，不阻塞 Qt 事件循环
- 回调中通过信号通知结果
- 使用智能指针管理 request 生命周期

```cpp
struct IOState {
    uint8_t digitalOutputH;   // DO 高字节 (8-15)
    uint8_t digitalOutputL;   // DO 低字节 (0-7)
    uint8_t digitalInputH;    // DI 高字节
    uint8_t digitalInputL;    // DI 低字节
    uint16_t analogInput[2];  // 模拟输入
    uint16_t analogOutput[2]; // 模拟输出
    uint8_t toolDigitalOutput;// 工具端 DO
    uint8_t toolDigitalInput; // 工具端 DI

    // 位操作辅助方法
    bool isDigitalOutputHigh(int bit) const {
        if (bit < 8) return (digitalOutputL >> bit) & 1;
        return (digitalOutputH >> (bit - 8)) & 1;
    }

    bool isDigitalInputHigh(int bit) const {
        if (bit < 8) return (digitalInputL >> bit) & 1;
        return (digitalInputH >> (bit - 8)) & 1;
    }
};
```

```cpp
void MfmsRosBridge::subscribeDevice(const OnlineDevice& device) {
    if (!m_node) {
        emit errorOccurred(RosBridgeError::ROS_NODE_NULL, "ROS节点未初始化");
        return;
    }

    // 取消现有订阅
    unsubscribe();

    m_currentDevice = device;
    QString topicName = device.topicName();

    // 根据设备类型创建对应的订阅
    switch (device.type) {
        case DeviceType::FrRobot:
            m_frSubscription = m_node->create_subscription<com_interfaces::msg::FrRobotState>(
                topicName.toStdString(), 10,
                [this](const com_interfaces::msg::FrRobotState::SharedPtr msg) {
                    handleFrRobotState(msg);
                });
            break;

        case DeviceType::HsRobot:
            m_hsSubscription = m_node->create_subscription<com_interfaces::msg::HsRobotState>(
                topicName.toStdString(), 10,
                [this](const com_interfaces::msg::HsRobotState::SharedPtr msg) {
                    handleHsRobotState(msg);
                });
            break;

        case DeviceType::SeerAgv:
            m_agvSubscription = m_node->create_subscription<com_interfaces::msg::SeerAgvState>(
                topicName.toStdString(), 10,
                [this](const com_interfaces::msg::SeerAgvState::SharedPtr msg) {
                    handleSeerAgvState(msg);
                });
            break;

        default:
            emit errorOccurred(RosBridgeError::UNSUPPORTED_DEVICE_TYPE,
                QString("不支持的设备类型: %1").arg(static_cast<int>(device.type)));
            return;
    }

    emit subscribeResult(device.id, true, "订阅成功");
}
```

**关键设计点**：
- 每次只订阅一个设备，简化状态管理
- 使用 lambda 捕获消息并转发到处理函数
- QoS 使用默认值 10，适合实时状态数据

## 6. 关键设计模式总结

### 6.1 事件驱动 vs 轮询

| 方面 | 传统轮询 | 本系统设计 |
|------|---------|-----------|
| 状态获取 | 定时查询状态表 | 触发器 → 事件表 → 轮询事件表 |
| 实时性 | 取决于轮询间隔 | 事件产生即可感知 |
| 数据库负载 | 每次查询全表 | 只查询新增事件 |
| 顺序保证 | 可能丢失中间状态 | 事件有序，不会丢失 |

### 6.2 上下位机解耦

```
上位机 ──写入──► 状态表 ──触发器──► 下位机事件表 ──轮询──► 下位机
                  ▲                                      │
                  └────────── 状态更新 ◄─────────────────┘
                              │
                              ▼ 触发器
                         上位机事件表 ──轮询──► 上位机
```

**优势**：
- 上下位机通过数据库解耦，无直接依赖
- 任一方重启不影响另一方
- 事件表可用于故障排查和审计

### 6.3 Qt 信号槽与 ROS 回调集成

```cpp
// ROS 回调线程
void handleFrRobotState(const FrRobotState::SharedPtr msg) {
    RobotRealtimeStatus status;
    // ... 转换消息到 status

    // 通过 Qt 信号跨线程传递到主线程
    QMetaObject::invokeMethod(this, [this, status]() {
        emit robotStatusUpdated(status);
    }, Qt::QueuedConnection);
}
```

**关键点**：
- ROS 回调在 ROS executor 线程执行
- 使用 `invokeMethod` + `QueuedConnection` 安全地跨线程发送信号
- 避免在回调中直接操作 Qt 对象
