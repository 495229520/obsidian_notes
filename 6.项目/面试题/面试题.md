```
                    │   用户请求       │
                    └────────┬────────┘
                             │
              ┌──────────────▼──────────────┐
              │     1. 设备指纹识别          │
              │  (Device Fingerprint)       │

              ┌──────────────▼──────────────┐
              │     2. 行为分析              │
              │  (Behavior Analysis)        │
```

|-----|------|------|

| | 行为轨迹 | 正常用户有浏览、停留，刷号直奔目标 |

### Q1: 设计一个文件系统

**问题类型**：系统设计题，考察对操作系统和数据结构的理解。

#### 文件系统核心组件

```
┌────────────────────────────────────────────────────────┐
│                    用户空间                             │
│   应用程序  ──►  open/read/write/close 系统调用         │
└────────────────────────────────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────┐
│                   VFS (虚拟文件系统)                    │
│            统一接口，屏蔽底层文件系统差异                  │
└────────────────────────────────────────────────────────┘
                           │
        ┌──────────────────┼──────────────────┐
        ▼                  ▼                  ▼
   ┌─────────┐       ┌─────────┐        ┌─────────┐
   │  ext4   │       │  NTFS   │        │   FAT   │
   └─────────┘       └─────────┘        └─────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────┐
│                    块设备层                             │
│                  磁盘/SSD/网络存储                       │
└────────────────────────────────────────────────────────┘
```

#### 核心数据结构

**1. 超级块（Superblock）**
```cpp
struct Superblock {
    uint32_t magic;              // 文件系统标识
    uint32_t block_size;         // 块大小（如4KB）
    uint32_t total_blocks;       // 总块数
    uint32_t free_blocks;        // 空闲块数
    uint32_t total_inodes;       // 总inode数
    uint32_t free_inodes;        // 空闲inode数
    uint32_t inode_table_start;  // inode表起始位置
    uint32_t data_block_start;   // 数据块起始位置
};
```

**2. inode（索引节点）**
```cpp
struct Inode {
    uint32_t mode;           // 文件类型 + 权限（rwx）
    uint32_t uid, gid;       // 所有者
    uint32_t size;           // 文件大小
    uint32_t atime, mtime, ctime;  // 时间戳
    uint32_t link_count;     // 硬链接数

    // 数据块索引（多级索引）
    uint32_t direct[12];     // 直接指针（12个）
    uint32_t indirect;       // 一级间接指针
    uint32_t double_indirect;// 二级间接指针
    uint32_t triple_indirect;// 三级间接指针
};
```

**3. 目录项（Directory Entry）**
```cpp
struct DirEntry {
    uint32_t inode_num;      // 指向的inode编号
    uint16_t rec_len;        // 本条目长度
    uint8_t  name_len;       // 文件名长度
    uint8_t  file_type;      // 文件类型
    char     name[256];      // 文件名
};
```

#### 多级索引图解

```
                        inode
                    ┌───────────┐
                    │ direct[0] │───► 数据块0
                    │ direct[1] │───► 数据块1
                    │    ...    │
                    │direct[11] │───► 数据块11
                    ├───────────┤
                    │ indirect  │───► ┌─────┐
                    ├───────────┤     │ ptr │───► 数据块
                    │  double   │     │ ptr │───► 数据块
                    │ indirect  │     │ ... │
                    ├───────────┤     └─────┘
                    │  triple   │
                    │ indirect  │───► (更深层索引)
                    └───────────┘

假设块大小4KB，指针4字节：
- 直接指针：12 × 4KB = 48KB
- 一级间接：1024 × 4KB = 4MB
- 二级间接：1024² × 4KB = 4GB
- 三级间接：1024³ × 4KB = 4TB
```

#### 核心操作实现思路

| 操作 | 实现步骤 |
|-----|---------|
| **创建文件** | 1. 分配inode 2. 初始化元数据 3. 在父目录添加目录项 |
| **读取文件** | 1. 通过路径找到inode 2. 根据偏移量定位数据块 3. 读取数据 |
| **写入文件** | 1. 找到inode 2. 分配新数据块（如需要）3. 写入数据 4. 更新inode |
| **删除文件** | 1. 减少link_count 2. 若为0则回收inode和数据块 3. 删除目录项 |

### Q2: 海量数据TopK问题

**问题**：10亿个数中找出最大的前K个（假设K=100）

#### 方法一：最小堆（单机内存不足时）

**核心思想**：维护一个大小为K的最小堆

```cpp
#include <queue>
#include <vector>

std::vector<int> topK(const std::vector<int>& nums, int k) {
    // 最小堆
    std::priority_queue<int, std::vector<int>, std::greater<int>> minHeap;

    for (int num : nums) {
        if (minHeap.size() < k) {
            minHeap.push(num);
        } else if (num > minHeap.top()) {
            minHeap.pop();
            minHeap.push(num);
        }
    }

    std::vector<int> result;
    while (!minHeap.empty()) {
        result.push_back(minHeap.top());
        minHeap.pop();
    }
    return result;
}
```

**复杂度**：
- 时间：O(N × logK)
- 空间：O(K)

#### 方法二：分治法（数据量极大时）

```
10亿数据（无法一次载入内存）
        │
        ▼
┌───────┴───────┬───────┬───────┐
│ 文件1(1亿)     │ 文件2 │  ...  │ 文件10
│     │         │       │       │
│     ▼         ▼       ▼       ▼
│  Top100    Top100  Top100  Top100
│     │         │       │       │
└─────┴─────────┴───────┴───────┘
              │
              ▼
        合并1000个数
              │
              ▼
          最终Top100
```

**实现步骤**：
1. 将大文件切分为多个小文件
2. 对每个小文件求TopK
3. 合并所有小文件的TopK结果，再求TopK

#### 方法三：快速选择（QuickSelect）

基于快排思想，平均O(N)时间复杂度找到第K大元素：

```cpp
int partition(std::vector<int>& nums, int left, int right) {
    int pivot = nums[right];
    int i = left;
    for (int j = left; j < right; ++j) {
        if (nums[j] > pivot) {  // 降序
            std::swap(nums[i++], nums[j]);
        }
    }
    std::swap(nums[i], nums[right]);
    return i;
}

int quickSelect(std::vector<int>& nums, int left, int right, int k) {
    if (left == right) return nums[left];

    int pivotIndex = partition(nums, left, right);

    if (pivotIndex == k - 1) {
        return nums[pivotIndex];
    } else if (pivotIndex > k - 1) {
        return quickSelect(nums, left, pivotIndex - 1, k);
    } else {
        return quickSelect(nums, pivotIndex + 1, right, k);
    }
}
```

#### 方法四：MapReduce（分布式场景）

```
           ┌─────────────────────────────────────┐
           │              Master                 │
           └───────────────┬─────────────────────┘
                           │ 分发任务
        ┌──────────────────┼──────────────────────┐
        ▼                  ▼                      ▼
   ┌─────────┐       ┌─────────┐           ┌─────────┐
   │ Worker1 │       │ Worker2 │    ...    │ WorkerN │
   │ Map阶段 │        │ Map阶段 │           │ Map阶段  │
   │ 局部TopK│        │ 局部TopK│           │ 局部TopK │
   └────┬────┘       └────┬────┘           └────┬────┘
        │                 │                     │
        └─────────────────┼─────────────────────┘
                          ▼
                   ┌─────────────┐
                   │  Reduce阶段 │
                   │  全局TopK   │
                   └─────────────┘
```

#### 方法对比

| 方法 | 时间复杂度 | 空间复杂度 | 适用场景 |
|-----|-----------|-----------|---------|
| 最小堆 | O(NlogK) | O(K) | K较小，数据可流式处理 |
| 快速选择 | O(N) 平均 | O(1) | 数据可全部载入内存 |
| 分治 | O(N) | O(K) | 单机内存不足 |
| MapReduce | O(N/M) | O(K) | 分布式集群 |

### Q3: 数据库怎么设计的，为什么要这么设计？

**回答框架**：围绕业务场景 → 设计决策 → 权衡取舍

#### 设计原则

| 原则 | 说明 |
|-----|------|
| **范式化** | 减少数据冗余，保证一致性（OLTP场景） |
| **反范式化** | 适当冗余，减少JOIN，提高查询性能 |
| **垂直拆分** | 按业务模块拆分表 |
| **水平拆分** | 按数据量拆分（分库分表） |

#### 典型设计决策及理由

```sql
-- 1. 主键设计：使用自增ID还是UUID？
-- 自增ID：有序插入，B+树效率高，但分布式环境有冲突
-- UUID：全局唯一，但无序导致页分裂
-- 推荐：分布式ID生成器（雪花算法）

-- 2. 索引设计
CREATE INDEX idx_user_status_time ON orders(user_id, status, create_time);
-- 理由：符合最左前缀原则，覆盖高频查询

-- 3. 字段类型选择
status TINYINT NOT NULL DEFAULT 0,  -- 用整数而非字符串
create_time DATETIME NOT NULL,       -- 避免使用字符串存时间
```

#### 回答示例

> "我们的订单表设计时，考虑到**查询模式**主要是按用户ID查询，所以将user_id作为分片键。为了避免**热点问题**，采用了用户ID取模的方式分散数据。同时为了满足**运营后台**按时间范围查询的需求，我们建立了时间维度的二级索引..."

## bilibili

### Q1: 四千万数据，优先分库分表还是单库分表？

**答案**：通常**优先单库分表**，当单库达到瓶颈时再分库。

#### 决策依据

```
数据量判断：
┌─────────────────────────────────────────────────────────┐
│  < 500万     │ 不需要分表，优化索引即可                    │
│  500万~2000万 │ 可以考虑单库分表                          │
│  2000万~1亿  │ 单库分表 + 读写分离                        │
│  > 1亿       │ 需要分库分表                              │
└─────────────────────────────────────────────────────────┘
```

#### 单库分表 vs 分库分表

| 对比项 | 单库分表 | 分库分表 |
|-------|---------|---------|
| **复杂度** | 低 | 高 |
| **事务支持** | 原生支持 | 需要分布式事务 |
| **JOIN查询** | 支持 | 困难 |
| **连接数** | 单库连接数限制 | 可横向扩展 |
| **运维成本** | 低 | 高 |
| **适用场景** | 数据量大但QPS不高 | 数据量大且QPS高 |

#### 四千万数据的建议方案

```
推荐：单库分表（按ID取模分16~64张表）

理由：
1. 4000万数据单表性能确实会下降
2. 但单库通常能支撑到上亿数据
3. 分库会引入分布式事务、跨库JOIN等复杂问题
4. 先把简单方案用到极限，再考虑复杂方案
```

**分表策略示例**：
```sql
-- 按user_id取模分表
SELECT * FROM orders_{user_id % 16} WHERE user_id = ?;

-- 使用中间件（ShardingSphere、MyCat）透明化分表逻辑
```

### Q2: 用户量增大后，如何提高系统效率？

#### 优化层次（由简到繁）

```
┌────────────────────────────────────────┐
│  1. 代码层优化（成本最低）                │
│    - 算法优化、减少不必要的计算            │
│    - N+1查询问题                        │
├────────────────────────────────────────┤
│  2. 数据库优化                          │
│    - 索引优化、SQL优化                   │
│    - 读写分离                           │
│    - 分库分表                           │
├────────────────────────────────────────┤
│  3. 缓存层                              │
```

### Q1: 三道算法题（含LRU）

#### LRU缓存实现

**核心思想**：`哈希表 + 双向链表`
- 哈希表：O(1) 查找
- 双向链表：O(1) 插入/删除，维护访问顺序

```cpp
class LRUCache {
    int capacity_;
    std::list<std::pair<int, int>> cache_;  // 双向链表：{key, value}
    std::unordered_map<int, std::list<std::pair<int, int>>::iterator> map_;

public:
    explicit LRUCache(int capacity) : capacity_(capacity) {}

    int get(int key) {
        auto it = map_.find(key);
        if (it == map_.end()) return -1;

        // 移动到链表头部（最近使用）
        cache_.splice(cache_.begin(), cache_, it->second);
        return it->second->second;
    }

    void put(int key, int value) {
        auto it = map_.find(key);
        if (it != map_.end()) {
            // key存在：更新值并移到头部
            it->second->second = value;
            cache_.splice(cache_.begin(), cache_, it->second);
            return;
        }

        // key不存在：检查容量
        if (cache_.size() == capacity_) {
            // 淘汰尾部（最久未使用）
            int old_key = cache_.back().first;
            cache_.pop_back();
            map_.erase(old_key);
        }

        // 插入头部
        cache_.emplace_front(key, value);
        map_[key] = cache_.begin();
    }
};
```

**时间复杂度**：get/put 均为 O(1)

### Q2: 如何用Redis实现分布式锁？

#### 基本实现

```bash
# 加锁（SET NX EX 原子操作）
SET lock_key unique_value NX EX 30

# 解锁（Lua脚本保证原子性）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

#### 关键要点

| 要点 | 说明 | 风险 |
|-----|------|------|
| **原子性加锁** | 必须用 `SET NX EX` 一条命令 | 分开执行可能导致死锁 |
| **设置过期时间** | 防止客户端崩溃后锁无法释放 | 业务执行时间 > 锁超时时间 |
| **唯一标识** | 解锁时验证是否是自己的锁 | 误删他人锁 |
| **Lua脚本解锁** | 保证"检查+删除"原子性 | 非原子操作可能误删 |

#### 进阶：Redlock算法

单Redis节点存在单点故障，Redlock使用多个独立Redis节点：

```
1. 获取当前时间戳T1
2. 依次向N个节点请求加锁（设置相同过期时间）
3. 如果在 (N/2 + 1) 个节点加锁成功，且总耗时 < 锁过期时间，则加锁成功
4. 加锁失败则向所有节点发送解锁请求
```

#### 更好的选择：Redisson

```java
// Redisson提供了开箱即用的分布式锁
RLock lock = redisson.getLock("myLock");
lock.lock();
try {
    // 业务逻辑
} finally {
    lock.unlock();
}
```

### Q4: Redis命令超时但执行了，数据不对怎么办？

**问题场景**：客户端发送命令后超时断开，但Redis实际执行了命令，导致数据不一致。

#### 解决方案

**1. 幂等性设计（最重要）**
```bash
# ❌ 非幂等操作
INCR counter  # 重试会导致多次自增

# ✅ 幂等操作
SET counter 100  # 重试结果相同
SETNX key value  # 已存在则不操作
```

**2. 业务流水号去重**
```python
def transfer_money(tx_id, amount):
    # 先检查是否已处理
    if redis.sismember("processed_tx", tx_id):
        return "already processed"

    # 执行业务逻辑
    redis.incrby("balance", amount)

    # 记录已处理
    redis.sadd("processed_tx", tx_id)
```

**3. Lua脚本保证原子性**
```lua
-- 检查 + 操作 原子执行
if redis.call("sismember", "processed", ARGV[1]) == 1 then
    return "duplicate"
end
redis.call("sadd", "processed", ARGV[1])
redis.call("incrby", "balance", ARGV[2])
return "ok"
```

**4. 对账机制**
- 定期对比Redis数据与数据库数据
- 发现不一致时触发修复逻辑

#### 常见瓶颈点

| 层次 | 瓶颈 | 表现 | 排查方法 |
|-----|------|------|---------|
| **数据库** | 慢查询、锁竞争 | RT飙高、超时 | 慢查询日志、`SHOW PROCESSLIST` |
