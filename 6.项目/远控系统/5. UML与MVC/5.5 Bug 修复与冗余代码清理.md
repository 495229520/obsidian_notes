## Bug 1：文件下载发送文件大小而非文件内容

> 📎 完整调试过程见 [[Debug-007 文件下载发送文件大小而非文件内容]]

### 问题分析

5.3 解耦重构将 `Send(pack)` 改为 `lstPacket.push_back()`。在 `DownloadFile` 的 `fread` 循环中，`CPacket` 的参数被误写：

```
DownloadFile 中有两种 CPacket：
  1. CPacket(4, (BYTE*)&data, 8)       ← 发送文件大小（正确用途）
  2. CPacket(4, (BYTE*)buffer, rlen)    ← 发送文件内容（正确用途）

重构时，循环体内的 ② 被错误地写成了 ① 的参数
```

### 修复前后对比

> 📁 `Command.h` : DownloadFile

```cpp
// ===== 修复前（5.3 版本，错误）=====
do {
    rlen = fread(buffer, 1, 1024, pFile);
    lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));   // ❌ 发送的是文件大小
} while (rlen >= 1024);
// ...
lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));       // ❌ 文件成功时也发结束包

// ===== 修复后（174aee7，正确）=====
do {
    rlen = fread(buffer, 1, 1024, pFile);
    // 读到的是buffer，但是传的是data，所以下载文件的时候会出问题
    lstPacket.push_back(CPacket(4, (BYTE*)buffer, rlen)); // ✅ 发送文件内容
} while (rlen >= 1024);
// ...
// ✅ 只在文件打开失败时发送零长度结束包
else
{
    lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));
}
```

### 修复后的完整 DownloadFile

```cpp
int DownloadFile(std::list<CPacket>& lstPacket, CPacket& inPacket)
{
    std::string strPath = inPacket.strData;
    long long data = 0;              // 文件大小（也用作"大小为0"的标记）
    FILE* pFile = NULL;
    errno_t err = fopen_s(&pFile, strPath.c_str(), "rb");
    if (err != 0)
    {
        // fopen 失败，发送文件大小 = 0 通知客户端
        lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));
        return -1;
    }
    if (pFile != NULL)
    {
        // ===== 1. 获取文件大小并发送 =====
        fseek(pFile, 0, SEEK_END);
        data = _ftelli64(pFile);
        lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));  // 第一个包：文件大小

        // ===== 2. 分块读取并发送文件内容 =====
        fseek(pFile, 0, SEEK_SET);
        char buffer[1024] = "";
        size_t rlen = 0;
        do {
            rlen = fread(buffer, 1, 1024, pFile);
            lstPacket.push_back(CPacket(4, (BYTE*)buffer, rlen));  // 后续包：文件内容
        } while (rlen >= 1024);

        fclose(pFile);
    }
    else
    {
        // pFile 为 NULL（理论上 fopen_s err==0 时不会到这里）
        lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));
    }
    return 0;
}
```

**数据流**：

```
客户端                              服务端
  │                                   │
  │  CPacket(sCmd=4, path)           │
  │ ─────────────────────────────→   │
  │                                   ├── fopen(path, "rb")
  │                                   ├── data = ftelli64() → 文件大小
  │   CPacket(4, &data, 8)          │
  │ ←─────────────────────────────   ├── 第 1 个响应包：文件大小
  │                                   │
  │   CPacket(4, buffer, rlen)       │
  │ ←─────────────────────────────   ├── 第 2..N 个响应包：文件内容（每包 ≤1024 字节）
  │   CPacket(4, buffer, rlen)       │
  │ ←─────────────────────────────   │
  │                                   ├── fclose(pFile)
```

## Bug 2：盘符显示遗漏最后一个驱动器

> 📎 完整调试过程见 [[Debug-008 盘符显示遗漏最后一个驱动器]]

### 修复前后对比

> 📁 `RemoteClientDlg.cpp` : OnBnClickedBtnFileinfo

```cpp
// ===== 修复前（hack 方式）=====
std::string drivers = pClient->GetPacket().strData;
drivers.push_back(',');  // hack：末尾加逗号，确保最后一项被处理
// 问题：空字符串时 "," 导致插入空项

// ===== 修复后（正确方式）=====
std::string drivers = pClient->GetPacket().strData;
// 不再 push_back(',')
std::string dr;
for (size_t i = 0; i < drivers.size(); i++)
{
    if (drivers[i] == ',')
    {
        dr += ":";
        HTREEITEM hTemp = m_Tree.InsertItem(dr.c_str(), TVI_ROOT, TVI_LAST);
        m_Tree.InsertItem(NULL, hTemp, TVI_LAST);
        dr = "";
        continue;
    }
    dr += drivers[i];
}
// 处理最后一个盘符（末尾无逗号的情况）
if (dr.size() > 0)
{
    dr += ":";
    HTREEITEM hTemp = m_Tree.InsertItem(dr.c_str(), TVI_ROOT, TVI_LAST);
    m_Tree.InsertItem(NULL, hTemp, TVI_LAST);
}
```

## Bug 4：锁机线程传 NULL 参数

> 📎 完整调试过程见 [[Debug-010 锁机线程传NULL参数]]

### 修复前后对比

> 📁 `Command.h` : LockMachine

```cpp
// ===== 修复前（5.2 引入，传 NULL）=====
_beginthreadex(NULL, 0, &CCommand::threadLockDlg, NULL, 0, &threadid);
//                                                 ^^^^
// threadLockDlg 内 (CCommand*)arg 为 NULL → 访问成员变量 dlg 是 UB

// ===== 修复后（传 this）=====
_beginthreadex(NULL, 0, &CCommand::threadLockDlg, this, 0, &threadid);
//                                                 ^^^^
// threadLockDlg 内 (CCommand*)arg 指向有效对象 → 正确访问 dlg
```

**static + void\* 回调模式的完整调用链**：

```
LockMachine(this)
  │
  └── _beginthreadex(threadLockDlg, this)
        │
        └── threadLockDlg(void* arg)         // static 函数
              │
              ├── CCommand* thiz = (CCommand*)arg;  // void* → CCommand*
              └── thiz->threadLockDlgMain();         // 通过正确的 this 访问 dlg
                    │
                    ├── dlg.Create(...)              // 需要 this 指针来定位 dlg 成员
                    ├── dlg.ShowWindow(...)
                    └── dlg.DestroyWindow();
```

## 冗余代码清理

### ServerSocket.h：移除废弃方法

[[5.3 解耦命令处理和网络模块]] 之后，CCommand 不再直接调用 CServerSocket 的方法，以下 3 个方法成为死代码：

| 移除的方法 | 原用途 | 替代方案（5.3 后） |
|-----------|--------|-------------------|
| `GetFilePath(std::string&)` | CCommand 获取文件路径 | 直接从 `inPacket.strData` 读取 |
| `GetMouseEvent(MOUSEEV&)` | CCommand 获取鼠标事件 | 直接从 `inPacket.strData` memcpy |
| `GetPacket()` | 外部获取解析后的包 | 数据通过回调参数 `CPacket& inPacket` 传递 |

> 📁 `ServerSocket.h` : 删除的代码

```cpp
// ===== 以下 3 个方法在 5.3 解耦后已无人调用，本次提交删除 =====

// 曾被 MakeDirectoryInfo/RunFile/DownloadFile/DeleteLocalFile 调用
bool GetFilePath(std::string& strPath)
{
    if (((m_packet.sCmd >= 2) && (m_packet.sCmd <= 4)) ||
        (m_packet.sCmd == 9))
    {
        strPath = m_packet.strData;
        return true;
    }
    return false;
}

// 曾被 MouseEvent 调用
bool GetMouseEvent(MOUSEEV& mouse)
{
    if (m_packet.sCmd == 5)
    {
        memcpy(&mouse, m_packet.strData.c_str(), sizeof(MOUSEEV));
        return true;
    }
    return false;
}

// 曾被 main() 和 SendCommandPacket 调用
CPacket& GetPacket()
{
    return m_packet;
}
```

### 删除前后的 CServerSocket 接口对比

| 方法 | 5.2（重构前） | 5.3（解耦后） | 5.5（本次清理） |
|------|-------------|-------------|---------------|
| `getInstance()` | public | public | public |
| `Run()` | 不存在 | **新增** public | public |
| `InitSocket()` | public | protected | protected |
| `AcceptClient()` | public | protected | protected |
| `DealCommand()` | public | protected | protected |
| `Send()` | public | protected | protected（待讨论） |
| `CloseClient()` | public | protected | protected |
| `GetFilePath()` | public | protected | **已删除** |
| `GetMouseEvent()` | public | protected | **已删除** |
| `GetPacket()` | public | protected | **已删除** |

> [!question] Send() 是否也应该删除？
> 代码中留下了注释：`// 为什么Send函数应该被删除？但是现在还没有处理完毕`
>
> 分析：`Send()` 目前被 `Run()` 内部调用（遍历 `lstPackets` 逐个发送）。它不是冗余的——它是 CServerSocket 内部的网络发送方法。但从 5.3 的解耦设计来看，外部已经不需要调用 `Send()`，它已经是 protected 的，所以**不需要删除**，只是不再暴露给外部。

## 易错点与调试

> [!warning] 重构后的常见 Bug 模式

### 1. 参数复制错误

```cpp
// ❌ 重构时复制了相邻行的参数
lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));      // ← 错误地复制了这行的参数
lstPacket.push_back(CPacket(4, (BYTE*)buffer, rlen));   // ← 应该用这行的参数
```

**防范**：重构后逐个函数回归测试，不要只看编译是否通过。

### 2. void* 回调参数遗漏

```cpp
// ❌ 忘记传 this
_beginthreadex(NULL, 0, &CCommand::threadLockDlg, NULL, 0, &threadid);

// ✅ 传 this
_beginthreadex(NULL, 0, &CCommand::threadLockDlg, this, 0, &threadid);
```

**防范**：每次使用 `static + void*` 模式时，检查 `void* arg` 传参点。

### 3. 查询 API 不等于操作 API

```cpp
// ❌ HitTest 只查询，不改变选中状态
HTREEITEM h = m_Tree.HitTest(pt);
// 后续 GetSelectedItem() 返回旧值！

// ✅ 查询后显式设置选中
HTREEITEM h = m_Tree.HitTest(pt);
m_Tree.SelectItem(h);
```

**防范**：使用 MFC 控件 API 时，区分"查询类"和"操作类"方法。
