## 关键修复点（只保留“改哪里”和“为什么”）

### 1) 服务端：`_findfirst` 句柄位宽（x64 截断）

`_findfirst` 返回 `intptr_t`，x64 下用 `int` 接收会截断句柄，后续 `_findnext` 访问冲突。

```cpp
// Command.h : MakeDirectoryInfo
// 修复：int -> intptr_t（x64 安全）
intptr_t hfind = 0;
```

> 过程与验证：[[Debug-001 获取目录信息崩溃与数据丢失]]

### 2) 客户端：DealCommand “先消费缓冲区，再 recv”

典型场景是服务端批量 send 后立刻 close，客户端第一次 `recv()` 可能把多个包都读进缓冲区。
如果下一次 `DealCommand()` 先 `recv()`，而对端已经关闭导致 `recv()==0`，就会误判失败并丢弃缓冲区里尚未解析的包。

修复策略：进入循环先尝试从 `buffer/index` 解析完整包，解析成功就直接返回，不再依赖新的 `recv()`。

> 过程与验证：[[Debug-001 获取目录信息崩溃与数据丢失]]

### 3) 下载：发送参数从 `&data` 修正为 `buffer`

解耦重构后 `DownloadFile` 通过 `lstPacket` 批量发送，循环体里如果仍然发送 `(BYTE*)&data, 8`，客户端实际拿到的是“文件大小的重复包”，不是内容。

```cpp
// Command.h : DownloadFile（核心修复点）
rlen = fread(buffer, 1, 1024, pFile);
lstPacket.push_back(CPacket(4, (BYTE*)buffer, rlen));
```

> 调试过程：[[Debug-007 文件下载发送文件大小而非文件内容]]

### 4) 锁机：线程入口参数 `NULL -> this`

`static threadLockDlg(void* arg)` 内部会把 `arg` 转成 `CCommand*` 来访问成员（例如 `dlg`）。
传 `NULL` 会导致未定义行为，修复为传 `this`。

```cpp
// Command.h : LockMachine（核心修复点）
_beginthreadex(NULL, 0, &CCommand::threadLockDlg, this, 0, &threadid);
```

> 调试过程：[[Debug-010 锁机线程传NULL参数]]

### 5) 文件浏览：树控件 HitTest 结果必须落到“选中态”

`HitTest` 只是查询，不会改变选中项。后续如果继续用 `GetSelectedItem()`，会拿到旧项导致路径拼接错误。

```cpp
// RemoteClientDlg.cpp : LoadFileInfo（核心修复点）
m_Tree.SelectItem(hTreeSelected);
```

> 调试过程：[[Debug-009 树控件未设置选中状态导致路径错误]]

### 6) 盘符显示：去掉“末尾补逗号”hack

盘符列表解析时，循环只在遇到 `,` 才 flush，末尾没有 `,` 会丢最后一个盘符。
修复为循环结束后处理残余 `dr`。

> 调试过程：[[Debug-008 盘符显示遗漏最后一个驱动器]]

## 冗余代码清理（解耦后的回归）

[[5.3 解耦命令处理和网络模块]] 之后，命令层不再从网络层取“路径/鼠标/包”，而是通过回调参数 `inPacket` 直接读。
因此 `ServerSocket.h` 中的以下方法变成死代码，本次提交移除：

- `GetFilePath(std::string&)`
- `GetMouseEvent(MOUSEEV&)`
- `GetPacket()`

这个清理的核心收益是：减少接口暴露面，让“解耦”在代码层面真正落地。
