## 各文件的函数与功能

### ServerSocket.h — 项目核心

包含两个类，所有逻辑以 inline 形式写在头文件中。

#### CPacket（协议包）

| 函数 | 功能 |
|------|------|
| `CPacket()` | 默认构造，所有字段置零 |
| `CPacket(const CPacket&)` | 拷贝构造 |
| `CPacket(const BYTE*, size_t&)` | **核心**：从字节流中解析一个包。`nSize` 既是输入（缓冲区大小）也是输出（已消费字节数，0 表示失败） |
| `operator=` | 赋值运算符 |

**协议包格式**：

```
偏移  0   2        6   8        8+N    8+N+2
     ┌────┬────────┬───┬────────┬──────┐
     │FEFF│nLength │Cmd│ Data   │ Sum  │
     │ 2B │  4B    │2B │  NB    │ 2B   │
     └────┴────────┴───┴────────┴──────┘
           ←─────── nLength ──────────→
```

| 字段 | 大小 | 作用 |
|------|------|------|
| `sHead` | 2B | 魔数 `0xFEFF`，在字节流中定位包头 |
| `nLength` | 4B | 包体长度（sCmd + strData + sSum） |
| `sCmd` | 2B | 命令码，决定服务器执行什么操作 |
| `strData` | NB | 可变长数据 |
| `sSum` | 2B | 校验和（strData 所有字节累加） |

#### CServerSocket（单例服务器）

| 函数                  | 可见性            | 功能                                            |
| ------------------- | -------------- | --------------------------------------------- |
| `getInstance()`     | public static  | 获取唯一实例（懒汉式）                                   |
| `InitSocket()`      | public         | bind 端口 9527 + listen，backlog=1               |
| `AcceptClient()`    | public         | 阻塞等待客户端连接，三次握手完成后返回                           |
| `DealCommand()`     | public         | recv 循环 → CPacket 解析 → memmove 处理粘包 → 返回 sCmd |
| `Send()`            | public         | 向客户端发送原始字节数据                                  |
| `CServerSocket()`   | private        | 构造：WSAStartup + 创建 TCP socket                 |
| `~CServerSocket()`  | private        | 析构：closesocket + WSACleanup                   |
| `InitSockEnv()`     | private        | WSAStartup(1.1) 封装                            |
| `releaseInstance()` | private static | delete 单例指针，供 CHelper 调用                      |
| 拷贝构造 / `operator=`  | private        | 禁止拷贝和赋值                                       |

#### CHelper（嵌套类，负责单例自动释放）

| 函数 | 功能 |
|------|------|
| `CHelper()` | 构造时调用 `getInstance()`，在 `main()` 前触发单例创建 |
| `~CHelper()` | 析构时调用 `releaseInstance()`，程序退出时自动释放 |

### ServerSocket.cpp — 静态成员初始化

```cpp
CServerSocket* CServerSocket::m_instance = NULL;     // 单例指针
CServerSocket::CHelper CServerSocket::m_helper;      // 触发构造
CServerSocket* pserver = CServerSocket::getInstance(); // 全局便捷指针
```

注意被注释掉的一行：`//CServerSocket server;`——这是最初的全局对象方案，被放弃后注释在这里，印证了 2.1 → 2.2 的演变。

### RemoteCtrl.cpp — main() 主循环

```cpp
CServerSocket* pserver = CServerSocket::getInstance();
int count = 0;
while (CServerSocket::getInstance() != NULL)
{
    if (pserver->InitSocket() == false)     // bind + listen
        exit(0);

    if (pserver->AcceptClient() == false) { // accept 阻塞
        if (count >= 3) exit(0);            // 连续失败 3 次放弃
        count++;
        continue;
    }

    int ret = pserver->DealCommand();       // recv 循环，返回 sCmd
    // TODO: 根据 ret 分发命令（此时还没实现）
}
```

这是**单连接、同步阻塞**模型：一次只能服务一个客户端，所有操作都在主线程上串行执行。

## 设计思路深入分析

### 为什么在 main() 之前初始化 Winsock？

利用 CRT 静态对象构造机制，在 `main()` 之前的单线程环境中初始化，天然线程安全：

```
CRT 启动 → 构造 m_helper → getInstance() → new CServerSocket() → WSAStartup()
    ↓
main() 开始（Winsock 已就绪）
    ↓
main() 返回
    ↓
CRT 析构 m_helper → releaseInstance() → ~CServerSocket() → WSACleanup()
```

RAII 保证任何退出路径都会自动清理，不需要在每个 `return` / `exit` 前手动调用 `WSACleanup()`。

### 为什么用单例而非全局对象？

全局对象可以被拷贝/赋值，导致多个对象持有同一个 socket 句柄：

```cpp
CServerSocket copy = server;  // ❌ 拷贝构造
// 析构时 copy 和 server 都调用 closesocket() → 重复关闭
```

单例通过私有化构造/拷贝/赋值从根本上杜绝了这个问题。

### 为什么需要 CPacket？

TCP 是字节流协议，不保留消息边界。连续发两个 5 字节消息，recv 可能得到：

| 情况 | recv 结果 | 名称 |
|------|-----------|------|
| 合并 | 一次收到 10 字节 | **粘包** |
| 截断 | 一次只收到 3 字节 | **拆包** |
| 混合 | 收到上个包尾部 + 下个包头部 | **混合** |

CPacket 的解决方案：
1. **魔数 `0xFEFF`**：在字节流中扫描定位包头（同步机制）
2. **`nLength` 长度字段**：知道要等多少字节才是完整包（解决拆包）
3. **`memmove` 滑动窗口**：解析完一个包后移除已消费数据（解决粘包）

### DealCommand 中的粘包处理机制

```cpp
size_t index = 0;  // 缓冲区中有效数据长度
while (true) {
    // 追加接收，不覆盖未处理数据
    size_t len = recv(m_client, buffer + index, BUFFER_SIZE - index, 0);
    index += len;
    len = index;

    m_packet = CPacket((BYTE*)buffer, len);

    if (len > 0) {
        // 解析成功，移除已消费数据
        memmove(buffer, buffer + len, BUFFER_SIZE - len);
        index -= len;
        return m_packet.sCmd;
    }
    // len == 0：包不完整，继续 recv 追加
}
```

## 已知 Bug 汇总

### Bug 1：赋值与比较混淆（`10d79cd`，在 `509bd5c` 修复）

```cpp
// ❌ 10d79cd 版本
if (m_client = -1)    // 赋值！永远为 true

// ✅ 509bd5c 修复
if (m_client == -1)   // 比较
```

### Bug 2：sCmd 重复读取（`509bd5c`）

```cpp
sCmd = *(WORD*)(pData + i);  i += 2;   // 第一次读，正确
// ... 完整性检查 ...
sCmd = *(WORD*)(pData + i);  i += 2;   // ❌ 又读了一次！此时 i 已越过 sCmd
                                        //    实际读到的是 strData 的前 2 字节
```

### Bug 3：校验和循环变量错误（`509bd5c`）

```cpp
// ❌ 原代码
for (size_t j = 0; i < strData.size(); j++) {
    sum += BYTE(strData[i]) & 0xFF;
}
// i 是包解析位置（值很大），几乎不可能 < strData.size()
// 循环一次都不执行，sum 始终为 0
// 几乎所有包都校验失败

// ✅ 应该是
for (size_t j = 0; j < strData.size(); j++) {
    sum += BYTE(strData[j]) & 0xFF;
}
```

### Bug 4：DealCommand 内存泄漏（`509bd5c`）

```cpp
char* buffer = new char[BUFFER_SIZE];   // 堆分配
// ...
return m_packet.sCmd;                   // ❌ 直接 return，没有 delete[] buffer
```

每次调用 `DealCommand()` 都泄漏 4096 字节。在 `e805494`（3.3 笔记）修复。

## 设计局限与后续改进方向

| 局限 | 后续改进 |
|------|---------|
| 单连接模型，一次只能服务一个客户端 | 后续引入多线程 |
