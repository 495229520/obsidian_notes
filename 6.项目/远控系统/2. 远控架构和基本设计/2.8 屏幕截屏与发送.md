# 2.8 屏幕截屏与发送

> 本节实现远程控制中的**屏幕监控**功能，通过 GDI 截取屏幕图像并转换为 PNG 格式，利用内存流高效传输到控制端。

---

## 1. 功能概述

### 1.1 需求分析

远程控制的核心功能是让控制端能够**实时查看被控端的屏幕**：

| 需求 | 技术要点 |
|------|---------|
| **截取屏幕** | 使用 GDI 的 BitBlt 函数拷贝屏幕像素 |
| **图像编码** | 将位图转换为 PNG/JPEG 等压缩格式 |
| **内存传输** | 避免保存到磁盘，直接在内存中完成编码和发送 |
| **网络发送** | 将图像数据封装到数据包中传输 |

### 1.2 设计思路

```
┌────────────────┐                    ┌────────────────┐
│   被控端        │                    │   控制端        │
├────────────────┤                    ├────────────────┤
│ 1. 获取屏幕DC   │                    │                │
│    GetDC(NULL) │                    │                │
│                │                    │                │
│ 2. BitBlt 截图 │                    │                │
│    → CImage    │                    │                │
│                │                    │                │
│ 3. 保存到内存流 │                    │                │
│    → IStream   │                    │                │
│    (PNG格式)   │                    │                │
│                │                    │                │
│ 4. 封装数据包   │ ── 6. 图像数据 ──→  │ 解码并显示     │
│    CPacket(6)  │    (PNG 字节流)    │                │
│                │                    │                │
│ 5. 网络发送     │                    │                │
│    Send()      │                    │                │
└────────────────┘                    └────────────────┘
```

## 2. 技术选型

### 2.1 为什么使用 CImage？

| 方案 | 优势 | 劣势 |
|------|------|------|
| **GDI（HBITMAP）** | 底层控制强 | 需要手动处理位图格式，不支持压缩 |
| **GDI+（Gdiplus::Bitmap）** | 支持多种格式，功能强大 | 需要额外初始化 `GdiplusStartup()` |
| **CImage（ATL）** | **简洁易用**，自动管理资源，支持格式转换 | 仅限 Windows MFC/ATL 项目 |

**选择 CImage 的原因**：
- 项目已使用 MFC 框架，无需额外依赖
- 提供 `Save()` 方法直接保存为 PNG/JPEG
- 自动管理 HDC 和 HBITMAP 资源
- 支持保存到 `IStream`（关键优势）

### 2.2 为什么使用 IStream？

传统方法：`CImage::Save(_T("screenshot.png"), ...)` 保存到磁盘
- ❌ 磁盘 I/O 慢（~10-50ms）
- ❌ 需要管理临时文件
- ❌ 磁盘可能已满

**内存流方案**：`CImage::Save(pStream, ...)` 直接保存到内存
- ✅ 零磁盘操作，性能高
- ✅ 数据直接可用于网络发送
- ✅ 自动管理内存生命周期

## 3. 核心实现

### 3.1 SendScreen() 函数

这是**首次实现**的屏幕截图功能，完整代码如下：

```cpp
int SendScreen()
{
    // ===== 第一步：创建 GDI 图像对象 =====
    CImage screen;  // ATL 的图像类，支持格式转换

    // ===== 第二步：获取屏幕设备上下文 =====
    // NULL 表示整个屏幕（所有显示器）
    HDC hScreen = ::GetDC(NULL);

    // 获取屏幕参数
    int nBitPerpixel = GetDeviceCaps(hScreen, BITSPIXEL);   // 颜色深度（如 32 位 ARGB）
    int nWidth = GetDeviceCaps(hScreen, HORZRES);            // 屏幕宽度（像素）
    int nHeight = GetDeviceCaps(hScreen, VERTRES);           // 屏幕高度（像素）

    // ===== 第三步：创建与屏幕兼容的位图 =====
    // 在 CImage 内部创建 HBITMAP 和 HDC ，HBITMAP 相当于一块用来存储像素的连续内存
    // 在内存里开辟一块位图缓冲区，并创建一个可以往里面画图的 DC，因为 Windows 不能直接往 bitmap 里画写
    screen.Create(nWidth, nHeight, nBitPerpixel);

    // ===== 第四步：截取屏幕内容 =====
    // BitBlt: 将源 DC 的像素块拷贝到目标 DC
    // SRCCOPY: 直接拷贝源像素，不做混合
    // ⚠️ Bug: 硬编码了 1920x1020，应该使用 nWidth x nHeight
    // 从 `hScreen`（整个屏幕的设备上下文）中，拷贝一块 **1920×1020** 的像素区域，到 `screen` 这个 
    // CImage 里。
    BitBlt(screen.GetDC(), 0, 0, 1920, 1020, hScreen, 0, 0, SRCCOPY);

    // 释放屏幕 DC（必须释放，否则资源泄漏）
    ReleaseDC(NULL, hScreen);

    // ===== 第五步：分配全局内存用于流 =====
    // GMEM_MOVEABLE: 可移动内存块，由系统管理碎片
    // 初始大小为 0，流写入时自动扩展
    HGLOBAL hMem = GlobalAlloc(GMEM_MOVEABLE, 0);
    if (hMem == NULL)
        return -1;  // 内存分配失败

    // ===== 第六步：创建基于内存的 IStream =====
    IStream* pStream = NULL;
    // TRUE: 流释放时自动 GlobalFree(hMem)
    // 把这块 HGLOBAL 包装成一个 COM 的 IStream，后续就可以进行 Write、Seek、Read
    HRESULT ret = CreateStreamOnHGlobal(hMem, TRUE, &pStream);

    if (ret == S_OK)
    {
        // ===== 第七步：将图像保存为 PNG 格式到流 =====
        // Gdiplus::ImageFormatPNG: GDI+ 的 PNG 编码器
        // 图像数据写入 pStream 指向的内存块
        screen.Save(pStream, Gdiplus::ImageFormatPNG);

        // ===== 第八步：重置流的读取位置 =====
        // 保存后，流的指针在末尾，需要重置到开头才能读取
        LARGE_INTEGER bg = { 0 };
        pStream->Seek(bg, STREAM_SEEK_SET, NULL);

        // ===== 第九步：锁定全局内存并获取数据指针 =====
        // GlobalLock: 将可移动内存固定，返回实际地址
        PBYTE pData = (PBYTE)GlobalLock(hMem);
        SIZE_T nSize = GlobalSize(hMem);  // 获取 PNG 数据的实际大小

        // ===== 第十步：封装为数据包并发送 =====
        // 命令码 6 表示屏幕截图
        CPacket pack(6, pData, nSize);
        CServerSocket::getInstance()->Send(pack);

        // 解锁内存（必须与 GlobalLock 配对）
        GlobalUnlock(hMem);
    }

    /*
    // ===== 性能测试代码（已注释） =====
    // 对比 PNG 和 JPEG 的编码速度
    TRACE("jpg %d\r\n", GetTickCount64() - tick);
    for (int i = 0; i < 10; i++)
    {
        DWORD tick = GetTickCount64();
        screen.Save(_T("test2020.png"), Gdiplus::ImageFormatPNG);
        TRACE("png %d\r\n", GetTickCount64() - tick);
        tick = GetTickCount();
        screen.Save(_T("test2020.jpg"), Gdiplus::ImageFormatJPEG);
        TRACE("jpg %d\r\n", GetTickCount64() - tick);
    }
    */

    // ===== 第十一步：清理资源 =====
    pStream->Release();       // 释放流对象，自动调用 GlobalFree(hMem)
    GlobalFree(hMem);         // ⚠️ Bug: 重复释放！pStream 已设置自动释放
    screen.ReleaseDC();       // 释放 CImage 的内部 DC

    return 0;
}
```

```
┌──────────────────────┐
│        CImage        │
│                      │
│  ┌────────────────┐  │
│  │    Memory DC   │◄──── screen.GetDC()
│  └────────────────┘  │
│          │           │
│          ▼           │
│  ┌────────────────┐  │
│  │    HBITMAP     │  │
│  │ (像素缓冲区)     │  │
│  └────────────────┘  │
└──────────────────────┘
```

**关键点解析**：

1. **屏幕 DC 获取**：`GetDC(NULL)` 获取整个虚拟屏幕，支持多显示器
2. **位图拷贝**：`BitBlt()` 是 GDI 的核心函数，直接从显存拷贝像素
3. **格式转换**：`CImage::Save()` 内部调用 GDI+ 编码器，将位图压缩为 PNG
4. **内存流**：避免磁盘 I/O，PNG 数据直接保存在 `hMem` 指向的内存中
5. **流位置重置**：保存后流指针在末尾，必须 `Seek(0)` 才能读取数据
6. **全局内存**：使用 `GlobalLock/GlobalUnlock` 固定和解锁可移动内存

> 📎 数据包封装使用 [[2.3 设计网络传输包协议]] 中定义的 `CPacket` 类，命令码 6 表示屏幕截图。

## 4. GDI 截屏原理

### 4.1 关键 API 详解

#### GetDC() - 获取设备上下文

```cpp
HDC GetDC(HWND hWnd);
```

| 参数 | 说明 |
|------|------|
| `NULL` | 返回整个屏幕的 DC（包括所有窗口） |
| `hWnd` | 返回指定窗口的客户区 DC |

**注意事项**：
- 必须调用 `ReleaseDC()` 释放，否则 DC 资源泄漏
- 屏幕 DC 数量有限（默认约 5 个），泄漏会导致系统无法绘图

#### GetDeviceCaps() - 获取设备能力

```cpp
int GetDeviceCaps(HDC hdc, int index);
```

| 参数 | 返回值 |
|------|--------|
| `HORZRES` | 屏幕宽度（像素） |
| `VERTRES` | 屏幕高度（像素） |
| `BITSPIXEL` | 每像素位数（如 32 位 ARGB） |

#### BitBlt() - 位图块传输

```cpp
BOOL BitBlt(
    HDC hdcDest,        // 目标 DC
    int xDest,          // 目标起始 x
    int yDest,          // 目标起始 y
    int width,          // 拷贝宽度
    int height,         // 拷贝高度
    HDC hdcSrc,         // 源 DC
    int xSrc,           // 源起始 x
    int ySrc,           // 源起始 y
    DWORD rop           // 光栅操作码
);
```

**光栅操作码 (ROP)**：

| 操作码 | 说明 |
|--------|------|
| `SRCCOPY` | 直接拷贝源像素（最常用） |
| `SRCAND` | 源 AND 目标（掩码） |
| `SRCINVERT` | 源 XOR 目标（透明效果） |

### 4.2 多显示器支持

当前实现使用 `GetDC(NULL)` 获取**虚拟屏幕**：

```
┌──────────────┬──────────────┐
│  显示器 1     │  显示器 2     │
│  (主屏)       │  (扩展)      │
│  1920x1080   │  1920x1080   │
└──────────────┴──────────────┘
       ↓
GetDC(NULL) 返回 3840x1080 的虚拟屏幕
```

如果只截取主屏幕：

```cpp
// 获取主屏幕区域
int nWidth = GetSystemMetrics(SM_CXSCREEN);   // 主屏宽度
int nHeight = GetSystemMetrics(SM_CYSCREEN);  // 主屏高度
```

### 5.1 IStream 工作原理
![[Pasted image 20260111021800.png]]

### 5.2 内存分配策略

```cpp
// 初始分配 0 字节
HGLOBAL hMem = GlobalAlloc(GMEM_MOVEABLE, 0);

// 写入时自动扩展
// PNG 编码器通过 IStream::Write() 写入数据
// 流内部调用 GlobalReAlloc() 扩展内存
```

**为什么不预分配大小？**
- ✅ 避免浪费（小图片不占用过多内存）
- ✅ 自动适应（大图片自动扩展）
- ⚠️ 可能导致多次重新分配（性能损失）

**优化方案**：预估大小

```cpp
// 1920x1080 屏幕，32位色，未压缩约 8MB
// PNG 压缩后约 1-3MB，预分配 4MB
HGLOBAL hMem = GlobalAlloc(GMEM_MOVEABLE, 4 * 1024 * 1024);
```

### 5.3 资源管理陷阱

**当前代码的 Bug**：

```cpp
HRESULT ret = CreateStreamOnHGlobal(hMem, TRUE, &pStream);
//                                         ↑
//                                   自动释放标志

// ...

pStream->Release();  // 释放流，内部已调用 GlobalFree(hMem)
GlobalFree(hMem);    // ❌ 重复释放！导致崩溃或内存损坏
```

**正确的做法**：

```cpp
// 方案 A：让流自动释放
CreateStreamOnHGlobal(hMem, TRUE, &pStream);
pStream->Release();  // ✅ 只释放流

// 方案 B：手动管理内存
CreateStreamOnHGlobal(hMem, FALSE, &pStream);  // 不自动释放
pStream->Release();
GlobalFree(hMem);  // ✅ 手动释放内存
```

> [!warning] 资源泄漏检测
> 使用 Visual Studio 的 **内存诊断工具** 或 **Valgrind**（Linux）检测重复释放。

## 6. 图像格式选择

### 6.1 PNG vs JPEG 对比

| 特性 | PNG | JPEG |
|------|-----|------|
| **压缩方式** | 无损压缩 | 有损压缩 |
| **文件大小** | 较大（1-3MB） | 较小（100-500KB） |
| **编码速度** | 较慢（50-100ms） | 较快（20-50ms） |
| **图像质量** | 完美保留细节 | 有损失，适合照片 |
| **适用场景** | 文字、图标、截图 | 照片、视频缩略图 |

### 6.2 实测数据

代码中注释的性能测试：

```cpp
// 测试环境：1920x1080 屏幕
screen.Save(_T("test2020.png"), Gdiplus::ImageFormatPNG);  // ~80ms
screen.Save(_T("test2020.jpg"), Gdiplus::ImageFormatJPEG); // ~30ms
```

**结论**：
- 如果需要**高帧率**（如实时监控 15+ FPS），使用 JPEG
- 如果需要**高质量**（如静态截图），使用 PNG

### 6.3 优化建议

**方案 1：动态格式选择**

```cpp
// 检测屏幕内容复杂度
bool isComplex = DetectComplexity(screen);  // 自定义函数
auto format = isComplex ? Gdiplus::ImageFormatJPEG : Gdiplus::ImageFormatPNG;
screen.Save(pStream, format);
```

**方案 2：可变质量**

```cpp
// 设置 JPEG 质量参数（0-100）
CLSID jpegClsid;
GetEncoderClsid(L"image/jpeg", &jpegClsid);

EncoderParameters params;
params.Count = 1;
params.Parameter[0].Guid = EncoderQuality;
params.Parameter[0].Type = EncoderParameterValueTypeLong;
params.Parameter[0].NumberOfValues = 1;
ULONG quality = 75;  // 质量 75（平衡大小和质量）
params.Parameter[0].Value = &quality;

Gdiplus::Bitmap* pBitmap = ...;  // 从 CImage 转换
pBitmap->Save(pStream, &jpegClsid, &params);
```

## 7. 性能分析

### 7.1 各阶段耗时

| 阶段 | 操作 | 耗时估算 |
|------|------|---------|
| 1 | 获取屏幕 DC | < 1ms |
| 2 | BitBlt 截图 | 5-15ms（取决于分辨率） |
| 3 | PNG 编码 | 50-100ms（1920x1080） |
| 4 | 内存操作 | < 1ms |
| 5 | 网络发送 | 10-200ms（取决于带宽） |
| **总计** | | **65-316ms** ≈ **3-15 FPS** |

### 7.2 性能瓶颈

```
┌─────────────────────────────────────────────────────┐
│              总耗时分布（PNG 格式）                   │
├─────────────────────────────────────────────────────┤
│ BitBlt 截图:     ████░░░░░░░░░░  15%                │
│ PNG 编码:        ██████████████████████  60%        │
│ 网络发送:        ███████░░░░░░░  25%                │
└─────────────────────────────────────────────────────┘
               最大瓶颈：PNG 编码
```

### 7.3 优化策略

| 问题 | 优化方案 | 效果 |
|------|---------|------|
| PNG 编码慢 | 改用 JPEG 或降低质量 | 编码时间减少 50-70% |
| 全屏截图大 | 只截取变化区域（差分） | 数据量减少 80-95% |
| 网络延迟高 | 使用 UDP 或压缩算法 | 延迟减少 30-50% |
| 帧率低 | 降低分辨率（如 1280x720） | FPS 提升 2-3 倍 |

**差分传输示例**（高级优化）：

```cpp
// 只发送与上一帧不同的区域
CImage previousFrame;
CImage currentFrame;

std::vector<RECT> changedRegions = DetectChanges(previousFrame, currentFrame);
for (const RECT& rect : changedRegions) {
    CImage patch = ExtractRegion(currentFrame, rect);
    SendPatch(patch, rect);
}
```

## 9. 代码缺陷分析

### Bug 1：硬编码的截图尺寸

**问题代码**（RemoteCtrl.cpp:275）：

```cpp
// ❌ 硬编码 1920x1020，无法适配其他分辨率
BitBlt(screen.GetDC(), 0, 0, 1920, 1020, hScreen, 0, 0, SRCCOPY);
```

**后果**：
- 如果屏幕是 **1280x720**，只截取部分区域
- 如果屏幕是 **3840x2160**（4K），只截取左上角一部分
- **导致功能失效或显示不完整**

**修正**：

```cpp
// ✅ 使用动态获取的屏幕尺寸
BitBlt(screen.GetDC(), 0, 0, nWidth, nHeight, hScreen, 0, 0, SRCCOPY);
```

### Bug 2：重复释放全局内存

**问题代码**（RemoteCtrl.cpp:307-308）：

```cpp
HRESULT ret = CreateStreamOnHGlobal(hMem, TRUE, &pStream);
//                                         ↑
//                                  fDeleteOnRelease = TRUE

// ...

pStream->Release();  // 内部调用 GlobalFree(hMem)
GlobalFree(hMem);    // ❌ 重复释放！
```

**后果**：
- **内存损坏**：释放已释放的内存块
- **程序崩溃**：下次内存操作时触发异常
- **不确定行为**：可能覆盖其他数据

**修正**：

```cpp
// 方案 A：只释放流（推荐）
pStream->Release();
// GlobalFree(hMem);  // 删除这一行

// 方案 B：改为手动管理
CreateStreamOnHGlobal(hMem, FALSE, &pStream);  // FALSE
pStream->Release();
GlobalFree(hMem);  // 保留这一行
```

### Bug 3：缺少错误处理

**问题代码**：

```cpp
screen.Save(pStream, Gdiplus::ImageFormatPNG);
// ⚠️ 未检查返回值

PBYTE pData = (PBYTE)GlobalLock(hMem);
// ⚠️ 未检查 NULL（内存可能已被移动或无效）
```

**改进建议**：

```cpp
// ✅ 检查保存是否成功
HRESULT hr = screen.Save(pStream, Gdiplus::ImageFormatPNG);
if (FAILED(hr)) {
    pStream->Release();
    return -2;  // 编码失败
}

// ✅ 检查内存锁定
PBYTE pData = (PBYTE)GlobalLock(hMem);
if (pData == NULL) {
    pStream->Release();
    return -3;  // 内存锁定失败
}
```

### Bug 4：未释放 CImage 的 DC

**问题代码**（RemoteCtrl.cpp:309）：

```cpp
screen.ReleaseDC();  // ✅ 释放了 DC
```

**实际情况**：
- `CImage::GetDC()` 需要与 `ReleaseDC()` 配对
- 但 `screen.GetDC()` 在 `BitBlt()` 中调用后，未显式释放
- 如果循环调用 `SendScreen()`，会逐渐**泄漏 DC 资源**

**修正**：

```cpp
// 保存 DC 引用，确保释放
HDC hImageDC = screen.GetDC();
BitBlt(hImageDC, 0, 0, nWidth, nHeight, hScreen, 0, 0, SRCCOPY);
screen.ReleaseDC();  // ✅ 立即释放
```

**详细步骤**：

1. **触发**：控制端发送命令码 6 的请求包

   - 调用 `CServerSocket::Send()` 发送

## 12. 关键要点总结

1. **技术栈**：使用 `CImage` + `IStream` 实现零磁盘 I/O 的屏幕截图
2. **GDI 核心**：`GetDC()` + `BitBlt()` 拷贝屏幕像素到内存位图
3. **格式选择**：PNG 无损但慢（60% 耗时），JPEG 有损但快（建议动态选择）
4. **内存流**：`CreateStreamOnHGlobal()` 创建可扩展的内存流，避免临时文件
5. **资源管理**：注意 DC 泄漏和重复释放问题
6. **性能瓶颈**：PNG 编码占总时间的 60%，可通过 JPEG、降分辨率或差分传输优化

## 13. 常见问题

### 问题 1：截图全黑或全白

**可能原因**：
1. `BitBlt()` 的尺寸参数错误（超出屏幕范围）
2. 屏幕 DC 未正确获取
3. 多显示器环境下坐标计算错误

**排查步骤**：
```cpp
// 打印参数验证
TRACE("Screen: %dx%d, BitsPerpixel: %d\n", nWidth, nHeight, nBitPerpixel);
```

### 问题 2：内存泄漏

**原因**：
- 未释放 `CImage` 的 DC
- 未释放屏幕 DC
- 重复调用未清理资源

**解决**：使用 RAII 封装

```cpp
class ScreenCapture {
    HDC m_hScreen;
    CImage m_image;
public:
    ScreenCapture() : m_hScreen(::GetDC(NULL)) {}
    ~ScreenCapture() {
        ReleaseDC(NULL, m_hScreen);
        m_image.ReleaseDC();
    }
    // ...截图方法...
};
```

### 问题 3：发送速度慢

**优化方案**：
1. 改用 JPEG 格式（编码速度提升 2-3 倍）
2. 降低分辨率（如 1280x720）
3. 使用多线程（截图线程 + 编码线程 + 发送线程）
4. 实现差分传输（只发送变化区域）
