# 9.3 UDPHole

> Summary：`ffc419c` 本身不是“完成 UDP 打洞”的提交，而是一次很小的日志文本收口：把 `RemoteCtrl.cpp` 里父进程创建子进程后输出的 `进程ID / 线程ID` 改成 `Process ID / Thread ID`。真正让程序入口转向 UDP 穿透实验骨架的是前一个提交 `b32ad78`：`main()` 从默认调用 `iocp()`，改成按 `argc` 拆分父进程、主客户端、子客户端三个角色。当前版本的准确状态是：**进程启动骨架已经出现，但 `udp_server()` / `udp_client()` 仍只是占位函数，尚未实现真实 UDP socket、地址交换或 NAT 打洞。**

---

## 1. 本次提交推进了什么

这篇笔记按当前最新提交 `ffc419c` 记录，但机制主线要放在 `b32ad78 → ffc419c` 的连续演进里看。

| 提交 | 变化性质 | 关键结论 |
|---|---|---|
| `b32ad78` / `UDP Penetration Framework` | 新增 UDP 穿透实验入口骨架 | `RemoteCtrl.cpp` 从 `main() -> iocp()` 改成 `main(argc, argv)`，并按命令行参数把程序分成父进程、客户端进程和服务端占位函数。 |
| `ffc419c` / `Translate UDP process trace output to English` | 日志文本清理 | 只把两处父进程 `TRACE` 输出从中文改成英文，并补上文件末尾换行；机制没有继续推进。 |

因此，`9.3 UDPHole` 不应写成“UDP 打洞已经完成”，而应写成：**远控项目开始从 TCP/IOCP 服务端实验，切出一个独立的 UDPHole 进程编排入口。**

---

## 2. 与前一版的关系

在前面的 IOCP 阶段，程序默认入口很直接：

```cpp
int main()
{
    if (!CEdoyunTool::Init()) return 1;
    iocp();
    return 0;
}
```

也就是说，启动 `RemoteCtrl.exe` 后，程序直接进入 `iocp()`，再创建 `EdoyunServer` 并调用 `StartService()`。这条链路属于 [[8.1 IOCP Server Architecture — EdoyunServer Initial Design]] 里的 TCP/IOCP 服务端模型。

`b32ad78` 之后，默认入口不再直接进入 IOCP，而是先看 `argc`：

- `argc == 1`：当前进程作为“父进程”，负责拉起两个子进程，然后自己进入 `udp_server()`。
- `argc == 2`：子进程进入 `udp_client()`，默认 `ishost = true`。
- `argc > 2`：子进程进入 `udp_client(false)`，代码注释里把它当成“child client”。

这说明项目正在尝试把“一个 exe 多角色启动”作为 UDPHole 的测试方式：同一个程序，根据命令行参数扮演不同角色。

## 3. 当前进程角色拆分

![[9.3-udphole-process-roles.svg]]

这张图要表达的不是“UDP 打洞成功”，而是当前入口层面的角色切换：

1. 旧版本默认跑 IOCP 服务端。
2. 新版本默认先变成父进程。
3. 父进程通过 `CreateProcessA` 拉起两个子进程。
4. 父进程继续调用 `udp_server()`。
5. 子进程按 `argc` 进入不同 `udp_client` 分支。
6. 但当前 `udp_server()` / `udp_client()` 还没有任何 UDP 网络逻辑。

## 4. 从启动到角色分配的时序

```mermaid
sequenceDiagram
    participant User as 用户/启动器
    participant Parent as 父进程 argc==1
    participant Create as CreateProcessA
    participant ChildA as 子进程A argc==2
    participant ChildB as 子进程B 当前仍可能argc==2
    participant Func as UDP占位函数

    User->>Parent: 启动 RemoteCtrl.exe
    Parent->>Parent: CEdoyunTool::Init()
    Parent->>Parent: GetCurrentDirectoryA(MAX_PATH, wstrDir)
    Parent->>Create: argv[0] + " 1"
    Create-->>ChildA: 创建第一个子进程
    ChildA->>Func: udp_client(true)

    Parent->>Create: strCmd += "2" → argv[0] + " 12"
    Create-->>ChildB: 创建第二个子进程
    ChildB->>Func: 当前大概率仍进入 udp_client(true)

    Note over Parent,ChildB: 代码注释想要第二个客户端走 udp_client(false)，但当前拼接方式没有产生第三个参数

    Parent->>Func: udp_server()
    Func-->>Parent: printf(__FILE__, __LINE__, __FUNCTION__)
    Func-->>Parent: getchar() 暂停服务端进程
```

这个时序图也暴露了一个关键点：**当前所谓 UDPHole 不是网络层打洞，而是进程层测试框架。** 它先解决“怎么同时启动服务端和两个客户端”的问题，还没有进入“客户端如何交换公网地址并互相发 UDP 包”的问题。

## 5. 核心实现：`main(argc, argv)` 如何分配角色

下面是当前版本中最核心的入口代码。注释是为了说明本项目里的角色分配，不是原始代码逐字注释。

```cpp
int main(int argc,char* argv[])
{
    if (!CEdoyunTool::Init()) return 1;

    // ===== 1. 无参数启动：当前进程作为父进程 =====
    // 正常双击或直接运行 RemoteCtrl.exe 时，argc 通常为 1。
    // 这个分支负责创建两个子进程，然后自己进入 udp_server()。
    if (argc == 1)
    {
        char wstrDir[MAX_PATH];
        GetCurrentDirectoryA(MAX_PATH,wstrDir);

        STARTUPINFOA si;
        PROCESS_INFORMATION pi;
        memset(&si, 0, sizeof(si));
        memset(&pi, 0, sizeof(pi));

        // ===== 2. 创建第一个子进程：RemoteCtrl.exe 1 =====
        // 预期作用：让第一个子进程进入 argc == 2 分支，即 udp_client(true)。
        string strCmd = argv[0];
        strCmd += " 1";
        BOOL bRet = CreateProcessA(
            NULL,
            (LPSTR)strCmd.c_str(),
            NULL,
            NULL,
            FALSE,
            0,
            NULL,
            wstrDir,
            &si,
            &pi
        );

        if (bRet)
        {
            CloseHandle(pi.hThread);
            CloseHandle(pi.hProcess);
            TRACE("Process ID:%d\r\n", pi.dwProcessId);
            TRACE("Thread ID:%d\r\n", pi.dwThreadId);

            // ===== 3. 创建第二个子进程：当前代码会变成 RemoteCtrl.exe 12 =====
            // 注意：这里是 strCmd += "2"，不是 strCmd += " 2"。
            // 因此它并没有新增一个命令行参数，只是把 " 1" 改成了 " 12"。
            // 结果很可能仍然是 argc == 2，而不是 argc > 2。
            strCmd += "2";
            CreateProcessA(NULL, (LPSTR)strCmd.c_str(), NULL, NULL,
                           FALSE, 0, NULL, wstrDir, &si, &pi);

            // ===== 4. 这里继续使用旧的 bRet =====
            // 第二次 CreateProcessA 的返回值没有写回 bRet，
            // 所以下面的 if (bRet) 仍然判断的是第一次创建是否成功。
            if (bRet)
            {
                CloseHandle(pi.hThread);
                CloseHandle(pi.hProcess);
                TRACE("Process ID:%d\r\n", pi.dwProcessId);
                TRACE("Thread ID:%d\r\n", pi.dwThreadId);

                // ===== 5. 父进程进入服务端占位函数 =====
                udp_server(); // server code
            }
        }
    }
    else if (argc == 2) // this is main client
    {
        udp_client();
    }
    else // this is child client
    {
        udp_client(false);
    }

    // iocp(); 当前默认入口已经不再调用 IOCP 服务端。
    return 0;
}
```

这段代码的系统意义是：项目正在把同一个可执行文件拆成多个运行角色。父进程像一个“启动器”，客户端和服务端都来自同一个 exe，只靠命令行参数区分职责。

这种方式适合早期实验，因为不用额外维护三个不同程序。但它也要求参数设计必须非常清楚，否则一个小的字符串拼接错误就会让角色分配失效。

## 6. `udp_server()` / `udp_client()` 当前只是占位

当前版本的 UDP 函数还没有进入 Winsock UDP 编程，只做了函数级 trace。

```cpp
void udp_server()
{
    printf("%s(%d):%s\r\n", __FILE__, __LINE__, __FUNCTION__);
    getchar();
}

void udp_client(bool ishost)
{
    if (ishost)
    {
        printf("%s(%d):%s\r\n", __FILE__, __LINE__, __FUNCTION__);
    }
    else
    {
        printf("%s(%d):%s\r\n", __FILE__, __LINE__, __FUNCTION__);
    }
}
```

这意味着：

- `udp_server()` 只是打印当前文件、行号、函数名，然后用 `getchar()` 卡住进程，方便开发者看见父进程仍在运行。
- `udp_client(true)` 和 `udp_client(false)` 目前输出完全一样，参数 `ishost` 还没有影响行为。
- 当前没有 `socket(AF_INET, SOCK_DGRAM, IPPROTO_UDP)`。
- 当前没有 `bind()`、`sendto()`、`recvfrom()`。
- 当前没有公网地址登记、对端地址交换、同时发包、保活包或超时重试。

所以这一版只能称为 **UDPHole 启动骨架**，不能称为 UDPHole 功能完成。

## 7. UDP Hole Punching 在这里应该解决什么

真实的 UDP 打洞通常要解决的是：两个客户端都在 NAT 后面，不能直接被公网主动连接，但它们都可以主动向一个公网协调者发 UDP 包。只要双方知道彼此在 NAT 外侧暴露出来的公网 `IP:Port`，并且几乎同时向对方发 UDP 包，NAT 表里就可能出现允许对端返回的映射。

放到本项目的远控语境里，未来比较合理的角色应该是：

| 角色 | 未来职责 | 当前状态 |
|---|---|---|
| 协调端 / rendezvous server | 接收两个客户端注册，记录它们的外侧地址，把 A 的地址发给 B，把 B 的地址发给 A。 | 目前 `udp_server()` 只是占位。 |
| 主客户端 / host client | 主动注册自己，等待协调端返回对端地址，然后向对端持续发送探测包。 | 目前 `udp_client(true)` 只是打印。 |
| 子客户端 / peer client | 同样注册自己，并与主客户端同时发包形成 NAT 映射。 | 目前 `udp_client(false)` 逻辑还没有真正走通。 |

这也解释了为什么当前提交要先做“多进程”：UDP 打洞测试需要至少三个角色同时存在。如果先不解决进程启动问题，就很难在一个本地调试环境里模拟完整链路。

## 8. Win32 / C++ 关键机制说明

| API / 机制 | 在本提交中的作用 | 易错点 |
|---|---|---|
| `int main(int argc, char* argv[])` | 用参数数量区分父进程、主客户端、子客户端。 | `argc` 包含程序自身路径，所以无额外参数时通常是 1。想走第三个分支，必须真正传入两个额外参数。 |
| `GetCurrentDirectoryA(MAX_PATH, wstrDir)` | 获取当前工作目录，作为子进程的 `lpCurrentDirectory`。 | 这里只给了 `MAX_PATH` 固定缓冲区；真实项目中要注意长路径和返回值。 |
| `CreateProcessA` | 从父进程启动同一个 exe 的另一个进程。 | 当前 `lpApplicationName` 为 `NULL`，命令行要自己处理路径空格、引号和参数拼接。 |
| `PROCESS_INFORMATION` | 接收新进程和主线程句柄，以及进程 ID、线程 ID。 | `CloseHandle(pi.hThread/pi.hProcess)` 只是关闭父进程持有的句柄，不会结束子进程。 |
| `TRACE` | MFC Debug 输出。`ffc419c` 只改了这里的输出文本。 | `TRACE` 在 Release 环境下可能不可见；不要把它当成功能逻辑。 |
| `getchar()` | 临时让 `udp_server()` 不立即退出。 | 这是调试占位，不是服务端事件循环。 |

## 9. 当前版本的易错点

### 9.1 第二个子进程参数拼接错误

当前代码：

```cpp
strCmd += " 1";
// 第一次：RemoteCtrl.exe 1

strCmd += "2";
// 第二次：RemoteCtrl.exe 12
```

如果目标是让第二个子进程进入 `argc > 2` 分支，那么这里没有做到。它只是把第一个参数从 `1` 改成了 `12`，没有新增第二个参数。

更清楚的方式是直接使用角色名，而不是靠参数数量：

```cpp
RemoteCtrl.exe server
RemoteCtrl.exe host
RemoteCtrl.exe peer
```

这样 `main()` 里判断 `argv[1]` 即可，读代码的人也能一眼看懂每个进程的身份。

### 9.2 第二次 `CreateProcessA` 的返回值没有保存

当前第二次创建进程后没有写回 `bRet`：

```cpp
CreateProcessA(NULL, (LPSTR)strCmd.c_str(), NULL, NULL, FALSE, 0, NULL, wstrDir, &si, &pi);
if (bRet)
{
    // 这里判断的仍然是第一次 CreateProcessA 的结果
}
```

这会导致第二次创建失败时，父进程仍可能按成功路径继续执行 `CloseHandle` 和 `TRACE`。如果要保持当前结构，应写成：

```cpp
bRet = CreateProcessA(...);
if (bRet)
{
    CloseHandle(pi.hThread);
    CloseHandle(pi.hProcess);
}
```

### 9.3 `argv[0]` 没有处理路径空格

当前命令行由 `argv[0]` 直接拼接而来：

```cpp
string strCmd = argv[0];
strCmd += " 1";
```

如果 exe 路径里带空格，且 `lpApplicationName` 又传 `NULL`，Windows 解析命令行时可能把第一个空格前面的片段当成程序名。更稳妥的写法是：

- `lpApplicationName` 明确传入 exe 路径；或
- 命令行里给 exe 路径加引号；或
- 用统一的启动函数封装参数拼接和转义。

### 9.4 `(LPSTR)strCmd.c_str()` 是一个危险信号

`c_str()` 的语义是返回字符串内部缓冲区的指针。这里强转成 `LPSTR`，本质上是在告诉 `CreateProcessA`：“你可以把它当作可写缓冲区”。这类写法可读性差，也容易在字符串生命周期、可写性或标准版本差异上踩坑。

更稳妥的思路是先准备一个明确可写的 buffer，再传给 `CreateProcessA`。

### 9.5 `udp_client(true)` / `udp_client(false)` 现在没有行为差异

`ishost` 这个参数现在只进入两个分支，但两个分支做的事情完全相同。后续实现时，至少应该让两个角色在日志、绑定端口、注册身份、目标对端上产生差异，否则无法判断打洞测试到底跑到了哪一步。

## 10. 当前版本的准确结论

| 项目 | 状态 | 说明 |
|---|---|---|
| 最新提交 `ffc419c` | 已完成 | 只做 trace 输出英文化和文件末尾换行。 |
| UDPHole 启动骨架 | 已出现 | `main(argc, argv)` 开始按参数区分角色。 |
| 多进程启动 | 部分可用 | 第一个子进程可进入 `argc == 2`；第二个子进程当前参数拼接有问题。 |
| `udp_server()` | 占位 | 只打印函数名并等待输入。 |
| `udp_client()` | 占位 | 只打印函数名，`ishost` 暂无实际语义。 |
| 真实 UDP 打洞 | 未实现 | 没有 UDP socket、地址交换、同时发包、保活和失败回退。 |
| IOCP 默认入口 | 暂停默认使用 | `iocp()` 函数还在，但当前 `main()` 默认不再调用它。 |
| 是否需要 Debug 笔记 | 暂不单独创建 | 当前是新骨架 + 明显风险点；等后续提交修复 `strCmd += "2"` 或 `bRet` 问题时，再拆成 `Debug-XXX` 更合适。 |

## 11. 下一步实现建议

下一步不要急着写复杂 NAT 逻辑，应该先把角色和日志做清楚。

建议顺序：

1. 把命令行参数从“数量判断”改成“角色名判断”。例如：`server`、`host`、`peer`。
2. 封装 `StartChildProcess(role)`，统一处理 exe 路径、引号、当前目录、返回值和句柄关闭。
3. 让 `udp_server()` 真正创建 UDP socket，并固定监听端口。
4. 让两个 `udp_client()` 都绑定本地 UDP socket，并向服务端注册自己的身份。
5. 服务端收到两个客户端后，把双方的外侧地址互相转发。
6. 两个客户端拿到对端地址后，在短时间窗口内重复 `sendto()`，形成 NAT 映射。
7. 加入超时、重试、失败原因日志，例如“未收到服务端地址”“未收到对端探测包”“疑似对称 NAT 或防火墙阻断”。
