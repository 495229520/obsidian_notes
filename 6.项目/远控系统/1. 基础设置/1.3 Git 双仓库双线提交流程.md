# 1.3 Git 双仓库双线提交流程

## 目标

- `remoteCtrl` 保留完整历史，给需要完整过程的老师查看。
- `newremoteCtrl` 只保留公开英文历史，给外国老师分析代码。
- 本地同时维护两条线，避免旧的中文提交历史暴露到公开仓库。

## 仓库与分支映射

- 完整历史仓库：`https://github.com/hosendovebelva-boop/remoteCtrl`
  - remote 名称：`github`
  - 本地分支：`design`
  - 远端分支：`design`
- 公开英文仓库：`https://github.com/hosendovebelva-boop/newremoteCtrl`
  - remote 名称：`public`
  - 本地分支：`public-en`
  - 远端分支：`main`

## 核心规则

- 所有日常开发先在 `design` 上完成。
- `design` 允许保留完整中文和英文历史，并持续推送到 `github/design`。
- 只有纯英文提交才允许同步到 `public-en`。
- 不要把中文提交、中文注释、中文 UI 文本同步到 `newremoteCtrl`。
- 不要在 `public-en` 上执行 `merge design` 或 `rebase design`，否则旧历史会被带入公开仓库。
- 公开仓库同步方式固定为：从 `design` 上把英文提交 `cherry-pick` 到 `public-en`。
- 如果 `newremoteCtrl/main` 只有初始化内容，可以用 `git push public public-en:main --force` 重建公开历史；如果远端已有需要保留的内容，先确认再覆盖。

## 日常操作流程

1. 在 `design` 上正常开发并提交。
2. 执行 `git push github design`，更新完整历史仓库。
3. 切换到 `public-en`。
4. 执行 `git cherry-pick <英文提交SHA>`，只同步需要公开的英文提交。
5. 执行 `git push public public-en:main`，更新公开英文仓库。
6. 切回 `design` 继续开发。
