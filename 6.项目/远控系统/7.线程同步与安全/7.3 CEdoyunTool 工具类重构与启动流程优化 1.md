# 7.3 CEdoyunTool 工具类重构与启动流程优化

> 基于提交 `7f1277456ad4d39be98e12b513dd6e72cb9057ac`（2026-04-03）整理。
> 这次提交没有新增用户可见的功能界面，也没有修改网络协议或线程模型；它的核心任务是把 [[7.1 管理员权限检测]] 和 [[7.2 管理员权限获取与提权重启]] 里散落在 `RemoteCtrl.cpp` 中的全局函数，统一收拢到 `CEdoyunTool` 静态工具类中，并顺带优化了 `main()` 的启动流程和 `RunAsAdmin()` 的实现细节。

> [!note]
> 如果你还不清楚“为什么要检测管理员权限”或“提权的基本原理”，建议先读 [[7.1 管理员权限检测]] 和 [[7.2 管理员权限获取与提权重启]]。本篇只讲：这些能力在代码组织上是怎么被重新整理的，以及启动流程因此变得多么清爽。

---

## 先看清楚，这次改了什么

| 变化点 | 代码表现 | 真正含义 |
|------|------|------|
| **函数归位** | 把 `ShowError`、`IsAdmin`、`RunAsAdmin`、`WriteStartupDir`、`WriteRegisterTable` 等从 `RemoteCtrl.cpp` 移到 `CEdoyunTool` | 这些功能本来就是“通用工具”，不应该和业务主文件混在一起 |
| **新增 `Init()`** | 把 `GetModuleHandle` + `AfxWinInit` 的启动前置逻辑也抽成静态方法 | `main()` 不再关心 MFC 初始化细节 |
| **`RunAsAdmin` 瘦身** | 删除 `LogonUser`，改为直接用 `CreateProcessWithLogonW` | 简化实现层级，去掉了一个没有真正参与创建进程的 `hToken` |
| **`main()` 大瘦身** | 嵌套层级从 4 层缩到 1 层，权限分支一眼就能看完 | 启动流程的可读性大幅提升 |
| **路径获取修正** | `RunAsAdmin` 从 `GetCurrentDirectory` 改成 `GetModuleFileName` | 新进程路径不再依赖“从哪启动”，而是始终指向真实的自己 |

---

## 和上一版是什么关系

> 上一版（`49d8651`）的核心结论是：启动前必须先过管理员权限这道门；但“门”旁边的辅助代码（检测、提权、错误显示、自启动写入）都还是乱七八糟堆在 `RemoteCtrl.cpp` 里的全局函数。

| 维度 | 上一版状态 | 这次状态 |
|------|------|------|
| 代码组织 | 全局函数散落在 `RemoteCtrl.cpp` | 全部归拢到 `CEdoyunTool` 静态类 |
| `main()` 长度 | ~50 行，嵌套深 | ~20 行，扁平清晰 |
| 提权实现 | `LogonUser` + `CreateProcessWithLogonW` | 直接用 `CreateProcessWithLogonW` |
| 新进程路径 | 基于当前工作目录拼接 | 基于模块真实路径 `GetModuleFileName` |
| 错误反馈 | 只有 `OutputDebugString` | 增加了 `MessageBox` 弹窗 |

## 先把主链路讲顺

这次改动不涉及网络通信，所以核心链路是**服务端的启动流程**。可以把它理解成一道“安检口”的整理：安检规则没变，但安检流程被梳理得更顺了。

1. `main()` 一进来就问 `CEdoyunTool::IsAdmin()`。
2. 如果是管理员：
   - 先 `CEdoyunTool::Init()` 做 MFC 初始化；
   - 再 `ChooseAutoInvoke()` 处理开机自启；
   - 最后进入 `CServerSocket::Run()` 启动网络服务。
3. 如果不是管理员：
   - 调用 `CEdoyunTool::RunAsAdmin()` 尝试拉起管理员版进程；
   - 当前普通权限进程直接返回，不再继续。

### 启动流程结构图

![[7.3-启动流程.svg]]

对比上一版，最大的视觉差异就是：`main()` 不再是俄罗斯套娃了。

## 核心实现

### 1. `CEdoyunTool::IsAdmin()` —— 权限检测入库

> `RemoteCtrl/RemoteCtrl/EdoyunTool.h` : `IsAdmin()`

```cpp
static bool IsAdmin()
{
    HANDLE hToken = NULL;
    if (!OpenProcessToken(GetCurrentProcess(), TOKEN_QUERY, &hToken))
    {
        ShowError();
        return false;
    }
    TOKEN_ELEVATION eve;
    DWORD len = 0;
    if (GetTokenInformation(hToken, TokenElevation, &eve, sizeof(eve), &len) == FALSE)
    {
        ShowError();
        return false;  // 注意：这里 hToken 仍未关闭
    }
    CloseHandle(hToken);

    if (len == sizeof(eve))
    {
        return eve.TokenIsElevated;
    }
    printf("length of tokeninformation is %d\r\n", len);
    return false;
}
```

**整体职责**：和上一版完全一致，负责查询当前进程令牌的提升状态。区别只在于它现在变成了 `CEdoyunTool` 的静态成员。

**关键步骤**：
1. `OpenProcessToken` 打开当前进程令牌；
2. `GetTokenInformation(..., TokenElevation, ...)` 读取提升标志；
3. 返回 `TokenIsElevated`。

**风险点**：`GetTokenInformation` 失败路径仍然漏了 `CloseHandle(hToken)`。这个问题从 `993c1ab` 带过来，本次重构**没有修复**它。

---

### 2. `CEdoyunTool::ShowError()` —— 错误信息对用户可见

> `RemoteCtrl/RemoteCtrl/EdoyunTool.h` : `ShowError()`

```cpp
static void ShowError()
{
    LPWSTR lpMessageBuf = NULL;
    FormatMessage(
        FORMAT_MESSAGE_FROM_SYSTEM | FORMAT_MESSAGE_ALLOCATE_BUFFER,
        NULL, GetLastError(), MAKELANGID(LANG_NEUTRAL, SUBLANG_DEFAULT),
        (LPWSTR)&lpMessageBuf, 0, NULL);
    OutputDebugString(lpMessageBuf);
    MessageBox(NULL, lpMessageBuf, _T("发生错误"), 0);

    LocalFree(lpMessageBuf);
}
```

**变化**：相比 `993c1ab` 和 `49d8651` 中的旧版 `ShowError()`，这里新增了 `MessageBox`。

**项目上下文里的意义**：
- 以前错误只输出到调试器，普通用户双击运行时根本看不到失败原因；
- 现在至少会弹出一个系统错误文本框，让非开发环境也能感知问题。

### 3. `CEdoyunTool::RunAsAdmin()` —— 精简实现，修正路径

> `RemoteCtrl/RemoteCtrl/EdoyunTool.h` : `RunAsAdmin()`

```cpp
static bool RunAsAdmin()
{
    STARTUPINFO si = { 0 };
    PROCESS_INFORMATION pi = { 0 };
    TCHAR sPath[MAX_PATH] = _T("");
    GetModuleFileName(NULL, sPath, MAX_PATH);
    BOOL ret = CreateProcessWithLogonW(
        _T("Administrator"), NULL, NULL,
        LOGON_WITH_PROFILE, NULL, sPath,
        CREATE_UNICODE_ENVIRONMENT, NULL, NULL, &si, &pi);
    if (!ret)
    {
        ShowError();
        MessageBox(NULL, sPath, _T("error"), 0);
        return false;
    }
    WaitForSingleObject(pi.hProcess, INFINITE);
    CloseHandle(pi.hProcess);
    CloseHandle(pi.hThread);
    return true;
}
```

**这段代码在整条链路中的位置**：非管理员分支的“最后一搏”——尝试用 `Administrator` 账户重新启动一个自己的副本。

**和上一版相比的关键变化**：

| 旧写法 | 新写法 | 为什么改 |
|------|------|---------|
| 先 `LogonUser` 拿 `hToken` | 直接 `CreateProcessWithLogonW` | 旧版的 `hToken` 虽然拿到了，但 `CreateProcessWithTokenW` 被注释掉了，实际用的是 `CreateProcessWithLogonW`，所以 `LogonUser` 成了无效中间步骤 |
| `GetCurrentDirectory` 拼路径 | `GetModuleFileName(NULL, ...)` | 工作目录 ≠ 程序真实位置；从快捷方式、命令行其他目录启动时，旧写法会找不到 exe |
| 失败时 `::exit(0)` | `return false` | 不再粗暴退出，把错误处理权交还给调用方 |

**风险点**：`si.cb = sizeof(si);` 仍然没写。虽然 `STARTUPINFO` 结构体被 `{ 0 }` 初始化，但严格来说使用 `CreateProcess*` 系列 API 时，`cb` 字段应该显式赋值。

### 4. `CEdoyunTool::Init()` —— 把 MFC 初始化也藏起来

> `RemoteCtrl/RemoteCtrl/EdoyunTool.h` : `Init()`

```cpp
static bool Init()
{
    HMODULE hModule = ::GetModuleHandle(nullptr);
    if (hModule == nullptr)
    {
        wprintf(L"错误 GetModuleHandle 失败\n");
        return false;
    }
    if (!AfxWinInit(hModule, nullptr, ::GetCommandLine(), 0))
    {
        wprintf(L"错误: MFC 初始化失败\n");
        return false;
    }
    return true;
}
```

**整体职责**：把 `main()` 里原本的前置初始化代码抽出来，让业务入口只关心“成功/失败”的布尔结果。

### 5. `main()` 的华丽瘦身

> `RemoteCtrl/RemoteCtrl/RemoteCtrl.cpp` : `main()`

```cpp
int main()
{
    if (CEdoyunTool::IsAdmin())
    {
        if (!CEdoyunTool::Init()) return 1;
        if (ChooseAutoInvoke(INVOKE_PATH_T))
        {
            CCommand cmd;
            int ret = CServerSocket::getInstance()->Run(&CCommand::RunCommand, &cmd);
            switch (ret)
            {
            case -1:
                MessageBox(NULL, _T("网络初始化异常，未能成功初始化，请检查网络状态！"), _T("网络初始化失败"), MB_OK | MB_ICONERROR);
                break;
            case -2:
                MessageBox(NULL, _T("无法正常接入用户，自动重试"), _T("接入用户失败"), MB_OK | MB_ICONERROR);
                break;
            }
        }
    }
    else
    {
        if (CEdoyunTool::RunAsAdmin() == false)
        {
            CEdoyunTool::ShowError();
            return 1;
        }
    }
    return 0;
}
```

**旧版 `main()` 的问题**：
- 先判断 `hModule != nullptr`；
- 再判断 `AfxWinInit`；
- 再判断 `IsAdmin`；
- 然后再分支到 `RunAsAdmin` 或继续服务……

嵌套层级深，读完一圈容易晕。

**新版 `main()` 的改善**：
- 扁平化：只有“管理员 / 非管理员”一个大分支；
- 每个子分支内部再做简短判断；
- 错误处理用 `return` 提前结束，不再继续往下 indent；
- `::exit(0)` 被全部删掉，统一用 `return` 返回。

这就是本次重构最直观的价值：**同样的安全边界，但读起来像大白话。**

### 6. `ChooseAutoInvoke` 的接口微调

> `RemoteCtrl/RemoteCtrl/RemoteCtrl.cpp` : `ChooseAutoInvoke()`

```cpp
bool ChooseAutoInvoke(const CString& strPath)
{
    // ...
    if (PathFileExists(strPath)) return true;
    // ...
    if (ret == IDYES)
    {
        if (!CEdoyunTool::WriteStartupDir(strPath))
        {
            MessageBox(NULL, _T("复制文件失败，是否权限不足？\r\n"), _T("错误"), MB_ICONERROR | MB_TOPMOST);
            return false;
        }
    }
    else if (ret == IDCANCEL)
    {
        return false;
    }
    return true;
}
```

**变化**：从 `void` 改成了 `bool`，调用方可以通过返回值判断“自启动策略是否成功设置”。

在 `main()` 里体现为：
```cpp
if (ChooseAutoInvoke(INVOKE_PATH_T))
{
    // 只有自启动流程没被取消，才进入网络服务
    CCommand cmd;
    CServerSocket::getInstance()->Run(...);
}
```

这相当于给启动流程又加了一层可选的短路点：如果用户点了“取消”，连网络服务都不启动。

## 当前版本的准确结论

### 已经做对的部分

- 把通用工具能力从业务主文件里彻底剥离，职责边界更清晰；
- `main()` 的启动流程扁平化，可读性和可维护性明显提升；
- `RunAsAdmin` 去掉了多余的 `LogonUser`，路径获取也更可靠；
- `ShowError` 增加了 `MessageBox`，让非调试环境也能看到错误；
- 统一用 `return` 替代 `::exit(0)`，更符合正常程序的退出语义。

### 还没完全收口的部分

- `IsAdmin()` 内部 `GetTokenInformation` 失败时仍然漏 `CloseHandle`；
- `RunAsAdmin()` 里 `STARTUPINFO::cb` 仍未显式赋值；
- 管理员进程和普通用户 Startup 目录的语义错位问题（见 [[7.2 管理员权限获取与提权重启]]）这次没有解决；
- `CreateProcessWithLogonW("Administrator", NULL, NULL, ...)` 这种传空密码的方式在大多数系统上无法成功，提权链路本身仍只是“有框架、难落地”。

> 所以本次提交的准确定位是：**安全边界的代码整理与入口可读性优化**，而不是管理员机制的“功能收官”。

## Win32 / MFC 关键机制

### `GetModuleFileName(NULL, ...)` vs `GetCurrentDirectory`

| API | 拿到的是什么 | 为什么新版选它 |
|------|------------|--------------|
| `GetCurrentDirectory` | 当前进程的工作目录 | 可以被人为修改，和 exe 本身可能不在同一处 |
| `GetModuleFileName(NULL, ...)` | 当前可执行模块的完整路径 | 不受工作目录影响，始终指向真实的自己 |

### `CreateProcessWithLogonW`

```cpp
BOOL CreateProcessWithLogonW(
    LPCWSTR lpUsername,      // 用户名（这里传 Administrator）
    LPCWSTR lpDomain,        // 域（NULL）
    LPCWSTR lpPassword,      // 密码（NULL）
    DWORD   dwLogonFlags,    // LOGON_WITH_PROFILE
    LPCWSTR lpApplicationName,
    LPWSTR  lpCommandLine,
    DWORD   dwCreationFlags, // CREATE_UNICODE_ENVIRONMENT
    LPVOID  lpEnvironment,
    LPCWSTR lpCurrentDirectory,
    LPSTARTUPINFOW lpStartupInfo,
    LPPROCESS_INFORMATION lpProcessInformation
);
```

**项目上下文**：这是新版 `RunAsAdmin` 的核心。它不再需要先用 `LogonUser` 拿令牌，而是把“登录”和“创建进程”合并成一步。但这也意味着用户、域、密码三个参数必须一次性给对。当前代码把密码传 `NULL`，在启用 UAC 或 Administrator 有密码的系统上基本会失败。

## 易错点与调试

### 1. 以为 `CEdoyunTool::Init()` 做了更多事

```cpp
// ❌ 错误理解
CEdoyunTool::Init(); // 以为它连网络、连窗口类都初始化了

// ✅ 正确理解
if (!CEdoyunTool::Init()) return 1; // 它只是做 MFC 的最小化前置初始化
```

`Init()` 的范围非常小：只保证 `AfxWinInit` 成功。真正的网络初始化还在 `CServerSocket::Run()` 里。

### 2. 把 `return` 和 `::exit(0)` 当一回事

旧版代码在多个错误分支里写了 `::exit(0)`。这会导致：
- 局部对象不会被析构；
- MFC 的清理流程可能被跳过；
- 调试时堆栈直接断掉，不好追踪。

新版统一用 `return`，虽然 `main` 里的局部对象也不多，但至少语义上是“正常结束当前函数调用链”。

### 3. 忽略 `RunAsAdmin`  still 依赖空密码 Administrator

```cpp
CreateProcessWithLogonW(_T("Administrator"), NULL, NULL, ...)
```

很多初学者看到这段代码会以为“管理员提权已经做完了”。实际上：
-  modern Windows 默认禁用内置 Administrator 账户；
- 即使启用，也通常有密码；
- 空密码在大多数场景下登录会直接失败。

所以这个函数目前的价值更多是**教学层面的流程演示**，而不是生产环境的可靠方案。
