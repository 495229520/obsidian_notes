## 1. IOCP 是什么

`IOCP` 是 `I/O Completion Port` 的缩写，也就是 Windows 内核提供的一种**高性能异步 I/O 通知机制**。

它的核心思想很简单：

- 不让线程为 I/O 原地等待
- 不让程序反复轮询“有没有完成”
- 而是由操作系统在 I/O 真正完成时，把结果投递给少量工作线程

这也是为什么 IOCP 会成为 Windows 平台高并发服务器的核心基础设施之一。

## 2. IOCP 要解决什么问题

传统 I/O 模型通常有两个常见问题：

- **阻塞模型**：一个连接占一个线程，线程大部分时间都在等待 I/O，连接数一多，线程数和上下文切换开销就会失控。
- **轮询模型**：线程不停检查某个 I/O 是否完成，会浪费大量 CPU 时间。

IOCP 的目标就是解决这两个扩展性问题：

- 线程在没有工作时保持休眠
- I/O 完成后由内核主动通知
- 用少量线程处理大量连接或大量异步请求

一句话理解就是：**线程只在真正有工作要做时才运行。**

## 3. 核心概念

### 3.1 完成端口对象

完成端口本质上是一个**内核对象**，可以把它理解成一个由内核维护的完成队列。你可以把套接字、文件、管道等句柄关联到它上面；当这些句柄上的异步 I/O 完成后，内核就会把“完成消息”投递到这个端口里。

### 3.2 完成包

完成端口里排队的不是业务数据本身，而是一个个**完成包**。一个完成包通常会携带：

- 本次传输了多少字节
- 操作是否出错
- 一个用户自定义上下文指针
- 与异步 I/O 绑定的 `OVERLAPPED` 结构

这意味着工作线程被唤醒后，不只是知道“有事发生了”，还知道“哪件事完成了、结果是什么、后续该处理哪块上下文”。

### 3.3 线程池与并发值

IOCP 天然适合和工作线程池配合使用。创建端口时可以指定一个 `concurrency value`，通常会设为 CPU 核数或与之接近的值。Windows 内核会尽量控制“同时真正运行的工作线程数”不要超过这个上限，从而减少无意义的线程争抢和上下文切换。

要注意，这个值限制的是**并发活跃线程数**，不是“线程总数”。

### 3.4 `OVERLAPPED`

在 Windows 的异步 I/O 模型里，`OVERLAPPED` 是非常关键的结构。它既是异步 I/O 的标识，也是回到业务上下文的重要入口。很多实际项目都会把 `OVERLAPPED` 放进更大的自定义结构体里，这样当 I/O 完成时，工作线程就能顺着这个指针找到连接对象、缓冲区和状态机。

## 4. IOCP 的四步生命周期

### 4.1 发起异步 I/O

线程调用 `ReadFile`、`WriteFile`、`WSARecv`、`WSASend` 等异步接口，并传入 `OVERLAPPED` 结构。调用会很快返回，线程不用傻等结果。

### 4.2 内核完成并投递完成包

当操作系统把这次 I/O 真正做完之后，会向 IOCP 内部队列投递一个完成包。这个包里会带上字节数、错误状态，以及指回你上下文的指针。

### 4.3 工作线程被唤醒

某个阻塞在 `GetQueuedCompletionStatus` 上的工作线程会被内核唤醒，并拿到这个完成包。内核会尽量保证同时运行的工作线程数量不超过设定的并发值。

### 4.4 应用处理结果并继续下一轮

工作线程处理本次结果，比如解析数据、回包、更新状态机、继续投递下一次 `WSARecv`，然后再次回到 `GetQueuedCompletionStatus` 等待。

## 5. 为什么 IOCP 快

- **没有轮询**：没有完成就休眠，有完成才唤醒。
- **线程数可控**：不需要“一个连接一个线程”。
- **内核负责唤醒调度**：减少用户态手搓线程池带来的额外争用。
- **统一抽象**：套接字、文件、命名管道、串口等都能走相似的完成端口模型。

IOCP 的核心优势不是“某个 API 特别快”，而是它把**等待成本、唤醒成本、线程数量和上下文切换成本**都压下来了。

---

## 6. 常用 API

| API | 作用 |
|---|---|
| `CreateIoCompletionPort` | 创建完成端口，或者把某个句柄关联到现有完成端口 |
| `GetQueuedCompletionStatus` | 阻塞等待完成包，并取出结果 |
| `PostQueuedCompletionStatus` | 人工向完成端口投递一个完成包，常用于任务投递或关闭哨兵 |
| `ReadFile` / `WriteFile` | 文件类异步 I/O |
| `WSARecv` / `WSASend` | 套接字异步 I/O |

---

## 7. IOCP 的两种常见用法

### 7.1 真正的异步 I/O 模式

这是最典型的 IOCP 用法：

- 套接字或文件句柄关联到完成端口
- 程序发起重叠 I/O
- 内核在 I/O 完成时自动投递完成包
- 工作线程通过 `GetQueuedCompletionStatus` 取到结果

高性能网络服务器通常就是这么用的。

### 7.2 纯队列模式

IOCP 还有一种很实用的变体：把它当成**内核维护的任务队列**来用。

这时会调用：

```cpp
CreateIoCompletionPort(INVALID_HANDLE_VALUE, NULL, NULL, 1);
```

这里的 `INVALID_HANDLE_VALUE` 表示当前不是去关联某个真实 I/O 设备，而是只创建一个“纯完成端口队列”。

然后程序自己调用：

```cpp
PostQueuedCompletionStatus(...)
```

手动往队列里投递任务；工作线程则继续使用：

```cpp
GetQueuedCompletionStatus(...)
```

来取任务。

[[7.5.1 基于完成端口的无锁队列]] 里分析的 `CEdoyunQueue<T>`，就是这种“**把 IOCP 当作线程安全内核队列**”的用法，而不是典型的网络异步收发用法。

## 8. IOCP 在实际工程中的应用

- **Node.js（Windows）**：`libuv` 在 Windows 上使用 IOCP 作为异步 I/O 后端。
- **IIS / ASP.NET**：HTTP 请求处理和异步管线大量依赖 IOCP。
- **SQL Server**：网络 I/O 和磁盘 I/O 都大量使用类似完成端口的模型。
- **高并发游戏服务器、交易系统、网关服务**：当你要用很少的线程支撑很多并发连接时，IOCP 是常见选择。
