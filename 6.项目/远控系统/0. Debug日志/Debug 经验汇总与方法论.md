## Bug 分类统计

### 按根因分类

| 分类 | Bug 编号 | 核心教训 |
|------|---------|---------|
| **TCP 网络** | 001, 002 | TCP 是流式协议，必须处理粘包；先消费缓冲区再 recv |
| **类型安全** | 001 | x64 下句柄/指针必须用 `intptr_t`，不能用 `int` |
| **语法陷阱** | 003 | `!=` vs `|=` 编译通过但语义不同，开启 `/W4` 警告 |
| **线程安全** | 004 | 线程必须有退出机制，先停线程再销毁资源 |
| **资源管理** | 005 | 每个 `new` 必须有对应 `delete`，优先用 RAII |
| **边界条件** | 006 | `>` vs `>=` 一个字符导致循环只执行一次 |
| **重构引入** | 007 | 批量替换时参数复制错误，重构后必须回归测试 |
| **字符串解析** | 008 | 分隔符分割时，最后一个元素后无分隔符需特殊处理 |
| **MFC 控件** | 009 | `HitTest` 只查询不设选中状态，需手动 `SelectItem` |
| **回调传参** | 010 | `static + void*` 模式中，必须传正确的对象指针，不能传 NULL |
| **空指针 + 成员偏移** | 011 | `getInstance()` 返回 nullptr，`this->成员` 变成偏移量地址导致崩溃 |
| **静态成员未定义** | 011 | 类内 `static` 声明 ≠ 定义，缺少 `.cpp` 中的定义导致 RAII 析构不执行 |
| **静态初始化时序** | 012 | 静态变量构造在 `main()` 前，此时 MFC 框架未初始化，不能创建含 MFC 对象的单例 |
| **多线程竞争** | 013 | 用 socket 状态代替线程存活状态判断，导致多条网络线程并发启动，共享容器无锁竞争 |
| **循环 break 缺失** | 014 | do-while 内 SetEvent 后未 break，isAutoClosed 命令收包后线程继续阻塞在 recv |
| **语法陷阱（= vs ==）** | 015 | `if (x = INVALID_HANDLE_VALUE)` 赋值结果为真，条件始终成立，线程每次都重建 |
| **消息分发 + 语法陷阱** | 016 | `ON_MESSAGE` 接错处理函数、`switch (pPacket != NULL)` 让 ACK 永远走错分支 |
| **资源管理 / 所有权约定** | 017 | `new CPacket(pack)` 通过消息回调交给窗口后，没有明确回收者，成功路径按包泄漏 |
| **消息分发 + 状态机断链** | 018 | `CWatchDialog::OnSendPackAck` 同款三连错 + `ret == 6` 永假导致 `m_isFull` 永远为 false，渲染永远跳过 |
| **资源管理 / 内联 new** | 019 | `PostThreadMessage` 参数里内联 `new PACKET_DATA`，失败时无变量引用，无法 delete |
| **TCP 缓冲区剩余数据搬运** | 020 | 解析完一个完整包后，剩余字节的起点必须是 `pBuffer + nLen`，不能把剩余长度误当成剩余起点 |
| **完成条件时机** | 021 | 分块传输的收尾条件要在“当前块消费之后”立即判断，不要把完成逻辑押到下一次 ACK |

### 按严重程度分类

```
🔴 崩溃/功能失效/卡死 (13个)
├── 001: _findfirst 句柄截断 → 服务端崩溃
```

## 调试方法论

### 第一步：观察现象，分类问题

```
现象观察
  │
  ├── 程序崩溃？
  │     ├── 看崩溃地址特征（见下表）
  │     └── 用调试器捕获异常位置
  │
  ├── 功能失效？
  │     ├── 加 TRACE/日志看中间变量值
  │     └── 对比预期值和实际值
  │
  ├── 数据丢失/不完整？
  │     ├── 两端加计数日志对比
  │     └── 确定丢失发生在发送端还是接收端
  │
  └── 资源泄漏？
        ├── 任务管理器观察内存/句柄增长
        └── 审查所有 return 路径的资源释放
```

### 第二步：分层排查，缩小范围

```
客户端 UI → 客户端网络 → TCP 传输 → 服务端网络 → 服务端逻辑
     ↑                                                ↑
   先排查这端                                      再排查这端
```

**原则**：从日志/现象判断问题在哪一端，然后集中排查。

### 第三步：定位根因

| 方法 | 适用场景 | 示例 |
|------|---------|------|
| **断点 + 单步** | 崩溃、逻辑错误 | Debug-001: 在崩溃函数打断点 |
| **TRACE 日志** | 数据流问题 | Debug-002: 两端加计数日志对比 |
| **代码审查** | 语法/类型错误 | Debug-003: 审查发现 `!=` vs `\|=` |
| **对比测试** | 环境相关问题 | Debug-001: x86 正常 x64 崩溃 |

## 高频错误模式

### 模式 1：一字之差

```cpp
nFlags != 0x20;   // 比较，结果丢弃（Debug-003）
nFlags |= 0x20;   // 按位或赋值 ✅

rlen > 1024;      // 永远为 false（Debug-006）
rlen >= 1024;     // 正确边界 ✅

int hfind;        // 32 位截断（Debug-001）
intptr_t hfind;   // 64 位安全 ✅
```

### 模式 2：资源生命周期

```cpp
// 内存泄漏（Debug-005）
char* buf = new char[SIZE];
if (error) return -1;  // ← 未释放！

// 线程悬挂（Debug-004）
for(;;) { ... }  // ← 无退出机制！

// Socket 不可重用（Debug-005）
connect(m_sock, ...);  // 失败后 m_sock 废了
```

### 模式 3：TCP 流式陷阱

```cpp
// 清零丢数据（Debug-002）
memset(buffer, 0, SIZE);  // ← 上次剩余数据丢失！

// 先 recv 后检查（Debug-001 Bug2）
recv(sock, ...);          // ← 对端关闭返回 0
return -1;                // ← 缓冲区数据丢弃！
```

### 模式 4：重构引入的参数错误

```cpp
// 复制相邻行参数（Debug-007）
lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));      // 文件大小
lstPacket.push_back(CPacket(4, (BYTE*)&data, 8));      // ← 应该是 buffer, rlen！

// void* 回调忘传对象指针（Debug-010）
_beginthreadex(NULL, 0, threadFunc, NULL, ...);  // ← 应该传 this！
```

### 模式 6：多线程竞争与循环逻辑缺陷

```cpp
// 用 socket 状态代替线程存活（Debug-013）
if (m_sock == INVALID_SOCKET)
    _beginthread(...);   // ← socket 关闭期间多次启动线程！

if (m_sock == INVALID_SOCKET && m_hThread == INVALID_HANDLE_VALUE)
    m_hThread = (HANDLE)_beginthread(...);  // ✅ 双重判断

// do-while 内 SetEvent 后缺少 break（Debug-014）
do {
    recv(...);
    if (isAutoClosed) SetEvent(hEvent);  // ← 没有 break，循环可能继续 recv 阻塞
} while (条件);

do {
    recv(...);
    if (isAutoClosed) { SetEvent(hEvent); break; }  // ✅ 收完立即退出
} while (条件);
```

### 模式 7：内联 new 的失败路径盲区

```cpp
// ❌ 内联 new：失败时无法清理（Debug-019）
PostThreadMessage(..., (WPARAM)new PACKET_DATA(...), ...);

// ✅ 先保存变量，再处理失败路径
PACKET_DATA* pData = new PACKET_DATA(...);
bool ok = PostThreadMessage(..., (WPARAM)pData, ...);
if (!ok) delete pData;
```

### 模式 8：状态机写入路径断链

```cpp
// ❌ 判断类型写错，写入路径永远不执行（Debug-018）
bool ret = sendCmd();    // 返回 bool
if (ret == 6)            // ← bool 最大值是 1，永假
    setState(true);      // ← 从未执行，状态永远是初始值

// ✅ 检查每个状态变量的所有赋值路径是否真的能执行到
if (ret == true)         // 或直接 if (ret)
    setState(true);
```

### 模式 5：MFC 控件 API 误解

```cpp
// HitTest 不等于 SelectItem（Debug-009）
HTREEITEM h = tree.HitTest(pt);     // 只查询，不改选中状态
tree.SelectItem(h);                  // 必须手动设置 ✅

// 字符串分割遗漏末尾（Debug-008）
for (...) { if (s[i] == ',') process(); }  // ← 最后一项无逗号，遗漏！
if (remaining.size() > 0) process();        // 循环后处理 ✅
```

## 调试工具箱

### VS 调试技巧

| 技巧 | 用途 |
|------|------|
| **跨进程调试** | 本机同时运行两端，IP 填 127.0.0.1 |
| **条件断点** | 右键断点 → 条件，如 `count == 27` |
| **数据断点** | 调试 → 新建断点 → 数据断点，监控变量被修改 |
| **TRACE 宏** | `TRACE("var=%d\r\n", var)` 输出到 VS 输出窗口 |
| **OutputDebugString** | 在 Release 版本中也能输出调试信息 |

### 日志对比法

```
服务端加日志：TRACE("server:count = %d", count);
客户端加日志：TRACE("client:count = %d", count);
两边数字不一致 → 问题在传输层
两边数字一致 → 问题在显示层
```

## 2026-03-27 新增条目（续）

- [[Debug-022 文件树目录节点插入后未自动展开]] — `InsertItem` 不触发展开，异步插入场景必须手动调 `Expand`，已展开状态下重复调用是幂等的。
- [[Debug-023 recv失败时ACK传NULL导致下载状态机无法识别错误命令]] — 消息传递所有权约定：失败路径也必须发合法 `CPacket*`，传 NULL 会让所有依赖 `sCmd` 分发的逻辑统统失效。
