# 6.2 优化 RemoteDlg 线程

> 将 `CRemoteClientDlg` 中的**屏幕监控线程**和**文件下载线程**迁移到 `CClientController`，彻底完成 View 层的瘦身。Dialog 不再管理任何工作线程，所有耗时操作由 Controller 统一调度。

---

## 功能概述

| 功能 | 说明 |
|------|------|
| **监控线程迁移** | `threadWatchData` 从 Dialog 移到 Controller，改名为 `threadWatchScreen` |
| **下载线程迁移** | `threadDownFile` 从 Dialog 移到 Controller，改名为 `threadDownloadFile` |
| **View 瘦身** | Dialog 删除 4 个线程相关方法 + `SendCommandPacket`，约 200 行代码 |
| **监控流程简化** | 不再通过 `SendMessage(WM_SEND_PACKET)` 回到 UI 线程发命令，直接调 Controller |
| **地址同步** | 新增 IP/Port 控件变化通知，实时同步到 Controller |

---

## 设计背景

### 上一步遗留的问题

在 [[6.1 初步完成控制层]] 中，Controller 已经接管了**网络操作门面**（`SendCommandPacket` 等），但 Dialog 中仍然保留着两个线程：

```
CRemoteClientDlg (重构前)
  │
  ├── threadWatchData()       ← 屏幕监控线程（还在 Dialog 里）
  │     └── SendMessage(WM_SEND_PACKET, 6)  ← 通过 UI 消息绕回主线程发命令
  │
  ├── threadDownFile()        ← 文件下载线程（还在 Dialog 里）
  │     └── SendMessage(WM_SEND_PACKET, 4)  ← 同样绕回主线程
  │
  └── SendCommandPacket()     ← 网络方法（已迁移到 Controller，但 Dialog 还保留着）
```

**核心矛盾**：Dialog（View 层）在管理工作线程，这违反了 [[5.7 MVC设计模式]] 的原则 —— **View 只负责 UI 展示和用户事件捕获**，业务逻辑和线程调度应该由 Controller 管理。

### 本次优化目标

| 目标 | 具体内容 |
|------|---------|
| 线程迁移 | 监控线程和下载线程移到 Controller |
| 删除冗余 | Dialog 中的 `SendCommandPacket` 彻底删除 |
| 简化通信 | 监控线程不再绕回 UI 线程，直接调 Controller 门面方法 |
| 完善通知 | IP/Port 变化时实时通知 Controller |

---

## 核心实现

### 1. 屏幕监控线程迁移

#### 启动流程

**View 层**（只剩两行）：

```cpp
void CRemoteClientDlg::OnBnClickedBtnStartWatch()
{
    CClientController::getInstance()->StartWatchScreen();
}
```

**Controller 层**（接管全部逻辑）：

```cpp
void CClientController::StartWatchScreen()
{
    m_isClosed = false;
    // 创建监控对话框，父窗口是主对话框
    CWatchDialog dlg(&m_remoteDlg);
    // 启动监控线程，使用 static 桥接模式（详见 [[6.1 初步完成控制层#5. 工作线程：static 桥接模式]]）
    m_hThreadWatch = (HANDLE)_beginthread(
        &CClientController::threadWatchScreen,  // static 桥接函数
        0,
        this    // 传递 this 指针
    );
    dlg.DoModal();              // 阻塞在这里，直到用户关闭监控窗口
    m_isClosed = true;          // 通知监控线程退出
    WaitForSingleObject(m_hThreadWatch, 500);  // 等待线程结束（最多 500ms）
}
```

**关键设计**：`dlg.DoModal()` 会阻塞当前线程（UI 线程），但监控线程在另一个线程中持续抓屏。当用户关闭对话框时，`DoModal()` 返回，设置 `m_isClosed = true` 通知线程退出。

#### 监控线程主体（重点：简化了通信路径）

```cpp
void CClientController::threadWatchScreen()
{
    Sleep(50);   // 等待对话框创建完成
    while (!m_isClosed)
    {
        // 检查 View 的图片缓存是否为空
        if (m_remoteDlg.isFull() == false)
        {
            // ✅ 直接调 Controller 的 SendCommandPacket，不再绕回 UI 线程
            int ret = SendCommandPacket(6);
            if (ret == 6)
            {
                // 通过工具类将字节流转为图片
                if (GetImage(m_remoteDlg.GetImage()) == 0)
                {
                    m_remoteDlg.SetImageStatus(true);  // 标记缓存有数据
                }
                else
                {
                    TRACE("获取图片失败！ret = %d \r\n", ret);
                }
            }
        }
        Sleep(1);  // 避免 CPU 空转
    }
}

// static 桥接函数（标准套路）
void CClientController::threadWatchScreen(void* arg)
{
    CClientController* thiz = (CClientController*)arg;
    thiz->threadWatchScreen();  // 调用同名成员函数
    _endthread();
}
```

#### 重构前后的通信路径对比

这是本次优化最重要的变化 —— 消除了一次不必要的跨线程消息传递：

```
重构前（绕回 UI 线程）:
  监控线程 ──SendMessage(WM_SEND_PACKET)──> UI 线程 ──SendCommandPacket──> 网络层
                  ↑                              │
                  └──────── 等待返回 ─────────────┘
  问题：SendMessage 是同步的，监控线程阻塞等待 UI 线程处理
        如果 UI 线程正忙（比如在处理用户操作），监控线程就卡住了

重构后（直接调用）:
  监控线程 ──SendCommandPacket──> 网络层
  优势：不依赖 UI 线程的消息循环，通信更直接
```

**为什么重构前要绕回 UI 线程？**

原来 `SendCommandPacket` 定义在 `CRemoteClientDlg` 中，而 MFC 的 `SendMessage` 会把消息投递到创建窗口的线程（即 UI 线程）来执行。这是因为 Dialog 是窗口对象，MFC 要求窗口操作在创建它的线程中进行。

现在 `SendCommandPacket` 搬到了 Controller（非窗口对象），在任何线程中调用都安全，不再需要绕回 UI 线程。

#### View 层新增的访问接口

为了让 Controller 的监控线程能操作 Dialog 的图片缓存，`CRemoteClientDlg` 新增了三个 public 方法：

```cpp
class CRemoteClientDlg : public CDialogEx
{
public:
    // 查询缓存状态
    bool isFull() const { return m_isFull; }
    // 获取图片缓存的引用（Controller 可以直接写入）
    CImage& GetImage() { return m_image; }
    // 设置缓存状态
    void SetImageStatus(bool isFull = false) { m_isFull = isFull; }
};
```

这三个方法在 [[6.1 初步完成控制层]] 中就已添加，本次实际投入使用。Controller 通过 `m_remoteDlg.isFull()` 和 `m_remoteDlg.GetImage()` 间接操作 View 的数据。

### 2. 文件下载线程迁移

#### 入口函数：DownFile

```cpp
int CClientController::DownFile(CString strPath)
{
    // ===== 1. 弹出"另存为"对话框 =====
    CFileDialog dlg(
        FALSE,          // FALSE = 保存对话框（TRUE 是打开对话框）
        NULL,           // 默认扩展名
        strPath,        // 默认文件名（用远程文件名）
        OFN_HIDEREADONLY | OFN_OVERWRITEPROMPT,  // 隐藏只读 + 覆盖提示
        NULL,           // 文件过滤器
        &m_remoteDlg    // 父窗口
    );

    if (dlg.DoModal() == IDOK)
    {
        // ===== 2. 保存路径信息到成员变量 =====
        m_strRemote = strPath;          // 远程文件路径
        m_strLocal = dlg.GetPathName(); // 用户选择的本地保存路径

        // ===== 3. 启动下载线程 =====
        m_hThreadDownload = (HANDLE)_beginthread(
            &CClientController::threadDownloadEntry, 0, this);

        // ===== 4. 检查线程是否启动成功 =====
        if (WaitForSingleObject(m_hThreadDownload, 0) != WAIT_TIMEOUT)
        {
            // 线程已经结束了（说明启动就失败了）
            return -1;
        }

        // ===== 5. 显示等待状态 =====
        m_remoteDlg.BeginWaitCursor();
        m_statusDlg.m_info.SetWindowText(_T("命令正在执行中"));
        m_statusDlg.ShowWindow(SW_SHOW);
        m_statusDlg.CenterWindow(&m_remoteDlg);
        m_statusDlg.SetActiveWindow();
    }

    return 0;
}
```

**关键设计**：路径通过成员变量 `m_strRemote` / `m_strLocal` 传递给线程，避免了在线程函数参数中传递复杂对象。

**线程启动检查技巧**：

```cpp
if (WaitForSingleObject(m_hThreadDownload, 0) != WAIT_TIMEOUT)
```

| 参数 / 返回值 | 含义 |
|------|------|
| `0`（超时时间） | 立即返回，不等待 |
| `WAIT_TIMEOUT` | 线程还在运行（正常） |
| `WAIT_OBJECT_0` | 线程已经结束（异常，说明启动就失败了） |

#### 下载线程主体

```cpp
void CClientController::threadDownloadFile()
{
    // ===== 1. 创建本地文件 =====
    FILE* pFile = fopen(m_strLocal, "wb+");
    if (pFile == NULL)
    {
        AfxMessageBox(_T("本地没有权限保存该文件，文件无法创建!"));
        m_statusDlg.ShowWindow(SW_HIDE);
        m_remoteDlg.EndWaitCursor();
        return;
    }

    CClientSocket* pClient = CClientSocket::getInstance();
    do
    {
        // ===== 2. 发送下载命令（cmd=4），不自动关闭连接 =====
        int ret = SendCommandPacket(4, false,
            (BYTE*)(LPCSTR)m_strRemote, m_strRemote.GetLength());

        // ===== 3. 读取文件总长度（服务端先发文件大小） =====
        long long nLenght = *(long long*)pClient->GetPacket().strData.c_str();
        if (nLenght == 0)
        {
            AfxMessageBox("文件长度为零或者无法读取文件！！！");
            break;
        }

        // ===== 4. 循环接收文件数据 =====
        long long nCount = 0;
        while (nCount < nLenght)
        {
            ret = pClient->DealCommand();  // 接收一个数据包
            if (ret < 0)
            {
                AfxMessageBox("传输失败！！");
                TRACE("传输失败：ret = %d\r\n", ret);
                break;
            }
            // 写入本地文件
            fwrite(pClient->GetPacket().strData.c_str(), 1,
                pClient->GetPacket().strData.size(), pFile);
            nCount += pClient->GetPacket().strData.size();
        }
    } while (false);  // do-while(false) 用于统一出口

    // ===== 5. 清理资源 =====
    fclose(pFile);
    pClient->CloseSocket();

    // ===== 6. 更新 UI 状态 =====
    m_statusDlg.ShowWindow(SW_HIDE);
    m_remoteDlg.EndWaitCursor();
    m_remoteDlg.MessageBox(_T("下载完成！"), _T("完成"));
}

// static 桥接函数
void CClientController::threadDownloadEntry(void* arg)
{
    CClientController* thiz = (CClientController*)arg;
    thiz->threadDownloadFile();
    _endthread();
}
```

**do-while(false) 惯用法**：

这是 C/C++ 中常见的**统一出口**写法。`break` 跳出 `do` 块后，统一执行 `fclose` 和 `CloseSocket` 清理。等效于 `goto cleanup`，但更结构化：

```cpp
// do-while(false) 展开后的逻辑等价于：
pFile = fopen(...);
// ... 处理逻辑，任何错误都 goto cleanup ...
cleanup:
    fclose(pFile);
    pClient->CloseSocket();
```

#### View 层的调用方式

```cpp
// 重构后的 OnDownloadFile —— View 只负责组装参数，逻辑全交给 Controller
void CRemoteClientDlg::OnDownloadFile()
{
    // 获取用户选中的文件名
    int nListSelected = m_List.GetSelectionMark();
    CString strFile = m_List.GetItemText(nListSelected, 0);
    // 拼接完整的远程路径
    HTREEITEM hSelected = m_Tree.GetSelectedItem();
    strFile = GetPath(hSelected) + strFile;
    // 一行搞定，Controller 处理其余所有事情
    int ret = CClientController::getInstance()->DownFile(strFile);

    if (ret != 0)
    {
        MessageBox(_T("下载失败！"));
        TRACE("下载失败 ret = %d\r\n", ret);
    }
}
```

**对比重构前**：原来 `OnDownloadFile()` 要启动线程、设置等待光标、显示状态对话框，这些 UI 状态管理和线程创建的代码全部散在 Dialog 中。现在 View 只做"组装远程路径 + 调 Controller"这一件事。

### 4. IP/Port 变化实时通知

新增两个消息处理函数，在用户修改 IP 地址或端口时，实时同步到 Controller：

```cpp
// IP 地址控件字段变化通知
void CRemoteClientDlg::OnIpnFieldchangedIpaddressServ(NMHDR* pNMHDR, LRESULT* pResult)
{
    LPNMIPADDRESS pIPAddr = reinterpret_cast<LPNMIPADDRESS>(pNMHDR);
    *pResult = 0;

void CRemoteClientDlg::OnEnChangeEditPort()
{

    CClientController* pController = CClientController::getInstance();
    pController->UpdateAddress(m_server_address, atoi((LPCTSTR)m_nPort));

```

**MFC 消息映射注册**：

```cpp
BEGIN_MESSAGE_MAP(CRemoteClientDlg, CDialogEx)

    ON_NOTIFY(IPN_FIELDCHANGED, IDC_IPADDRESS_SERV,

    ON_EN_CHANGE(IDC_EDIT_PORT,
```

|-----|---------|---------|
| `ON_NOTIFY(IPN_FIELDCHANGED, ...)` | IP 控件通知 | IP 地址任一字段修改后 |

## 线程函数的 static 桥接回顾

本次新增的两个线程同样使用了 **static 桥接模式**（在 [[6.1 初步完成控制层#5. 工作线程：static 桥接模式]] 中已详细讲解）：

| 线程 | static 桥接函数 | 实际执行函数 |
|------|----------------|-------------|
| 监控 | `threadWatchScreen(void* arg)` | `threadWatchScreen()` |
| 下载 | `threadDownloadEntry(void* arg)` | `threadDownloadFile()` |
| 消息循环 | `threadEntry(void* arg)` | `threadFunc()` |

三者都是同一个套路：static 函数接收 `void* arg` → 转型为 `CClientController*` → 调用同名/对应的成员函数 → `_endthread()`。

注意监控线程用的是 `_beginthread` 而非 `_beginthreadex`：

| API | 返回值 | 线程函数签名 | 关闭方式 |
|-----|--------|-------------|---------|
| `_beginthread` | `uintptr_t` | `void func(void*)` | 自动关闭句柄，退出用 `_endthread()` |
| `_beginthreadex` | `uintptr_t` | `unsigned __stdcall func(void*)` | 需手动 `CloseHandle`，退出用 `_endthreadex()` |

两个 API 都可以用，但 `_beginthreadex` 更安全（不会自动关闭句柄，避免竞态条件）。消息循环线程用 `_beginthreadex`，监控和下载线程用 `_beginthread`，属于项目中的混用。

## 易错点与调试

### 1. 监控线程直接操作 MFC 控件的风险

```cpp
// ⚠️ Controller 线程中操作 View 的数据
m_remoteDlg.SetImageStatus(true);
```

`SetImageStatus` 只是设置一个 `bool` 标志位，没有操作 MFC 控件，所以是安全的。但如果在非 UI 线程中直接调用 MFC 控件方法（如 `SetWindowText`、`InsertItem`），会导致未定义行为或崩溃，因为 MFC 控件只能在创建它们的线程中操作。

```cpp
// ❌ 危险：非 UI 线程直接操作控件
m_remoteDlg.m_List.InsertItem(0, "test");

// ✅ 安全：只操作普通成员变量
m_remoteDlg.SetImageStatus(true);
```

### 2. 下载线程中路径传递方式

```cpp
// ✅ 通过成员变量传递路径（线程安全前提：DownFile 中先赋值再启线程）
m_strRemote = strPath;
m_strLocal = dlg.GetPathName();
m_hThreadDownload = (HANDLE)_beginthread(&CClientController::threadDownloadEntry, 0, this);
```

赋值在 `_beginthread` **之前**，线程启动后只读 `m_strRemote` / `m_strLocal`，不会再修改。这种"先写后读"的时序保证了线程安全（存在隐式的 happens-before 关系：线程创建点之前的写操作对新线程可见）。

但如果用户在下载过程中再次点击下载另一个文件，`m_strRemote` / `m_strLocal` 会被覆盖，导致第一次下载的路径信息丢失。当前设计**不支持并发下载**。

### 3. WaitForSingleObject 超时值

```cpp
// StartWatchScreen 中
WaitForSingleObject(m_hThreadWatch, 500);  // 500ms 超时
```

如果监控线程在 500ms 内没有退出（比如正在等待网络响应），`WaitForSingleObject` 会超时返回，线程可能还在运行。这是一个潜在问题 —— 线程可能在 Controller 已经认为它结束后继续访问成员变量。
