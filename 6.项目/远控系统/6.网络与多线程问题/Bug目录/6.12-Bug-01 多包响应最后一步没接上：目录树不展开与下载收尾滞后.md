# 6.12-Bug-01 多包响应最后一步没接上：目录树不展开与下载收尾滞后

> 这次提交里最表层的两个现象，一个是“目录项明明回来了，但树节点看起来像没展开”，另一个是“最后一块文件已经写完了，但下载流程没有立刻结束”。  
> 它们背后其实是同一种问题：**多包响应已经到达窗口层，但最后一步 UI 落地或状态收尾没有接上。**

---

## 现象

1. 单击目录节点后，请求已经发出，也能收到 `FILEINFO` 包，但树控件的可见展开状态不对，用户会误以为“目录没加载出来”。
2. 文件下载时，最后一块数据已经 `fwrite` 到本地文件，但等待光标和完成提示不一定立刻结束。

---

## 根因拆开看

### 1. 目录链路的问题不在“没收到包”，而在“收到后没把可见状态补完”

`OnSendPackAck(case 2)` 已经能把目录项插到父节点下面，但如果父节点不主动展开：

- 占位子节点虽然被删掉了
- 真实子项虽然也插进去了
- 用户仍然可能在单击后看不到变化

所以这次补了一句：

```cpp
m_Tree.Expand((HTREEITEM)lParam, TVE_EXPAND);
```

这句修的不是协议，而是“ACK 到了以后，UI 最后一跳怎么让用户看见”。

### 2. 下载链路的问题不在“没写入最后一块”，而在“完成条件判断得太晚”

旧逻辑大意是：

```cpp
if (length == 0) {
    // 读总长度
}
else if (length > 0 && index >= length) {
    // 只有下一次回调进来，才会收尾
}
else {
    fwrite(...);
    index += head.strData.size();
}
```

问题在于：

- 当前块写完后，`index` 可能已经刚好达到 `length`
- 但这一轮回调不会再次判断
- 收尾只能指望“下一次 ACK”再进来一次

这就把“下载已完成”错误地绑定到了下一轮回调，而不是当前块的真实写入结果。

---

## 这次怎么修

### 目录链路

- `LoadFileInfo()` 只负责发送 `cmd=2`
- 发送时把 `HTREEITEM` 一起带出去
- ACK 回来后按 `lParam` 把目录项插到正确树节点
- 插完立刻 `Expand`

### 下载链路

在 `fwrite` 之后立刻判断：

```cpp
FILE* pFile = (FILE*)lParam;
fwrite(head.strData.c_str(), 1, head.strData.size(), pFile);
index += head.strData.size();
if (index >= length)
{
    fclose((FILE*)lParam);
    length = 0;
    index = 0;
    CClientController::getInstance()->DownloadEnd();
}
```

这等于把“完成”重新定义成：

> 当前业务目标已经完成，而不是下一次网络事件碰巧又来了。
