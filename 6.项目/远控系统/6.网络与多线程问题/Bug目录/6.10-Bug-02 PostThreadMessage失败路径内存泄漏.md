# 6.10-Bug-02 PostThreadMessage 失败路径内存泄漏

> `CClientSocket::SendPacket` 里，`new PACKET_DATA` 写在 `PostThreadMessage` 的参数里。
> 如果 `PostThreadMessage` 失败，分配的堆对象永远泄漏，因为连引用它的变量名都没有。

---

## 问题代码

```cpp
// ❌ 旧版本 CClientSocket::SendPacket
bool CClientSocket::SendPacket(HWND hWnd, const CPacket& pack, bool isAutoClosed, WPARAM wParam)
{
    UINT nMode = isAutoClosed ? CSM_AUTOCLOSE : 0;
    std::string strOut;
    pack.Data(strOut);
    bool ret = PostThreadMessage(
        m_nThreadID,
        WM_SEND_PACK,
        (WPARAM)new PACKET_DATA(strOut.c_str(), strOut.size(), nMode, wParam),  // ← 内联 new
        (LPARAM)hWnd
    );
    return ret;
}
```

`new PACKET_DATA(...)` 直接嵌在函数调用的参数里，没有赋值给任何变量。

---

## 为什么会泄漏

`PostThreadMessage` 不是一个"永远成功"的函数。它可能因为以下原因返回 `false`：

| 失败原因 | 触发场景 |
|---------|---------|
| 目标线程消息队列已满 | 短时间内发包过多 |
| 目标线程 ID 无效 | 线程已经退出 |
| 系统资源不足 | 极少见但可能 |

正常情况下，`PostThreadMessage` 成功，消息被投递到网络线程，网络线程的 `SendPack` 函数里会 `delete (PACKET_DATA*)wParam`，所有权交接完成。

失败时：
- 消息没有被投递
- 没有任何代码会处理这个 `PACKET_DATA`
- 旧代码里连 `pData` 变量都没有，**你甚至不知道去 delete 谁**
- `PACKET_DATA` 就这样永远留在堆上

---

## 修复方案

先保存指针，再传给 `PostThreadMessage`，失败时自己清理：

```cpp
// ✅ 新版本
bool CClientSocket::SendPacket(HWND hWnd, const CPacket& pack, bool isAutoClosed, WPARAM wParam)
{
    UINT nMode = isAutoClosed ? CSM_AUTOCLOSE : 0;
    std::string strOut;
    pack.Data(strOut);

    // 先保存指针
    PACKET_DATA* pData = new PACKET_DATA(strOut.c_str(), strOut.size(), nMode, wParam);
    bool ret = PostThreadMessage(m_nThreadID, WM_SEND_PACK, (WPARAM)pData, (LPARAM)hWnd);

    if (ret == false)
    {
        // 发送失败，没有接收方，自己负责回收
        delete pData;
    }
    return ret;
}
```

所有权规则变得清晰：
- `PostThreadMessage` 成功 → 所有权转移给网络线程，由 `SendPack` 负责 `delete`
- `PostThreadMessage` 失败 → 所有权留在 `SendPacket`，自己 `delete`

---

## 内联 new 的通用风险

这个 bug 的根源是一种写法习惯：**把 `new` 嵌在函数参数里**。

```cpp
func((WPARAM)new Foo(...));
```

这种写法有三个问题：

**1. 失败路径无法清理**

函数失败时，你没有变量名来引用这个对象，无法 `delete`。

**2. 异常场景更危险**

在有异常的代码里，如果函数抛出异常，内联 `new` 的对象会直接泄漏（C++ 函数参数求值顺序不确定，可能在构造和传递之间抛出）。

**3. 语义不清晰**

读代码的人需要搜索函数定义才能确认"谁负责 delete"，而内联写法遮掩了所有权转移的发生点。

**通用建议**：

```cpp
// 任何时候，new 的结果都应该先保存到具名变量
Foo* pFoo = new Foo(...);
// 再处理失败路径
bool ok = func((WPARAM)pFoo);
if (!ok) delete pFoo;
```
