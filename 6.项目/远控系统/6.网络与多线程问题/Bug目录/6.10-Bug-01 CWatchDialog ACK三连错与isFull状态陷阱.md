# 6.10-Bug-01 CWatchDialog ACK 三连错与 isFull 状态陷阱

> 监控对话框打开之后画面一直黑屏。网络没问题，包也发了，响应也回来了，就是不显示图像。
> 根因：`CWatchDialog::OnSendPackAck` 里有四处错误同时生效，任何一处都能让渲染逻辑失效。

---

## 现象

1. 点击"远程监控"，`CWatchDialog` 对话框弹出
2. `threadWatchScreen` 开始循环发送 cmd=6 截图请求
3. 服务端返回截图数据，`WM_SEND_PACK_ACK` 消息正常到达
4. 但画面始终是空白的，没有任何图像更新

---

## 根因分析

`CWatchDialog::OnSendPackAck` 里存在四个独立的问题，任一一个都会让图像渲染失效，四个叠在一起稳稳黑屏。

### 问题一：switch 分发依据是布尔值

```cpp
// ❌ 旧代码
switch (pPacket != NULL)
{
case 5:
case 6:
    ...
```

`pPacket != NULL` 的结果是 `bool`，只会是 `0`（空）或 `1`（非空）。`case 6` 永远不会命中——`switch` 只有 `case 0` 和 `case 1` 是有效的。

这个 bug 和 [[Bug目录/6.9-Bug-02 ACK 回调三连错：接线、分发与内存泄漏]] 第二错完全一样，复制代码时一起带过来了。

### 问题二：case 5 fall-through 到 case 6

```cpp
// ❌ 旧代码
case 5:         // 没有 break！
case 6:
{
    // 鼠标 ACK 会执行截图渲染逻辑（逻辑错乱）
}
```

鼠标事件的响应（cmd=5）和截图响应（cmd=6）应该是独立逻辑。没有 `break` 让两者混在一起。

> 注意：由于问题一的存在，`case 5/6` 在旧代码里根本走不到（`switch` 条件是布尔值，`1` 对应的是 `case 1` 而非 `case 5` 或 `case 6`）。所以这两个问题叠加后，效果是：**什么也不执行**。

### 问题三：m_isFull 状态链断裂（核心）

即使前两个问题都修好，渲染还是进不去，因为：

```cpp
case 6:
{
    if (m_isFull)       // ← m_isFull 永远是 false！
    {
        // 渲染代码
        m_isFull = false;
    }
}
```

`m_isFull = true` 的路径在 `threadWatchScreen` 里：

```cpp
// ClientController.cpp — 旧版 threadWatchScreen
int ret = SendCommandPacket(m_watchDlg.GetSafeHwnd(), 6, true, NULL, 0);
if (ret == 6)           // ← SendCommandPacket 返回 bool，最大值是 1，永远不等于 6
{
    if (CEdoyunTool::Bytes2Image(...) == 0)
    {
        m_watchDlg.SetImageStatus(true);   // ← 这里从来执行不到
    }
}
```

`SendCommandPacket` 返回 `bool`（0 或 1），`ret == 6` 是永假条件。`SetImageStatus(true)` 一次都没被调用，`m_isFull` 从构造函数的 `false` 一路保持到关闭。

**状态链断裂的完整路径**：

```
构造函数：m_isFull = false
threadWatchScreen：
    ret = SendCommandPacket(...)   // 返回 bool
    if (ret == 6)                  // 永假 → 跳过
        SetImageStatus(true)       // 从未执行
OnSendPackAck case 6：
    if (m_isFull)                  // 永远是 false → 跳过
        渲染图像                   // 永远不执行 ←── 黑屏在这
```

三个组件之间靠 `m_isFull` 协调，但负责"设置 true"的那段代码因为类型判断错误从来没运行，整条状态链静悄悄地断掉了，没有任何报错。

### 问题四：CPacket 堆对象泄漏

```cpp
// ❌ 旧代码
CPacket* pPacket = (CPacket*)wParam;
// ... 用完之后，没有 delete pPacket
```

网络线程在 `CClientSocket::SendPack` 里通过 `new CPacket(pack)` 分配对象，所有权通过 `wParam` 传给 `CWatchDialog`。旧代码没有 `delete`，每来一个响应泄漏一个 `CPacket`。

和 [[Bug目录/6.9-Bug-02 ACK 回调三连错：接线、分发与内存泄漏]] 第三错完全一样。

---

## 修复方案

```cpp
// ✅ 修复后 CWatchDialog::OnSendPackAck
CPacket* pPacket = (CPacket*)wParam;
if (pPacket != NULL)
{
    // 1. 先拷贝到栈变量，后续操作栈对象，生命周期清晰
    CPacket head = *(CPacket*)wParam;
    // 2. 立即释放堆对象，所有权在入口处统一回收
    delete (CPacket*)wParam;

    // 3. 用命令号正确分发
    switch (head.sCmd)
    {
    case 5:
        TRACE("mouse event ack\r\n");
        break;   // 4. case 5 独立处理，不再 fall-through

    case 6:
    {
        // 5. 移除 if (m_isFull)，直接渲染
        CEdoyunTool::Bytes2Image(m_image, head.strData);
        CRect rect;
        m_picture.GetWindowRect(rect);
        m_nObjWidth = m_image.GetWidth();
        m_nObjHeight = m_image.GetHeight();

        m_image.StretchBlt(
            m_picture.GetDC()->GetSafeHdc(), 0, 0,
            rect.Width(), rect.Height(), SRCCOPY);
        m_picture.InvalidateRect(NULL);
        TRACE("更新图片完成%d %d %08X\r\n", m_nObjWidth, m_nObjHeight, (HBITMAP)m_image);
        m_image.Destroy();
        break;
    }
    default:
        break;
    }
}
```

同时在 `threadWatchScreen` 里移除了基于 `isFull` 的图像处理逻辑，改为纯粹的"定时发请求"，图像渲染全部交给 `OnSendPackAck` 回调：

```cpp
// ✅ 修复后 threadWatchScreen（截图线程只负责发请求）
int ret = SendCommandPacket(m_watchDlg.GetSafeHwnd(), 6, true, NULL, 0);
if (ret == 1)   // bool true == 1
{
    TRACE("成功发送请求图片 %08X\r\n");
}
```

---

## 四处问题对比

| 问题 | 根因类型 | 修法 |
|------|------|------|
| `switch (pPacket != NULL)` | 布尔值当命令号用 | 改为 `switch (head.sCmd)` |
| case 5 无 break | 漏写 break | 补上 break，单独处理鼠标 ACK |
| m_isFull 永远 false | bool 函数和整数常量误比较 | 移除 m_isFull 条件，直接渲染 |
| CPacket 未 delete | 所有权未约定 | 入口处先拷贝后立即 delete |

---

## 经验总结

**1. 修完一处，要检查同名函数是否有副本**

`OnSendPackAck` 在 `RemoteClientDlg` 和 `CWatchDialog` 里各有一份。上次只修了主对话框，监控对话框原封不动。做 bug 修复时，用 Grep 全搜同名函数，逐一确认。

**2. 状态机调试要顺着写入路径走**

`m_isFull` 问题不会崩溃，不会报错，只是静默地什么都不做。碰到"代码逻辑好像没问题但就是不执行"的情况，去找状态变量的所有赋值路径，逐一验证每个赋值是否真的能被执行到。

**3. bool 返回值不要和非 0/1 的常量比较**

```cpp
bool ret = someFunc();
if (ret == 6)   // ❌ 永假，而且编译不报警
if (ret == true) // ✅ 或直接 if (ret)
```
