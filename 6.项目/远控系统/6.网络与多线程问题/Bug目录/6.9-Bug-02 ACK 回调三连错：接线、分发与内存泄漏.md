# 6.9-Bug-02 ACK 回调三连错：接线、分发与内存泄漏

> 这次提交在 `OnSendPackAck()` 里修了三个彼此独立的问题，但它们都指向同一件事：**消息机制的接收侧每个环节都没写对**。错了三处，任何一处出问题都会导致整条 ACK 链路失效，或者悄悄漏内存。

---

## 第一错：消息映射接错类

```cpp
// ❌ 旧版本 RemoteClientDlg.cpp
BEGIN_MESSAGE_MAP(CRemoteClientDlg, CDialogEx)
    ...
    ON_MESSAGE(WM_SEND_PACK_ACK, &CWatchDialog::OnSendPackAck)
END_MESSAGE_MAP()
```

`BEGIN_MESSAGE_MAP(CRemoteClientDlg, ...)` 声明的是 `CRemoteClientDlg` 的消息表，
但 `ON_MESSAGE` 却绑到了 `CWatchDialog` 的成员函数。

**效果**：`WM_SEND_PACK_ACK` 发给主窗口，但主窗口的消息表告诉 MFC 去找 `CWatchDialog::OnSendPackAck`，而主窗口实例本身根本不是 `CWatchDialog`，行为是未定义的。`CRemoteClientDlg` 自己写的 `OnSendPackAck()` 完全没被调用到。

**修法**：

```cpp
// ✅ 修复后
ON_MESSAGE(WM_SEND_PACK_ACK, &CRemoteClientDlg::OnSendPackAck)
```

**记忆点**：MFC 的 `BEGIN_MESSAGE_MAP(A, B)` 里，`ON_MESSAGE` 绑的函数指针必须是 `A::` 的成员函数，不能绑别的类。

---

## 第二错：switch 分发依据写成了布尔值

```cpp
// ❌ 旧版本 OnSendPackAck
CPacket* pPacket = (CPacket*)wParam;
if (pPacket != NULL)
{
    CPacket& head = *pPacket;
    switch (pPacket != NULL)   // ← 这里分发依据是"指针是否非空"
    {
    case 1:  // ← 恰好 true == 1，所以 cmd=1 的路径"看起来"有效
        ...
    case 2:
    case 4:
        ...
    }
}
```

`switch (pPacket != NULL)` 只会得到两个值：
- 指针为空 → `0`
- 指针非空 → `1`

这意味着：
- `case 1` 在指针非空时**误打误撞可以命中**（因为 `true` 的整数值就是 `1`）
- `case 2`、`case 4` 等所有其他分支**永远不会命中**
- 消息机制看起来"部分有用"，实际上只有一个命令号偶然工作

这种 bug 最难缠：不是完全失效（完全失效反而容易发现），而是"某个路径恰好能用"，掩盖了整体的错误。

**修法**：

```cpp
// ✅ 修复后
CPacket head = *(CPacket*)wParam;
delete (CPacket*)wParam;
switch (head.sCmd)   // 分发依据是命令号，这才是正确的
{
case 1:
    ...
case 2:
    ...
case 4:
    ...
}
```

## 第三错：堆上的 CPacket 没人回收

网络线程在 `SendPack()` 里发 ACK 时：

```cpp
// CClientSocket.cpp — SendPack 函数
::SendMessage(hWnd, WM_SEND_PACK_ACK, (WPARAM)new CPacket(pack), data.wParam);
```

`new CPacket(pack)` 在堆上分配了对象，所有权通过 `wParam` 传给了主窗口。

问题：旧版本主窗口拿到 `wParam` 里的指针之后，用完不 `delete`。

后果：
- 每来一个成功响应，泄漏一个 `CPacket`
- 文件树请求会来很多包，下载也会来很多包，远程监控持续刷图更是连续泄漏
- 内存会随着使用时间累积增长

**修法**：

```cpp
// ✅ 修复后：在入口处统一回收
if (wParam != NULL)
{
    CPacket head = *(CPacket*)wParam;   // 先拷贝到栈
    delete (CPacket*)wParam;            // 立即回收堆对象
    switch (head.sCmd)
    { ... }
}
```

**为什么要先拷贝再 delete，而不是 delete 后继续用指针？**

```cpp
// ❌ 危险：delete 后还在用原指针
CPacket* p = (CPacket*)wParam;
delete p;
switch (p->sCmd)   // use-after-free
```

先把数据拷贝到栈变量 `head`，然后 `delete` 原始指针，后面只操作 `head`——栈对象的生命周期由编译器管理，不会出现 use-after-free。

---

## 三个问题的关系

这三个问题其实是**相互独立**的——它们碰巧出现在同一次提交的修复里，但根因各不相同：

| 问题 | 根因类型 | 出现原因 |
|------|------|------|
| `ON_MESSAGE` 接错类 | 复制粘贴时没有改类名 | 从 `CWatchDialog` 复制代码过来，忘了改 |
| `switch (pPacket != NULL)` | 条件写错 | 应该写 `pPacket->sCmd`，写成了存在性判断 |
| 不 `delete` 堆对象 | 所有权没有约定 | 收到指针后没有意识到自己是回收方 |

三个问题任何一个单独存在，都会让消息机制出问题；三个叠在一起，整条 ACK 链路从外到里都是错的。

---

## 经验总结

| 教训 | 说明 |
|------|------|
| MFC `ON_MESSAGE` 必须绑当前类 | `BEGIN_MESSAGE_MAP(A, ...)` 里只能用 `&A::func` |
| `switch` 条件一定要是业务字段 | 布尔值退化会让多数分支永远不执行，且不报错 |
| 跨线程传堆指针，必须明确谁 delete | 所有权约定不能靠默契，要在文档或代码里写清楚 |
