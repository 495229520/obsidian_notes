# 6.9 线程启动握手模式与跨线程对象所有权

> 基于提交 `f8d695b33bfac7eed7b496f6c6079d04efea9542`（2026-03-24）。
> 这次提交修了三类问题：网络线程的启动时机、`SendPacket` 的职责边界、以及 ACK 回调里的接线错误和内存泄漏。单看每一条都不复杂，但它们背后藏着两个值得单独讲的设计原则：**"线程启动握手"** 和 **"跨线程对象所有权"**。本篇围绕这两个主题展开。

## 一、线程启动握手：把"假设"换成"确认"

### 为什么旧做法不稳

旧版本的逻辑是：第一次调用 `SendPacket()` 时，发现线程还没启动，就在里面临时创建。
看起来合理，但这里有个 Win32 的隐性前提：

> **`_beginthreadex` 返回线程句柄，不等于线程消息队列已经准备好。**

只有当线程真正执行到 `GetMessage()` 的那一刻，系统才会为它创建消息队列。
在这之前调用 `PostThreadMessage()`，消息可能直接丢失，也可能报错 `ERROR_INVALID_THREAD_ID`。

所以旧版本存在一个竞态窗口：

```
UI 线程：创建网络线程 → 立刻 PostThreadMessage(WM_SEND_PACK)
                                              ↑
                            网络线程：还没跑到 GetMessage
                                      消息队列还不存在
                                      首包可能就这样丢了
```

另外旧版本还有一个写法 bug，不是今天的重点，详见 [[Bug目录/6.9-Bug-01 线程懒启动的消息队列竞态]]。

### 修复思路：用事件做启动握手

这次提交的解法是：在 `CClientSocket` 构造时启动线程，并且**等线程真正进入消息泵之后，构造函数才继续往下走**。

用的工具是 Win32 的 `HANDLE` 事件（`CreateEvent`）。流程如下：

```
CClientSocket 构造函数
│
├── CreateEvent(m_eventInvoke)     ← 创建一个"线程已就绪"的信号
├── _beginthreadex → threadFunc2   ← 启动网络线程
├── WaitForSingleObject(100ms)     ← 等线程来通知
└── CloseHandle(m_eventInvoke)     ← 握手完成，事件句柄用完就关

            ↓ 同时进行 ↓

网络线程 threadFunc2()
│
├── SetEvent(m_eventInvoke)        ← 我已经准备好了！
└── GetMessage 消息泵 ← 开始工作
```

握手完成之后，后续任何时候调用 `PostThreadMessage()`，目标线程的消息队列已经稳稳在那里了。

### 看代码

```cpp
// CClientSocket.cpp — 构造函数片段

CClientSocket::CClientSocket() :
    m_nIP(INADDR_ANY),
    m_nPort(0),
    m_sock(INVALID_SOCKET),
    m_bAutoClose(true),
    m_hThread(INVALID_HANDLE_VALUE)
{
    if (InitSockEnv() == FALSE)
    {
        MessageBox(NULL, ...);
        exit(0);
    }

    // === 启动握手开始 ===
    // 1. 先建一个"手动复位"的事件，初始状态为"未触发"
    m_eventInvoke = CreateEvent(NULL, TRUE, FALSE, NULL);

    // 2. 启动网络线程
    m_hThread = (HANDLE)_beginthreadex(NULL, 0, &CClientSocket::threadEntry, this, 0, &m_nThreadID);

    // 3. 等网络线程进入消息泵后来 SetEvent，最多等 100ms
    //    超时说明线程启动出了问题，TRACE 一条警告
    if (WaitForSingleObject(m_eventInvoke, 100) == WAIT_TIMEOUT)
    {
        TRACE("网络消息处理线程启动失败了！\r\n");
    }

    // 4. 握手完成，事件句柄不再需要，立即关掉
    CloseHandle(m_eventInvoke);
    // === 启动握手结束 ===

    m_buffer.resize(BUFFER_SIZE);
    memset(m_buffer.data(), 0, BUFFER_SIZE);
    // ... 消息处理函数注册等初始化
}
```

```cpp
// CClientSocket.cpp — threadFunc2 片段

void CClientSocket::threadFunc2()
{
    // 第一件事：告诉主线程"我的消息队列已经就绪了"
    SetEvent(m_eventInvoke);

    MSG msg;
    while (::GetMessage(&msg, NULL, 0, 0))
    {
        TranslateMessage(&msg);
        DispatchMessage(&msg);
        TRACE("Get Message:%08X \r\n", msg.message);
        if (m_mapFunc.find(msg.message) != m_mapFunc.end())
        {
            (this->*m_mapFunc[msg.message])(msg.message, msg.wParam, msg.lParam);
        }
    }
}
```

**关键细节**：

- `CreateEvent(NULL, TRUE, FALSE, NULL)` 的第二个参数 `TRUE` 是"手动复位"。
  这里要手动复位的原因是：这是一次性的"启动确认"，不需要自动复位，等主线程 `WaitForSingleObject` 返回后就不再用这个事件了。

- `CloseHandle(m_eventInvoke)` 要在构造函数里关，而不是析构函数里关。
  因为事件只用于启动握手，握手完就没用了。如果留着，析构的时候还需要判断有没有关过，多余的状态管理。

- `WaitForSingleObject` 超时设 100ms 而不是 `INFINITE`。
  如果等 `INFINITE`，万一系统资源不足线程创建失败，构造函数会永久卡住。100ms 是一个合理的容错边界。

### 这个模式的本质

这里的 `CreateEvent / SetEvent / WaitForSingleObject` 是在做一件非常经典的事：

**让调用者能"感知"到异步操作的完成状态**。

线程创建本质上是异步的——你调用 `_beginthreadex`，系统去创建线程，什么时候准备好你不知道。事件机制就是给这个"不知道"加上了一个确定性的通知点。

这个模式不只在远控项目里有用，只要你有"A 启动 B，A 需要等 B 就绪后再继续"的场景，都可以用这个套路。

## 二、SendPacket 的职责单一化

### 职责边界是什么

把线程创建移出 `SendPacket()` 之后，这个函数变得很干净：

```cpp
bool CClientSocket::SendPacket(HWND hWnd, const CPacket& pack, bool isAutoClosed, WPARAM wParam)
{
    UINT nMode = isAutoClosed ? CSM_AUTOCLOSE : 0;
    std::string strOut;
    pack.Data(strOut);  // 序列化数据包
    bool ret = PostThreadMessage(
        m_nThreadID,
        WM_SEND_PACK,
        (WPARAM)new PACKET_DATA(strOut.c_str(), strOut.size(), nMode, wParam),
        (LPARAM)hWnd
    );
    return ret;
}
```

它只做两件事：
1. 序列化 `CPacket` 为二进制数据
2. 通过 `PostThreadMessage` 投递给网络线程

之前的版本里，`SendPacket` 还要负责"判断线程有没有在跑、不在就启动"，等于一个函数同时管了**网络线程的生命周期**和**数据包的发送**。

这种混合往往是 bug 的温床——两个不同关注点的逻辑纠缠在一起，每次改一个地方都容易影响另一个。这次提交把线程生命周期收到构造函数，`SendPacket` 只剩发包，职责变清晰了，出问题的可能性也低了。

## 三、跨线程对象所有权：谁 new、谁 delete、怎么约定

### 问题是怎么来的

在这套消息机制里，有一条很具体的数据路径：

```
网络线程 SendPack()
  └── recv 到响应包
  └── new CPacket(pack)                      ← 在堆上分配
  └── SendMessage(hWnd, WM_SEND_PACK_ACK,
          (WPARAM)new CPacket(pack),          ← 指针作为 wParam 传出去
          data.wParam)

                ↓ 跨越线程边界 ↓

主窗口 OnSendPackAck(wParam, lParam)
  └── 拿到 wParam 里的 CPacket*
  └── 用完之后... 应该 delete，但旧版本没 delete
```

这里存在一个跨线程的对象所有权传递。

网络线程 `new` 了一个 `CPacket`，然后把裸指针放进消息的 `wParam` 里发给主窗口。这个时刻，所有权就被"转移"出去了——但转移到谁手里？怎么回收？旧版本没有明确约定，也没有写回收代码。

结果就是：每来一个成功响应，堆上就多一个 `CPacket` 对象，没人清理，一直泄漏。
在持续接收场景（远程监控、文件分包下载）里，这条泄漏会不断积累。

详见 [[Bug目录/6.9-Bug-02 ACK 回调三连错：接线、分发与内存泄漏]]。

### 修复：在回调入口统一回收

```cpp
LRESULT CRemoteClientDlg::OnSendPackAck(WPARAM wParam, LPARAM lParam)
{
    if (lParam != 0)
    {
        // lParam != 0 通常表示网络异常，wParam 此时可能为 NULL
        ...
        return 0;
    }
    else
    {
        if (wParam != NULL)
        {
            // 1. 先把数据拷贝到栈上
            CPacket head = *(CPacket*)wParam;
            // 2. 立即释放堆上的原始对象
            delete (CPacket*)wParam;
            // 3. 后续只操作栈上的 head，生命周期简单清晰
            switch (head.sCmd)
            {
            case 1:
                ...
            case 2:
                ...
            case 4:
                ...
            }
        }
    }
    return 0;
}
```

**为什么要先拷贝再 delete，而不是直接拿指针用？**

```cpp
// 方案 A：直接用指针（不推荐）
CPacket* pHead = (CPacket*)wParam;
delete pHead;          // 删了
pHead->sCmd;           // ← 用了已释放的内存！

// 方案 B：先拷贝再删（这次用的做法）
CPacket head = *(CPacket*)wParam;   // 值拷贝到栈上
delete (CPacket*)wParam;            // 删堆上的
head.sCmd;                          // 安全，操作栈上的副本
```

方案 B 的好处很明确：`delete` 之后，你就不再持有那块堆内存的引用了，后续操作都走栈对象 `head`，不会出现"删了之后还用"的生命周期问题。
同时，`delete` 放在入口处，不是放在某个 `case` 里，不管哪个命令分支走，都不会忘记回收。

### 跨线程传对象的通用原则

在 Windows 消息机制里，`wParam` / `lParam` 本质上就是一个 `uintptr_t`。你可以把堆上的对象地址塞进去，对面接收的时候强转回来用。

但这里有一个隐含的设计决策需要明确：

| 场景 | 所有权怎么走 |
|------|------|
| 发送方 `new`，接收方 `delete` | 所有权完整转移，接收方负责回收 |
| 发送方 `new`，自己 `delete` | 所有权不转移，但必须保证接收方用完之前对象还活着 |
| 双方都不 `delete` | 泄漏 |

这套机制里选择的是"所有权完整转移"：网络线程 `new`，主窗口 `delete`。这个约定只要明确出来，就不容易出问题。真正出问题的往往是"没有明确约定"——两边都以为另一边会删，或者都不确定该不该删。

## 小结：这一版做了什么

| 改动 | 解决的问题 | 核心原则 |
|------|------|------|
| 构造阶段启动线程 + 事件握手 | `PostThreadMessage` 在消息队列就绪前被调用 | 把"假设"换成"确认" |
| `SendPacket` 精简 | 线程创建逻辑混入发包函数 | 单一职责 |
| `ON_MESSAGE` 修正 | ACK 绑到错误类的 handler | MFC 消息映射必须接自己的成员函数 |
| `switch(head.sCmd)` | 分发依据退化成布尔值 | 分发条件必须是业务字段 |
| `delete (CPacket*)wParam` | 堆对象跨线程传递后无人回收 | 所有权要有明确约定 |

## Win32 API 速查

### `CreateEvent`

```cpp
HANDLE CreateEvent(
    LPSECURITY_ATTRIBUTES lpEventAttributes,  // 通常 NULL
    BOOL bManualReset,   // TRUE：手动复位；FALSE：自动复位
    BOOL bInitialState,  // TRUE：初始已触发；FALSE：初始未触发
    LPCTSTR lpName       // 命名事件，NULL 表示匿名
);
```

| 参数 | 这里的意义 |
|------|------|
| `bManualReset = TRUE` | 一次性的启动确认，主线程 `Wait` 返回后无需自动复位 |
| `bInitialState = FALSE` | 初始未触发，主线程需要等网络线程来 `SetEvent` |

### `WaitForSingleObject`

```cpp
DWORD WaitForSingleObject(
    HANDLE hHandle,    // 要等待的对象句柄（这里是事件）
    DWORD dwMilliseconds  // 等待时限，INFINITE 表示无限等
);
// 返回 WAIT_OBJECT_0: 对象已触发
//      WAIT_TIMEOUT:  超时
//      WAIT_FAILED:   失败
```

### `PostThreadMessage`

```cpp
BOOL PostThreadMessage(
    DWORD idThread,   // 目标线程 ID（不是线程句柄！）
    UINT Msg,         // 消息 ID
    WPARAM wParam,
    LPARAM lParam
);
```

**前提**：目标线程必须已经调用过 `GetMessage` 或 `PeekMessage`，否则消息队列不存在，函数会失败。
