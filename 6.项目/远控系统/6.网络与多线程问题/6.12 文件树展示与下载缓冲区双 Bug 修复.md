## 一、文件树：目录节点插入后未展开

### 问题

文件树采用**异步 ACK 驱动**的插入模式——每收到一个 `FILEINFO` 包就往树里插一条目录项。但 `InsertItem` 只改变数据结构，不改变节点的展开状态。节点展开与否完全由用户交互或代码显式控制。

旧代码插入子节点后什么都没做，父节点保持折叠，用户看不到任何新内容。

### 修复

```cpp
// ✅ case 2：获取文件信息，每条目录 ACK 处理
HTREEITEM hTemp = m_Tree.InsertItem(pInfo->szFileName, (HTREEITEM)lParam, TVI_LAST);
m_Tree.InsertItem("", hTemp, TVI_LAST);        // 为子目录预留展开占位
m_Tree.Expand((HTREEITEM)lParam, TVE_EXPAND);  // ← 插入后立即展开父节点
```

**技术栈**：
- `CTreeCtrl::Expand(hItem, nCode)`：强制改变指定节点的展开状态
- `TVE_EXPAND`：展开；对已展开节点重复调用是幂等的

**为什么每条 ACK 都调一次 Expand？**  
文件列表是分批返回的，不知道哪条是"最后一条"。每收到一条就展开，安全且无副作用（重复展开是幂等操作）。

> 📎 详见 [[Debug-022 文件树目录节点插入后未自动展开]]

## 二、下载缓冲区：memmove 参数顺序错误

### 背景

`CClientSocket::SendPack` 的 recv 循环里，每次收到数据后用 `CPacket` 解析出一个完整包（`nLen` 为消耗字节数），然后把缓冲区里**剩余的未消耗字节**移到头部，等待下次 recv 继续填充。

### 错误代码

```cpp
// 解包后：
// nLen  = 本次消耗的字节数（已处理）
// index = 剩余字节数（index -= nLen 之后）
index -= nLen;
memmove(pBuffer, pBuffer + index, nLen);
//                ↑ 起点错了！   ↑ 长度也错了！
// src = pBuffer + 剩余量 = 把"剩余"的偏移量当成了起点
// len = 消耗量           = 把"消耗"的长度当成了移动量
```

**执行结果**：把消耗过的（已无效）数据重新搬回缓冲区头部，剩余的有效数据被覆盖或丢失，导致下次解析重复读到旧数据。这正是 [[Debug-020 单击目录一次后重复解析同一FILEINFO导致日志刷屏]] 的根本原因。

### 修复代码

```cpp
index -= nLen;
memmove(pBuffer, pBuffer + nLen, index);
//                ↑ 正确起点：跳过已消耗字节
//                              ↑ 正确长度：剩余字节数
```

**记忆口诀**：`memmove(目标, 源头, 字节数)`

```
已消耗 nLen 字节 │ 剩余 index 字节
  0             nLen           nLen+index
  └────────────┘└─────────────┘
     已处理，丢弃   ↑ 这段移到头部
                  src = pBuffer + nLen
                  len = index (剩余)
```

> 📎 详见 [[Debug-020 单击目录一次后重复解析同一FILEINFO导致日志刷屏]]

## 三、下载完成：最后一包写入后立即检测收尾

### 问题

`OnSendPackAck` case 4 的完成检测旧逻辑：

```
if (length == 0)       → 处理首包（文件大小）
else if (index >= length) → 收尾（但这条是在下次 ACK 进来时才触发！）
else                   → 写数据包
```

写完最后一块数据后，`index` 刚好等于 `length`，但收尾逻辑要等**下一次 ACK 来临**才能执行。  
服务端发完数据就不再发包，下一次 ACK 永远不来 → 界面永远不收尾。

### 修复代码

```cpp
else
{
    FILE* pFile = (FILE*)lParam;
    fwrite(head.strData.c_str(), 1, head.strData.size(), pFile);
    index += head.strData.size();
    TRACE("index = %d\r\n", index);
    if (index >= length)          // ← 新增：写完立即判断是否完成
    {
        fclose((FILE*)lParam);
        length = 0;
        index = 0;
        CClientController::getInstance()->DownloadEnd();
    }
}
```

> 📁 `RemoteClient/RemoteClientDlg.cpp` : `OnSendPackAck` case 4（行 492-502）

**关键原则**：**分块传输的收尾条件要在"消费当前块之后"立即判断，不能留给下一次进入。**  
这也是 [[Debug-021 下载完成依赖下一次ACK导致最后一包后界面不收尾]] 总结的核心教训。

> 📎 详见 [[Debug-021 下载完成依赖下一次ACK导致最后一包后界面不收尾]]

## 四、recv 失败时 ACK 携带有效命令号

### 问题

`SendPack` 中 recv 失败路径旧代码：

```cpp
CloseSocket();
::SendMessage(hWnd, WM_SEND_PACK_ACK, NULL, 1);  // wParam = NULL
```

`OnSendPackAck` 对 `wParam == NULL` 直接跳过整个 switch，下载状态机（静态变量 `length / index`）不重置，行为混乱。

### 修复代码

```cpp
// 在 SendPack 入口处缓存原始命令包
size_t nTemp = data.strData.size();
CPacket current((BYTE*)data.strData.c_str(), nTemp);   // ← 保存原始命令

// recv 失败路径
TRACE("recv failed length %d index %d cmd %d\r\n", length, index, current.sCmd);
CloseSocket();
::SendMessage(hWnd, WM_SEND_PACK_ACK,
              (WPARAM)new CPacket(current.sCmd, NULL, 0), 1);  // ← 空数据但有命令号
```

> 📁 `RemoteClient/CClientSocket.cpp` : `SendPack`（行 290-328）

**设计原则**：通过消息传递 `CPacket*` 的所有权时，错误路径也必须发送合法对象，不能发 `NULL`，否则接收方的命令分发机制会全部失效。

> 📎 详见 [[Debug-023 recv失败时ACK传NULL导致下载状态机无法识别错误命令]]

## 五、LoadFileInfo 重构

### 背景

旧版 `LoadFileInfo` 在发送 `cmd=2` 后，跟着一个同步的 `lstPackets` 循环——但这个循环里的 `lstPackets` 从来没有被填充过（文件列表改为异步 ACK 回调之后遗留的死代码）：

```cpp
// ❌ 旧代码：lstPackets 永远是空的，这段完全不执行
std::list<CPacket> lstPackets;
int nCmd = SendCommandPacket(GetSafeHwnd(), 2, false, ...);
if (lstPackets.size() > 0)   // ← 永远为 false
{
    // 解析目录列表...（23 行死代码）
}
```

### 修复后

```cpp
// ✅ 新代码：只做一件事——发送命令，ACK 由 OnSendPackAck 异步处理
void CRemoteClientDlg::LoadFileInfo()
{
    m_List.DeleteAllItems();
    CString strPath = GetPath(hTreeSelected);
    TRACE("hTreeSelected %08X\r\n", hTreeSelected);
    CClientController::getInstance()->SendCommandPacket(GetSafeHwnd(), 2, false,
        (BYTE*)(LPCTSTR)strPath, strPath.GetLength(), (WPARAM)hTreeSelected);
}
```

同时，`LoadFileInfo` 从 `private` 改为 `public`，使 `threadDownloadFile` 在下载完成后可以调用它刷新文件树：

```cpp
// ClientController.cpp : threadDownloadFile 末尾
m_remoteDlg.MessageBox(_T("下载完成！"), _T("完成"));
m_remoteDlg.LoadFileInfo();   // ← 新增：下载完成后自动刷新文件树
```
