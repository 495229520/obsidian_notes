# 146. LRU 缓存

## LRU 简介

LRU 是 **Least Recently Used** 的缩写，意思是**最近最少使用**。

它是一种常见的缓存淘汰策略：当缓存容量满了，又要插入新数据时，优先淘汰最久没有被访问过的数据。

### 核心思想

把缓存中的数据按“最近使用时间”排序：

- 头部：最近使用的数据，也叫 MRU
- 尾部：最久未使用的数据，也叫 LRU
- 每次访问一个数据，都把它移动到头部
- 插入新数据时，也把它放到头部
- 容量满时，删除尾部数据

## 为什么需要哈希表 + 双向链表

题目要求 `get` 和 `put` 的平均时间复杂度都是 `O(1)`。

单独使用一种数据结构很难同时满足两个要求：

| 数据结构 | 优点 | 缺点 |
|----------|------|------|
| 哈希表 | 可以 `O(1)` 查找 key | 不能维护最近使用顺序 |
| 链表 | 可以 `O(1)` 移动、插入、删除节点 | 不能 `O(1)` 查找 key |

所以需要二者结合：

- `unordered_map`：通过 key 快速找到链表节点
- `list`：维护从 MRU 到 LRU 的使用顺序

> [!tip]
> 在 C++ 中，`list<pair<int, int>>::iterator` 可以直接指向链表中的某个节点，因此哈希表可以保存 `key -> 链表节点迭代器` 的映射。

---

## 题目描述

请设计并实现一个满足 LRU 缓存约束的数据结构。

实现 `LRUCache` 类：

- `LRUCache(int capacity)`：以正整数作为容量初始化 LRU 缓存
- `int get(int key)`：如果关键字 `key` 存在于缓存中，则返回关键字的值，否则返回 `-1`
- `void put(int key, int value)`：如果关键字 `key` 已经存在，则更新其值；如果不存在，则插入该组 `key-value`

当 `put` 操作导致缓存容量超过 `capacity` 时，必须淘汰最久未使用的关键字。

---

## 解题思路

### 数据结构设计

```cpp
int capacity;
list<pair<int, int>> cache;
unordered_map<int, list<pair<int, int>>::iterator> map;
```

含义：

- `capacity`：缓存容量
- `cache`：双向链表，头部是最近使用，尾部是最久未使用
- `map`：哈希表，记录 `key` 对应的链表节点位置

链表中保存的是 `{key, value}`，这样淘汰尾部节点时，可以拿到被淘汰的 `key`，再从哈希表中删除它。

---

## get 操作

如果 `key` 不存在，直接返回 `-1`。

如果 `key` 存在：

1. 通过哈希表找到链表节点
2. 取出节点中的 `value`
3. 删除原位置的节点
4. 把 `{key, value}` 插入链表头部
5. 更新哈希表中 `key` 对应的新迭代器
6. 返回 `value`

### 为什么要移动到头部

因为 `get(key)` 本身也是一次访问。只要被访问过，它就变成“最近使用”的数据。

---

## put 操作

### key 已经存在

如果 `key` 已经存在，先删除旧节点。后面再统一把新 `{key, value}` 插入头部。

### key 不存在且缓存已满

如果缓存已经满了，需要淘汰尾部节点：

1. `cache.back()` 找到尾部节点
2. 从哈希表中删除尾部节点的 `key`
3. `cache.pop_back()` 删除尾部节点

### 插入新节点

最后把 `{key, value}` 插入链表头部，并更新哈希表。

---

## 代码实现

```cpp
class LRUCache {
public:
    LRUCache(int capacity) {
        this->capacity = capacity;
    }

    int get(int key) {
        if (map.find(key) == map.end()) {
            return -1;
        }

        auto it = map[key];
        int value = it->second;

        cache.erase(it);
        cache.push_front({key, value});
        map[key] = cache.begin();

        return value;
    }

    void put(int key, int value) {
        if (map.find(key) != map.end()) {
            cache.erase(map[key]);
        } else if (cache.size() == capacity) {
            auto last = cache.back();
            map.erase(last.first);
            cache.pop_back();
        }

        cache.push_front({key, value});
        map[key] = cache.begin();
    }

private:
    int capacity;
    list<pair<int, int>> cache;
    unordered_map<int, list<pair<int, int>>::iterator> map;
};
```

---

## 示例流程

容量为 `2`：

| 操作 | 结果 | 缓存状态（左边是最近使用） |
|------|------|----------------------------|
| `put(1, 1)` | `null` | `[1]` |
| `put(2, 2)` | `null` | `[2, 1]` |
| `get(1)` | `1` | `[1, 2]` |
| `put(3, 3)` | `null` | `[3, 1]`，淘汰 `2` |
| `get(2)` | `-1` | `[3, 1]` |
| `put(4, 4)` | `null` | `[4, 3]`，淘汰 `1` |
| `get(1)` | `-1` | `[4, 3]` |
| `get(3)` | `3` | `[3, 4]` |
| `get(4)` | `4` | `[4, 3]` |

---

## 复杂度

- `get`：平均 `O(1)`
- `put`：平均 `O(1)`
- 空间复杂度：`O(capacity)`

原因：

- 哈希表查找、插入、删除平均是 `O(1)`
- 双向链表在已知迭代器时，删除节点是 `O(1)`
- 链表头部插入和尾部删除都是 `O(1)`

---

## 易错点

1. `get` 命中后也要移动到链表头部
2. `put` 更新已有 key 时，也要刷新最近使用顺序
3. 淘汰尾部节点时，要同时从 `map` 中删除旧 key
4. 链表里必须保存 `key`，否则删除尾部时不知道该从 `map` 删除哪个 key
5. `map[key]` 保存的是链表迭代器，不是 value 本身
