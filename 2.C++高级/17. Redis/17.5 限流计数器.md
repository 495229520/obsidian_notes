# 限流计数器

> 限流是保护系统的重要手段，Redis 的原子操作和过期机制使其成为实现分布式限流的理想选择。

---

## 概念定义

**限流（Rate Limiting）** 是控制系统接受请求速率的机制，防止系统被过量请求压垮。常见场景：

- **API 限流**：每用户每秒最多 10 次请求
- **短信验证码**：每手机号每天最多 5 条
- **登录防护**：连续失败 5 次锁定 30 分钟

核心目标：**保护系统稳定性，保证公平使用**

---

## 限流算法对比

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **固定窗口** | 时间窗口内计数 | 简单 | 边界突刺问题 |
| **滑动窗口** | 多个小窗口组合 | 平滑 | 实现复杂 |
| **漏桶** | 固定速率流出 | 平滑 | 无法应对突发 |
| **令牌桶** | 定速生成令牌 | 允许突发 | 实现较复杂 |

---

## 固定窗口计数器

### 原理

将时间划分为固定长度的窗口，每个窗口独立计数。

```
时间线：
|-------- 窗口1 --------|-------- 窗口2 --------|
   1  2  3  4  5          1  2  3  ...
   ↑                      ↑
 窗口开始，计数器=0      新窗口，重置计数器
```

### Redis 实现

利用 INCR 原子递增 + EXPIRE 自动过期。

```bash
# 用户 1001 在当前分钟的请求计数
SET rate:1001:202312241530 1 EX 60 NX   # 第一次请求
INCR rate:1001:202312241530              # 后续请求

# key 格式：rate:{用户ID}:{时间窗口}
# 过期时间 = 窗口长度，自动清理
```

### C++ 实现

```cpp
#include <hiredis/hiredis.h>
#include <string>
#include <chrono>

class FixedWindowRateLimiter {
public:
    FixedWindowRateLimiter(redisContext* ctx, int max_requests, int window_seconds)
        : ctx_(ctx)
        , max_requests_(max_requests)
        , window_seconds_(window_seconds)
    {}

    // 检查是否允许请求
    bool allow(const std::string& key) {
        std::string redis_key = buildKey(key);

        // 使用 Lua 脚本保证原子性
        const char* lua_script = R"(
            local current = redis.call('INCR', KEYS[1])
            if current == 1 then
                redis.call('EXPIRE', KEYS[1], ARGV[1])
            end
            return current
        )";

        auto* reply = static_cast<redisReply*>(
            redisCommand(ctx_, "EVAL %s 1 %s %d",
                lua_script, redis_key.c_str(), window_seconds_)
        );

        int64_t count = 0;
        if (reply && reply->type == REDIS_REPLY_INTEGER) {
            count = reply->integer;
        }

        if (reply) freeReplyObject(reply);

        return count <= max_requests_;
    }

    // 获取当前计数
    int64_t getCount(const std::string& key) {
        std::string redis_key = buildKey(key);

        auto* reply = static_cast<redisReply*>(
            redisCommand(ctx_, "GET %s", redis_key.c_str())
        );

        int64_t count = 0;
        if (reply && reply->type == REDIS_REPLY_STRING) {
            count = std::stoll(reply->str);
        }

        if (reply) freeReplyObject(reply);
        return count;
    }

private:
    redisContext* ctx_;
    int max_requests_;
    int window_seconds_;

    std::string buildKey(const std::string& key) {
        // 计算当前窗口的时间标识
        auto now = std::chrono::system_clock::now();
        auto seconds = std::chrono::duration_cast<std::chrono::seconds>(
            now.time_since_epoch()
        ).count();

        int64_t window = seconds / window_seconds_;
        return "rate:" + key + ":" + std::to_string(window);
    }
};
```

**关键点**：
- Lua 脚本保证 INCR 和 EXPIRE 的原子性
- 只在第一次请求时设置过期时间（`current == 1`）
- key 中包含时间窗口标识，自然实现窗口切换

### 边界突刺问题

```
窗口1（限制10次）     窗口2（限制10次）
    ...  9  10            1   2  ...
           ↑   ↑
         这里可能瞬间 20 次请求！
```

在窗口边界，用户可能在窗口1末尾发 10 次，窗口2开头再发 10 次，瞬间压力翻倍。

## 滑动窗口计数器

### 原理

将大窗口划分为多个小窗口，计算最近 N 个小窗口的请求总和。

```
|--小窗口1--|--小窗口2--|--小窗口3--|--小窗口4--|
    3           5           2           4

当前时间在小窗口4，统计窗口2+3+4 = 11 次
```

### Redis 实现

使用 Hash 存储各小窗口计数。

```bash
# 每 10 秒一个小窗口，统计最近 60 秒
HINCRBY rate:user:1001 1703001600 1   # 小窗口 timestamp
HINCRBY rate:user:1001 1703001610 1
# ...

# 查询时累加最近 6 个小窗口的值
```

### C++ 实现

```cpp
class SlidingWindowRateLimiter {
public:
    SlidingWindowRateLimiter(redisContext* ctx,
                             int max_requests,
                             int window_seconds,
                             int bucket_seconds = 1)
        : ctx_(ctx)
        , max_requests_(max_requests)
        , window_seconds_(window_seconds)
        , bucket_seconds_(bucket_seconds)
        , bucket_count_(window_seconds / bucket_seconds)
    {}

    bool allow(const std::string& key) {
        auto now = std::chrono::system_clock::now();
        auto now_seconds = std::chrono::duration_cast<std::chrono::seconds>(
            now.time_since_epoch()
        ).count();

        // 当前小窗口
        int64_t current_bucket = now_seconds / bucket_seconds_;
        std::string redis_key = "sliding:" + key;

        // Lua 脚本：清理过期桶 + 递增当前桶 + 计算总数
        const char* lua_script = R"(
            local key = KEYS[1]
            local current_bucket = tonumber(ARGV[1])
            local bucket_count = tonumber(ARGV[2])
            local expire_bucket = current_bucket - bucket_count

            -- 删除过期的桶
            local fields = redis.call('HKEYS', key)
            for i, field in ipairs(fields) do
                if tonumber(field) <= expire_bucket then
                    redis.call('HDEL', key, field)
                end
            end

            -- 递增当前桶
            redis.call('HINCRBY', key, current_bucket, 1)

            -- 设置整个 key 的过期时间
            redis.call('EXPIRE', key, ARGV[3])

            -- 计算总请求数
            local total = 0
            local values = redis.call('HVALS', key)
            for i, v in ipairs(values) do
                total = total + tonumber(v)
            end

            return total
        )";

        auto* reply = static_cast<redisReply*>(
            redisCommand(ctx_, "EVAL %s 1 %s %lld %d %d",
                lua_script, redis_key.c_str(),
                current_bucket, bucket_count_, window_seconds_ * 2)
        );

        int64_t total = 0;
        if (reply && reply->type == REDIS_REPLY_INTEGER) {
            total = reply->integer;
        }

        if (reply) freeReplyObject(reply);

        return total <= max_requests_;
    }

private:
    redisContext* ctx_;
    int max_requests_;
    int window_seconds_;
    int bucket_seconds_;
    int bucket_count_;
};
```

**设计思路**：
- 使用 Hash 的 field 存储小窗口时间戳，value 存储计数
- Lua 脚本原子执行：清理过期桶 → 递增当前桶 → 累加统计
- 整个 key 设置过期时间，防止用户不再活跃后数据残留

## 令牌桶算法

### 原理

以固定速率向桶中添加令牌，请求时消耗令牌；桶满则丢弃新令牌。

```
令牌生成: 每秒 10 个
桶容量: 100 个

正常请求 ──► 消耗 1 个令牌 ──► 放行
突发请求 ──► 消耗桶中存量 ──► 放行（允许短时突发）
令牌耗尽 ──► 拒绝请求
```

**优势**：允许一定程度的突发流量，同时保证平均速率。

### C++ 实现

```cpp
class TokenBucketRateLimiter {
public:
    TokenBucketRateLimiter(redisContext* ctx,
                           int capacity,           // 桶容量
                           double refill_rate)     // 每秒填充令牌数
        : ctx_(ctx)
        , capacity_(capacity)
        , refill_rate_(refill_rate)
    {}

    bool allow(const std::string& key, int tokens = 1) {
        std::string redis_key = "token:" + key;

        // Lua 脚本实现令牌桶
        const char* lua_script = R"(
            local key = KEYS[1]
            local capacity = tonumber(ARGV[1])
            local refill_rate = tonumber(ARGV[2])
            local now = tonumber(ARGV[3])
            local requested = tonumber(ARGV[4])

            -- 获取当前状态
            local data = redis.call('HMGET', key, 'tokens', 'last_time')
            local tokens = tonumber(data[1]) or capacity
            local last_time = tonumber(data[2]) or now

            -- 计算应该填充的令牌数
            local elapsed = now - last_time
            local refill = elapsed * refill_rate
            tokens = math.min(capacity, tokens + refill)

            -- 检查是否有足够令牌
            if tokens >= requested then
                tokens = tokens - requested
                redis.call('HMSET', key, 'tokens', tokens, 'last_time', now)
                redis.call('EXPIRE', key, 3600)  -- 1小时过期
                return 1  -- 允许
            else
                redis.call('HMSET', key, 'tokens', tokens, 'last_time', now)
                redis.call('EXPIRE', key, 3600)
                return 0  -- 拒绝
            end
        )";

        auto now = std::chrono::duration_cast<std::chrono::milliseconds>(
            std::chrono::system_clock::now().time_since_epoch()
        ).count() / 1000.0;

        auto* reply = static_cast<redisReply*>(
            redisCommand(ctx_, "EVAL %s 1 %s %d %f %f %d",
                lua_script, redis_key.c_str(),
                capacity_, refill_rate_, now, tokens)
        );

        bool allowed = false;
        if (reply && reply->type == REDIS_REPLY_INTEGER) {
            allowed = reply->integer == 1;
        }

        if (reply) freeReplyObject(reply);
        return allowed;
    }

    // 获取当前令牌数（调试用）
    double getTokens(const std::string& key) {
        std::string redis_key = "token:" + key;

        auto* reply = static_cast<redisReply*>(
            redisCommand(ctx_, "HGET %s tokens", redis_key.c_str())
        );

        double tokens = 0;
        if (reply && reply->type == REDIS_REPLY_STRING) {
            tokens = std::stod(reply->str);
        }

        if (reply) freeReplyObject(reply);
        return tokens;
    }

private:
    redisContext* ctx_;
    int capacity_;
    double refill_rate_;
};
```

**关键点**：
- 使用 Hash 存储令牌数和最后更新时间
- 每次请求时计算应填充的令牌：`elapsed * refill_rate`
- 令牌数不超过桶容量：`math.min(capacity, tokens + refill)`

### 使用示例

```cpp
void rateLimitDemo(redisContext* ctx) {
    // 配置：桶容量 100，每秒填充 10 个令牌
    TokenBucketRateLimiter limiter(ctx, 100, 10.0);

    std::string user_id = "user:1001";

    // 模拟突发请求
    for (int i = 0; i < 120; ++i) {
        if (limiter.allow(user_id)) {
            std::cout << "请求 " << i << ": 允许\n";
        } else {
            std::cout << "请求 " << i << ": 拒绝（限流）\n";
        }
    }

    // 等待令牌恢复
    std::this_thread::sleep_for(std::chrono::seconds(5));

    std::cout << "5秒后令牌数: " << limiter.getTokens(user_id) << '\n';
}
```

## 漏桶算法

### 原理

请求进入桶中排队，以固定速率流出处理。

```
请求入口 ──► [请求队列] ──► 固定速率流出 ──► 处理
               │
         桶满则拒绝新请求
```

**特点**：无论入口流量多大，出口速率恒定，适合对下游系统有严格速率要求的场景。

### 与令牌桶的区别

| 特性 | 令牌桶 | 漏桶 |
|------|-------|------|
| 突发流量 | 允许（消耗存量令牌） | 不允许（恒定流出） |
| 出口速率 | 可变 | 恒定 |
| 适用场景 | API 限流 | 平滑流量整形 |

## 实际应用：API 限流中间件

```cpp
class ApiRateLimitMiddleware {
public:
    ApiRateLimitMiddleware(redisContext* ctx)
        : global_limiter_(ctx, 10000, 10.0)   // 全局：10000容量，每秒10000
        , user_limiter_(ctx, 100, 10.0)        // 用户：100容量，每秒10
        , ip_limiter_(ctx, 200, 20.0)          // IP：200容量，每秒20
    {}

    enum class Result {
        Allowed,
        GlobalLimit,
        UserLimit,
        IpLimit
    };

    Result check(const std::string& user_id, const std::string& ip) {
        // 1. 检查全局限流
        if (!global_limiter_.allow("global")) {
            return Result::GlobalLimit;
        }

        // 2. 检查用户限流
        if (!user_limiter_.allow("user:" + user_id)) {
            return Result::UserLimit;
        }

        // 3. 检查 IP 限流
        if (!ip_limiter_.allow("ip:" + ip)) {
            return Result::IpLimit;
        }

        return Result::Allowed;
    }

private:
    TokenBucketRateLimiter global_limiter_;
    TokenBucketRateLimiter user_limiter_;
    TokenBucketRateLimiter ip_limiter_;
};
```

**分层限流策略**：
1. **全局限流**：保护系统整体
2. **用户限流**：防止单用户滥用
3. **IP 限流**：防止爬虫/攻击

## 最佳实践

| 推荐 | 避免 |
|-----|------|
| 使用 Lua 脚本保证原子性 | 多个 Redis 命令分开执行 |
| 分层限流（全局+用户+IP） | 单一限流策略 |
| 令牌桶用于 API 限流 | 漏桶用于 API（不允许突发） |
| 限流后返回明确错误码 | 静默丢弃请求 |
| 设置合理的 key 过期时间 | 限流数据永不过期 |

## 限流响应处理

```cpp
// HTTP 429 Too Many Requests
struct RateLimitResponse {
    int status_code = 429;
    std::string message = "Too Many Requests";
    int retry_after_seconds;  // 建议客户端等待时间

    std::string toJson() const {
        return R"({"error":"rate_limited","retry_after":)"
             + std::to_string(retry_after_seconds) + "}";
    }
};
```
