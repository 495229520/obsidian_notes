# 热点数据缓存

> 将频繁访问的数据存储在 Redis 内存中，避免重复查询数据库，是 Redis 最基础也是最重要的应用场景。

---

## 概念定义

**热点数据缓存** 是一种将高频访问数据从慢速存储（数据库）复制到快速存储（内存）的策略。核心目标是：

- **降低延迟**：内存读取 μs 级 vs 磁盘 IO ms 级
- **减轻数据库压力**：避免数据库成为瓶颈
- **提升吞吐量**：支撑更高的并发请求

---

## 缓存架构模式

### Cache-Aside（旁路缓存）

最常用的缓存模式，应用程序同时管理缓存和数据库。

### 为什么是「删除」而不是「更新」缓存？

- **避免并发写问题**：两个请求同时更新，可能导致缓存与数据库不一致
- **延迟计算**：有些缓存值需要复杂计算，不必每次都更新
- **简化逻辑**：删除是幂等操作，更容易保证一致性

## C++ 实现

### 缓存封装类

```cpp
#include <hiredis/hiredis.h>
#include <string>
#include <optional>
#include <functional>
#include <memory>

class RedisCache {
public:
    explicit RedisCache(const std::string& host = "127.0.0.1", int port = 6379)
        : ctx_(redisConnect(host.c_str(), port), redisFree)
    {
        if (!ctx_ || ctx_->err) {
            throw std::runtime_error("Redis 连接失败");
        }
    }

    // 获取缓存，未命中返回 nullopt
    std::optional<std::string> get(const std::string& key) {
        auto reply = execCommand("GET %s", key.c_str());
        if (reply && reply->type == REDIS_REPLY_STRING) {
            return std::string(reply->str, reply->len);
        }
        return std::nullopt;
    }

    // 设置缓存，带过期时间（秒）
    bool set(const std::string& key, const std::string& value, int ttl_seconds = 0) {
        std::unique_ptr<redisReply, decltype(&freeReplyObject)> reply(nullptr, freeReplyObject);

        if (ttl_seconds > 0) {
            reply.reset(execCommand("SET %s %s EX %d",
                key.c_str(), value.c_str(), ttl_seconds));
        } else {
            reply.reset(execCommand("SET %s %s", key.c_str(), value.c_str()));
        }

        return reply && reply->type == REDIS_REPLY_STATUS;
    }

    // 删除缓存
    bool del(const std::string& key) {
        auto reply = execCommand("DEL %s", key.c_str());
        return reply && reply->type == REDIS_REPLY_INTEGER && reply->integer > 0;
    }

    // Cache-Aside 模式：带回源查询
    template<typename Loader>
    std::string getOrLoad(const std::string& key, Loader&& loader, int ttl_seconds = 300) {
        // 1. 先查缓存
        if (auto cached = get(key)) {
            return *cached;
        }

        // 2. 缓存未命中，从数据源加载
        std::string value = loader();

        // 3. 写入缓存
        set(key, value, ttl_seconds);

        return value;
    }

private:
    std::unique_ptr<redisContext, decltype(&redisFree)> ctx_;

    redisReply* execCommand(const char* format, ...) {
        va_list args;
        va_start(args, format);
        auto* reply = static_cast<redisReply*>(redisvCommand(ctx_.get(), format, args));
        va_end(args);
        return reply;
    }
};
```

### 使用示例

```cpp
#include <iostream>

// 模拟数据库查询
std::string queryUserFromDB(int user_id) {
    std::cout << "[DB] 查询用户 " << user_id << '\n';
    // 实际项目中这里是 MySQL 查询
    return R"({"id":)" + std::to_string(user_id) + R"(,"name":"张三"})";
}

int main() {
    RedisCache cache;

    // 第一次查询：缓存未命中，查数据库
    std::string user = cache.getOrLoad(
        "user:1001",
        []{ return queryUserFromDB(1001); },
        300  // 缓存 5 分钟
    );
    std::cout << "用户数据: " << user << '\n';

    // 第二次查询：缓存命中，不查数据库
    user = cache.getOrLoad(
        "user:1001",
        []{ return queryUserFromDB(1001); },
        300
    );
    std::cout << "用户数据: " << user << '\n';

    return 0;
}
```

**输出**：
```
[DB] 查询用户 1001
用户数据: {"id":1001,"name":"张三"}
用户数据: {"id":1001,"name":"张三"}
```

第二次没有 `[DB]` 输出，说明命中了缓存。

## 缓存常见问题

### 1. 缓存穿透

**问题**：查询不存在的数据，每次都穿透到数据库。

![[Pasted image 20251224095256.png]]

**解决方案**：

```cpp
// 方案1：缓存空值
std::string getOrLoadWithNull(const std::string& key, auto&& loader) {
    if (auto cached = get(key)) {
        if (*cached == "__NULL__") {
            return "";  // 空值标记
        }
        return *cached;
    }

    std::string value = loader();
    if (value.empty()) {
        set(key, "__NULL__", 60);  // 空值缓存较短时间
    } else {
        set(key, value, 300);
    }
    return value;
}

// 方案2：布隆过滤器（拦截不存在的 key）
// 需要额外引入 bloom filter 库
```

### 2. 缓存击穿

**问题**：热点 key 过期瞬间，大量请求同时打到数据库。

![[Pasted image 20251224095541.png]]

**解决方案**：互斥锁

```cpp
#include <mutex>
#include <shared_mutex>
#include <unordered_map>

class CacheWithLock {
public:
    std::string getOrLoad(const std::string& key, auto&& loader, int ttl) {
        // 1. 先查缓存（读锁）
        {
            std::shared_lock lock(mutex_);
            if (auto cached = cache_.get(key)) {
                return *cached;
            }
        }

        // 2. 获取该 key 的互斥锁
        std::unique_lock key_lock(getKeyMutex(key));

        // 3. Double-check：可能其他线程已经加载
        if (auto cached = cache_.get(key)) {
            return *cached;
        }

        // 4. 只有一个线程执行加载
        std::string value = loader();
        cache_.set(key, value, ttl);
        return value;
    }

private:
    RedisCache cache_;
    std::shared_mutex mutex_;
    std::unordered_map<std::string, std::mutex> key_mutexes_;
    std::mutex map_mutex_;

    std::mutex& getKeyMutex(const std::string& key) {
        std::lock_guard lock(map_mutex_);
        return key_mutexes_[key];
    }
};
```

### 3. 缓存雪崩

**问题**：大量 key 同时过期，或 Redis 宕机。

**解决方案**：

```cpp
// 方案1：过期时间加随机值，避免同时过期
int randomTTL(int base_ttl) {
    static std::mt19937 gen(std::random_device{}());
    std::uniform_int_distribution<> dist(0, base_ttl / 10);
    return base_ttl + dist(gen);
}

cache.set(key, value, randomTTL(300));  // 300 ± 30 秒

// 方案2：多级缓存（本地缓存 + Redis）
// 方案3：Redis 集群 + 哨兵（高可用）
```

## 缓存更新策略

| 策略 | 一致性 | 复杂度 | 适用场景 |
|------|-------|--------|---------|
| **Cache-Aside** | 最终一致 | 低 | 通用场景 |
| **Write-Through** | 强一致 | 中 | 写少读多 |
| **Write-Behind** | 最终一致 | 高 | 写多场景 |
| **Refresh-Ahead** | 近实时 | 中 | 热点数据 |

---

## 过期淘汰策略

Redis `maxmemory-policy` 配置：

| 策略 | 说明 |
|------|------|
| `noeviction` | 内存满时拒绝写入（默认） |
| `allkeys-lru` | 所有 key 中淘汰最近最少使用的 |
| `volatile-lru` | 仅淘汰设置了过期时间的 key（LRU） |
| `allkeys-random` | 随机淘汰 |
| `volatile-ttl` | 淘汰 TTL 最短的 key |

**推荐**：缓存场景使用 `allkeys-lru`

---

## 最佳实践

| 推荐 | 避免 |
|-----|------|
| 设置合理的过期时间 | 缓存永不过期（数据可能过时） |
| Key 命名规范：`业务:对象:ID` | 随意命名，如 `a`, `temp` |
| 使用连接池 | 每次请求新建连接 |
| 序列化用 JSON/Protobuf | 存储复杂对象用字符串拼接 |
| 监控缓存命中率 | 上线后不关注缓存效果 |
