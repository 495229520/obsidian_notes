# 14. volatile 关键字

## 1. 概述

### 什么是 volatile？

`volatile` 是 C/C++ 中的类型修饰符（type qualifier），用于告诉编译器：**这个变量的值可能在程序控制流之外被改变**，因此编译器不应对其进行优化。

```cpp
volatile int flag = 0;
```

当一个变量被声明为 `volatile` 时，编译器会：

- **每次访问都从内存读取**：不会将变量缓存到寄存器中
- **每次修改都写回内存**：不会延迟写入操作
- **保持访问顺序**：不会对该变量的读写进行重排序

### volatile 的核心作用

`volatile` 主要解决两个问题：

1. **防止[[11.2 编译器基本概念#7. 编译器优化|编译器优化]]**：编译器不会将 volatile 变量优化到寄存器中，也不会进行[[10.3 C和C++的const区别#5. 常量折叠的底层原理|常量折叠]]等优化
2. **保证内存可见性**：确保对变量的读写直接作用于内存

---

## 2. 为什么需要 volatile？

### 2.1 编译器优化导致的问题

考虑以下代码：

```cpp
int flag = 0;

void wait_for_flag() {
    while (flag == 0) {
        // 等待 flag 变为非零
    }
    printf("Flag is set!\n");
}
```

在高优化级别（如 `-O2`、`-O3`）下，编译器可能会将其优化为：

```cpp
void wait_for_flag() {
    // 编译器认为循环内 flag 不会改变，优化为无限循环
    if (flag == 0) {
        while (true) { }  // 死循环！
    }
    printf("Flag is set!\n");
}
```

编译器这样优化是因为它没有看到循环内有任何修改 `flag` 的代码，因此认为 `flag` 的值不会改变。

### 2.2 使用 volatile 解决

```cpp
volatile int flag = 0;

void wait_for_flag() {
    while (flag == 0) {
        // 每次循环都会真正从内存读取 flag
    }
    printf("Flag is set!\n");
}
```

使用 `volatile` 后，编译器每次都会从内存中读取 `flag` 的值，不会进行优化。

## 3. volatile 的使用场景

### 3.1 硬件寄存器映射（最常见场景）

在嵌入式编程中，硬件寄存器通常被映射到特定的内存地址。这些寄存器的值可能被硬件改变：

```cpp
// 硬件状态寄存器映射到地址 0x40000000
volatile uint32_t* const STATUS_REG = (volatile uint32_t*)0x40000000;

void wait_for_hardware_ready() {
    // 必须用 volatile，否则编译器可能优化掉读取操作
    while ((*STATUS_REG & 0x01) == 0) {
        // 等待硬件就绪位
    }
}
```

### 3.2 中断服务程序（ISR）共享变量

中断处理程序和主程序之间共享的变量必须声明为 volatile：

```cpp
volatile bool data_ready = false;
volatile int received_data = 0;

// 中断服务程序
void ISR_UART_Handler() {
    received_data = UART_DATA_REG;  // 读取硬件数据
    data_ready = true;               // 设置标志
}

// 主程序
int main() {
    while (true) {
        if (data_ready) {
            process(received_data);
            data_ready = false;
        }
    }
}
```

### 3.3 信号处理程序共享变量

与信号处理程序共享的变量也需要 volatile：

```cpp
#include <signal.h>

volatile sig_atomic_t shutdown_requested = 0;

void signal_handler(int signum) {
    shutdown_requested = 1;
}

int main() {
    signal(SIGINT, signal_handler);

    while (!shutdown_requested) {
        do_work();
    }

    cleanup();
    return 0;
}
```

> **注意**：`sig_atomic_t` 是 C 标准保证可以原子访问的整数类型。

### 3.4 内存映射 I/O (MMIO)

```cpp
// 定义一个指向 volatile 内存的指针
volatile uint8_t* const GPIO_PORT = (volatile uint8_t*)0x20000000;

void toggle_led() {
    *GPIO_PORT |= 0x01;   // 开灯
    delay(1000);
    *GPIO_PORT &= ~0x01;  // 关灯
}
```

## 4. volatile 的限制

### 4.1 volatile 不保证原子性

```cpp
volatile int counter = 0;

void increment() {
    counter++;  // 这不是原子操作！
    // 实际上是: tmp = counter; tmp = tmp + 1; counter = tmp;
}
```

多线程同时调用 `increment()` 会导致数据竞争（data race）。

### 4.2 volatile 不提供线程间同步

`volatile` 只能防止编译器优化，但**不能**：

- 防止 CPU 指令重排序
- 提供内存屏障（memory barrier）
- 保证多核 CPU 之间的缓存一致性

---

## 5. volatile 与 std::atomic 的区别

在多线程编程中，应使用 `std::atomic` 而不是 `volatile`：

| 特性 | `volatile` | `std::atomic` |
|------|------------|---------------|
| **防止编译器优化** | ✅ 是 | ✅ 是 |
| **保证原子性** | ❌ 否 | ✅ 是 |
| **提供内存屏障** | ❌ 否 | ✅ 是 |
| **线程安全** | ❌ 否 | ✅ 是 |
| **适用场景** | 硬件编程、ISR | 多线程同步 |

### 代码对比

```cpp
#include <atomic>

// 错误：多线程中使用 volatile
volatile int unsafe_counter = 0;

// 正确：多线程中使用 atomic
std::atomic<int> safe_counter{0};

void thread_func() {
    unsafe_counter++;      // 数据竞争！
    safe_counter++;        // 线程安全，原子操作
    safe_counter.fetch_add(1, std::memory_order_relaxed);  // 显式内存序
}
```

### 选择建议

```
问：这个变量会被硬件/信号处理程序/ISR 修改吗？
├── 是 → 使用 volatile
└── 否 → 问：这个变量会被多个线程访问吗？
          ├── 是 → 使用 std::atomic
          └── 否 → 不需要特殊处理
```

## 6. volatile 与 const 的组合

`volatile` 和 [[4. const、指针和引用|const]] 可以同时使用：

```cpp
// 只读的硬件状态寄存器
const volatile uint32_t* STATUS = (const volatile uint32_t*)0x40000000;

// volatile：值可能被硬件改变，每次都要从内存读取
// const：程序不应该修改它
```

### 语法规则

类似于 const，volatile 也遵循"右边规则"：

```cpp
volatile int* p1;        // p1 指向 volatile int
int volatile* p2;        // 同上，等价写法
int* volatile p3;        // p3 本身是 volatile 的
volatile int* volatile p4;  // 两者都是 volatile
```

## 7. 实际示例：避免优化问题

### 7.1 延时函数

```cpp
// 错误：可能被优化掉
void bad_delay(int count) {
    for (int i = 0; i < count; i++) {
        // 编译器可能认为这个循环没有副作用，直接删除
    }
}

// 正确：使用 volatile 防止优化
void good_delay(int count) {
    for (volatile int i = 0; i < count; i++) {
        // volatile 防止编译器优化掉循环
    }
}
```

### 7.2 读取内存以触发副作用

```cpp
// 确保读取操作真正发生（如清除硬件中断标志）
void clear_interrupt_flag() {
    volatile uint32_t* flag = (volatile uint32_t*)0x40000100;
    (void)*flag;  // 读取操作会清除中断标志
}
```

---

## 8. 关键要点

1. **核心作用**：`volatile` 告诉编译器变量可能在程序外被修改，禁止相关优化
2. **使用场景**：硬件寄存器、中断处理、信号处理程序
3. **不适用于多线程**：`volatile` 不提供原子性和内存屏障，多线程应使用 `std::atomic`
4. **与 const 的关系**：两者可以组合使用，`const volatile` 表示只读但可能被外部修改的变量
5. **编译器行为**：使用 volatile 后，每次访问都会从内存读取，不会缓存到寄存器
