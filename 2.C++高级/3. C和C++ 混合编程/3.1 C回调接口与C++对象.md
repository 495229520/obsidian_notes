# 3.1 C 回调接口与 C++ 对象

> C 回调接口要求传入纯粹的 C 函数指针，但 C++ 成员函数有隐含的 `this` 参数和不同的调用约定，两者**类型不兼容**。这是 C/C++ 混合编程中最常见的问题之一，解决方案是 **static 桥接模式**。

---

## 什么是 C 回调接口

C 回调接口就是：一个 C 语言库/API 说"你给我一个函数指针，我在适当的时候帮你调用它"。

操作系统和标准库中大量使用这种设计（参见 [[9.1.5 函数指针#回调函数 (Callback Functions)]]）：

```c
// 线程创建：你告诉我线程要执行什么函数
int pthread_create(..., void*(*start_routine)(void*), void* arg);

// Windows 线程：
_beginthreadex(..., unsigned(__stdcall*)(void*), void* arg, ...);

// 排序：你告诉我怎么比较两个元素
void qsort(..., int(*compare)(const void*, const void*));

// 信号处理：你告诉我收到信号后干什么
signal(SIGINT, void(*handler)(int));
```

**共同点**：这些 API 都要求传入一个**纯粹的 C 函数指针**——一个裸地址。

---

## 为什么 C++ 成员函数不能当 C 回调用

### 调用约定不兼容

C 的函数指针就是一个裸地址，调用时直接 `call 那个地址 + 参数压栈` 就行。但 C++ 非静态成员函数使用 `__thiscall` 调用约定（参见 [[9.1.10 函数约定调用]]），需要**通过 ECX 寄存器传递 this 指针**：

```
C 的普通函数（__cdecl 或 __stdcall）：

    栈:  [arg2] [arg1] [返回地址]
    调用: call 函数地址


C++ 非静态成员函数（__thiscall）：

    ECX寄存器: this 指针          ← 多了这个！
    栈:  [arg2] [arg1] [返回地址]
    调用: call 函数地址
```

C 的 API 内部调用回调时，**根本不知道要往 ECX 里塞一个 this 指针**，因为 C 语言连"对象"是什么都不知道。

### 如果强行调用会怎样

```
_beginthreadex 内部（C 代码）：
    push arg          ; 压入 void* 参数
    call threadFunc   ; 调用函数

    问题：ECX 寄存器里是什么？随机垃圾值！

    threadFunc 内部：
    mov eax, [ecx+4]  ; 尝试访问 this->m_data
                       ; ecx 是垃圾 → 访问违规 → 崩溃！
```

### 类型本质不同

```cpp
// C 函数指针：一个裸地址
using C_Callback = unsigned (__stdcall*)(void*);

// C++ 成员函数指针：不是裸地址，包含额外信息
using CPP_MemberFunc = unsigned (__stdcall Controller::*)(void*);

// 两种指针大小可能都不一样：
sizeof(C_Callback)      // 4 字节（32位）或 8 字节（64位）
sizeof(CPP_MemberFunc)  // 可能 8~16 字节！（含虚表偏移等信息）
```

成员函数指针可能包含虚表偏移、多重继承调整等额外信息（参见 [[9. 指向类成员的指针]]）。C 的 `void*` 根本装不下这些东西。

---

## 为什么这个矛盾普遍存在

### 历史原因：操作系统 API 都是 C 写的

```
1972  ── C 语言诞生
1979  ── Unix 系统调用全部用 C 定义
1983  ── C++ 诞生（"C with Classes"）
1985  ── Windows 1.0（Win32 API 用 C 定义）
1995  ── POSIX 标准化（pthread 等，全部 C 接口）
2011  ── C++11（终于有了 std::thread）
```

**关键事实**：
1. 操作系统内核是 C 写的 → 系统调用是 C 接口
2. 标准库（pthread、socket、signal）是 C 写的 → 回调是 C 函数指针
3. 大量第三方库（SQLite、OpenSSL、libcurl）是 C 写的 → 回调是 C 函数指针

C++ 程序员**必须调用这些 C 库**，但又想用面向对象的方式组织代码。矛盾就此产生：

![[3_1_1.svg]]

**只要用 C++ 写应用、却需要调用 C 接口，这个矛盾就永远存在。**

---

## 解决方案：Static 桥接模式

### 核心思想

用 static 成员函数做"翻译官"，把 `this` 指针藏进 `void* arg` 参数中传递：

![[3_1_2.svg]]

**static 成员函数是"双面人"**（参见 [[12.1.10 静态成员函数]]）：
- **对 C 来说**：它是一个普通函数（没有 this，`__stdcall` 调用约定）→ C 认识它
- **对 C++ 来说**：它在类的作用域内，可以访问类的 private/protected 成员 → C++ 也认识它

### 代码模式

```cpp
class MyClass {
public:
    void Init() {
        // 第一步：把 this 藏进 void* arg
        _beginthreadex(NULL, 0, &MyClass::threadEntry, this, 0, &m_tid);
    }

private:
    int m_data = 42;

    void threadFunc() {
        // 第三步：正常使用成员变量
        printf("%d\n", m_data);   // this->m_data，正常工作
    }

    // 第二步：static 桥接函数，还原 this
    static unsigned __stdcall threadEntry(void* arg) {
        MyClass* thiz = static_cast<MyClass*>(arg);
        thiz->threadFunc();       // thiz 充当 this
        _endthreadex(0);
        return 0;
    }

    unsigned m_tid;
};
```

### 常见应用场景

| 场景 | C 回调接口 | `void*` 参数传 `this` |
|------|-----------|----------------------|
| Windows 线程 | `_beginthreadex(... func, arg ...)` | `arg = this` |
| Linux 线程 | `pthread_create(... func, arg)` | `arg = this` |
| Windows 定时器 | `SetTimer` / `TimerProc` | 通过 `LPARAM` 传 |
| Windows 窗口过程 | `WNDPROC` | 通过 `SetWindowLongPtr` 存 `this` |
| C 排序 | `qsort(... compare)` | 无 `void*` 参数，需要全局/static 变量 |

---

## Modern C++ 的改进

C++11 的 `std::thread` 在内部帮你做了桥接，不需要手动写 static 函数：

```cpp
// C++11：直接传成员函数 + 对象指针
auto t = std::thread(&MyClass::threadFunc, this);

// 或者用 lambda 捕获 this
auto t = std::thread([this]() { this->threadFunc(); });
```

但 `std::thread` 的内部实现**仍然使用了相同的桥接技巧**——只是标准库帮你封装好了。在需要直接对接 C API 的场景（如 Win32、MFC、嵌入式）中，仍然需要手动写 static 桥接。

---

## 关键要点

1. **C 回调只认裸函数指针**：`call 地址 + 参数压栈`，不知道 `this` 的存在
2. **C++ 成员函数多了 this**：`__thiscall` 调用约定，通过 ECX 寄存器传递，与 C 不兼容
3. **矛盾的根源**：操作系统和基础库都是 C 写的，C++ 必须与之互操作
4. **Static 桥接**：static 函数没有 this → 兼容 C 签名；通过 `void*` 传 this → 还原对象指针
5. **Modern C++**：`std::thread` / lambda 在内部封装了桥接，但底层原理不变
