# 工厂模式

> 工厂模式是一种**创建型设计模式**，将对象的创建逻辑封装到工厂类中，客户端无需关心具体创建细节。

---

## 概念定义

工厂模式（Factory Pattern）通过定义一个创建对象的接口，让子类决定实例化哪一个类。它将对象的**实例化推迟到子类**，实现了创建逻辑与使用逻辑的分离。

工厂模式有三种变体：

| 类型 | 复杂度 | 适用场景 |
|------|--------|---------|
| **简单工厂** | 低 | 产品种类少，不需要扩展 |
| **工厂方法** | 中 | 需要扩展产品类型 |
| **抽象工厂** | 高 | 需要创建产品族 |

---

## 为什么需要

### 问题：直接 `new` 的弊端

```cpp
// ❌ 客户端直接创建对象
void processDocument(const std::string& type) {
    Document* doc;
    if (type == "pdf") {
        doc = new PdfDocument();  // 直接依赖具体类
    } else if (type == "word") {
        doc = new WordDocument();
    }
    // 如果新增 Excel 类型，必须修改此处
}
```

**问题分析**：
1. **违反开闭原则**：新增类型需要修改客户端代码
2. **高耦合**：客户端直接依赖具体实现类
3. **重复代码**：多处创建逻辑散落在代码各处
4. **难以测试**：无法注入 mock 对象

### 解决：工厂封装创建逻辑

```cpp
// ✅ 使用工厂模式
auto doc = DocumentFactory::create("pdf");  // 客户端只依赖接口
```

## 简单工厂

### 设计思路

简单工厂将所有创建逻辑集中到一个工厂类中，通过参数决定创建哪种产品。

```
┌─────────────┐      creates      ┌──────────────┐
│   Factory   │─────────────────▶│   Product    │ (抽象基类)
└─────────────┘                   └──────┬───────┘
                                         │
                          ┌──────────────┼──────────────┐
                          ▼              ▼              ▼
                   ┌──────────┐   ┌──────────┐   ┌──────────┐
                   │ProductA  │   │ProductB  │   │ProductC  │
                   └──────────┘   └──────────┘   └──────────┘
```

### 实现示例

首先定义产品的抽象接口：

```cpp
// 产品抽象基类
class Document {
public:
    virtual ~Document() = default;
    virtual void open() = 0;
    virtual void save() = 0;
};
```

然后实现具体产品类：

```cpp
// 具体产品：PDF 文档
class PdfDocument : public Document {
public:
    void open() override {
        std::cout << "Opening PDF document\n";
    }
    void save() override {
        std::cout << "Saving PDF document\n";
    }
};

// 具体产品：Word 文档
class WordDocument : public Document {
public:
    void open() override {
        std::cout << "Opening Word document\n";
    }
    void save() override {
        std::cout << "Saving Word document\n";
    }
};
```

**关键点**：产品类继承自同一抽象基类，确保接口一致。

最后实现简单工厂：

```cpp
// 简单工厂
class DocumentFactory {
public:
    // 使用枚举代替字符串，提供类型安全
    enum class Type { Pdf, Word };

    static std::unique_ptr<Document> create(Type type) {
        switch (type) {
            case Type::Pdf:
                return std::make_unique<PdfDocument>();
            case Type::Word:
                return std::make_unique<WordDocument>();
            default:
                throw std::invalid_argument("Unknown document type");
        }
    }
};
```

**设计要点**：
- 使用 `std::unique_ptr` 管理对象生命周期，遵循 RAII 原则
- 使用枚举代替字符串，避免拼写错误
- 静态方法便于调用，无需实例化工厂

### 使用示例

```cpp
int main() {
    auto pdf = DocumentFactory::create(DocumentFactory::Type::Pdf);
    pdf->open();
    pdf->save();
    // unique_ptr 自动释放资源
}
```

### 简单工厂的局限

| 优点 | 缺点 |
|-----|------|
| 实现简单 | 违反开闭原则（新增产品需修改工厂） |
| 集中管理创建逻辑 | 工厂类职责过重 |
| 客户端无需知道具体类名 | 不适合产品种类多的场景 |

## 工厂方法

### 设计思路

工厂方法模式定义一个创建对象的接口，但由**子类决定实例化哪个类**。每种产品对应一个工厂子类。

```
┌─────────────────┐              ┌─────────────────┐
│  Creator        │              │  Product        │
│  + create()     │─────────────▶│                 │
└────────┬────────┘              └────────┬────────┘
         │                                │
    ┌────┴────┐                     ┌─────┴─────┐
    ▼         ▼                     ▼           ▼
┌────────┐ ┌────────┐         ┌─────────┐ ┌─────────┐
│CreatorA│ │CreatorB│         │ProductA │ │ProductB │
└────────┘ └────────┘         └─────────┘ └─────────┘
    │           │                  ▲           ▲
    └───creates─┼──────────────────┘           │
                └──────────────────────────────┘
```

### 实现示例

定义抽象工厂接口：

```cpp
// 抽象工厂
class DocumentCreator {
public:
    virtual ~DocumentCreator() = default;

    // 工厂方法：由子类实现
    virtual std::unique_ptr<Document> createDocument() = 0;

    // 模板方法：定义通用流程
    void processDocument() {
        auto doc = createDocument();  // 调用工厂方法
        doc->open();
        // ... 其他处理
        doc->save();
    }
};
```

**关键点**：`createDocument()` 是纯虚函数，强制子类提供实现。

实现具体工厂：

```cpp
// PDF 文档工厂
class PdfCreator : public DocumentCreator {
public:
    std::unique_ptr<Document> createDocument() override {
        return std::make_unique<PdfDocument>();
    }
};

// Word 文档工厂
class WordCreator : public DocumentCreator {
public:
    std::unique_ptr<Document> createDocument() override {
        return std::make_unique<WordDocument>();
    }
};
```

**设计要点**：
- 每种产品对应一个工厂类
- 新增产品只需新增工厂子类，无需修改现有代码
- 符合开闭原则

### 使用示例

```cpp
void clientCode(DocumentCreator& creator) {
    // 客户端只依赖抽象工厂，不关心具体类型
    creator.processDocument();
}

int main() {
    PdfCreator pdfCreator;
    WordCreator wordCreator;

    clientCode(pdfCreator);   // 处理 PDF
    clientCode(wordCreator);  // 处理 Word
}
```

## 抽象工厂

### 设计思路

抽象工厂提供一个创建**一系列相关对象**的接口，而无需指定它们的具体类。适用于产品族场景。

```
┌───────────────────────┐
│   AbstractFactory     │
│  + createButton()     │
│  + createTextBox()    │
└───────────┬───────────┘
            │
     ┌──────┴──────┐
     ▼             ▼
┌─────────┐   ┌─────────┐
│WinFactory│   │MacFactory│
└─────────┘   └─────────┘
     │             │
     ▼             ▼
 WinButton     MacButton      (产品族 A)
 WinTextBox    MacTextBox     (产品族 B)
```

### 实现示例

首先定义产品族的抽象接口：

```cpp
// 产品族 A：按钮
class Button {
public:
    virtual ~Button() = default;
    virtual void render() = 0;
};

// 产品族 B：文本框
class TextBox {
public:
    virtual ~TextBox() = default;
    virtual void render() = 0;
};
```

实现 Windows 风格产品：

```cpp
class WinButton : public Button {
public:
    void render() override {
        std::cout << "Rendering Windows button\n";
    }
};

class WinTextBox : public TextBox {
public:
    void render() override {
        std::cout << "Rendering Windows textbox\n";
    }
};
```

实现 Mac 风格产品（结构类似，省略）。

定义抽象工厂接口：

```cpp
// 抽象工厂
class GUIFactory {
public:
    virtual ~GUIFactory() = default;
    virtual std::unique_ptr<Button> createButton() = 0;
    virtual std::unique_ptr<TextBox> createTextBox() = 0;
};
```

**关键点**：抽象工厂定义了一组相关产品的创建方法。

实现具体工厂：

```cpp
// Windows 工厂：创建 Windows 风格的整套 UI 组件
class WinFactory : public GUIFactory {
public:
    std::unique_ptr<Button> createButton() override {
        return std::make_unique<WinButton>();
    }
    std::unique_ptr<TextBox> createTextBox() override {
        return std::make_unique<WinTextBox>();
    }
};

// Mac 工厂：创建 Mac 风格的整套 UI 组件
class MacFactory : public GUIFactory {
public:
    std::unique_ptr<Button> createButton() override {
        return std::make_unique<MacButton>();
    }
    std::unique_ptr<TextBox> createTextBox() override {
        return std::make_unique<MacTextBox>();
    }
};
```

### 使用示例

```cpp
void buildUI(GUIFactory& factory) {
    auto button = factory.createButton();
    auto textBox = factory.createTextBox();

    button->render();
    textBox->render();
    // 保证 UI 风格一致性
}

int main() {
    // 根据平台选择工厂
    #ifdef _WIN32
        WinFactory factory;
    #else
        MacFactory factory;
    #endif

    buildUI(factory);
}
```

**设计要点**：
- 抽象工厂保证**产品族的一致性**
- 切换整套产品只需更换工厂实例
- 适合跨平台 UI、数据库驱动等场景

---

## 三种工厂模式对比

| 特性 | 简单工厂 | 工厂方法 | 抽象工厂 |
|------|---------|---------|---------|
| **创建方式** | 一个工厂创建所有产品 | 每种产品一个工厂 | 每个产品族一个工厂 |
| **扩展性** | 差（需修改工厂） | 好（新增工厂子类） | 好（新增工厂子类） |
| **复杂度** | 低 | 中 | 高 |
| **适用场景** | 产品少、稳定 | 产品需要扩展 | 产品族、跨平台 |
| **开闭原则** | 违反 | 遵守 | 遵守 |

## Modern C++ 实践

### 使用 `std::variant` 简化简单工厂

C++17 的 `std::variant` 可以避免虚函数开销：

```cpp
using DocumentVariant = std::variant<PdfDocument, WordDocument>;

class ModernFactory {
public:
    static DocumentVariant create(std::string_view type) {
        if (type == "pdf") return PdfDocument{};
        if (type == "word") return WordDocument{};
        throw std::invalid_argument("Unknown type");
    }
};

// 使用 std::visit 处理
auto doc = ModernFactory::create("pdf");
std::visit([](auto& d) { d.open(); }, doc);
```

**优点**：无虚函数调用、值语义、类型安全。

### 使用模板工厂

```cpp
template<typename T, typename... Args>
std::unique_ptr<T> make(Args&&... args) {
    return std::make_unique<T>(std::forward<Args>(args)...);
}

// 使用
auto pdf = make<PdfDocument>();
```

### 注册式工厂（消除 switch-case）

```cpp
class DocumentRegistry {
    using Creator = std::function<std::unique_ptr<Document>()>;
    std::unordered_map<std::string, Creator> registry_;

public:
    void registerType(const std::string& name, Creator creator) {
        registry_[name] = std::move(creator);
    }

    std::unique_ptr<Document> create(const std::string& name) {
        auto it = registry_.find(name);
        if (it == registry_.end()) {
            throw std::invalid_argument("Unknown type: " + name);
        }
        return it->second();
    }
};

// 使用
DocumentRegistry registry;
registry.registerType("pdf", []{ return std::make_unique<PdfDocument>(); });
registry.registerType("word", []{ return std::make_unique<WordDocument>(); });

auto doc = registry.create("pdf");
```

**优点**：完全消除 switch-case，支持运行时注册新类型。

---

## 最佳实践

| 推荐 | 避免 | 原因 |
|-----|------|------|
| 返回 `unique_ptr` | 返回裸指针 | RAII 自动管理生命周期 |
| 使用枚举/注册表 | 字符串硬编码 | 类型安全，避免拼写错误 |
| 工厂方法返回抽象类型 | 返回具体类型 | 保持多态性 |
| 根据扩展需求选择模式 | 一律用抽象工厂 | 避免过度设计 |
