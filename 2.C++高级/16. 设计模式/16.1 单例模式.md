# 单例模式

> 单例模式是一种**创建型设计模式**，确保一个类只有一个实例，并提供该实例的全局访问点。

---

## 概念定义

单例模式（Singleton Pattern）控制实例数量，保证全局唯一性。它适用于需要**集中管理共享资源**的场景。

核心要素：

| 要素 | 实现方式 |
|------|---------|
| **唯一实例** | 私有静态成员或局部静态变量 |
| **禁止外部构造** | 私有构造函数 |
| **全局访问点** | 公有静态方法 `getInstance()` |

```
┌─────────────────────────────┐
│         Singleton           │
├─────────────────────────────┤
│ - instance: Singleton*      │
│ - Singleton()               │  ← 私有构造
├─────────────────────────────┤
│ + getInstance(): Singleton& │  ← 全局访问点
└─────────────────────────────┘
```

---

## 为什么需要

### 适用场景

| 场景        | 原因           |
| --------- | ------------ |
| **配置管理器** | 全局配置需要统一访问   |
| **日志系统**  | 避免多个日志实例竞争文件 |
| **连接池**   | 集中管理数据库连接    |
| **硬件访问**  | 打印机、显卡等物理资源  |
| **缓存**    | 全局共享缓存数据     |

### 问题：全局变量的弊端

```cpp
// ❌ 全局变量的问题
Logger globalLogger;  // 初始化顺序不确定

// 另一个翻译单元
extern Logger globalLogger;
globalLogger.log("message");  // 可能尚未初始化
```

**问题分析**：
1. **初始化顺序不确定**：跨翻译单元的全局变量初始化顺序未定义
2. **无法控制构造时机**：程序启动时立即构造
3. **难以进行延迟初始化**：无法实现按需创建

---

## 实现方式

### 1. 饿汉式（Eager Initialization）

程序启动时立即创建实例：

```cpp
class EagerSingleton {
    static EagerSingleton instance_;  // 类外初始化

    EagerSingleton() = default;

public:
    EagerSingleton(const EagerSingleton&) = delete;
    EagerSingleton& operator=(const EagerSingleton&) = delete;

    static EagerSingleton& getInstance() {
        return instance_;
    }
};

// 类外初始化（必须）
EagerSingleton EagerSingleton::instance_;
```

| 优点 | 缺点 |
|-----|------|
| 实现简单 | 程序启动即初始化，浪费资源 |
| 天然线程安全 | 无法传递构造参数 |
| 无锁，访问效率高 | 初始化顺序问题（SIOF） |
#### 1.1 编译阶段：内存布局确定

当编译器处理 `EagerSingleton` 类时：

- 它发现 `instance_` 是一个 **静态成员变量**。
    
- 静态变量不占用类实例的空间，而是存储在程序的 **全局/静态存储区（Data Segment）**。
    
- 编译器会为该变量预留空间，并记录下它需要调用构造函数进行初始化。
    
#### 1.2 程序启动阶段：静态初始化（关键点）

这是“饿汉式”得名的原因。在操作系统加载程序并调用 `main()` 函数之前，C++ 运行时库会负责初始化所有的全局变量和静态变量。

- **调用构造函数**：执行到 `EagerSingleton EagerSingleton::instance_;` 这一行时，系统调用 `EagerSingleton` 的默认构造函数。
    
- **内存分配**：此时 `instance_` 对象已经在全局内存中创建完毕。
    
- **线程安全性**：在单线程环境下（程序启动初期），这种初始化由编译器和链接器保证只执行一次，因此天然避开了多线程竞态条件（Race Condition）的问题。
    
#### 1.3 运行阶段：访问实例

当你的代码在 `main` 函数或其他地方调用 `EagerSingleton::getInstance()` 时：

1. **直接返回**：由于 `instance_` 已经在启动阶段创建好了，函数只需简单地返回该对象的引用。
    
2. **无需加锁**：不需要像懒汉式（Lazy Initialization）那样判断 `if (instance == nullptr)` 或使用 `std::mutex`，执行效率极高。

#### 静态初始化顺序问题（SIOF）

**Static Initialization Order Fiasco**：跨翻译单元的静态变量初始化顺序是**未定义**的。

```cpp
// === file_a.cpp ===
class ConfigManager {
    static ConfigManager instance_;
public:
    static ConfigManager& getInstance() { return instance_; }
    std::string getValue(const std::string& key);
};
ConfigManager ConfigManager::instance_;  // 何时初始化？

// === file_b.cpp ===
class Logger {
    static Logger instance_;
    Logger() {
	    // 有另一个全局静态对象尝试调用静态方法 static ConfigManager& getInstance() 
        // ❌ 危险：ConfigManager 可能尚未初始化
        auto level = ConfigManager::getInstance().getValue("log_level");
    }
public:
    static Logger& getInstance() { return instance_; }
};
Logger Logger::instance_;  // 可能先于 ConfigManager 初始化
```

**问题本质**：
- 同一翻译单元内，静态变量按**定义顺序**初始化
- 跨翻译单元，初始化顺序由**链接器决定**，标准未规定

> [!tip] 解决方案
> 使用 Meyers' Singleton（局部静态变量），将初始化推迟到首次访问时。

### 2. 懒汉式（Lazy Initialization）- 经典但有问题

首次调用时创建实例：

```cpp
// ❌ 线程不安全版本
class LazySingleton {
    static LazySingleton* instance_;

    LazySingleton() = default;

public:
    LazySingleton(const LazySingleton&) = delete;
    LazySingleton& operator=(const LazySingleton&) = delete;

    static LazySingleton* getInstance() {
        if (instance_ == nullptr) {      // 检查
            instance_ = new LazySingleton();  // 创建
        }
        return instance_;
    }
};
LazySingleton* LazySingleton::instance_ = nullptr;
```

#### 线程安全问题分析

**竞态条件时序图**：

```
Thread A                    Thread B                    instance_
   │                           │                           │
   ├─ if(instance_==nullptr)   │                        nullptr
   │  [true]                   │                           │
   │                           ├─ if(instance_==nullptr)   │
   │                           │  [true]                   │
   ├─ new LazySingleton() ─────┼───────────────────────► obj1
   │                           │                           │
   │                           ├─ new LazySingleton() ──► obj2 (内存泄漏!)
   │                           │                           │
```

**问题**：两个线程可能同时通过 `nullptr` 检查，创建多个实例。

#### 简单加锁版本（性能差）

```cpp
// ⚠️ 线程安全但效率低
class LazySingletonLocked {
    static LazySingletonLocked* instance_;
    static std::mutex mutex_;

public:
    static LazySingletonLocked* getInstance() {
        std::lock_guard<std::mutex> lock(mutex_);  // 每次都加锁
        if (instance_ == nullptr) {
            instance_ = new LazySingletonLocked();
        }
        return instance_;
    }
};
```

**性能问题**：实例创建后，每次访问仍需获取锁，而此时锁是**完全不必要**的。

| 方案 | 线程安全 | 性能 | 问题 |
|-----|:-------:|:----:|-----|
| 无锁版本 | ❌ | 高 | 竞态条件 |
| 全锁版本 | ✅ | **低** | 每次访问都加锁 |
| DCLP | ✅ | 高 | 实现复杂，需正确使用内存序 |

### 3. 双重检查锁定（DCLP）

**Double-Checked Locking Pattern**：尝试在保证线程安全的同时避免每次都加锁。

#### 朴素 DCLP（有问题）

```cpp
// ❌ 错误实现 - 在 C++11 之前（甚至之后不使用 atomic）都有问题
class NaiveDCLP {
    static NaiveDCLP* instance_;
    static std::mutex mutex_;

public:
    static NaiveDCLP* getInstance() {
        if (instance_ == nullptr) {           // 第一次检查（无锁）
            std::lock_guard<std::mutex> lock(mutex_);
            if (instance_ == nullptr) {       // 第二次检查（有锁）
                instance_ = new NaiveDCLP();  // ⚠️ 问题所在
            }
        }
        return instance_;
    }
};
```

**设计思路**：
- **第一次检查**：无锁快速路径，实例存在时直接返回
- **加锁**：只有首次创建时才需要
- **第二次检查**：防止多个线程同时通过第一次检查后重复创建

看起来完美，但实际上有严重的内存模型问题。

#### DCLP 失败的根本原因

`new` 操作在编译器/CPU层面可能被分解为三个步骤：

```cpp
instance_ = new NaiveDCLP();
// 实际可能被分解为：
// 1. 分配内存：temp = operator new(sizeof(NaiveDCLP))
// 2. 构造对象：new(temp) NaiveDCLP()
// 3. 赋值指针：instance_ = temp
```

**问题**：编译器和 CPU 可能对步骤 2 和步骤 3 进行**重排序**：

```
重排序后：
1. 分配内存：temp = operator new(sizeof(NaiveDCLP))
3. 赋值指针：instance_ = temp     ← 先赋值！
2. 构造对象：new(temp) NaiveDCLP() ← 后构造
```

**灾难性后果**：

```
Thread A                           Thread B
   │                                  │
   ├─ if(instance_==nullptr) [true]   │
   ├─ lock()                          │
   ├─ if(instance_==nullptr) [true]   │
   ├─ 分配内存 temp                    │
   ├─ instance_ = temp ───────────────┼──► instance_ 非空但未构造!
   │  (对象尚未构造完成!)               │
   │                                  ├─ if(instance_==nullptr) [false]
   │                                  ├─ return instance_  ← 返回未初始化对象!
   │                                  ├─ instance_->method() ← 未定义行为!
   ├─ 构造对象                         │
   ├─ unlock()                        │
```

Thread B 拿到了一个**指针非空但对象未构造完成**的实例。

#### 为什么会发生重排序？

| 层级 | 重排序来源 | 原因 |
|-----|-----------|------|
| **编译器** | 优化重排 | 编译器认为重排不影响单线程语义 |
| **CPU** | 乱序执行 | 现代 CPU 为提高性能会乱序执行指令 |
| **缓存** | Store Buffer | 写操作可能在缓存中重排 |

> [!important] 关键理解
> 单线程下，重排序不影响程序正确性（as-if 规则）。
> 多线程下，其他线程可能**观察到**中间状态，导致问题。

#### 正确的 DCLP 实现（C++11）

使用 `std::atomic` 配合正确的内存序：

```cpp
class DCLPSingleton {
    static std::atomic<DCLPSingleton*> instance_;
    static std::mutex mutex_;

    DCLPSingleton() = default;

public:
    static DCLPSingleton* getInstance() {
        // acquire：保证后续读写不会被重排到此之前
        DCLPSingleton* tmp = instance_.load(std::memory_order_acquire);
        if (tmp == nullptr) {
            std::lock_guard<std::mutex> lock(mutex_);
            tmp = instance_.load(std::memory_order_relaxed);  // 锁内可用 relaxed
            if (tmp == nullptr) {
                tmp = new DCLPSingleton();
                // release：保证之前的写操作对其他线程可见
                instance_.store(tmp, std::memory_order_release);
            }
        }
        return tmp;
    }
};

std::atomic<DCLPSingleton*> DCLPSingleton::instance_{nullptr};
std::mutex DCLPSingleton::mutex_;
```

#### 内存序解释

```
                    Thread A (写)                Thread B (读)
                         │                            │
          ┌──────────────┼────────────────────────────┼──────────────┐
          │   构造函数写入成员变量                       │              │
          │   obj->data_ = 42;                        │              │
          │              │                            │              │
          │              ▼                            │              │
          │   ════════ release barrier ════════      │              │
          │              │                            │              │
          │   instance_.store(obj, release)          │              │
          │              │                            │              │
          │              └──────────同步──────────────►│              │
          │                                           │              │
          │                        instance_.load(acquire)           │
          │                                           │              │
          │                        ════════ acquire barrier ════════ │
          │                                           │              │
          │                                           ▼              │
          │                              obj->data_ 保证可见 (42)     │
          └──────────────────────────────────────────────────────────┘
```

| 内存序                    | 作用   | 保证                       |
| ---------------------- | ---- | ------------------------ |
| `memory_order_release` | 写端屏障 | 之前所有写操作对"配对的 acquire"可见  |
| `memory_order_acquire` | 读端屏障 | 能看到"配对的 release"之前的所有写操作 |
| `memory_order_relaxed` | 无屏障  | 仅保证原子性，锁内使用即可            |

#### DCLP 总结

| 实现方式 | 正确性 | 说明 |
|---------|:------:|------|
| 普通指针 | ❌ | 有重排序问题 |
| `volatile` 指针 | ❌ | C++ 的 `volatile` 不提供线程安全保证 |
| `std::atomic` + 默认内存序 | ✅ | 正确但可能过度同步 |
| `std::atomic` + acquire/release | ✅ | **最优实现** |

> [!warning] 实践建议
> DCLP 实现复杂且容易出错。除非有特殊需求（如需要指针语义），**推荐使用 Meyers' Singleton**。

### 4. Meyers' Singleton（推荐）

**C++11 后最佳实践**：利用局部静态变量的线程安全初始化：

```cpp
class Singleton {
    Singleton() = default;
    ~Singleton() = default;

public:
    // 禁止拷贝和赋值
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static Singleton& getInstance() {
        static Singleton instance;  // C++11 保证线程安全
        return instance;
    }

    void doSomething() {
        std::cout << "Singleton operation\n";
    }
};
```

**为什么这是最佳实践**：

| 特性 | 说明 |
|------|------|
| **线程安全** | C++11 标准保证局部静态变量初始化是线程安全的 |
| **延迟初始化** | 首次调用 `getInstance()` 时才构造 |
| **自动销毁** | 程序结束时自动调用析构函数 |
| **代码简洁** | 无需手动管理指针和锁 |

**C++11 标准保证**（[stmt.dcl] §6.7）：
> "If control enters the declaration concurrently while the variable is being initialized, the concurrent execution shall wait for completion of the initialization."

### 5. 使用 `std::call_once`

另一种线程安全的实现方式：

```cpp
class CallOnceSingleton {
    static std::unique_ptr<CallOnceSingleton> instance_;
    static std::once_flag initFlag_;

    CallOnceSingleton() = default;

public:
    CallOnceSingleton(const CallOnceSingleton&) = delete;
    CallOnceSingleton& operator=(const CallOnceSingleton&) = delete;

    static CallOnceSingleton& getInstance() {
        std::call_once(initFlag_, []() {
            instance_.reset(new CallOnceSingleton());
        });
        return *instance_;
    }
};

std::unique_ptr<CallOnceSingleton> CallOnceSingleton::instance_;
std::once_flag CallOnceSingleton::initFlag_;
```

**适用场景**：需要更细粒度控制初始化时机。

## 实现对比

| 实现方式 | 线程安全 | 延迟初始化 | 复杂度 | 性能 | 推荐度 |
|---------|:-------:|:---------:|:-----:|:----:|:-----:|
| 饿汉式 | ✅ | ❌ | 低 | 高 | ⭐⭐ |
| 懒汉式（无锁） | ❌ | ✅ | 低 | 高 | ❌ |
| 懒汉式（全锁） | ✅ | ✅ | 低 | **低** | ⭐ |
| 朴素 DCLP | ❌ | ✅ | 中 | 高 | ❌ |
| DCLP + atomic | ✅ | ✅ | **高** | 高 | ⭐⭐⭐ |
| **Meyers' Singleton** | ✅ | ✅ | **低** | 高 | ⭐⭐⭐⭐⭐ |
| `call_once` | ✅ | ✅ | 中 | 高 | ⭐⭐⭐⭐ |

**选择建议**：
- **默认选择**：Meyers' Singleton（简单、安全、高效）
- **需要指针语义**：DCLP + atomic 或 `call_once`
- **启动时初始化可接受**：饿汉式

## 带参数初始化

有时单例需要初始化参数：

```cpp
class ConfigSingleton {
    std::string configPath_;

    explicit ConfigSingleton(std::string path)
        : configPath_(std::move(path)) {}

public:
    ConfigSingleton(const ConfigSingleton&) = delete;
    ConfigSingleton& operator=(const ConfigSingleton&) = delete;

    // 首次调用必须传参
    static ConfigSingleton& getInstance(const std::string& path = "") {
        static ConfigSingleton instance(path);
        return instance;
    }

    const std::string& getConfigPath() const { return configPath_; }
};

// 使用
ConfigSingleton::getInstance("/etc/app.conf");  // 初始化
ConfigSingleton::getInstance().getConfigPath(); // 后续访问
```

> [!warning] 注意
> 局部静态变量只初始化一次，后续调用传入的参数会被忽略。

## 单例的销毁问题

### 析构顺序问题

多个单例之间可能存在依赖关系：

```cpp
class Logger {
public:
    static Logger& getInstance() {
        static Logger instance;
        return instance;
    }
    void log(const std::string& msg) { /* ... */ }
};

class Database {
public:
    static Database& getInstance() {
        static Database instance;
        return instance;
    }
    ~Database() {
        Logger::getInstance().log("Database closed");  // ⚠️ Logger 可能已销毁
    }
};
```

**问题**：静态变量以构造的逆序销毁，如果 `Logger` 先于 `Database` 销毁，析构时访问 `Logger` 会导致未定义行为。

### 解决方案：Phoenix Singleton

```cpp
class PhoenixSingleton {
    static PhoenixSingleton* instance_;
    static bool destroyed_;

    PhoenixSingleton() = default;
    ~PhoenixSingleton() {
        instance_ = nullptr;
        destroyed_ = true;
    }

public:
    static PhoenixSingleton& getInstance() {
        if (instance_ == nullptr) {
            if (destroyed_) {
                // 重新创建（凤凰重生）
                new(instance_) PhoenixSingleton();
                destroyed_ = false;
            } else {
                static PhoenixSingleton theInstance;
                instance_ = &theInstance;
            }
        }
        return *instance_;
    }
};
```

**实践建议**：尽量避免单例之间的依赖，或使用依赖注入替代单例。

## 单例的测试问题

单例模式使测试变得困难：

```cpp
// ❌ 难以测试的代码
void processOrder(const Order& order) {
    Logger::getInstance().log("Processing order");
    Database::getInstance().save(order);
}
```

### 解决方案：依赖注入

```cpp
// ✅ 可测试的代码
class OrderProcessor {
    Logger& logger_;
    Database& database_;

public:
    OrderProcessor(Logger& logger, Database& db)
        : logger_(logger), database_(db) {}

    void process(const Order& order) {
        logger_.log("Processing order");
        database_.save(order);
    }
};

// 测试时可注入 mock 对象
class MockLogger : public Logger { /* ... */ };
class MockDatabase : public Database { /* ... */ };

void test() {
    MockLogger mockLogger;
    MockDatabase mockDb;
    OrderProcessor processor(mockLogger, mockDb);
    // 测试...
}
```

## 单例模板

可以创建通用的单例模板：

```cpp
template<typename T>
class Singleton {
protected:
    Singleton() = default;
    ~Singleton() = default;

public:
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    static T& getInstance() {
        static T instance;
        return instance;
    }
};

// 使用
class Logger : public Singleton<Logger> {
    friend class Singleton<Logger>;  // 允许基类访问私有构造
    Logger() = default;

public:
    void log(const std::string& msg) {
        std::cout << msg << "\n";
    }
};

// 调用
Logger::getInstance().log("Hello");
```

**CRTP（Curiously Recurring Template Pattern）** 让派生类自动获得单例能力。

## 陷阱与误区

> [!warning] 常见问题

### 1. 忘记禁用拷贝

```cpp
// ❌ 可被拷贝
Singleton s = Singleton::getInstance();  // 创建了副本

// ✅ 禁用拷贝
Singleton(const Singleton&) = delete;
Singleton& operator=(const Singleton&) = delete;
```

### 2. 返回指针而非引用

```cpp
// ❌ 返回指针
static Singleton* getInstance() { ... }
// 用户可能 delete 指针

// ✅ 返回引用
static Singleton& getInstance() { ... }
```

### 3. 多线程初始化竞争

```cpp
// ❌ C++11 之前的局部静态变量不是线程安全的
// ✅ C++11 后 Meyers' Singleton 是安全的
```

### 4. 滥用单例

单例不应该成为"全局变量的包装器"。如果发现代码中有大量单例，考虑重构为依赖注入。

## 最佳实践

| 推荐 | 避免 | 原因 |
|-----|------|------|
| 使用 Meyers' Singleton | 手动 new/delete | 简洁且线程安全 |
| 返回引用 | 返回指针 | 防止误删 |
| 考虑依赖注入替代 | 滥用单例 | 提高可测试性 |
| 禁用拷贝和赋值 | 允许拷贝 | 保证唯一性 |
