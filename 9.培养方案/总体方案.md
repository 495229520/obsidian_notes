# 一、培养目标

12 个月结束时，你应该达到这几个结果：

**能力目标**

1. 能独立实现一个中小型 C++ 服务端项目，不是 demo，而是有分层、日志、配置、测试、压测、指标。
    
2. 能做一个可讲清楚的 `LLM Gateway`，支持 provider 抽象、streaming、timeout/retry、限流、trace、配置化。

5. 能投递这几类岗位：

    - Agent Infra / Agent Runtime
        
    - LLM Serving / Gateway
        
    - C++ 后端 / 推理服务 / 智能体平台相关岗位
        

**项目目标**  
最后至少沉淀 3 个主项目：

- `cpp-service-core`
    
- `llm-gateway`
    
- `mini-agent-runtime`
    

**表达目标**

- 每个项目都能讲 3 分钟版、8 分钟版

# 二、这版方案相对原方案怎么改

原方案好的地方我保留：

- 每周必须有代码和输出物
    
- 每月必须有可展示成果
    
- 最终落到 GitHub、README、架构图、简历 bullet point
    

但我做 4 个大改动：

**第一，把基础期压缩。**  
原方案前 6 个月才完成 Linux/网络/线程/epoll 基础，对你现在的起点太慢。  
我会压成前 **2.5–3 个月**。

**第二，把 Agent Infra 主线前移。**  
原方案第 10 个月才做 Gateway，第 13 个月才做 Runtime。  
我会在 **第 5 个月开始 Gateway，第 7 个月开始 Runtime**。

**第三，把工程化前移。**  
原方案把 Docker/K8s 放得偏后。  
我会在 **第 6–8 个月** 就加入容器化、部署、观测、压测。

**第四，把“应用层 AI 学习”降权。**  
你不是去走“Prompt 工程 + RAG 应用开发”的路线。  
你要学的是：

- 请求生命周期
    
- 多 provider 抽象
    
- 流式协议
    
- 调度与执行
    
- 状态管理
    
- 可观测性
    
- 故障处理
    
- 工程边界

# 十一、技术选型清单（补充）

> 原方案缺少具体技术选型，实际执行时容易反复纠结。以下是推荐选型。

| 工具 | 用途 | 说明 |

|-----|------|---------|

| **yaml-cpp** | 配置文件 | 比 JSON 更适合做配置 |
| **GoogleTest** | 单元测试 | 工业标准 |
| **Google Benchmark** | 性能测试 | 配合 GoogleTest 使用 |

## 进阶依赖（第 5 月起）

| 库 | 用途 | 说明 |
|-----|------|------|
| **gRPC + protobuf** | Tool 服务化协议 | 第 9 月 Tool 服务化必用 |
| **Boost.Asio** | 异步网络（可选） | 如果想替代手写 epoll |
| **prometheus-cpp** | 指标暴露 | 配合观测面板 |

## C++20 特性路线

| 特性 | 何时学 | 用途 |
|------|--------|------|
| concepts | 第 3 月 | 模板约束，让接口更清晰 |

| coroutines | 第 5 月 | streaming 处理、异步 IO |
| std::format | 第 1 月 | 替代 printf/iostream 拼接 |

# 十二、卡点回退策略（补充）

> 原方案缺少”卡住了怎么办”的指导。

## 通用原则

**30 分钟规则**：一个问题卡超 30 分钟，切换策略：
1. 先看 muduo / 参考项目的实现
2. 简化问题，写最小复现
3. 记录卡点，换一个任务推进，隔天回来

## 各阶段常见卡点

| 阶段 | 卡点 | 回退方案 |
|------|------|---------|
| 第 1 月 CMake | 配置报错 | 先用 vcpkg 的 CMake integration 模板 |
| 第 2 月 epoll | reactor 写不出来 | 先读 muduo 的 EventLoop 源码，再仿写 |
| 第 3 月 分层设计 | 不知道怎么拆模块 | 参考 muduo 的目录结构，先抄再改 |
| 第 5 月 streaming | SSE 解析不明白 | 先用 curl 手动请求 OpenAI API，观察原始数据流 |
| 第 7 月 状态机 | 状态转换混乱 | 先画状态图（Mermaid），再写代码 |

## 验收不通过时

- **基础验收不通过**：不要跳到下一月，用 1 周补齐最薄弱的 1-2 个点
- **项目验收不通过**：降低功能范围，先确保核心链路跑通，再逐步加功能
