> 面向：中台开发者、Qt 客户端开发者、下位机开发者  
> 基线工程：`/home/cyi/Document/fix/new/transplant_test`  
> 设计时间：2026-04-07  
> 文档性质：架构与实施设计，不含代码实现

## 1. 文档目的

本文档用于在 **不推翻现有 MFMS 数据中台架构** 的前提下，基于仙工底盘设备包，为上位机补充以下能力：

1. 上位机指定单个站点，复合机器人运动到指定站点。
2. 上位机指定一系列站点，复合机器人按顺序运动到路径上的每个站点。
3. 保留现有法奥机械臂链路与数据库状态机，不影响原有 UI 和设备接入方式。

本文档不讨论地图算法、全局路径求解或复合机器人协同工艺逻辑；重点只放在 **数据中台如何增量接入仙工底盘接口**。

## 3. 现状问题总结

结合现有工程和下位机提供的设备包，当前仙工底盘链路存在 5 个关键问题。

### 3.1 顶层已预留 AGV 控制接口，但底层未实现

当前 `MfmsCommandService::sendAgvMotion()` 与 `stopAgv()` 仍直接返回“AGV 控制功能已禁用”。  
这意味着：

- 现有 UI 到 Worker 的 AGV 按钮链路只是半成品。
- 不能在现有 `AgvMotionCommand` 上直接堆导航功能。

### 3.2 当前 AGV 模型只覆盖基础运动，不覆盖导航任务

当前 `AgvMotionCommand` 只能描述：

- 线速度
- 距离
- 转动角度
- 转动速度

但仙工底盘设备包实际支持的能力不止这些，设备包命令号已明确包含：

- `SEERAGV_MOVE_POINT`
- `SEERAGV_MOVE_POINT_LIST`
- `SEERAGV_MOVE_PATH`
- `SEERAGV_CHECK_STATION`
- `SEERAGV_CURRENT_STATION`
- `SEERAGV_CHECK_GUIDE`

因此，当前中台必须将 **基础运动** 和 **站点导航** 分开建模。

### 3.3 当前 AGV 状态回传在 Worker 层被严重压缩

现有 `CommunicationWorker::handleRobotStatus()` 对 AGV 仅保留了少数字段：

- 名称
- 版本
- IP
- 位姿
- 置信度
- 速度
- blocked

而下位机发布侧实际已具备更完整的状态：

- `current_station`
- `last_station`
- 电池信息
- 阻挡/减速原因
- 告警等级和告警 JSON

如果不补状态通路，中台无法稳定支持导航任务跟踪。

### 3.4 项目内 `srv/msg` 与下位机样例存在契约版本不一致

当前项目内的 `src/com_interfaces/srv/SeerAgvCmdInterface.srv` 和 `src/com_interfaces/msg/SeerAgvState.msg` 版本落后于下位机提供的样例：

- 旧版 `srv` 缺少导航查询返回字段：
  - `task_status`
  - `task_type`
  - `target_id`
  - `msg`
- 旧版 `msg` 缺少：
  - 电池信息
  - 告警等级
  - 告警 JSON

因此在正式开发前，必须先统一接口契约版本。

### 3.5 当前客户端接口仍是“设备控制”思维，不是“导航任务”思维

客户端现有 AGV 接口更像：

- 前进
- 后退
- 左转
- 右转

但新需求实际需要的是：

- 获取站点列表
- 指定站点导航
- 指定站点序列导航
- 查询导航进度
- 暂停/继续/取消导航

因此本次中台改造的重点，是补一层 **任务级导航接口**。

## 4. 改造目标

### 4.1 业务目标

中台新增以下仙工底盘能力：

1. 基础运动
   - 平移
   - 转动
   - 停止
2. 查询能力
   - 查询站点列表
   - 查询当前站点
   - 查询电量
   - 查询导航状态
3. 单点导航
   - 上位机选择一个站点
   - 中台下发导航
   - 中台持续跟踪并回报任务结果
4. 多站点顺序执行
   - 上位机选择一个站点序列
   - 中台下发路径任务
   - 中台回报每一站的进度与最终结果

### 4.2 架构目标

1. 保留现有 `MfmsGateway` 三服务结构。
2. 保留现有设备连接/断开状态机。
3. 在中台内部新增仙工底盘专属命令与状态适配，不污染法奥机械臂链路。
4. 给客户端暴露“可直接使用”的导航接口，而不是底层 `srv/msg` 细节。
5. 为后续复合机器人动作编排预留扩展点。

## 5. 设计原则

### 5.1 原架构不推翻，只做增量

本次不新增新的中台主骨架，不重做通信模式，只在以下模块增量扩展：

- `com_interfaces`
- `MfmsCommandService`
- `MfmsRosBridge`
- `CommunicationWorker`
- `CommunicationInterface`

### 5.2 基础运动与导航任务分层

必须区分两类控制语义：

- **基础运动控制**
  - 适合点动、调试、手动微调
  - 使用速度、距离、角度等参数
- **导航任务控制**
  - 适合到站、顺序路线执行
  - 使用站点名、站点序列、任务号等参数

### 5.3 中台只做编排，不做底盘算法

中台负责：

- 命令参数校验
- 服务下发
- 状态汇聚
- 任务跟踪
- 异常回报

中台不负责：

- 地图求解
- 路径搜索
- 障碍避让算法
- 导航控制算法

这些能力仍由仙工控制器/下位机负责。

### 5.4 客户端面向任务，中台内部面向设备协议

对客户端暴露：

- 站点查询
- 单点到站
- 多站点顺序执行
- 导航进度

对中台内部实现：

- `SEERAGV_MOVE_POINT`
- `SEERAGV_MOVE_POINT_LIST`
- `SEERAGV_MOVE_PATH`
- `SEERAGV_CHECK_GUIDE`

## 6. 目标架构

### 6.1 增量后的总体结构

```text
Qt Client / UI
  -> CommunicationInterface
  -> CommunicationInterfaceImpl
  -> CommunicationWorker
  -> MfmsGatewayImpl
      -> MfmsDbService         (保留原有设备连接状态机)
      -> MfmsRosBridge         (增加 Seer 原始状态直通)
      -> MfmsCommandService    (增加 Seer 专属命令组)
  -> 下位机 vehicle 设备包
      -> Seer_linker_server
      -> Seer_linker_publisher
```

### 6.2 分层职责

| 层 | 模块 | 本次职责 |
|---|---|---|
| 客户端门面层 | `CommunicationInterface` | 向 Qt/客户端暴露仙工导航接口 |
| 编排层 | `CommunicationWorker` | 管理导航任务上下文、命令回执、进度跟踪 |
| 聚合层 | `MfmsGatewayImpl` | 继续聚合 DB / ROS / Command 三类服务 |
| 命令层 | `MfmsCommandService` | 封装仙工底盘基础运动、查询、导航命令 |
| 状态层 | `MfmsRosBridge` | 订阅并转发 Seer 原始状态消息 |
| 设备层 | `vehicle` 包 | 与仙工控制器通信并发布状态/服务 |

## 8. 模块改造设计

### 8.1 `com_interfaces` 改造

#### 8.1.1 改造目标

统一仙工底盘的 ROS 契约版本，为中台和下位机建立一致的协议。

#### 8.1.2 改造内容

1. 将 `src/com_interfaces/srv/SeerAgvCmdInterface.srv` 对齐至新版样例。
2. 将 `src/com_interfaces/msg/SeerAgvState.msg` 对齐至新版样例。
3. 保持包名、命名风格、放置位置遵循《向系统添加新设备指南》。

#### 8.1.3 结果

改造完成后，中台才能稳定拿到：

- 导航状态
- 当前目标站点
- 电量
- 告警

### 8.2 `MfmsCommandService` 改造

#### 8.2.1 改造目标

让中台真正具备仙工底盘命令下发能力。

#### 8.2.2 设计方案

保留现有 `RobotProxyAdapter` 和法奥机械臂控制逻辑不动；新增仙工底盘专属命令发送分支。

建议引入一个内部适配对象，逻辑上可以命名为：

- `SeerAgvCommandAdapter`

其职责是：

- 构造 `SeerAgvCmdInterface` 请求
- 管理对应 ROS Service client
- 统一处理错误码、超时和响应数据

#### 8.2.3 方法职责

查询方法：

- 负责将响应中的 `station_list/current_station/battery_level/task_status/task_type/target_id` 回传给 Worker。

基础运动方法：

- 负责向下位机发送 `MOVE_LINEAR/MOVE_TURN`。
- 不进入导航任务上下文。

导航方法：

- 负责向下位机发送 `MOVE_POINT/MOVE_POINT_LIST/MOVE_PATH`。
- 只做“任务受理”语义，不直接判断任务完成。
- 任务完成判断交由 Worker 的任务跟踪逻辑。

#### 8.2.4 命令结果语义

命令层只报告两类结果：

1. **命令受理结果**
   - 服务是否调用成功
   - 下位机是否接受该命令
2. **查询结果**
   - 当前导航状态
   - 当前站点
   - 站点列表

导航最终成功/失败不在命令层直接决定。

### 8.3 `MfmsRosBridge` 改造

#### 8.3.1 改造目标

保留现有统一状态桥接能力，同时补一条仙工原始状态链路。

#### 8.3.2 设计方案

保持当前订阅逻辑不变：

- `FrRobotState`
- `HsRobotState`
- `SeerAgvState`

在 `SeerAgvState` 分支上新增两条输出：

1. 原有输出
   - 继续转换成 `RobotRealtimeStatus`
   - 兼容老逻辑
2. 新增输出
   - 直接转发完整 `com_interfaces::msg::SeerAgvState`
   - 供 Worker 的导航任务上下文使用

#### 8.3.3 设计收益

- 旧 UI 可继续使用原有通道。
- 新导航能力可拿到完整状态字段。
- 不需要推翻现有 `RobotRealtimeStatus` 设计。

### 8.4 `CommunicationWorker` 改造

#### 8.4.1 改造目标

让 Worker 成为“仙工导航任务编排层”。

#### 8.4.2 新增上下文模型

建议 Worker 内部维护 `SeerNavigationContext`，字段建议如下：

| 字段 | 含义 |
|---|---|
| `deviceId` | 当前导航设备 |
| `taskId` | 中台生成的任务号 |
| `mode` | 单点/多站点/连续路径 |
| `stationList` | 目标站点列表 |
| `currentIndex` | 当前已完成到第几个站点 |
| `targetStation` | 当前目标站点 |
| `lastStation` | 最近确认到达的站点 |
| `taskStatus` | 当前导航状态 |
| `accepted` | 下位机是否受理 |
| `startTime` | 开始时间 |
| `finished` | 是否完成 |
| `failed` | 是否失败 |
| `errorCode` | 错误码 |
| `errorMessage` | 错误信息 |

#### 8.4.3 Worker 需要承担的职责

1. 校验客户端请求参数。
2. 生成并维护 `taskId`。
3. 调用 `MfmsCommandService` 发送仙工命令。
4. 接收命令回执。
5. 监听 `SeerAgvState` 原始状态。
6. 必要时轮询 `querySeerGuide()` 补充导航状态。
7. 将底层状态整合成客户端可理解的任务进度。

#### 8.4.4 到站判定策略

建议采用“双条件判定”：

1. 主条件：`CHECK_GUIDE` 返回任务完成。
2. 辅条件：`current_station == targetStation`。

设计原因：

- 只看 `current_station`，可能存在靠近站点但任务未真正结束的情况。
- 只看 `CHECK_GUIDE`，可能存在状态刷新延迟。

因此建议：

- 若两个条件都满足，立即判定完成。
- 若其中一个满足，进入短暂确认窗口。
- 确认窗口超时则按失败或待确认状态上报。

#### 8.4.5 多站点进度推进

对于 `MOVE_POINT_LIST` 模式：

- 当检测到 `current_station` 进入 `stationList[currentIndex]` 时，推进 `currentIndex`。
- 每推进一次，向客户端发出 `seerNavigationProgress`。
- 当 `currentIndex == stationList.size()` 且导航状态完成时，发 `seerNavigationFinished`。

#### 8.4.6 异常处理

导航过程中至少处理以下异常：

- 服务调用超时
- 下位机拒绝命令
- 站点不存在
- 当前状态未连接
- blocked 持续超时
- 告警级别进入 `fatal/error`
- 导航状态长期不变化
- 当前站点与目标站点长期不一致

### 8.5 `CommunicationInterface` 改造

#### 8.5.1 改造目标

为客户端提供明确、可落地的仙工底盘接口。

#### 8.5.2 设计要求

1. 所有仙工接口显式带 `deviceId`。
2. 不再允许“若当前设备为空则回退到第一个设备”的隐式行为。
3. 基础运动接口与导航接口分开。
4. 导航结果按任务回调，不混入原有基础运动结果信号。

#### 8.5.3 兼容策略

保留现有：

- `agvMoveForward`
- `agvMoveBackward`
- `agvTurnLeft`
- `agvTurnRight`

但新功能优先通过新增 Seer 专属接口接入。  
旧接口主要用于调试，不再作为导航主入口。

## 9. 关键业务流程

### 9.1 站点列表查询流程

```text
Client
  -> requestSeerStationList(deviceId)
  -> CommunicationWorker
  -> MfmsCommandService::querySeerStations(deviceId)
  -> Seer Service: SEERAGV_CHECK_STATION
  -> 返回 station_list
  -> Worker 校验/缓存
  -> seerStationListReceived(deviceId, stationIds)
```

### 9.2 单点到站流程

```text
Client
  -> requestSeerMoveToStation(deviceId, stationId)
  -> Worker 校验设备状态/站点合法性
  -> Worker 生成 taskId
  -> MfmsCommandService::navigateToStation(deviceId, stationId, taskId)
  -> Seer Service: SEERAGV_MOVE_POINT
  -> 命令受理成功
  -> seerNavigationAccepted(deviceId, taskId)
  -> Worker 开始跟踪:
       - 订阅 SeerAgvState
       - 必要时查询 SEERAGV_CHECK_GUIDE
  -> 到达目标站点
  -> seerNavigationFinished(deviceId, taskId)
```

### 9.3 多站点顺序执行流程

```text
Client
  -> requestSeerExecuteRoute(deviceId, [A, B, C, D])
  -> Worker 校验列表
  -> Worker 生成 taskId
  -> MfmsCommandService::navigateByStationList(deviceId, [A, B, C, D], taskId)
  -> Seer Service: SEERAGV_MOVE_POINT_LIST
  -> 命令受理成功
  -> Worker 持续跟踪:
       current_station = A -> progress 1/4
       current_station = B -> progress 2/4
       current_station = C -> progress 3/4
       current_station = D -> progress 4/4
  -> 导航完成
  -> seerNavigationFinished(deviceId, taskId)
```

### 9.4 暂停/继续/取消流程

```text
Client
  -> requestSeerPauseNavigation(deviceId)
  -> Worker
  -> MfmsCommandService::pauseSeerNavigation(deviceId)
  -> Seer Service: SEERAGV_GUDIE_PAUSE

Client
  -> requestSeerResumeNavigation(deviceId)
  -> MfmsCommandService::resumeSeerNavigation(deviceId)
  -> Seer Service: SEERAGV_GUDIE_CONT

Client
  -> requestSeerCancelNavigation(deviceId)
  -> MfmsCommandService::cancelSeerNavigation(deviceId)
  -> Seer Service: SEERAGV_GUDIE_CANCEL
  -> Worker 将任务置为 cancelled
```

## 10. 状态机设计

### 10.1 设备连接状态机

设备连接状态机保持原样，不做架构变更：

```text
offline -> load -> online -> connected
connected -> unload -> offline
```

仙工底盘接入必须继续遵守该机制，避免破坏现有中台生命周期管理。

### 10.2 导航任务状态机

新增导航任务状态机：

```text
idle
  -> validating
  -> accepted
  -> running
  -> paused
  -> running
  -> finished

异常分支:
accepted/running/paused -> failed
accepted/running/paused -> cancelled
```

状态说明：

- `idle`
  - 无任务
- `validating`
  - 校验设备、站点、连接状态
- `accepted`
  - 下位机已接受导航命令
- `running`
  - 导航进行中
- `paused`
  - 导航暂停
- `finished`
  - 正常完成
- `failed`
  - 异常失败
- `cancelled`
  - 客户端或系统取消

## 11. 任务号与幂等策略

### 11.1 任务号生成

建议 `taskId` 由中台生成，而不是客户端生成。  
生成策略可采用：

- 当前时间戳 + 本地递增序号
- 或 Worker 内部原子递增整型

### 11.2 幂等约束

同一设备同一时刻只允许一个活动导航任务。

规则建议如下：

- 若设备已有 `running/paused/accepted` 任务，则拒绝新导航任务。
- 基础运动命令与导航任务互斥。
- 客户端如需强制切换任务，应先发送取消。

## 12. 错误处理与超时策略

### 12.1 错误分类

建议统一分为：

1. 参数错误
   - 站点为空
   - 站点列表为空
   - 站点不在合法列表中
2. 状态错误
   - 设备未连接
   - 导航任务冲突
3. 通信错误
   - ROS service 超时
   - topic 断流
4. 设备错误
   - 下位机返回错误码
   - 告警级别 `error/fatal`
5. 业务超时
   - 导航长时间无进度
   - blocked 持续时间过长

### 12.2 超时建议

| 场景 | 建议超时 |
|---|---:|
| 查询类 service | 3~5 秒 |
| 导航命令受理 | 5 秒 |
| 导航状态无变化 | 15~30 秒 |
| 长时间 blocked | 30~60 秒 |

说明：

- 具体值以后续现场测试为准。
- 本阶段设计上只定义“存在超时与失败回报机制”。

## 13. 对客户端开发的建议接口模型

客户端不应直接理解命令号，而应按下面的业务对象使用中台。

### 13.1 设备对象

```text
SeerDeviceSummary
  - deviceId
  - name
  - ip
  - state
  - currentStation
  - batteryLevel
  - hasError
```

### 13.2 导航任务对象

```text
SeerNavigationTask
  - taskId
  - deviceId
  - mode
  - stationList
  - currentIndex
  - currentStation
  - targetStation
  - status
  - errorCode
  - errorMessage
```

### 13.3 客户端使用建议

1. 先查询站点列表。
2. 再发起单点或多点导航。
3. 以 `seerNavigationProgress` 作为进度刷新主来源。
4. 以 `seerNavigationFinished/Failed` 作为任务结束信号。

## 15. 验收标准

### 15.1 基础验收

- 设备可通过现有数据库状态机完成连接和断开。
- 客户端可查询仙工站点列表。
- 客户端可查询当前站点和电量。
- 客户端可接收完整仙工状态。

### 15.2 单点导航验收

- 客户端可指定一个合法站点。
- 中台可下发命令并收到受理结果。
- 中台可在导航过程中持续回报进度。
- 到站后可正确回报完成。
- 异常时可回报失败原因。

### 15.3 多站点导航验收

- 客户端可指定一个合法站点序列。
- 中台可顺序跟踪每个站点的到达情况。
- 可回报当前执行到第几个站点。
- 最终站点到达后可回报任务完成。

## 16. 风险与待确认事项

### 16.1 必须确认的协议问题

1. `SeerAgvCmdInterface.srv` 是否以根目录新版为准。
2. `SeerAgvState.msg` 是否以根目录新版为准。
3. 下位机 `CHECK_GUIDE` 的状态码语义是否固定。
4. `MOVE_POINT_LIST` 与 `MOVE_PATH` 的现场语义是否与注释一致。

### 16.2 必须确认的业务问题

1. 多站点任务一期是否只支持“每站停留”模式。
2. 是否要求路径中途可插入机械臂动作。
3. 是否允许导航与机械臂动作并发。
4. 是否需要任务持久化和恢复。

### 16.3 实施风险

- 接口契约不统一会导致中台与下位机编译或运行不一致。
- 旧 `CommunicationWorker` AGV 状态封装逻辑会吞掉关键字段。
- 若客户端继续使用旧 AGV 按钮接口，会绕过新任务接口。

## 17. 最终结论

本次改造的正确方向不是重做中台，而是：

1. 保留现有 MFMS 数据中台骨架和数据库驱动状态机。
2. 统一仙工底盘 `srv/msg` 契约版本。
3. 在 `MfmsCommandService` 中补齐仙工专属命令组。
4. 在 `MfmsRosBridge` 中补齐仙工原始状态直通。
5. 在 `CommunicationWorker` 中新增导航任务上下文与任务状态机。
6. 在 `CommunicationInterface` 中补齐面向客户端的站点/导航接口。

这样可以在不破坏原有法奥机械臂链路的前提下，把仙工底盘的基础运动、站点查询、单点导航、多站点顺序执行增量接入到现有中台中。
