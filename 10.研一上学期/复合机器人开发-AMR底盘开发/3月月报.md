# 周报

## 2. 总体结论

当前项目相较 `transplant_test`，不是简单修补，而是一次明显的工程化重构。核心变化是：

- 将旧项目中集中在 `src/mfms_server/mfms_core` 和 `src/mfms_server/qt_file` 内的能力拆分成多个独立 ROS 2 包。
- 将通信接口、命令服务、数据库、ROS bridge 从 Qt 客户端中解耦出来，形成更清晰的数据中台结构。
- 新增设计文档、集成说明、测试程序和 mock，用更接近“平台层 + 客户端层”的方式组织代码。
- 同时引入了 Blockly 可视化编程相关能力，并重做了 RViz/模型展示部分。

## 3. 目录与模块重构

### 3.1 旧项目到当前项目的主要映射关系

| 旧项目路径 | 当前项目路径 | 变化说明 |
| --- | --- | --- |
| `src/inspection_robot` | `src/myviz` | 可视化与模型展示包重命名并重构 |
| `src/mfms_server/com_interfaces` | `src/com_interfaces` | 通信接口包上移为独立 ROS 2 包 |
| `src/mfms_server/mfms_fr_adapter` | `src/mfms_fr_adapter` | FR 机械臂适配层上移为独立包 |
| `src/mfms_server/qt_file` | `src/qt_file` | Qt 客户端保留，但不再承载全部核心逻辑 |
| `src/mfms_server/mfms_core` | `src/mfms_server/cmd_service`、`src/mfms_server/interface`、`src/mfms_server/mfms_db`、`src/mfms_server/ros_bridge`、`src/mfms_server/common` | 单体核心库被拆分为多个职责单一的模块 |

### 3.2 顶层工作区结构变化

- 当前项目把 `com_interfaces`、`mfms_fr_adapter`、`qt_file`、`myviz` 都直接放到 `src/` 根下，更符合 ROS 2 workspace 的常见组织方式。
- 旧项目中多个包嵌套在 `src/mfms_server/` 下，当前项目进行了拆包和平铺，降低了包间路径耦合。
- 当前项目新增 `src/Blockly_test`，引入 Blockly 前端及自定义 block/generator 代码。
- 当前项目将 `MFMS_BASE.sql`、`MFMS_BASE_core.sql` 收敛到 `src/mfms_server/` 下集中管理。

## 4. 具体修改了什么

### 4.1 `mfms_core` 被拆成数据中台式分层

旧项目的 `mfms_core` 同时承担了数据库、ROS bridge、命令服务、gateway 等职责。当前项目将其拆分为：

- `cmd_service`
  负责命令编排、`RobotProxyAdapter`、`AgvProxyAdapter`
- `interface`
  负责 `CommunicationInterface`、`CommunicationInterfaceImpl`、`CommunicationWorker`
- `mfms_db`
  负责数据库模型与 `MfmsDbService`
- `ros_bridge`
  负责 ROS 消息桥接与设备状态转换
- `common`
  负责公共头文件

这意味着旧项目里“一个大库装下全部业务”的组织方式，被替换成“按职责拆包”的结构。

### 4.2 Qt 客户端从“带后端逻辑的 GUI”变成“依赖中台接口的 GUI”

旧项目 `src/mfms_server/qt_file/CMakeLists.txt` 直接依赖 `mfms_core`，并把下面这些通信层实现一起编进 GUI：

- `CommunicationInterface.h`
- `CommunicationInterfaceImpl.*`
- `CommunicationWorker.*`

当前项目中，这部分代码被迁移到 `src/mfms_server/interface/`，Qt 客户端改为依赖 `mfms_server_interface`。

这说明当前项目把通信接口从 UI 包中抽离出来了，GUI 不再直接承载通信线程和核心业务逻辑，层次更清楚。

### 4.3 命令服务从内嵌实现变成独立可测试模块

旧项目中 `MfmsCommandService`、`RobotProxyAdapter`、`AgvProxyAdapter` 都属于 `mfms_core` 的一部分。当前项目将其放入独立的 `cmd_service` 包，并新增：

- `tests/test_command_service.cpp`
- `tests/test_fr_command_mock.cpp`
- `tests/mock_fr_adapter_server.cpp`
- `INTEGRATION_GUIDE.md`
- `README.md`

此外，当前项目在 `cmd_service/CMakeLists.txt` 中对 `hyrms_export` 做了显式存在性检查，缺失时给出 warning，而不是默认强依赖。这比旧项目更利于在不同机器上构建和调试。

### 4.4 数据库与 ROS bridge 被独立出来

旧项目中数据库和 ROS bridge 都只是 `mfms_core` 的内部组成。当前项目把它们独立成：

- `src/mfms_server/mfms_db`
- `src/mfms_server/ros_bridge`

对应新增了：

- `test_db_service.cpp`
- `test_ros_bridge.cpp`
- `MfmsDbService_Design.md`
- `MfmsRosBridge_Design.md`
- `ROS_BRIDGE_INTEGRATION_TEST_REPORT.md`

这使数据库逻辑和 ROS 逻辑可以分别构建、分别测试、分别演进。

### 4.5 `com_interfaces` 从内嵌接口包变成公共接口包

旧项目的接口定义位于 `src/mfms_server/com_interfaces`。当前项目把它提升为 `src/com_interfaces`，角色更清晰，定位也更像跨模块共享接口。

同时存在以下差异：

- 当前项目保留了 FR、HS、Kossi、HK、Seer、VirtDev 等接口定义。
- 旧项目中的 `SiemensPlcState.msg` 与 `SiemensPlcCmdInterface.srv` 在当前项目中已删除。
- `ForceTorque.msg` 也从当前项目中移除。

这说明当前项目在接口层上做了收敛，优先保留当前数据中台所需的设备类型。

### 4.6 FR 机械臂适配层被独立并调整依赖布局

`mfms_fr_adapter` 已从 `src/mfms_server/` 下移到 `src/` 根目录，继续保留：

- `FrAdapterPublisher.cpp`
- `FrAdapterServer.cpp`
- `libfairino.so` 等供应商库

同时，当前项目把开发头文件布局切换为：

- `HyRMS_export_202601251449_bszydxh-HP/hyrms_export`
- `dev_export/robot`
- `dev_export/virt`

这比旧项目直接依赖 `HyRMS_export`、`dev_export.zip` 的方式更明确，也更方便本地解包后直接编译。

### 4.7 可视化展示包从 `inspection_robot` 演进为 `myviz`

当前项目将原有 `inspection_robot` 重构为 `myviz`，具体变化包括：

- 启动文件从 `realdisplay.launch.py` 变为 `display.launch.py`、`qtdisplay.launch.py`
- 机器人模型从 `InspectionRobot.urdf/.xacro` 切换为 `auboi16.urdf`
- mesh 资源整体替换，新增 `base.STL`
- `qt_file` 中的 RViz 集成类从 `MyRviz` 演进为 `Myviz`

这说明当前项目已经从“巡检机器人展示”转向“更贴合当前机械臂/系统集成场景”的展示模型。

### 4.8 Qt 客户端功能侧的增删

当前项目在 `qt_file` 中新增或强化了：

- `BlocklyTest.cpp/.h`
- `Myviz.cpp/.h`
- `VisualProgram.*` 的大幅修改
- `Control.*`、`SystemManagement.*`、`database_realSubject.*` 等多个核心页面/逻辑重写

同时移除了旧项目中的：

- `CommunicationInterface.*` 在 Qt 包内的实现
- `CommunicationWorker.*` 在 Qt 包内的实现
- `UpdateUiPool.*`
- `lidarView.*`
- `RvizEmbedWidget.*`
- 旧的 `MyRviz.*`
- `test_arm_control.py`、`run_arm_tests.bash`、`test_gateway.cpp`、`test_communication.cpp`

这说明当前客户端更聚焦业务 UI、可视化编程和中台调用，不再自己承载底层通信/网关实现。

### 4.9 文档体系明显增强

旧项目除少量说明文件外，设计资料较少。当前项目在 `src/mfms_server/design/` 下新增了完整文档体系：

- `MfmsCommandService_Design.md`
- `MfmsDbService_Design.md`
- `MfmsRosBridge_Design.md`
- `QUICK_START_FR_TEST.md`
- `ROS_BRIDGE_INTEGRATION_TEST_REPORT.md`
- `TEST_PLAN_FR_COMMAND.md`
- `数据中台全链路展示.md`

这部分是当前项目最明显的工程化提升之一。

## 5. 改进了什么

### 5.1 架构分层更清晰

- 旧项目中 `mfms_core` 过于集中，数据库、ROS、命令、网关逻辑都混在一个库里。
- 当前项目按职责拆成独立模块，更适合长期维护，也更符合“数据中台 + 客户端”的架构方向。

### 5.2 UI 与核心服务解耦

- 旧项目 `qt_file` 同时承担 UI、通信线程和业务入口。
- 当前项目将 `CommunicationInterface/Worker` 外提为独立接口层，Qt 只通过中台接口访问业务能力。

这降低了 GUI 改动对核心服务的影响范围。

### 5.3 可测试性增强

旧项目测试主要集中在 `qt_file/test`，而当前项目把测试拆到对应模块旁边，新增了：

- `cmd_service/tests/test_command_service.cpp`
- `cmd_service/tests/test_fr_command_mock.cpp`
- `cmd_service/tests/mock_fr_adapter_server.cpp`
- `interface/tests/test_agv_state_smoke.cpp`
- `mfms_db/test_db_service.cpp`
- `ros_bridge/test_ros_bridge.cpp`

相比旧项目，这种布局更利于按模块回归验证。

### 5.4 文档和交付物更完整

- 当前项目不仅有设计文档，还有集成指南、测试计划、联调报告。
- 对于新成员接手、联调排障、阶段性汇报都更友好。

### 5.5 构建与依赖管理更务实

- `cmd_service` 对 `hyrms_export` 做显式路径检查和降级处理。
- 多个包各自声明依赖，而不是把所有能力都绑在单一库上。
- 接口包、适配层、可视化包、Qt 客户端的边界比旧项目清楚。

## 6. 需要注意的差异

这些变化不一定是缺点，但在后续推进中需要持续确认：

- 旧项目存在 `mfms_seer_adapter`，当前项目未保留该独立包，Seer 能力更多体现在消息/服务定义和 `AgvProxyAdapter` 中，是否完全覆盖旧链路需要再核验。
- 旧项目中的 Siemens PLC 接口已删除，如果后续仍需 PLC 接入，需要重新补齐接口定义和适配层。
- 当前项目新增了 Blockly 相关目录和依赖，这带来了更强的可视化编程能力，但也增加了前端资源、Node 依赖和维护成本。
- 当前项目 `.gitignore` 相比旧项目更简化，后续需要留意是否会把不该入库的本地产物带进版本库。

## 7. 结论

当前项目相对 `transplant_test` 的核心价值，不在于“多了几个文件”，而在于完成了从“单体式业务工程”向“数据中台化、模块化、可测试化工程”的迁移：

- 核心服务被拆包，职责边界更清楚。
- Qt 客户端与中台服务解耦。
- FR 命令链、数据库、ROS bridge 都有更明确的独立实现和测试入口。
- 文档、测试、演示链路都比旧项目更完整。

如果从周报角度总结，这一版最主要的改进可以概括为三点：

- 完成数据中台核心能力拆分
- 完成 GUI 与核心服务解耦
- 完成测试与设计文档补齐
