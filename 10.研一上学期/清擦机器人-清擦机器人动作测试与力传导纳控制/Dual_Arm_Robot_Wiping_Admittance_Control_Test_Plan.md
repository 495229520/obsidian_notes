# 双臂机器人清擦动作测试与末端力传导纳控制方案

## 1. 文档目的

本文档用于为双臂机器人清擦动作测试构建一套可落地的实验方案，重点覆盖以下内容：

- 明确双臂机器人当前实际的数据采集链路；
- 基于 `【User manual】KWR75-RS422.pdf` 和现场沟通，梳理 `KWR75B` 六维力传感器的 `RS485 + Modbus RTU` 接入方式；
- 给出“通过控制箱 `RS485` 口读取力传数据”的具体实施步骤；
- 规划清擦动作测试、导纳控制验证和后续工程化落地路径。

本文不是采购文件，也不是最终控制软件设计说明，而是用于研发阶段验证“清擦动作 + 末端力感知 + 导纳控制”可行性的测试方案文档。

## 2. 当前项目边界与已知事实

根据当前任务资料、现场图片和 2026 年 3 月 31 日的沟通记录，当前方案按以下条件展开：

- 当前机器人平台为法奥机械臂系统。
- 六维力传感器型号固定为 `KWR75B`。
- 之前曾通过调用机械臂 SDK 获取装在机械臂上的力传数据。
- 当前工程师要求改为通过控制箱上的 `RS485` 口读取力传数据。
- 数据读取位置不再默认放在机械臂 SDK 内部，而是放到控制箱内电脑或其可访问的串口设备上。
- 当前已提供并重点参考的资料为：
  - [【User manual】KWR75-RS422.pdf](<【User manual】KWR75-RS422.pdf>)
  - [cx5100_en.pdf](<cx5100_en.pdf>)

> [!note]
> `cx5100_en.pdf` 在本文中只保留为“可选上位 PLC / IPC”参考，不再作为本轮力传接入的主链路依据。当前主链路以“法奥控制箱 `RS485` 口 + 控制箱内电脑读取”为准。

## 3. 关键结论

### 3.1 当前控制平台的角色

根据现场沟通，当前系统中应这样理解各模块职责：

- 法奥机械臂控制箱负责机械臂本体控制、驱动、基础 I/O 和外部设备串口接入。
- 控制箱内电脑或其可访问的上位计算单元负责打开串口、执行 `Modbus RTU` 主站轮询、解析六维力数据。
- 力传数据后续再提供给导纳控制、数据记录和测试程序使用。
- 机械臂 SDK 可以保留作对照或兼容方案，但不再是本轮主方案的数据入口。

### 3.2 `KWR75B` 传感器本体结论

根据已有手册整理，`KWR75B` 的关键特性如下：

- `KWR75` 系列是应变式六维力/力矩传感器；
- 可输出 `Fx`、`Fy`、`Fz`、`Mx`、`My`、`Mz` 六个分量；
- `KWR75B` 额定量程：
  - `Fx, Fy, Fz = 200 N`
  - `Mx, My, Mz = 8 Nm`
- 采样频率：`1 kHz`
- 分辨率：`24 bit`
- 精度：`0.1% FS`
- 准确度：`0.5% FS`
- 供电：标准版按 `9~24 VDC` 理解，现场优先按 `24 VDC` 配置；
- 通信接口系列可选：`RS422 / RS485 / ModBUS / CAN / Ethernet / EtherCAT / USB`。

### 3.3 当前主通信方案结论

本轮不再按 `CAN` 版本推进，而是按 `RS485 + Modbus RTU` 版本推进。

当前主方案为：

- `KWR75B-RS485`
- 通过控制箱上的 `RS485` 口接入
- 由控制箱内电脑读取串口
- 电脑侧以 `Modbus RTU` 主站身份轮询传感器
- 将读取出的六维力数据提供给导纳控制和测试程序

### 3.4 接线颜色结论

根据现场截图，`RS485` 版本的线色定义为：

- 蓝：电源 `+`
- 白：电源 `-`
- 粉：`485 A+`
- 棕：`485 B-`
- 屏蔽：屏蔽层
- 灰：`RS485` 通信产品不接
- 黑：`RS485` 通信产品不接

### 3.5 通信参数结论

根据现场截图中可见内容，当前至少可确认：

- 物理层：`RS485`
- 协议：`Modbus RTU`
- 默认波特率：`460800`

当前尚需在实施前再次核对的内容：

- 设备地址 `slave id`
- 寄存器起始地址
- 寄存器长度
- 数据类型是 `int`、`uint` 还是 `float`
- 字节序 / 字序
- 是否需要 CRC 以外的特殊校验设置

> [!warning]
> 当前文档还没有把 `RS485` 版本的完整寄存器表抄录出来。因此在真正编码前，必须回到 [【User manual】KWR75-RS422.pdf](<【User manual】KWR75-RS422.pdf>) 逐项确认 `Modbus` 寄存器定义。

## 4. 为什么不再走机械臂 SDK

### 4.1 之前方案的问题

之前通过机械臂 SDK 获取力传数据，虽然可以快速看到数据，但存在几个问题：

- 数据链路依赖机械臂厂家的软件封装；
- 不利于直接掌控串口参数、轮询频率和寄存器映射；
- 调试时不容易区分是“传感器问题”还是“SDK 转发问题”；
- 后续若要把力数据接入控制箱内部程序或独立控制链路，复用性不足。

### 4.2 当前方案的优点

改为控制箱 `RS485` 口后，链路更清晰：

- 传感器就是标准串口从站；
- 控制箱内电脑直接做 `Modbus RTU` 主站；
- 数据链路不再依赖机械臂 SDK 封装；
- 便于后续把数据直接接入导纳控制、日志记录和上层算法。

## 5. 推荐系统架构

### 5.1 主方案

本文主方案采用：

- 机械臂运动控制继续走法奥控制箱原有链路；
- 力传感器通过 `RS485 + Modbus RTU` 接入控制箱；
- 控制箱内电脑负责打开串口并轮询寄存器；
- 电脑端程序输出 `Fx, Fy, Fz, Mx, My, Mz`；
- 导纳控制外环优先放在电脑端验证；
- 清擦轨迹仍由原有机器人运动控制链路执行。

### 5.2 数据链路

```text
KWR75B-RS485
  -> 控制箱 24 V 供电
  -> 控制箱 RS485 A/B
  -> 控制箱内电脑串口设备
  -> Modbus RTU 主站程序
  -> 解析 Fx/Fy/Fz/Mx/My/Mz
  -> 导纳控制 / 数据记录 / 可视化
  -> 发送补偿后的运动命令给机器人控制程序
```

### 5.3 控制职责划分

- `KWR75B`
  - 负责输出 6 轴原始力/力矩数据
- 控制箱 `RS485` 口
  - 负责提供稳定的串口物理接入通道
- 控制箱内电脑
  - 负责作为 `Modbus RTU` 主站轮询设备
  - 负责时间戳、滤波、记录和对外发布
- 机器人控制程序
  - 负责轨迹执行
  - 负责把导纳补偿叠加到动作命令上

## 6. 具体怎么做

这一章对应工程师真正落地时的操作步骤。

### 6.1 第一步：先确认控制箱上的口真的是 `RS485`

在接线前，必须先确认控制箱上准备使用的端子是：

- `RS485 A/B`
- 或 `D+ / D-`
- 或 `485+ / 485-`

而不是：

- 普通 `DI`
- 普通 `DO`
- 模拟量输入
- 其他功能端子

如果端子定义不明确，必须先看控制箱电气图或法奥接口说明。不要把普通 I/O 端子误接成 `RS485`。

### 6.2 第二步：完成硬件接线

如果控制箱确实提供标准 `2 线半双工 RS485`，建议按下面接：

- 传感器蓝线 -> 控制箱 `24V+`
- 传感器白线 -> 控制箱 `24V- / GND`
- 传感器粉线 -> 控制箱 `RS485 A+ / D+`
- 传感器棕线 -> 控制箱 `RS485 B- / D-`
- 屏蔽层 -> 机柜保护地，建议单端接地
- 灰线 -> 不接
- 黑线 -> 不接

实施注意事项：

- `RS485` 通信线和供电线都要接，不能只接 `A/B` 两根线；
- `485` 口一般不负责给传感器供电；
- 若 `A/B` 接反，通常不会损坏设备，但会完全收不到数据；
- 若控制箱内 `RS485` 不带隔离，现场要特别注意地线和屏蔽处理；
- 若控制箱没有可直接被电脑访问的串口设备，可在控制箱内部增加一只工业级 `USB-to-RS485` 转换器作为过渡方案。

### 6.3 第三步：确认串口设备

控制箱内电脑必须能看到串口设备。

常见形式：

- Windows：`COM3`、`COM4` 等
- Linux：`/dev/ttyS*`、`/dev/ttyUSB*`、`/dev/ttyAMA*`

这一阶段要做的事情：

- 确认电脑是否识别到串口；
- 确认该串口就是控制箱 `RS485` 对应的设备；
- 做最简单的串口打开测试；
- 不要一开始就直接上控制算法。

### 6.4 第四步：配置串口与 `Modbus RTU`

建议先按以下参数起步：

- 波特率：`460800`
- 数据位：`8`
- 停止位：`1`
- 校验位：先按 `N` 理解，最终以手册为准
- 协议：`Modbus RTU`
- 超时：`20 ~ 50 ms`

还需要确认：

- 设备地址 `slave id`
- 功能码是读保持寄存器还是输入寄存器
- 六维数据对应的寄存器分布
- 是否存在状态寄存器或错误码寄存器

### 6.5 第五步：写一个独立的力传采集服务

建议不要把串口细节直接写死在主控制程序里，而是单独做一个“力传采集服务”。

推荐结构：

1. 打开串口
2. 轮询 `Modbus` 寄存器
3. 解析出 6 轴数据
4. 加时间戳
5. 做零点修正和必要滤波
6. 通过本地接口发布给导纳控制和可视化程序

对上层程序的输出形式可以选：

- 本地 TCP
- UDP
- 共享内存
- 本地消息队列
- 直接写入进程内共享对象

### 6.6 第六步：做最小闭环验证

在接入力控算法前，先做这些基础验证：

- 串口能稳定打开
- 设备能持续回应
- 6 个通道数值能正常变化
- 手推传感器时，力和力矩方向变化符合预期
- 空载零点稳定
- 连续运行 `10 min` 不掉线

## 7. 测试前准备

### 7.1 机械与安装准备

根据 `KWR75` 系列使用要求：

- 传感器安装面和过渡法兰接触必须均匀；
- 应按对角顺序逐步锁紧螺钉；
- 应使用定位销保证重复安装一致性；
- 工具法兰方向必须与后续控制建模的坐标定义一致；
- 上电后建议预热 `30 min` 再做零点和精度类测试。

实施要求：

- 工具端固定标准擦拭模块，例如海绵垫或百洁布板；
- 记录擦拭模块质量、质心、厚度和接触面尺寸；
- 明确传感器坐标系与工具坐标系的旋转关系；
- 为后续重力补偿建立工具质量模型。

### 7.2 电气与通信准备

- `KWR75B-RS485` 优先采用控制箱提供的稳定 `24 VDC` 辅助电源；
- 接线完成后先空载上电，不带控制算法验证链路；
- 串口先用 `460800` 配置尝试；
- 若无法通信，优先排查：
  - `A/B` 是否接反
  - 设备地址是否正确
  - 校验位是否正确
  - 控制箱口是否真是 `RS485`
- 若控制箱直接读取一时未打通，可临时用 `USB-to-RS485` 转换器在电脑上做台架验证，但这只作为调试手段，不作为最终量产接法。

### 7.3 控制与软件准备

- 控制箱内电脑上运行独立的力传采集进程；
- 力传采集、轨迹命令、接触状态、报警状态统一打时间戳；
- 导纳控制外环先在电脑端完成验证；
- 与机械臂运动程序的接口先保持简单清晰，优先传递：
  - `Fx, Fy, Fz, Mx, My, Mz`
  - 时间戳
  - 通信状态
  - 故障标志

## 8. 导纳控制实现细节

### 8.1 信号处理链

建议流程：

1. 读取原始 `Fx, Fy, Fz, Mx, My, Mz`
2. 做零点修正
3. 做工具重力补偿
4. 转换到工具坐标系或接触坐标系
5. 做一阶低通滤波
6. 进入导纳控制器
7. 输出位移或速度补偿
8. 与原始清擦轨迹叠加

### 8.2 推荐初始参数

以下为工程建议初值，不是手册给出的固定参数。

#### 法向导纳外环初值

- `F_ref`: `5 N -> 10 N -> 15 N -> 20 N` 分级测试
- `M_a`: `0.5 ~ 2.0 kg`
- `B_a`: `80 ~ 200 N*s/m`
- `K_a`: `1500 ~ 5000 N/m`

#### 姿态导纳初值

- 第二阶段再启用
- `Mx/My` 对应姿态微调
- 建议初值：
  - `J_a`: `0.01 ~ 0.05 kg*m^2`
  - `D_a`: `0.5 ~ 2.0 N*m*s/rad`
  - `K_theta`: `20 ~ 80 N*m/rad`

#### 采样与轮询建议

- 传感器内部采样：按 `1 kHz` 理解
- 串口侧初始轮询频率：先 `100 Hz`
- 稳定后再提高到 `200 Hz`
- 第一阶段不建议一上来就追求极限轮询频率
- 先以“稳定、可持续、可校验”为优先

### 8.3 安全阈值建议

虽然 `KWR75B` 额定量程较大，但清擦实验初期不要直接逼近额定量程。

建议首轮调试软限位：

- `|Fx|, |Fy|, |Fz| <= 30 N`
- `|Mx|, |My|, |Mz| <= 1.5 Nm`

建议实验急停阈值：

- 任一力分量 `> 50 N`
- 任一力矩分量 `> 3 Nm`
- 连续无新数据 `> 100 ms`
- 导纳补偿位移超出预设上限

## 9. 清擦动作测试流程

### 9.1 阶段 A：传感器链路单体验证

#### 目标

- 验证 `KWR75B-RS485` 供电、通信、零点和轮询周期是否稳定。

#### 步骤

1. 接通传感器供电，预热 `30 min`
2. 打开控制箱对应串口
3. 使用 `Modbus RTU` 读取设备在线寄存器或状态寄存器
4. 读取 6 轴数据寄存器
5. 记录空载 `10 min` 数据

#### 关注指标

- 串口是否稳定在线
- 是否有 CRC 错误或超时
- 6 个通道是否均为有效数值
- 零漂是否稳定
- 数据时间戳是否连续

#### 通过标准

- `10 min` 内无链路中断
- 无持续超时
- 无明显异常跳变
- 手动施力时各轴有合理响应

### 9.2 阶段 B：末端安装与静态标定

#### 目标

- 验证安装质量、坐标方向和工具重力补偿。

#### 步骤

1. 将传感器固定到机器人末端
2. 固定清擦工具
3. 按不同姿态采集空载值
4. 拟合工具重量和质心补偿项
5. 校验静态姿态下理论重力与测量结果是否一致

#### 通过标准

- 传感器坐标与工具坐标映射已明确
- 重力补偿后静态偏置显著降低

### 9.3 阶段 C：单轴法向接触测试

#### 目标

- 仅在垂直平面或水平平面上验证法向接触控制，不做擦拭。

#### 步骤

1. 使用平整刚性测试板
2. 低速接近表面
3. 依次测试 `5N / 10N / 15N / 20N`
4. 记录接触过程中的超调、稳定误差和振荡

#### 建议通过标准

- 峰值超调 < `30%`
- 稳态法向力误差 < `+-2 N`
- 无持续自激振荡

### 9.4 阶段 D：单臂直线清擦测试

#### 目标

- 在稳定法向力条件下完成沿直线的清擦轨迹。

#### 记录内容

- 位置轨迹
- 6 轴力/力矩
- 导纳补偿量
- 轨迹跟踪误差
- 清擦前后表面效果

### 9.5 阶段 E：表面扰动适应测试

#### 目标

- 验证导纳控制对表面不平、台阶和倾斜的适应能力。

#### 关注指标

- 力峰值是否受控
- 导纳补偿是否平稳
- 姿态力矩是否显著增大
- 是否发生擦拭跳动或拖拽

### 9.6 阶段 F：双臂协同清擦

#### 目标

- 在单臂测试稳定后，验证双臂协同动作。

#### 推荐策略

- 第一轮：
  - 一只手执行清擦
  - 另一只手只做工件扶持、拖线整理或邻域辅助
- 第二轮：
  - 两只手在不同区域同步清擦
- 暂不建议第一轮就做双臂共压同一区域

## 10. 推荐控制状态机

建议状态机如下：

1. `INIT`
   - 设备上电、自检、通信建立
2. `WARMUP`
   - 力传预热
3. `ZERO_CAL`
   - 零点采集、工具重力补偿
4. `APPROACH`
   - 低速接近待擦表面
5. `CONTACT_BUILD`
   - 建立目标法向接触力
6. `WIPE_RUN`
   - 执行切向清擦轨迹，法向导纳在线工作
7. `LIFT_OFF`
   - 平稳退出接触
8. `FAULT`
   - 超力、超矩、无新数据、位置异常、互锁触发时进入

## 11. 风险与对策

| 风险 | 原因 | 对策 |
| --- | --- | --- |
| 控制箱端子并不是真正的 `RS485` 口 | 端子定义识别错误 | 先核对控制箱电气图和端子定义，再接线 |
| 传感器有通信但无法稳定上电 | 只接了 `A/B` 没接供电 | 明确 `24 V+` 和 `24 V-` 单独供电 |
| 完全收不到数据 | `A/B` 接反、地址错误或串口参数不一致 | 先排查 `A/B`、地址、校验位、波特率 |
| 寄存器读通但数值不对 | 字节序、比例因子或数据类型理解错误 | 回到 `RS485 / Modbus` 手册核对寄存器定义 |
| 串口轮询导致数据时延过大 | 轮询频率过高或程序阻塞 | 先从 `100 Hz` 起步，稳定后再提高 |
| 力数据抖动导致擦拭发振 | 末端接触刚度高、滤波不足、导纳参数过硬 | 先做 1 自由度法向导纳，限制带宽，逐步调参 |
| 双臂互相干涉 | 轨迹规划未分区 | 单臂先验证，双臂阶段增加空间互锁 |
| 传感器输出不可信 | 安装面不平、锁紧不均、未预热 | 严格按手册安装，预热 30 分钟 |

## 12. 最终建议

### 12.1 本轮测试主方案

本轮建议采用以下实现路径：

- 机器人平台：法奥机械臂控制箱
- 力传：`KWR75B-RS485`
- 协议：`Modbus RTU`
- 接入方式：`KWR75B-RS485 -> 控制箱 RS485 口 -> 控制箱内电脑 -> Modbus 轮询 -> 六维力数据输出`
- 控制算法：先做 `Fz` 单轴导纳，再扩展到 `Fz + Mx/My`
- 测试顺序：链路验证 -> 静态标定 -> 单轴接触 -> 单臂清擦 -> 扰动适应 -> 双臂协同

### 12.2 推荐开发顺序

后续开发建议分四步推进：

1. 先打通 `RS485 + Modbus` 链路；
2. 再完成 6 轴数据解析、零点修正和时间戳记录；
3. 然后做单轴法向导纳闭环；
4. 最后扩展到姿态补偿和双臂协同。

### 12.3 后续更优的工程化方向

如果后续项目对实时性要求更高，或者希望把力传更紧地并入控制闭环，可进一步评估：

- 更稳定的控制箱内部串口服务化方案；
- 或直接评估 `EtherCAT` 版本力传感器。

但在当前阶段，最务实的方案仍然是：

- 先把 `RS485 + Modbus` 版本跑通；
- 把数据稳定读出来；
- 再基于现有机械臂控制链路做导纳控制验证。
