# IO 缓冲

## 问题引入

TCP [[7.6 套接字|套接字]] 的数据收发是**无边界**的。服务器端即使调用 1 次 `write()` 函数传输 40 字节的数据，客户端也可能通过 4 次 `read()` 函数调用每次读取 10 字节。但这也带来一些疑问：服务器端一次性传输了 40 字节，而客户端逐步分批接收。客户端接收 10 字节后，剩下的 30 字节在何处等待呢？是不是像飞机等待着陆而在空中盘旋一样，剩下 30 字节也在网络中徘徊并等待接收呢？

实际上，**`write()` 函数调用后并非立即传输数据，`read()` 函数调用后也并非马上接收数据**。更准确地说，如下图所示，`write()` 函数调用瞬间，数据将移至**输出缓冲**；`read()` 函数调用瞬间，从**输入缓冲**中读取数据。

---

## TCP 套接字 IO 缓冲

要理解 TCP 套接字 IO 缓冲，你需要把它想象成一个**临时存储区**，它存在于操作系统的内核中，紧邻着每个 TCP [[7.6 套接字|套接字]]。每个 [[7.5 TCP协议基础#TCP 的核心特性|全双工]] 的 TCP 连接都有两个这样的缓冲区：

### 发送缓冲区（Send Buffer）

当你调用 [[8.3 TCP客户端#send 函数|send()]] 或 `write()` 函数发送数据时，数据**并不是立即通过网络发送出去的**，而是先被复制到这个发送缓冲区中。

### 接收缓冲区（Receive Buffer）

当网络上接收到数据包时，这些数据会先被存放到这个接收缓冲区中，然后等待应用程序通过 [[8.3 TCP客户端#recv 函数|recv()]] 或 `read()` 函数来读取。

这种设计是 TCP [[7.5 TCP协议基础#TCP 的核心特性|全双工（full-duplex）]] 特性的体现，即数据可以同时双向流动。

### 缓冲区示意图

```
                    ┌─────────────────────────────────────┐
                    │            主机 A                    │
                    │  ┌─────────────┐  ┌─────────────┐   │
        read() ←────┼──│  输入缓冲    │  │  输出缓冲    │←──┼──── write()
                    │  │   (data)    │  │   (data)    │   │
                    │  └──────┬──────┘  └──────┬──────┘   │
                    └─────────┼────────────────┼──────────┘
                              │                │
                              ↓                ↓
                    ════════════════════════════════════════
                              网络传输通道
                    ════════════════════════════════════════
                              ↓                ↓
                    ┌─────────┼────────────────┼──────────┐
                    │  ┌──────┴──────┐  ┌──────┴──────┐   │
       write() ────→┼──│  输出缓冲    │  │  输入缓冲    │──→┼───→ read()
                    │  │   (data)    │  │   (data)    │   │
                    │  └─────────────┘  └─────────────┘   │
                    │            主机 B                    │
                    └─────────────────────────────────────┘
```

## IO 缓冲的特点

| 特点 | 说明 |
|:---|:---|
| **独立存在** | I/O 缓冲在每个 TCP 套接字中单独存在 |
| **自动创建** | I/O 缓冲在创建套接字时自动生成 |
| **输出缓冲继续发送** | 即使关闭套接字也会继续传递输出缓冲中遗留的数据 |
| **输入缓冲数据丢失** | 关闭套接字将丢失输入缓冲中的数据 |

> **注意**：关闭套接字时，输出缓冲中的数据会尽量发送完毕（优雅关闭），但输入缓冲中未读取的数据将被丢弃。

## 缓冲的作用

IO 缓冲的存在是为了解耦应用程序的读写速度和网络传输速度，从而提高效率和可靠性。它主要有以下几个作用：

### 1. 提高效率

如果没有缓冲区，每次应用程序调用 `send()` 发送一个字节，网络层就必须立即为这个字节构建一个 TCP/IP 包头，然后发送出去。这会导致大量的网络流量和系统开销。

有了缓冲区，应用程序可以一次性写入大量数据，而操作系统会根据网络状况、**MSS**（Maximum Segment Size）和拥塞控制等因素，智能地将缓冲区中的数据打包成一个个高效的 TCP 段（segment）再发送。

### 2. 流量控制（Flow Control）

接收方利用接收缓冲区来实现 [[8.8 流控制|流量控制]]。当接收方的接收缓冲区快要满时，它会在 TCP 报文的**窗口大小（Window Size）**字段中通知发送方减慢发送速度，甚至停止发送，直到它处理完数据并腾出空间。这有效地防止了发送方发送过快导致数据丢失。

### 3. 可靠性

TCP 的可靠传输机制很大程度上依赖于发送缓冲区。发送到缓冲区的数据，在收到对方的 ACK（确认）之前，都会一直保存在发送缓冲区中。如果超时没有收到 ACK，TCP 会认为数据包丢失，然后从缓冲区中重传这些数据。

## 缓冲区大小

### 查看缓冲区大小

在 Linux 系统中，可以通过以下命令查看和设置缓冲区大小：

```bash
# 查看默认的发送缓冲区大小
cat /proc/sys/net/core/wmem_default

# 查看默认的接收缓冲区大小
cat /proc/sys/net/core/rmem_default

# 查看最大的发送缓冲区大小
cat /proc/sys/net/core/wmem_max

# 查看最大的接收缓冲区大小
cat /proc/sys/net/core/rmem_max
```

### 通过代码设置缓冲区大小

可以使用 [[10.1 套接字选项#setsockopt 函数|setsockopt()]] 函数设置缓冲区大小：

```c
#include <sys/socket.h>

int sockfd = socket(AF_INET, SOCK_STREAM, 0);

// 设置发送缓冲区大小
int send_buf_size = 65536;  // 64KB
setsockopt(sockfd, SOL_SOCKET, SO_SNDBUF, &send_buf_size, sizeof(send_buf_size));

// 设置接收缓冲区大小
int recv_buf_size = 65536;  // 64KB
setsockopt(sockfd, SOL_SOCKET, SO_RCVBUF, &recv_buf_size, sizeof(recv_buf_size));

// 获取实际的缓冲区大小
int actual_size;
socklen_t len = sizeof(actual_size);
getsockopt(sockfd, SOL_SOCKET, SO_SNDBUF, &actual_size, &len);
printf("实际发送缓冲区大小: %d\n", actual_size);
```

> **注意**：Linux 内核通常会将设置的值翻倍，以留出空间给内核数据结构。详见 [[10.1 套接字选项#SO_SNDBUF / SO_RCVBUF]]。

## 关键要点

1. **缓冲区位置**：IO 缓冲存在于操作系统内核中，每个 TCP 套接字独立拥有
2. **write/send 不等于发送**：数据先进入输出缓冲区，由内核决定何时发送
3. **read/recv 从缓冲区读取**：数据先进入输入缓冲区，再被应用程序读取
4. **关闭套接字的影响**：
   - 输出缓冲：尽量发送完毕
   - 输入缓冲：数据丢失
5. **缓冲区的作用**：提高效率、流量控制、保证可靠性
