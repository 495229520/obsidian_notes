# 流控制

## 简介

流控制（Flow Control）是一种通信协议机制，它的核心目的是**防止发送方发送数据的速度过快，导致接收方来不及处理而丢包**。

想象一下你和一个朋友在互相扔球，你的朋友接球的速度比你扔球的速度慢。如果你一直不停地扔，最终他会因为接不住而让球掉到地上。流控制就是为了避免这种"球掉到地上"的情况发生。

在计算机网络中，发送方是生产者，接收方是消费者。流控制确保了**消费者的处理能力能够跟上生产者的发送速度**。

TCP 实现了自己的流控制机制，而 UDP 则没有。

---

## TCP 流控制

TCP 的流控制是通过**滑动窗口协议（Sliding Window Protocol）**实现的。

### 工作原理

接收方在每个 ACK（确认）报文中都包含一个**窗口大小（Window Size）**字段。这个字段告诉发送方，它当前还有多少可用的 [[8.7 IO缓冲#接收缓冲区（Receive Buffer）|接收缓冲区]] 空间。

### 实现过程

```
     发送方                                        接收方
        │                                            │
        │                                   [缓冲区空闲]
        │   ────────→ 数据 (1000字节) ────────→       │
        │                                   [处理数据中...]
        │   ←──── ACK, Window=8000 ←────────         │
        │   (可以继续发送 8000 字节)                   │
        │                                            │
        │   ────────→ 数据 (5000字节) ────────→       │
        │                                   [缓冲区快满]
        │   ←──── ACK, Window=3000 ←────────         │
        │   (只能再发送 3000 字节)                     │
        │                                            │
        │   ────────→ 数据 (3000字节) ────────→       │
        │                                   [缓冲区已满]
        │   ←──── ACK, Window=0 ←───────────         │
        │   (暂停发送！)                              │
        │                                            │
        │   [等待...]                                 │
        │                                   [处理完数据，腾出空间]
        │   ←──── ACK, Window=4000 ←────────         │
        │   (可以继续发送了)                           │
        │                                            │
```

### 滑动窗口详解

| 步骤 | 描述 |
|:---:|:---|
| 1 | 发送方收到 ACK 后，会读取其中的窗口大小 |
| 2 | 发送方只能发送窗口大小范围内的数据 |
| 3 | 如果接收方的缓冲区满了，它会把窗口大小设置为零，告诉发送方暂停发送 |
| 4 | 当接收方处理完数据，腾出空间后，它会发送一个带非零窗口大小的 ACK，告诉发送方可以继续发送 |

通过这种动态调整窗口大小的方式，TCP 能够确保发送方的发送速率永远不会超过接收方的处理速率，从而避免了丢包，保证了连接的 [[7.5 TCP协议基础#TCP 可靠性机制|可靠性]]。

### 零窗口探测

当接收方发送 `Window=0` 后，发送方会进入等待状态。为了防止死锁（接收方的窗口更新 ACK 丢失），发送方会定期发送**零窗口探测报文（Zero Window Probe）**，询问接收方是否有可用空间。

```c
// 零窗口探测的伪代码逻辑
while (window_size == 0) {
    sleep(probe_interval);  // 等待一段时间
    send_zero_window_probe();  // 发送探测报文
    // 接收方会回复当前的窗口大小
}
```

## UDP 流控制

**UDP 本身不实现流控制。**

UDP 的设计哲学是**牺牲可靠性来换取速度**。它只管将数据包发出去，根本不关心接收方是否能处理过来。正因为如此，UDP 的数据报到达目的地时可能会因为接收方缓冲区溢出而被丢弃。

### 应用层实现流控制

如果你需要对 UDP 实现流控制，你必须**在应用层自己实现**。这通常意味着：

```
┌─────────────────────────────────────────────────────────┐
│                    应用层流控制                           │
├─────────────────────────────────────────────────────────┤
│  1. 在发送的数据包中加入序列号                              │
│  2. 接收方在收到数据包后，发送应用层的确认包（ACK）给发送方     │
│  3. 如果发送方在超时时间内没有收到确认，就认为数据包丢失，      │
│     需要进行重传                                          │
│  4. 接收方还可以通过确认包告诉发送方，自己还有多少处理能力，    │
│     从而实现类似 TCP 的流控制                              │
└─────────────────────────────────────────────────────────┘
```

但这部分逻辑完全由程序员自己编写，而不是由 UDP 协议本身提供。

### 常见的应用层协议

一些基于 UDP 的应用层协议实现了自己的流控制和可靠性机制：

| 协议 | 说明 |
|:---|:---|
| **QUIC** | Google 开发，HTTP/3 的底层协议，实现了可靠传输和流控制 |
| **RTP/RTCP** | 用于实时音视频传输，RTCP 提供反馈机制 |
| **TFTP** | 简单文件传输协议，使用停等协议实现可靠传输 |

## TCP 与 UDP 流控制对比

| 特性 | TCP | UDP |
|:---|:---|:---|
| **流控制** | 内置（滑动窗口） | 无 |
| **实现层** | 传输层 | 需应用层自行实现 |
| **窗口通告** | 通过 ACK 报文中的 Window 字段 | 不适用 |
| **零窗口处理** | 自动探测 | 不适用 |
| **适用场景** | 需要可靠传输的场景 | 实时性要求高的场景 |

## 拥塞控制 vs 流控制

流控制容易与**拥塞控制（Congestion Control）**混淆，但它们解决的是不同的问题：

| 机制 | 目的 | 关注点 |
|:---|:---|:---|
| **流控制** | 防止发送方淹没接收方 | 端到端（接收方的处理能力） |
| **拥塞控制** | 防止发送方淹没网络 | 网络路径（路由器、交换机的处理能力） |

TCP 同时实现了这两种机制：
- **流控制**：通过滑动窗口（接收方通告的窗口大小）
- **拥塞控制**：通过拥塞窗口（cwnd）和算法（慢启动、拥塞避免、快重传、快恢复）

实际发送窗口 = min(接收方窗口, 拥塞窗口)

---

## 关键要点

1. **流控制的目的**：防止发送方发送过快，导致接收方来不及处理
2. **TCP 流控制**：通过滑动窗口协议实现，接收方在 ACK 中通告窗口大小
3. **零窗口**：当接收方缓冲区满时，窗口设为 0，发送方暂停发送
4. **UDP 无流控制**：需要在应用层自行实现
5. **流控制 ≠ 拥塞控制**：流控制关注端到端，拥塞控制关注网络路径
