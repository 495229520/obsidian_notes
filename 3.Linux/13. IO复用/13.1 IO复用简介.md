# IO 复用简介

## 1. Linux "一切皆文件" 的理念

在 Linux 系统中，"一切皆文件" 是一个核心设计理念。这意味着几乎所有的资源都可以通过统一的[[4.1 打开、读取、写入、关闭#文件描述符（File Descriptor）|文件描述符]]接口来访问：

| 资源类型 | 示例 |
|:---|:---|
| 普通文件 | 文本文件、二进制文件 |
| 设备文件 | `/dev/null`、`/dev/sda` |
| [[11.3 管道通讯\|管道]] | 进程间通信 |
| [[7.6 套接字\|套接字（Socket）]] | 网络通信 |

这种统一的抽象使得我们可以用相同的 `read()`/`write()` 接口操作不同类型的资源。

## 2. 五种 I/O 模型

在网络编程中，I/O 操作（特别是网络数据的读写）是最核心的部分。根据数据准备和数据复制两个阶段的处理方式，Unix/Linux 系统提供了五种 I/O 模型：

### 2.1 阻塞 I/O（Blocking I/O）

```
应用进程                         内核
    │                             │
    │──── recvfrom() ────────────►│
    │                             │ 等待数据准备
    │        (进程阻塞)            │ ...
    │                             │ 数据准备完成
    │                             │ 复制数据到用户空间
    │◄─────── 返回数据 ───────────│
    │                             │
```

**特点**：
- 调用 I/O 函数后，进程被阻塞，直到数据准备完成并复制到用户空间
- **最简单**的 I/O 模型，编程复杂度低
- 单进程/线程只能同时处理一个连接
- 典型应用：简单的单客户端程序

### 2.2 非阻塞 I/O（Non-blocking I/O）

```
应用进程                         内核
    │                             │
    │──── recvfrom() ────────────►│ 数据未准备好
    │◄─────── EWOULDBLOCK ────────│
    │                             │
    │──── recvfrom() ────────────►│ 数据未准备好
    │◄─────── EWOULDBLOCK ────────│
    │         (轮询)               │
    │──── recvfrom() ────────────►│ 数据准备完成
    │                             │ 复制数据到用户空间
    │◄─────── 返回数据 ─────────── │
```

**特点**：
- 调用 I/O 函数后立即返回，如果数据未准备好返回错误码 `EAGAIN`/`EWOULDBLOCK`
- 需要应用程序**不断轮询**检查数据是否准备好
- **CPU 占用率高**（忙等待）
- 可通过 `fcntl()` 设置 `O_NONBLOCK` 标志启用

```c
// 设置非阻塞模式
int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);
```

### 2.3 I/O 复用（I/O Multiplexing）

```
应用进程                         内核
    │                             │
    │──── select()/poll() ───────►│
    │                             │ 监控多个fd
    │        (进程阻塞)            │ ...
    │                             │ 某个fd数据准备好
    │◄─────── 返回就绪fd ──────────│
    │                             │
    │──── recvfrom() ────────────►│
    │                             │ 复制数据到用户空间
    │◄─────── 返回数据 ───────────│
```

**特点**：
- 单个进程/线程可以**同时监控多个文件描述符**
- 只有当至少一个 fd 就绪时才进行实际的 I/O 操作
- 常用接口：`select()`、`poll()`、`epoll()`
- **本章重点讨论的模型**

### 2.4 信号驱动 I/O（Signal-driven I/O）

```
应用进程                         内核
    │                             │
    │──── sigaction(SIGIO) ──────►│ 注册信号处理
    │◄─────── 立即返回 ────────────│
    │                             │
    │   (进程可做其他事情)           │ 等待数据准备
    │                             │ ...
    │◄─────── 发送 SIGIO ──────────│ 数据准备完成
    │                             │
    │──── recvfrom() ────────────►│
    │                             │ 复制数据到用户空间
    │◄─────── 返回数据 ───────────│
```

**特点**：
- 通过信号机制通知应用程序数据准备就绪
- 不需要轮询，减少 CPU 占用
- 信号处理的复杂性较高，使用较少

### 2.5 异步 I/O（Asynchronous I/O）

```
应用进程                         内核
    │                             │
    │──── aio_read() ────────────►│
    │◄─────── 立即返回 ────────────│
    │                             │
    │   (进程可做其他事情)          │ 等待数据准备
    │                             │ ...
    │                             │ 数据准备完成
    │                             │ 复制数据到用户空间
    │◄─────── 信号/回调通知 ───────│ 全部完成
```

**特点**：
- 真正的异步：数据准备和复制都由内核完成
- 应用程序只需在操作完成后处理结果
- Linux 原生支持较弱，可使用 `io_uring`（Linux 5.1+）或 `libaio`

### 2.6 五种模型对比

| 模型 | 等待数据阶段 | 复制数据阶段 | 特点 |
|:---|:---|:---|:---|
| 阻塞 I/O | 阻塞 | 阻塞 | 最简单 |
| 非阻塞 I/O | 非阻塞（轮询） | 阻塞 | CPU 占用高 |
| **I/O 复用** | 阻塞（在 select/poll/epoll 上） | 阻塞 | 可监控多 fd |
| 信号驱动 | 非阻塞 | 阻塞 | 使用复杂 |
| 异步 I/O | 非阻塞 | 非阻塞 | 真正异步 |

> **注意**：前四种模型在数据复制阶段都是阻塞的，只有异步 I/O 是真正的全程非阻塞。

## 3. I/O 复用详解

### 3.1 什么是 I/O 复用

**I/O 复用**（I/O Multiplexing）是指单个进程/线程可以同时监控多个文件描述符的 I/O 状态，当其中任意一个或多个文件描述符就绪（可读、可写或异常）时，能够得到通知并进行相应处理。

### 3.2 为什么需要 I/O 复用

考虑一个需要同时处理多个客户端连接的服务器：

**不使用 I/O 复用的方案**：
- [[11.8 基于多任务的多进程并发服务器|多进程]]：每个连接 fork 一个子进程处理，资源开销大
- [[12.6 多线程并发服务器案例|多线程]]：每个连接创建一个线程，线程数量受限

**使用 I/O 复用的方案**：
- 单进程/线程同时监控所有连接
- 只处理有数据到达的连接
- 资源利用率高，可处理大量并发连接

### 3.3 I/O 复用的工作流程

```
                    ┌─────────────────────────────────┐
                    │         设置文件描述符集合         │
                    │    (需要监控的 socket fd)         │
                    └───────────────┬─────────────────┘
                                    │
                                    ▼
                    ┌─────────────────────────────────┐
                    │         设置监控事件             │
                    │   (可读/可写/异常)               │
                    └───────────────┬─────────────────┘
                                    │
                                    ▼
                    ┌─────────────────────────────────┐
                    │      调用 select/poll/epoll     │
                    │        阻塞等待事件就绪           │
                    └───────────────┬─────────────────┘
                                    │
                                    ▼
                    ┌─────────────────────────────────┐
                    │         遍历就绪的 fd            │
                    │       进行相应的 I/O 操作        │
                    └───────────────┬─────────────────┘
                                    │
                                    ▼
                                  循环
```

### 3.4 Linux 提供的 I/O 复用机制

Linux 系统提供了三种 I/O 复用机制：

| 机制 | 特点 | 详细说明 |
|:---|:---|:---|
| **select** | 最早的实现，跨平台性好 | 见 [[13.2 select模型]] |
| **poll** | 改进 select 的限制 | 见 [[13.3 poll模型]] |
| **epoll** | Linux 特有，性能最优 | 见 [[13.4 epoll模型]] |

---

## 4. I/O 复用的应用场景

1. **高并发服务器**：如 Web 服务器（Nginx）、数据库（Redis）
2. **网络代理**：需要同时处理多个连接的转发
3. **聊天服务器**：同时管理大量用户连接
4. **实时监控系统**：监控多个数据源

---

## 5. 关键要点

1. **五种 I/O 模型**：阻塞、非阻塞、I/O复用、信号驱动、异步 I/O
2. **I/O 复用的核心价值**：单进程/线程高效处理多连接
3. **Linux I/O 复用接口**：select、poll、epoll，其中 epoll 性能最优
4. **适用场景**：高并发、多连接的网络服务
