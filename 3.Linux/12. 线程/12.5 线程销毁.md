# 线程销毁

线程的生命周期管理是多线程编程中的重要环节。与 [[12.2 线程的创建与运行|线程的创建]] 相对应，线程销毁有多种方式，每种方式都有其适用场景和注意事项。

---

## 自然退出

这是最推荐、最安全的方式。当线程的入口函数（你用 `pthread_create` 传入的函数）执行完毕并返回时，线程就会自然终止。

### 实现方式

- **使用 return 语句**：在线程函数内部，你可以使用 `return` 语句或 `pthread_exit()` 函数来结束线程
- **pthread_exit()**：这个函数允许你指定一个返回值，其他线程可以通过 `pthread_join()` 来获取这个值

```c
#include <pthread.h>

void *thread_routine(void *arg) {
    // 线程任务 ...
    pthread_exit(NULL);  // 线程在此处退出
}
```

### 优点

这种方式能确保所有线程内部的资源（如局部变量、[[1.2 函数栈帧建立与销毁原理|栈空间]]）都得到正确释放，避免了[[6.3 动态内存分配的风险|内存泄漏]]。

---

## 强制终止

这是一种不推荐的方式，因为它非常危险，可能会导致不可预测的后果。你可以在一个线程中，强制终止另一个线程的执行。

### 函数

```c
#include <pthread.h>

int pthread_cancel(pthread_t thread);
```

### pthread_cancel()

这个函数会向指定线程发送一个终止信号。然而，被终止的线程并非立即停止，它需要在代码中设置**取消点（cancellation point）**。这些取消点通常是阻塞的系统调用，如 `read()`、`write()`、`sleep()` 等。当线程执行到取消点时，它才会检查是否有挂起的终止信号并退出。

### 危险性

1. **资源泄漏**：如果线程在持有锁、分配了内存或打开了文件的情况下被强制终止，这些资源可能永远不会被释放，导致**死锁**或[[6.3 动态内存分配的风险|内存泄漏]]
2. **数据不一致**：线程可能在修改共享数据的过程中被终止，使得数据处于一个不完整或不一致的状态，引发严重的数据错误

因此，除非你非常清楚你在做什么，并且已经为强制终止设计了完善的清理机制，否则应该避免使用此方法。

---

## 分离（Detach）

这是另一种管理线程生命周期的方式，它主要用于**自动回收资源**，而不是手动终止线程。

### 线程的两种状态

一个线程在被创建后，有两种状态：

- **可汇合（Joinable）**：这是默认状态。当线程结束时，它的资源不会被立即释放，而是等待其他线程调用 `pthread_join()` 来回收。如果忘记调用 `pthread_join()`，就会造成**僵尸线程**，占用系统资源
- **分离（Detached）**：在这种状态下，线程在结束时，其资源会自动被系统回收，不需要任何线程来调用 `pthread_join()`

### 函数

```c
#include <pthread.h>

int pthread_detach(pthread_t thread);
```

### pthread_detach()

可以在线程创建后立即调用此函数来将其设置为分离状态，或者在创建时通过属性（`pthread_attr_t`）来设置。

```c
#include <pthread.h>

void *background_task(void *arg) {
    // 后台任务 ...
    return NULL;
}

int main() {
    pthread_t thread;
    pthread_create(&thread, NULL, background_task, NULL);

    // 将线程设置为分离状态
    pthread_detach(thread);

    // 主线程继续执行其他任务
    // 分离线程结束后会自动回收资源

    return 0;
}
```

> **注意**：分离后的线程无法再被 `pthread_join()`，也无法获取其返回值。

---

## 三种方式对比

| 方法 | 核心函数 | 优点 | 缺点 | 适用场景 |
|:---|:---|:---|:---|:---|
| **自然退出** | `return` / `pthread_exit()` | 安全、资源清理完善 | 无法从外部强制停止 | 大多数常规任务 |
| **强制终止** | `pthread_cancel()` | 可以从外部立即停止线程 | 极易导致资源泄漏和死锁 | 紧急情况，如用户强制关闭程序 |
| **分离** | `pthread_detach()` | 自动回收资源，无须 `pthread_join()` | 无法获取线程返回值 | 后台运行的、不依赖主线程的独立任务 |

---

## 与 C++ std::thread 的对比

在 C++11 中，`std::thread` 提供了类似的线程生命周期管理机制，详见 [[15.2 thread|std::thread]]：

| POSIX pthread | C++ std::thread | 说明 |
|:---|:---|:---|
| `pthread_join()` | `thread.join()` | 等待线程结束并回收资源 |
| `pthread_detach()` | `thread.detach()` | 分离线程，自动回收资源 |
| `pthread_exit()` | 函数返回 | 线程自然退出 |
| `pthread_cancel()` | 无直接对应 | C++ 标准库不提供强制终止 |

---

## 关键要点

1. **首选自然退出**：让线程函数正常返回是最安全的方式

2. **谨慎使用 pthread_cancel()**：
   - 只在紧急情况下使用
   - 必须考虑资源清理问题
   - 需要正确设置取消点

3. **分离适用于独立任务**：
   - 后台任务
   - 不需要返回值的任务
   - 生命周期独立于主线程的任务

4. **避免僵尸线程**：对于 Joinable 状态的线程，必须调用 `pthread_join()` 回收资源
