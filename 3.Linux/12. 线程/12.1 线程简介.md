# 12.1 线程简介

## 1. 什么是线程

> **线程是操作系统调度的最小单位**，一个[[11.1 进程简介|进程]]最少有一个线程。
>
> 线程 (Thread) 是在[[11.1 进程简介|进程]]中产生的一个执行单元，是操作系统能够进行调度的最小单位。在同一个进程中与多个线程并行运行，共享[[1.3 进程的虚拟地址空间|进程内的资源]]，比如内存、地址空间、打开的文件等。

你可以把一个进程比作一个公司，而这个公司的每个员工就是一个线程。一个公司可以只有一个员工（单线程），也可以有很多员工一起工作（多线程）。

---

## 2. 线程与进程的关系

| 特性 | 进程 | 线程 |
|------|------|------|
| **资源共享** | 同一个进程中的所有线程共享该进程的[[1.3 进程的虚拟地址空间|内存地址空间]]、文件描述符、全局变量、打开的文件、信号等资源。这就好比一个公司里的所有员工都使用同一栋办公楼和同一台打印机 | 不同进程拥有独立的地址空间 |
| **独立执行** | 尽管线程共享资源，但每个线程都有自己独立的执行路径。每个线程都有自己的**程序计数器**、**栈**、**局部变量**和**寄存器**，这就像公司的每个员工都有自己的工作任务和待办清单 | 进程之间完全独立 |
| **轻量级** | 线程比进程更"轻"。创建、终止和切换线程的开销远小于创建进程的开销，这是因为线程不需要像进程那样为新的地址空间分配资源 | 进程切换开销更大 |

### 2.1 线程独有的资源

每个线程拥有自己独立的：

- **程序计数器 (Program Counter)**：指示线程下一条要执行的指令的地址
- **寄存器 (Registers)**：CPU 寄存器中的值，包括通用寄存器、栈指针等
- **栈 (Stack)**：[[1.3 进程的虚拟地址空间|线程的栈空间]]，用于存储局部变量和函数调用信息
- **其他状态信息**：如内存管理信息、I/O 状态等，这些信息会被存储在**线程控制块 (Thread Control Block, TCB)** 中，以便在将来恢复该线程的执行

### 2.2 进程与线程的比较

```
┌─────────────────────────────────────────────────┐
│                    进程                          │
│  ┌────────────────────────────────────────────┐ │
│  │          共享资源（堆、全局变量、文件等）     │ │
│  └────────────────────────────────────────────┘ │
│                                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │  线程 1   │  │  线程 2   │  │  线程 3   │      │
│  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │      │
│  │ │ 栈   │ │  │ │ 栈   │ │  │ │ 栈   │ │      │

```

## 3. 线程的上下文切换

**线程的上下文切换 (Context Switch)** 是指操作系统（内核）将 CPU 的执行权从一个线程（或进程）转移到另一个线程（或进程）的过程。这个过程是实现多任务和并发的核心机制。

在单核 CPU 上，CPU 在任何给定的时刻只能执行一个线程。为了让多个线程看起来像是在"同时"运行，操作系统会为每个线程分配一个短暂的 CPU 时间片。当一个线程的时间片用完，或者它因为某些原因（如等待 I/O、等待互斥锁）而无法继续执行时，内核就会介入，暂停当前线程的执行，并切换到另一个就绪的线程。

### 3.1 切换过程的三个阶段

线程的上下文切换通常涉及以下三个步骤：

| 阶段 | 操作 | 说明 |
|------|------|------|
| **1. 保存 (Save)** | 内核会保存当前正在运行的线程的**上下文信息** | 这些信息包括程序计数器、寄存器值、栈指针等，会被存储在该线程的 TCB 中 |
| **2. 调度 (Schedule)** | 操作系统内核的调度器会运行，根据某种调度算法（例如时间片轮转、优先级调度）选择下一个要执行的线程 | 选择最适合当前运行的线程 |
| **3. 恢复 (Restore)** | 内核会从新选择的线程的 TCB 中加载其保存的上下文信息 | 一旦上下文信息被加载到 CPU 的寄存器中，新线程就可以从它上次被中断的地方继续执行，就好像它从未停止过一样 |

### 3.2 线程与进程上下文切换的区别

| 对比项 | 线程上下文切换 | 进程上下文切换 |
|--------|----------------|----------------|
| **开销更小** | 线程的上下文切换比进程的上下文切换开销小得多 | 需要切换整个地址空间 |
| **共享资源** | 线程属于同一个进程，它们共享相同的地址空间。因此，在进行线程上下文切换时，**内核不需要切换内存地址空间**（如页表、虚拟内存映射），这大大减少了开销 | 需要切换页表等 |
| **独占资源** | 进程上下文切换除了要保存和恢复线程的所有信息外，还需要切换整个**虚拟内存空间**。这涉及到更新 CPU 的页表寄存器，这个操作的开销是很大的 | 全部资源都需要切换 |

## 4. 使用多线程的作用

使用多线程的主要目的是为了提高程序的**并发性**和**响应性**。

### 4.1 提高并发性 (Concurrency) 和并行性 (Parallelism)

- **并发性**：指的是程序在同一时间段内能够处理多个任务的能力。即使在单核 CPU 上，操作系统也能通过快速的上下文切换，让多个线程轮流执行，从宏观上看就像是同时在运行，这使得程序可以同时处理多个请求或事件
- **并行性**：这是在多核 CPU 上才能实现的真正意义上的"同时"执行。如果一个程序有 4 个线程，而系统有 4 个 CPU 核心，那么这 4 个线程可以真正地在 4 个核心上同时执行，这种硬件级别的并行可以极大地提高计算密集型任务的执行时间

### 4.2 提升资源利用率 (Resource Utilization)

多线程可以更有效地利用系统的资源，特别是当程序涉及到 I/O 密集型任务时。

- **CPU 与 I/O 的解耦**：当一个线程执行 I/O 操作（例如，从磁盘读取文件、通过网络下载数据、等待数据库查询结果）时，它通常会进入阻塞状态，等待外部设备完成操作。在单线程模型下，CPU 会在这段时间内空闲下来，造成资源浪费
- **CPU 时间片的利用**：在多线程模型下，当一个线程阻塞时，操作系统调度器可以立即将 CPU 的执行权切换给另一个就绪的线程。这样，CPU 就不会闲置，而是继续执行其他有意义的计算任务，这大大提高了 CPU 的利用效率

**案例**：一个 Web 服务器需要处理多个客户端的请求：
- **单线程**：如果一个请求需要访问数据库（一个耗时的 I/O 操作），主线程会一直等待数据库返回结果。在这段时间内，所有其他客户端的请求都会被阻塞，服务器无法响应
- **多线程**：每个客户端请求都由一个独立的线程来处理。当一个线程因为等待数据库而阻塞时，其他线程可以继续处理新的客户端请求。这使得服务器能够同时服务多个客户端，提升了吞吐量和用户体验

## 5. 多线程的缺点

| 缺点 | 说明 |
|------|------|
| **同步和死锁问题** | 由于多个线程共享同一块内存，对共享资源的访问必须小心，否则会导致**数据竞争**。这需要使用[[12.3 互斥量|同步机制]]（如互斥锁、[[12.4 信号量|信号量]]）来保护共享数据，如果同步处理不当，还可能导致**死锁** |
| **调试难度** | 由于线程的执行顺序不确定，多线程程序的调试要比单线程程序复杂得多，难以重现和定位 bug |

---

## 6. 关键要点

1. **线程是操作系统调度的最小单位**，比进程更轻量
2. **同一进程的线程共享资源**（堆、全局变量、文件描述符），但各有独立的栈和寄存器
3. **上下文切换**是实现多任务的核心机制，线程切换比进程切换开销更小
4. **多线程的优势**：提高并发性、并行性和资源利用率
5. **多线程的挑战**：需要处理同步、死锁和调试困难等问题
