# Nagle 算法

## 简介

Nagle 算法是 [[7.5 TCP协议基础|TCP]] 协议中的一种优化算法，用于减少网络中小数据包的数量。

**基本思想**：当一个 TCP 连接中有数据待发送时，它不会立即发送每个小的数据段。相反，它会等待，直到：

1. **待发送的数据量达到一个 TCP 的最大段大小**（Maximum Segment Size, MSS）
2. **或者，上一个已发送的数据包的确认（ACK）到达**

简单来说，Nagle 算法将多个小数据块"攒"在一起，直到它们足够大，或者收到上一个包的确认，再一起发送。这就像是把多封信件装进一个信封再寄出，而不是每写完一句话就寄一封信。

---

## 优点

### 1. 减少小包数量，提高网络效率

这是 Nagle 算法最核心的优点。通过将多个小数据块合并成一个大的数据段，它减少了网络中的数据包数量。每个数据包都需要包含 TCP/IP 头部，这个头部通常有 40 字节或更多。如果发送 1 字节的数据，却要发送 40 字节的头部，这是非常低效的。Nagle 算法可以显著减少这种"头部开销"。

```
不使用 Nagle 算法：
┌────────────────┐ ┌────────────────┐ ┌────────────────┐
│ TCP头(40B) + H │ │ TCP头(40B) + e │ │ TCP头(40B) + l │  ...
└────────────────┘ └────────────────┘ └────────────────┘
      发送 3 个包，总计 120+ 字节头部

使用 Nagle 算法：
┌──────────────────────┐
│ TCP头(40B) + Hello   │  合并发送
└──────────────────────┘
      发送 1 个包，只需 40 字节头部
```

### 2. 缓解网络拥塞

减少数据包数量直接有助于缓解网络拥塞，因为它降低了网络上的流量，减少了路由器和交换机的处理负担。

---

## 缺点

### 1. 可能导致延迟

Nagle 算法的"等待"机制是它的主要缺点。如果一个应用需要实时响应，例如游戏或远程登录（如 SSH），等待上一个数据包的确认再发送下一个数据包可能会引入明显的延迟。用户可能会感觉输入卡顿，因为每个按键的输入都需要等待前一个数据包的确认才能发送。

```
场景：SSH 远程终端输入
用户输入: a → 等待 ACK → b → 等待 ACK → c → 等待 ACK
                ↓              ↓              ↓
            明显的输入延迟感
```

### 2. 与某些应用冲突

Nagle 算法与 TCP 延迟确认（Delayed ACK）机制结合时，可能会产生更长的延迟。延迟确认是另一个优化 TCP 的机制，它会延迟发送 ACK，以期能与发送端发来的数据一起回传。如果发送端正在等待 ACK 才能继续发送，而 ACK 又被延迟了，就会导致双方互相等待，形成"死锁"般的延迟。

```
Nagle + Delayed ACK 的问题：

发送方                              接收方
   │                                   │
   │ ─── 小数据包 ───→                  │
   │                                   │
   │ 等待 ACK...                        │ 等待更多数据或超时再发 ACK（Delayed ACK）
   │                                   │
   │         ...互相等待...             │
   │                                   │
   │ ←─── ACK（超时后发送）────          │
   │                                   │
   │ ─── 下一个数据包 ───→              │
```

---

## 禁用 Nagle 算法

对于需要低延迟的应用，可以通过设置 `TCP_NODELAY` [[10.1 套接字选项|套接字选项]] 来禁用 Nagle 算法。

### 代码示例

```c
#include <stdio.h>
#include <sys/socket.h>
#include <netinet/tcp.h>

int main() {
    int sock = socket(PF_INET, SOCK_STREAM, 0);

    // 禁用 Nagle 算法
    int nodelay = 1;
    if (setsockopt(sock, IPPROTO_TCP, TCP_NODELAY, &nodelay, sizeof(nodelay)) < 0) {
        perror("setsockopt TCP_NODELAY failed");
        return -1;
    }

    printf("Nagle 算法已禁用\n");

    // ... 后续的网络操作

    close(sock);
    return 0;
}
```

### 获取当前状态

```c
int nodelay;
socklen_t optlen = sizeof(nodelay);

getsockopt(sock, IPPROTO_TCP, TCP_NODELAY, &nodelay, &optlen);
printf("TCP_NODELAY = %d\n", nodelay);  // 1 表示已禁用 Nagle，0 表示启用
```

---

## 适用场景分析

| 场景 | 是否禁用 Nagle | 原因 |
|:---|:---|:---|
| **SSH/Telnet** | 是（禁用） | 需要实时响应键盘输入 |
| **在线游戏** | 是（禁用） | 需要低延迟的实时交互 |
| **HTTP/REST API** | 通常不需要 | 请求/响应通常足够大 |
| **文件传输** | 否（启用） | 大块数据传输，Nagle 有助于效率 |
| **数据库连接** | 视情况 | 小查询可能需要禁用 |
| **视频流** | 否（启用） | 数据包通常较大 |

## Nagle 算法伪代码

```
if 有新数据要发送:
    if 数据长度 >= MSS:
        立即发送
    else if 有未确认的数据（在途数据）:
        将数据加入缓冲区，等待
    else:
        立即发送这个小数据包

当收到 ACK 时:
    if 缓冲区有等待的数据:
        发送缓冲区中的数据
```

## 与其他机制的关系

### 与滑动窗口的关系

Nagle 算法工作在应用层和 TCP 层之间，而 [[7.5 TCP协议基础#3. 滑动窗口|滑动窗口]] 是 TCP 层的流量控制机制。两者共同工作以优化网络传输。

### 与 Cork 选项的关系

Linux 提供了 `TCP_CORK` 选项，它比 Nagle 更激进：

| 选项 | 行为 |
|:---|:---|
| **Nagle（默认）** | 等待 ACK 或数据达到 MSS 后发送 |
| **TCP_CORK** | 完全阻止发送，直到显式取消 CORK 或缓冲区满 |
| **TCP_NODELAY** | 完全禁用 Nagle，立即发送所有数据 |

---

## 关键要点

1. **Nagle 算法目的**：减少网络中的小数据包数量，提高网络效率
2. **工作原理**：等待数据达到 MSS 或收到 ACK 后再发送
3. **优点**：减少网络拥塞，降低头部开销
4. **缺点**：可能增加延迟，与 Delayed ACK 可能冲突
5. **禁用方法**：使用 [[10.1 套接字选项#setsockopt 函数|setsockopt()]] 设置 `TCP_NODELAY` 选项
6. **适用场景**：实时应用（游戏、SSH）应禁用，批量传输应启用
