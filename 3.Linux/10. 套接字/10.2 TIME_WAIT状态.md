# TIME_WAIT 状态

## 概述

在 [[7.5 TCP协议基础#TCP 四次挥手（断开连接）|TCP 四次挥手]] 过程中，**主动关闭连接的一方**在发送最后一个 ACK 后会进入 **TIME_WAIT** 状态。这个状态会持续 **2MSL**（Maximum Segment Lifetime，报文最大生存时间），通常为 1-4 分钟。

TIME_WAIT 状态是 TCP 协议设计中的重要机制，但在服务器开发中经常会导致 "Address already in use" 错误。

---

## TIME_WAIT 产生的场景

```
       主机A（主动关闭）                          主机B（被动关闭）
            │                                        │
            │         ─── FIN (seq=7000) ───→        │
       [FIN_WAIT_1]                          [CLOSE_WAIT]
            │                                        │
            │         ←── ACK (ack=7001) ────        │
       [FIN_WAIT_2]                                  │
            │                                        │
            │         ←── FIN (seq=5001) ────        │
            │                                [LAST_ACK]
       [TIME_WAIT]                                   │
            │         ─── ACK (ack=5002) ───→        │
            │                                [CLOSED]
            │                                        │
        2MSL 等待                                     │
            │                                        │
       [CLOSED]                                      │
```

**关键点**：TIME_WAIT 状态只出现在**主动关闭连接**的一方。

## TIME_WAIT 存在的原因

### 1. 确保最后的 ACK 能够到达

在四次挥手的最后一步，主动关闭方发送 ACK 给被动关闭方。如果这个 ACK 丢失：

```
       主动关闭方                              被动关闭方
            │                                      │
       [TIME_WAIT]                            [LAST_ACK]
            │    ─── ACK ───→ ✗（丢失）            │
            │                                      │
            │    ←── FIN（重传）────               │  超时重传 FIN
            │                                      │
            │    ─── ACK（重发）───→               │  重发 ACK
            │                                [CLOSED]
```

如果没有 TIME_WAIT 状态，主动关闭方会直接进入 CLOSED 状态，无法响应被动关闭方重传的 FIN，导致被动关闭方一直停留在 LAST_ACK 状态。

### 2. 让旧连接的报文消失

TIME_WAIT 持续 2MSL 的目的是确保本次连接的所有报文都从网络中消失。

**假设没有 TIME_WAIT**：

```
旧连接：192.168.1.1:8080 <-> 192.168.1.2:12345

1. 旧连接关闭
2. 新连接立即使用相同的四元组建立
3. 旧连接的延迟报文（seq=1000）到达
4. 新连接错误地接收了旧数据！
```

2MSL 的等待时间确保了旧连接的任何报文都已经在网络中超时消失。

---

## TIME_WAIT 带来的问题

### 问题描述

当服务器主动关闭连接后，如果立即重启，会遇到 "Address already in use" 错误：

```
服务端                                        客户端
   │                                            │
   │  ←───────── 客户端连接 ──────────            │
   │                                            │
   │  ─────────── 关闭服务器 ────────→            │
   │                                            │
[TIME_WAIT]                                     │
   │                                            │
   │  重启服务器                                 │
   │  bind() 失败！                              │
   │  "Address already in use"                  │
```

![[Pasted image 20251218085447.png]]

**原因**：端口仍然被处于 TIME_WAIT 状态的旧套接字占用。

### 实际场景

```bash
# 第一次启动服务器
$ ./server 9527
服务器启动成功，监听端口 9527

# 关闭服务器后立即重启
$ ./server 9527
bind failed: Address already in use
```

---

## SO_REUSEADDR 解决方案

通过设置 `SO_REUSEADDR` [[10.1 套接字选项|套接字选项]]，可以允许绑定处于 TIME_WAIT 状态的地址。

### 原理

`SO_REUSEADDR` 选项告诉内核：即使端口处于 TIME_WAIT 状态，也允许新的套接字绑定到该端口。

### 代码示例

```c
#include <stdio.h>
#include <string.h>
#include <unistd.h>
#include <sys/socket.h>
#include <netinet/in.h>
#include <arpa/inet.h>

void server() {
    int sock;
    struct sockaddr_in addr, cli;
    socklen_t addrlen = sizeof(addr);
    char message[256];

    // 1. 创建套接字
    sock = socket(PF_INET, SOCK_STREAM, 0);

    // 2. 【关键】设置 SO_REUSEADDR 选项
    int optval = 1;
    setsockopt(sock, SOL_SOCKET, SO_REUSEADDR, &optval, sizeof(optval));
    printf("SO_REUSEADDR = %d\n", optval);

    // 再次确认设置成功
    optval = 0;
    setsockopt(sock, SOL_SOCKET, SO_REUSEADDR, &optval, sizeof(optval));
    setsockopt(sock, SOL_SOCKET, SO_REUSEADDR, &optval, sizeof(optval));
    printf("SO_REUSEADDR = %d\n", optval);

    // 3. 设置服务器地址
    memset(&addr, 0, addrlen);
    addr.sin_family = AF_INET;
    addr.sin_addr.s_addr = inet_addr("127.0.0.1");
    addr.sin_port = htons(9527);
    addrlen = sizeof(addr);

    // 4. 绑定地址（现在即使端口处于 TIME_WAIT 也能成功）
    if (bind(sock, (struct sockaddr*)&addr, addrlen) == -1) {
        perror("bind failed");
        return;
    }

    // 5. 监听和接受连接
    listen(sock, 3);

    int client = accept(sock, (struct sockaddr*)&cli, &addrlen);
    read(client, message, sizeof(message));

    // 6. 关闭连接
    close(client);
    close(sock);
}
```

### 客户端代码

```c
void client() {
    int client = socket(PF_INET, SOCK_STREAM, 0);

    struct sockaddr_in servaddr;
    memset(&servaddr, 0, sizeof(servaddr));
    servaddr.sin_family = AF_INET;
    servaddr.sin_addr.s_addr = inet_addr("127.0.0.1");
    servaddr.sin_port = htons(9527);

    int ret = connect(client, (struct sockaddr*)&servaddr, sizeof(servaddr));
    while (ret == 0) {
        char buffer[256] = "";
        fgets(buffer, sizeof(buffer), stdin);

        if ((strcmp(buffer, "q\n") == 0) || (strcmp(buffer, "Q\n") == 0)) {
            break;
        }

        size_t len = strlen(buffer);
        size_t send_len = 0;

        // 分包发送
        while (len > 0) {
            ssize_t ret = write(client, buffer + send_len, len);
            if (ret <= 0) {
                fputs("write failed!\n", stdout);
                close(client);
                std::cout << "client done!" << std::endl;
                return;
            }
            send_len += (size_t)ret;
        }

        write(client, buffer, strlen(buffer));
        memset(buffer, 0, sizeof(buffer));

        // 接收响应
        size_t read_len = 0;
        while (read_len < len) {
            ssize_t ret = read(client, buffer + read_len, len - read_len);
            if (ret <= 0) {
                fputs("read failed!\n", stdout);
                close(client);
                std::cout << "client done!" << std::endl;
                return;
            }
            read_len += (size_t)ret;
        }

        std::cout << "from server: " << buffer << std::endl;
    }

    close(client);
    std::cout << "client done!" << std::endl;
}
```

## 运行效果对比

### 不使用 SO_REUSEADDR

```bash
# 终端1：启动服务端
$ ./server
服务器启动成功

# 终端2：启动客户端并发送消息后退出
$ ./client
输入消息: hello
^C

# 终端1：服务端意外中断，第一个服务器处于 TIME_WAIT
# 再次启动服务端
$ ./server
bind failed: Address already in use
```

### 使用 SO_REUSEADDR

```bash
# 终端1：启动服务端（设置了 SO_REUSEADDR）
$ ./server
SO_REUSEADDR = 1
服务器启动成功

# 终端2：客户端连接后退出
$ ./client

# 终端1：重启服务端（即使端口处于 TIME_WAIT）
$ ./server
SO_REUSEADDR = 1
服务器启动成功  # 成功！
```

## 内存与 CPU 视角

### TIME_WAIT 对资源的影响

| 资源 | 影响 |
|:---|:---|
| **内存** | 每个 TIME_WAIT 连接会占用少量内核内存（约 200-300 字节） |
| **端口** | 端口被占用，无法被新连接使用（除非设置 SO_REUSEADDR） |
| **文件描述符** | TIME_WAIT 不占用文件描述符（套接字已关闭） |

### 高并发场景下的问题

在高并发服务器中，如果服务器主动关闭大量连接，可能会积累大量 TIME_WAIT 状态的连接，导致：

1. 端口耗尽（可用端口数有限）
2. 内核内存占用增加

**优化建议**：
- 让客户端主动关闭连接（TIME_WAIT 在客户端产生）
- 使用连接池复用连接
- 调整系统参数（如 `tcp_tw_reuse`、`tcp_tw_recycle`）

---

## 关键要点

1. **TIME_WAIT 产生**：只在主动关闭连接的一方产生
2. **持续时间**：2MSL（通常 1-4 分钟）
3. **存在原因**：
   - 确保最后的 ACK 能够重传
   - 让旧连接的报文在网络中消失
4. **解决方案**：使用 [[10.1 套接字选项#setsockopt 函数|setsockopt()]] 设置 `SO_REUSEADDR` 选项
5. **设置时机**：必须在 [[7.6 套接字#bind 函数|bind()]] 之前设置
