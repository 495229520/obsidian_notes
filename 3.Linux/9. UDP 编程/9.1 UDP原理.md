# UDP 原理

## 简介

UDP，全称 **User Datagram Protocol**，中文是**用户数据报协议**，它是互联网 [[7.4 协议|协议]] 栈中一个核心的传输层协议，与 [[7.5 TCP协议基础|TCP]] 是并列的关系（详细对比请参见 [[7.4 协议#TCP 与 UDP 对比|TCP与UDP对比]]）。

UDP 的设计哲学非常简洁："**快，但不可靠**"。它只管把数据报发出去，完全不关心对方是否收到，也不关心数据报的顺序。

## 基本原理

UDP 的工作方式可以类比为传统的**寄信服务**：

| 步骤 | 说明 |
|:---:|:---|
| **打包** | 你把信件（数据）放进一个信封里，写上收件人地址 |
| **发送** | 你把信交给邮局，邮局把它发出去 |
| **完成** | 你的工作到此结束。你无法得知信件是否到达，也不知道它会不会丢失，或者是否会比后寄的信先到达 |

UDP 协议只是简单地发送数据报，只负责把数据发送到目标地址，但不会建立连接，也不会确认收到。它没有**握手机制**、**重传机制**，也不保证数据的顺序和完整性。

---

## 特点

正是这种简洁的设计，赋予了 UDP 以下几个核心特点：

### 1. 无连接（Connectionless）

- **原理**：UDP 发送数据前，不需要像 [[7.5 TCP协议基础#TCP 三次握手（建立连接）|TCP 那样建立三次握手]]，它直接将数据报发送到目标地址
- **优点**：省去了建立和断开连接的开销，传输效率高
- **缺点**：无法保证数据顺序和接收状态，每个数据报都是独立的

### 2. 不可靠（Unreliable）

- **原理**：UDP 不提供任何可靠性保证机制。它没有确认传输机制，也没有确认机制（ACK），如果数据报在传输过程中丢失，UDP 不会知道，也不会重新发送
- **优点**：没有重传、确认等复杂机制，传输更轻量
- **缺点**：不适合对数据完整性要求高的场景，如文件传输

### 3. 无序（Unorder）

- **原因**：由于每个数据报都是独立的，它们在传输中可能走不同的路径，这种特性可能会造成数据的乱序。到达顺序可能与发送顺序不一致
- **说明**：在某些应用中，如果数据报的乱序不重要，这不是问题；但如果应用是有序的，那可能需要在应用层自己处理乱序问题
- **对策**：对于需要保证数据顺序的应用，可在应用层自行添加序号

### 4. 高效（Efficient）

- **原因**：UDP 头部只有 **8 字节**，目没有复杂的流量控制和拥塞控制机制。无需建立连接，开始传输快，传输时开销小
- **优点**：传输延迟低，开始传输快，非常适合需要快速传输的应用
- **缺点**：对于需要可靠性或拥塞控制的场景可能不太适合

## 适用场景

基于上述特性，UDP 适用于以下类型的应用：

| 场景 | 说明 |
|:---|:---|
| **实时传输** | 如视频聊天、在线游戏、IP 电话（VoIP）。在这些应用中，偶尔丢失几个包可以接受，但包丢弃后可以接受，因为重传反而导致延迟 |
| **广播和多播** | UDP 支持一对多和一对所有人的通信模式。主机发送一个请求，可以使用广播或组播的方式获取响应 |
| **DNS 查询** | DNS（域名系统）查询通常使用 UDP。主要因为查询和响应数据量小，使用 UDP 双发更高效。如果需要，可以使用 TCP |
| **简单请求-响应** | 在简单媒体流中，可以用 UDP 传输音视频流数据，再由应用层处理丢包恢复。典型应用如视频会议软件 |

## UDP 陷阱

UDP 本身**不实现可靠性**，正因为此，UDP 的数据报若丢失可能会造成问题：

| 问题 | 说明 |
|:---|:---|
| **接收方在收到数据报后立即发出响应** | 接收方发出的响应，若发送方无法确认收到，发送方会重发请求 |
| **发送方在收到确认前会重发数据** | 如果需要可靠，UDP 的数据报丢失后，需要应用层实现 ACK 检测和重发机制 |
| **如果确认丢失** | 如期望接收的数据报丢失，发送方可能重发，从而造成 TCP 的顺序混乱。如同 TCP 一样，需要序列号来区分不同的消息 |

> 这些问题远没有想完，只是表示用 UDP 协议开发时自己需要小心，不需要用 UDP 协议来实现。

## 为什么服务器需要 bind？

服务器需要 [[7.6 套接字#bind 函数|bind]] 的核心原因是：**它必须提供一个固定端口，让所有的客户端都能找到它并和它进行通信**。

| 要点       | 说明                                                          |                    |                                                                      |
| :------- | :---------------------------------------------------------- | ------------------ | -------------------------------------------------------------------- |
| **固定入口** | bind() 函数将服务器绑定在指定的 [[7.2 IP地址                              | IP 地址]] 和 [[7.3 端口 | 端口]] 上。服务器需要绑定到一个固定的 IP 和端口，以便客户端知道"电话号码"，然后发送请求到这个地址上，而且是一直是这一固定的地址 |
| **唯一标识** | 比如一台服务器服务 80 端口就是"宿舍"。如果端口号变了，就相当于把门牌号换了，客户端就找不到了，每次都不同，找不到 |                    |                                                                      |
| **监听准备** | 如果没有 bind()，操作系统无法知道你想监听哪个端口。你自己都不知道地址，别人怎么可能知道呢            |                    |                                                                      |

---

## 为什么客户端不用 bind？

客户端通常**不需要** bind 的原因在于：**它不关心自己使用哪个本地端口，它只需要发送数据即可**。

| 要点         | 说明                                                                                                           |
| :--------- | :----------------------------------------------------------------------------------------------------------- |
| **无需固定端口** | 客户端不需要提供一个固定的地址让其他设备来找。当客户端调用 sendto() 函数时，操作系统会自动为它分配一个临时端口，可用的 IP 地址和端口号                                   |
| **自动分配**   | sendto() 函数不仅负责发送数据，还负责自动绑定地址。当服务器接收到数据，会自动获取发送者的地址信息。客户端 recvfrom() 等待服务器的响应时，服务器正是通过这个自动分配的"临时端口"将响应发回给客户端 |
| **临时性质**   | 服务器确定能通过这个通道给你回复消息，用后释放。当客户端退出或发起一次新的请求时，这个"临时端口"的资源可能会被释放或重新分配                                              |

---

## 关键要点

1. **UDP vs TCP**：UDP 是无连接、不可靠的协议；[[7.5 TCP协议基础|TCP]] 是面向连接、可靠的协议
2. **UDP 特点**：无连接、不可靠、无序、高效（8字节头部）
3. **适用场景**：实时音视频、在线游戏、DNS 查询、广播/多播
4. **服务器必须 bind**：提供固定 [[7.3 端口|端口]] 让客户端能够找到
5. **客户端通常不 bind**：操作系统自动分配临时端口
6. **编程实践**：参见 [[9.2 UDP服务端代码|服务端代码]] 和 [[9.3 UDP客户端代码|客户端代码]]
