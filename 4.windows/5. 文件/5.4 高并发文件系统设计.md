# 5.4 高并发文件系统设计

> 本文介绍如何在 Windows 平台设计一个支持高并发访问的文件系统服务，核心技术包括异步 IO、IOCP 和线程池。

---

## 1. 问题背景

### 1.1 同步 IO 的瓶颈

在 [[3.2 读取文件案例]] 中，我们使用了同步 IO 模型。这种模型有一个根本问题：**每个 IO 操作都会阻塞当前线程**。

```
同步模型的执行流程：
线程1: [发起读取] ──等待──等待──等待── [读取完成] [处理数据]
线程2: [发起读取] ──等待──等待──等待── [读取完成] [处理数据]
线程3: [发起读取] ──等待──等待──等待── [读取完成] [处理数据]
```

**问题分析**：
- 如果要同时处理 1000 个文件读取请求，就需要 1000 个线程
- 每个线程默认占用 1MB 栈空间，1000 个线程就是 1GB 内存
- 线程切换带来大量上下文切换开销

### 1.2 异步 IO 的优势

异步 IO 的核心思想是：**发起 IO 操作后立即返回，不等待操作完成**。

```
异步模型的执行流程：
线程1: [发起读取A] [发起读取B] [发起读取C] ... [处理A完成] [处理B完成] ...
```

**优势分析**：
- 一个线程可以同时管理多个 IO 操作
- 线程不会因为 IO 等待而阻塞
- 大幅减少所需线程数量和内存消耗

## 2. 核心架构设计

### 2.1 整体架构图

```
┌─────────────────────────────────────────────────────────┐
│                    客户端请求                            │
│          (可能有成百上千个并发文件操作请求)               │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐
│              IO 完成端口 (IOCP)                          │
│                                                         │
│  作用：                                                  │
│  1. 接收所有异步 IO 完成通知                             │
│  2. 将完成通知分发给工作线程                             │
│  3. 自动负载均衡                                        │
│                                                         │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐       │
│  │ Worker1 │ │ Worker2 │ │ Worker3 │ │ Worker4 │       │
│  │ (空闲)  │ │ (处理中) │ │ (空闲)  │ │ (处理中) │       │
│  └────┬────┘ └────┬────┘ └────┬────┘ └────┬────┘       │
└───────┼──────────┼──────────┼──────────┼───────────────┘
        │          │          │          │
        └──────────┴──────────┴──────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐
│                    文件系统                              │
└─────────────────────────────────────────────────────────┘
```

### 2.2 设计思路

整个系统基于 **Proactor 模式**，这是 Windows 异步 IO 的标准模式：

| 组件 | 职责 | 类比 |
|------|------|------|
| **IOCP** | 事件分发器，管理所有 IO 完成事件 | 餐厅的叫号系统 |
| **工作线程** | 处理完成的 IO 操作 | 服务员 |
| **IOContext** | 携带单次 IO 操作的所有信息 | 点菜单 |
| **FileContext** | 管理单个文件的状态 | 餐桌 |

**为什么选择 IOCP？**

1. **内核级优化**：IOCP 是 Windows 内核直接支持的机制，效率最高
2. **自动负载均衡**：系统自动将完成事件分发给空闲线程
3. **LIFO 唤醒策略**：优先唤醒最近使用的线程，提高 CPU 缓存命中率

## 3. 关键技术详解

### 3.1 OVERLAPPED 结构

`OVERLAPPED` 是 Windows 异步 IO 的核心数据结构，它告诉系统"这是一个异步操作"。

```cpp
typedef struct _OVERLAPPED {
    ULONG_PTR Internal;      // 系统内部使用，存储 IO 状态
    ULONG_PTR InternalHigh;  // 系统内部使用，存储传输字节数
    union {
        struct {
            DWORD Offset;      // 文件操作的起始偏移（低32位）
            DWORD OffsetHigh;  // 文件操作的起始偏移（高32位）
        };
        PVOID Pointer;
    };
    HANDLE hEvent;           // 可选的事件句柄，用于通知完成
} OVERLAPPED;
```

**关键点**：
- `Offset/OffsetHigh`：指定从文件的哪个位置开始读写
- 异步操作时，这个结构必须保持有效直到操作完成
- 可以通过继承扩展这个结构，携带自定义数据

### 3.2 异步文件操作流程

**步骤 1：以异步模式打开文件**

```cpp
HANDLE hFile = CreateFileA(
    "test.txt",
    GENERIC_READ,
    FILE_SHARE_READ,
    nullptr,
    OPEN_EXISTING,
    FILE_FLAG_OVERLAPPED,  // 关键！必须指定这个标志
    nullptr
);
```

`FILE_FLAG_OVERLAPPED` 告诉系统：这个文件句柄将用于异步操作。如果不设置这个标志，即使传入 `OVERLAPPED` 结构，操作仍然是同步的。

**步骤 2：发起异步读取**

```cpp
OVERLAPPED overlapped = {0};
overlapped.Offset = 0;        // 从文件开头读取
overlapped.OffsetHigh = 0;

char buffer[4096];
BOOL result = ReadFile(
    hFile,
    buffer,
    sizeof(buffer),
    nullptr,        // 异步模式下，这个参数被忽略
    &overlapped     // 传入 OVERLAPPED 结构表示异步操作
);
```

**步骤 3：判断操作状态**

```cpp
if (result) {
    // 操作立即完成（数据可能在系统缓存中）
    // 可以直接使用 buffer 中的数据
} else {
    DWORD error = GetLastError();
    if (error == ERROR_IO_PENDING) {
        // 正常情况：操作正在后台进行
        // 稍后会通过 IOCP 收到完成通知
    } else {
        // 真正的错误
    }
}
```

### 3.3 IO 完成端口 (IOCP)

IOCP 是 Windows 提供的高性能 IO 完成通知机制。

**创建 IOCP**

```cpp
// 方式一：创建一个新的 IOCP
HANDLE hIOCP = CreateIoCompletionPort(
    INVALID_HANDLE_VALUE,  // 不关联任何句柄
    nullptr,               // 创建新的 IOCP
    0,                     // 完成键（此时无意义）
    4                      // 允许同时运行的线程数
);
```

最后一个参数 `NumberOfConcurrentThreads` 很重要：
- 设为 0 表示使用 CPU 核心数
- 这个值限制的是**同时运行**的线程数，不是总线程数
- 如果一个线程阻塞了，系统会自动唤醒另一个线程

**关联文件句柄到 IOCP**

```cpp
// 方式二：将文件句柄关联到已存在的 IOCP
CreateIoCompletionPort(
    hFile,           // 要关联的文件句柄
    hIOCP,           // 已存在的 IOCP
    (ULONG_PTR)ctx,  // 完成键：当该文件的 IO 完成时，会返回这个值
    0                // 此参数被忽略
);
```

**完成键 (Completion Key)** 的作用：
- 当文件的 IO 操作完成时，会返回这个值
- 通常用来传递文件相关的上下文信息
- 可以是指向自定义结构的指针

**工作线程等待完成事件**

```cpp
DWORD bytesTransferred;
ULONG_PTR completionKey;
LPOVERLAPPED pOverlapped;

BOOL result = GetQueuedCompletionStatus(
    hIOCP,             // IOCP 句柄
    &bytesTransferred, // 传输的字节数
    &completionKey,    // 完成键（之前关联时设置的）
    &pOverlapped,      // 指向 OVERLAPPED 结构的指针
    INFINITE           // 等待时间（INFINITE = 永久等待）
);
```

这个函数会阻塞，直到有 IO 操作完成。返回时：
- `bytesTransferred`：实际读/写的字节数
- `completionKey`：关联句柄时设置的值，用于识别是哪个文件
- `pOverlapped`：指向发起操作时传入的 `OVERLAPPED` 结构

## 4. 数据结构设计

### 4.1 设计思路

我们需要设计两个核心数据结构：

1. **IOContext**：描述单次 IO 操作的所有信息
2. **FileContext**：描述单个文件的状态

**为什么要扩展 OVERLAPPED？**

Windows API 只返回 `OVERLAPPED*`，但我们需要更多信息（缓冲区、回调函数等）。通过继承，我们可以通过指针类型转换获取完整信息。

### 4.2 IOContext 结构

```cpp
// 操作类型枚举
enum class IOOperation {
    Read,
    Write
};

// 扩展的 OVERLAPPED 结构
// 继承自 OVERLAPPED，可以安全地进行指针转换
struct IOContext : public OVERLAPPED {
    IOOperation operation;      // 操作类型：读还是写
    std::vector<char> buffer;   // 数据缓冲区
    DWORD bytesTransferred;     // 实际传输的字节数
    std::function<void(IOContext*, DWORD)> callback;  // 完成回调

    IOContext(size_t bufferSize = 4096)
        : buffer(bufferSize), bytesTransferred(0)
    {
        // 必须初始化 OVERLAPPED 的所有成员
        Internal = 0;
        InternalHigh = 0;
        Offset = 0;
        OffsetHigh = 0;
        hEvent = nullptr;  // 使用 IOCP 时不需要事件
    }
};
```

**设计要点**：

1. **继承 OVERLAPPED**：这样 `IOContext*` 可以直接传给 Windows API
2. **buffer 使用 vector**：自动管理内存，避免手动 new/delete
3. **callback 使用 std::function**：支持 lambda，使用更灵活

### 4.3 FileContext 结构

```cpp
struct FileContext {
    HANDLE hFile;                    // 文件句柄
    std::string filePath;            // 文件路径（用于调试）
    std::atomic<int> pendingOps{0};  // 待完成的操作数
};
```

**为什么需要 pendingOps？**

- 跟踪该文件有多少个 IO 操作正在进行
- 关闭文件前必须等待所有操作完成
- 使用 `atomic` 保证线程安全

## 5. 服务类实现

### 5.1 类的整体结构

```cpp
class HighConcurrencyFileService {
private:
    HANDLE m_hIOCP;                      // IOCP 句柄
    std::vector<HANDLE> m_workerThreads; // 工作线程句柄
    std::atomic<bool> m_running{false};  // 运行状态标志
    DWORD m_threadCount;                 // 工作线程数量

public:
    HighConcurrencyFileService(DWORD threadCount = 0);
    ~HighConcurrencyFileService();

    bool Start();   // 启动服务
    void Stop();    // 停止服务

    std::unique_ptr<FileContext> OpenFile(const std::string& path,
                                          DWORD access = GENERIC_READ);
    bool AsyncRead(...);
    bool AsyncWrite(...);

private:
    static DWORD WINAPI WorkerThread(LPVOID lpParam);
};
```

### 5.2 构造函数：确定线程数量

```cpp
HighConcurrencyFileService(DWORD threadCount = 0) {
    if (threadCount == 0) {
        // 获取 CPU 核心数
        SYSTEM_INFO sysInfo;
        GetSystemInfo(&sysInfo);
        // IO 密集型任务，使用 2 倍核心数
        m_threadCount = sysInfo.dwNumberOfProcessors * 2;
    } else {
        m_threadCount = threadCount;
    }
}
```

**线程数量的选择依据**：

| 任务类型 | 推荐线程数 | 原因 |
|----------|-----------|------|
| CPU 密集型 | N | 超过核心数会增加切换开销 |
| IO 密集型 | 2N | IO 等待时可以处理其他任务 |
| 混合型 | 1.5N | 折中方案 |

### 5.3 启动服务

```cpp
bool Start() {
    // 第一步：创建 IOCP
    m_hIOCP = CreateIoCompletionPort(
        INVALID_HANDLE_VALUE,  // 不关联句柄，只创建 IOCP
        nullptr,               // 新建 IOCP
        0,                     // 完成键
        m_threadCount          // 并发线程数
    );

    if (m_hIOCP == nullptr) {
        return false;
    }

    m_running = true;

    // 第二步：创建工作线程池
    for (DWORD i = 0; i < m_threadCount; ++i) {
        HANDLE hThread = CreateThread(
            nullptr,        // 默认安全属性
            0,              // 默认栈大小
            WorkerThread,   // 线程函数
            this,           // 传递 this 指针
            0,              // 立即运行
            nullptr         // 不需要线程 ID
        );

        if (hThread != nullptr) {
            m_workerThreads.push_back(hThread);
        }
    }

    return !m_workerThreads.empty();
}
```

### 5.4 停止服务

停止服务需要优雅地关闭所有工作线程：

```cpp
void Stop() {
    m_running = false;

    // 第一步：通知所有工作线程退出
    // PostQueuedCompletionStatus 可以手动投递一个完成事件
    for (size_t i = 0; i < m_workerThreads.size(); ++i) {
        PostQueuedCompletionStatus(
            m_hIOCP,
            0,        // 传输字节数
            0,        // 完成键 = 0 作为退出信号
            nullptr   // OVERLAPPED = nullptr 作为退出信号
        );
    }

    // 第二步：等待所有线程退出
    if (!m_workerThreads.empty()) {
        WaitForMultipleObjects(
            static_cast<DWORD>(m_workerThreads.size()),
            m_workerThreads.data(),
            TRUE,      // 等待所有线程
            INFINITE
        );

        // 第三步：清理线程句柄
        for (auto& hThread : m_workerThreads) {
            CloseHandle(hThread);
        }
        m_workerThreads.clear();
    }

    // 第四步：关闭 IOCP
    if (m_hIOCP != nullptr) {
        CloseHandle(m_hIOCP);
        m_hIOCP = nullptr;
    }
}
```

**关键点**：使用 `PostQueuedCompletionStatus` 发送退出信号，这样工作线程可以优雅地退出循环。

### 5.5 打开文件

```cpp
std::unique_ptr<FileContext> OpenFile(const std::string& path,
                                      DWORD access = GENERIC_READ) {
    // 第一步：以异步模式打开文件
    HANDLE hFile = CreateFileA(
        path.c_str(),
        access,
        FILE_SHARE_READ,   // 允许其他进程读取
        nullptr,
        OPEN_EXISTING,
        FILE_FLAG_OVERLAPPED,  // 关键：启用异步模式
        nullptr
    );

    if (hFile == INVALID_HANDLE_VALUE) {
        return nullptr;
    }

    // 第二步：创建文件上下文
    auto context = std::make_unique<FileContext>();
    context->hFile = hFile;
    context->filePath = path;

    // 第三步：将文件句柄关联到 IOCP
    // 完成键设为 FileContext 指针，这样完成时可以找到对应的文件
    if (CreateIoCompletionPort(
            hFile,
            m_hIOCP,
            reinterpret_cast<ULONG_PTR>(context.get()),  // 完成键
            0) == nullptr) {
        CloseHandle(hFile);
        return nullptr;
    }

    return context;
}
```

### 5.6 异步读取

```cpp
bool AsyncRead(FileContext* fileCtx,
               DWORD offset,
               DWORD size,
               std::function<void(IOContext*, DWORD)> callback) {
    // 第一步：创建 IO 上下文（堆分配，因为要跨越异步边界）
    auto ioCtx = new IOContext(size);
    ioCtx->operation = IOOperation::Read;
    ioCtx->Offset = offset;           // 设置读取起始位置
    ioCtx->callback = std::move(callback);

    // 第二步：增加待处理计数
    fileCtx->pendingOps++;

    // 第三步：发起异步读取
    BOOL result = ReadFile(
        fileCtx->hFile,
        ioCtx->buffer.data(),
        size,
        nullptr,      // 异步模式下忽略
        ioCtx         // IOContext 继承自 OVERLAPPED，可以直接传入
    );

    // 第四步：检查结果
    if (!result && GetLastError() != ERROR_IO_PENDING) {
        // 发生真正的错误
        fileCtx->pendingOps--;
        delete ioCtx;
        return false;
    }

    // 成功：操作正在进行，完成后会收到 IOCP 通知
    return true;
}
```

### 5.7 工作线程

工作线程是整个系统的核心，它不断从 IOCP 获取完成事件并处理：

```cpp
static DWORD WINAPI WorkerThread(LPVOID lpParam) {
    auto service = static_cast<HighConcurrencyFileService*>(lpParam);

    while (service->m_running) {
        DWORD bytesTransferred = 0;
        ULONG_PTR completionKey = 0;
        LPOVERLAPPED pOverlapped = nullptr;

        // 等待 IO 完成事件
        // 这里会阻塞，直到有事件到来
        BOOL result = GetQueuedCompletionStatus(
            service->m_hIOCP,
            &bytesTransferred,
            &completionKey,
            &pOverlapped,
            INFINITE
        );

        // 检查是否是退出信号
        // Stop() 中发送的信号：completionKey=0, pOverlapped=nullptr
        if (completionKey == 0 && pOverlapped == nullptr) {
            break;  // 收到退出信号，退出循环
        }

        // 还原上下文指针
        auto fileCtx = reinterpret_cast<FileContext*>(completionKey);
        auto ioCtx = static_cast<IOContext*>(pOverlapped);

        if (ioCtx != nullptr) {
            // 记录实际传输的字节数
            ioCtx->bytesTransferred = bytesTransferred;

            // 执行用户回调
            if (ioCtx->callback) {
                DWORD error = result ? 0 : GetLastError();
                ioCtx->callback(ioCtx, error);
            }

            // 减少待处理计数
            fileCtx->pendingOps--;

            // 释放 IO 上下文（在 AsyncRead 中 new 的）
            delete ioCtx;
        }
    }

    return 0;
}
```

**工作线程的执行流程**：

```
┌──────────────────────────────────────────────────────────────┐
│                      工作线程循环                             │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ GetQueuedCompletionStatus() - 阻塞等待              │    │
│  └─────────────────────┬───────────────────────────────┘    │
│                        │                                     │
│                        ▼                                     │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 收到完成事件                                         │    │
│  │ - bytesTransferred: 实际传输字节数                   │    │
│  │ - completionKey: FileContext 指针                   │    │
│  │ - pOverlapped: IOContext 指针                       │    │
│  └─────────────────────┬───────────────────────────────┘    │
│                        │                                     │
│                        ▼                                     │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 执行用户回调 callback(ioCtx, error)                  │    │
│  │ 用户可以在回调中处理读取到的数据                      │    │
│  └─────────────────────┬───────────────────────────────┘    │
│                        │                                     │
│                        ▼                                     │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 清理资源，返回循环开始                               │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 6. 使用示例

```cpp
#include <iostream>
#include <atomic>

int main() {
    // 创建服务（使用默认线程数 = CPU核心数 × 2）
    HighConcurrencyFileService fileService;

    // 启动服务
    if (!fileService.Start()) {
        std::cerr << "Failed to start file service" << std::endl;
        return 1;
    }

    // 打开文件
    auto fileCtx = fileService.OpenFile("test.txt");
    if (!fileCtx) {
        std::cerr << "Failed to open file" << std::endl;
        return 1;
    }

    // 并发读取计数器
    std::atomic<int> completedReads{0};
    const int NUM_READS = 100;

    // 发起 100 个并发读取请求
    for (int i = 0; i < NUM_READS; ++i) {
        fileService.AsyncRead(
            fileCtx.get(),
            i * 1024,      // 每次读取不同的偏移位置
            1024,          // 每次读取 1KB
            [&completedReads, i](IOContext* ctx, DWORD error) {
                if (error == 0) {
                    // 成功读取
                    // ctx->buffer.data() 包含读取的数据
                    // ctx->bytesTransferred 是实际读取的字节数
                    std::cout << "Read #" << i << " completed: "
                              << ctx->bytesTransferred << " bytes\n";
                } else {
                    std::cerr << "Read #" << i << " failed: " << error << "\n";
                }
                completedReads++;
            }
        );
    }

    // 等待所有读取完成
    while (completedReads < NUM_READS) {
        Sleep(10);
    }

    std::cout << "All " << NUM_READS << " reads completed!" << std::endl;

    // 关闭文件
    CloseHandle(fileCtx->hFile);

    // 服务会在析构时自动停止
    return 0;
}
```

## 7. 性能对比

### 7.1 同步 vs 异步

| 指标 | 同步 IO | 异步 IO (IOCP) |
|------|---------|----------------|
| 1000 并发所需线程 | 1000 | 8-16 |
| 内存消耗 | ~1GB | ~16MB |
| 上下文切换 | 频繁 | 很少 |
| 代码复杂度 | 简单 | 较复杂 |
| 适用场景 | 低并发、简单应用 | 高并发、服务器 |

### 7.2 实际测试数据参考

## 8. 常见陷阱与解决方案

### 8.1 忘记设置异步标志

```cpp
// ❌ 错误：没有 FILE_FLAG_OVERLAPPED
HANDLE hFile = CreateFileA(path, GENERIC_READ, 0, nullptr,
                           OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, nullptr);

// ✅ 正确：必须设置 FILE_FLAG_OVERLAPPED
HANDLE hFile = CreateFileA(path, GENERIC_READ, 0, nullptr,
                           OPEN_EXISTING, FILE_FLAG_OVERLAPPED, nullptr);
```

### 8.2 OVERLAPPED 生命周期问题

```cpp
// ❌ 错误：OVERLAPPED 在栈上，函数返回后被销毁
void BadAsyncRead(HANDLE hFile) {
    OVERLAPPED overlapped = {0};
    ReadFile(hFile, buffer, size, nullptr, &overlapped);
}  // overlapped 被销毁，但 IO 可能还在进行！

// ✅ 正确：使用堆分配，在回调中释放
void GoodAsyncRead(HANDLE hFile) {
    auto* overlapped = new OVERLAPPED{};
    ReadFile(hFile, buffer, size, nullptr, overlapped);
    // 在 IOCP 回调中 delete
}
```

### 8.3 错误处理

```cpp
BOOL result = ReadFile(hFile, buffer, size, nullptr, &overlapped);

if (!result) {
    DWORD error = GetLastError();
    if (error == ERROR_IO_PENDING) {
        // 这不是错误！这是正常的异步操作状态
    } else {
        // 这才是真正的错误
        HandleError(error);
    }
}
```

## 9. 关键要点总结

1. **IOCP 是 Windows 高并发 IO 的最佳选择**：内核级支持，自动负载均衡
2. **FILE_FLAG_OVERLAPPED 是必须的**：没有它就不是真正的异步
3. **OVERLAPPED 必须保持有效**：直到 IO 操作完成
4. **线程数选择**：IO 密集型用 2×CPU 核心数
5. **完成键用于识别上下文**：关联文件时设置，完成时返回
