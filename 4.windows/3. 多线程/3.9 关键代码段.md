# 3.9 关键代码段 (CRITICAL_SECTION)

## 简介

在 Windows API 中，除了[[3.3 线程同步|内核对象]]（如 [[3.4 互斥量|Mutex]]、[[3.6 事件对象与线程同步|Event]]、[[3.7 信号量|Semaphore]]）之外，还提供了一种用户模式的同步对象——**临界区对象 (或称为 关键代码段对象)**，其优势在于速度极快，因为它避免了频繁进入内核模式的开销。因此在编写多线程程序并且需要实现线程同步时，首先关键代码段。

---

## 概念与特性

1. **仅用于进程内同步**：`CRITICAL_SECTION` 对象只能用于同步同一个进程内部的多个线程，不能用于跨进程同步（这是互斥量 Mutex 的主要用途之一）

2. **用户模式优先**：当线程尝试进入一个未被锁定的临界区时，它是完全在用户模式下完成，效率极高。只有当临界区已经被锁定时，等待线程才会切换到内核模式进行等待

3. **递归性**：允许同一线程多次获取它已拥有的临界区锁

---

## 关键函数

### 初始化关键代码段

```cpp
InitializeCriticalSection(
    _Out_ LPCRITICAL_SECTION lpCriticalSection
);
```

### 进入关键代码段

```cpp
VOID
WINAPI
EnterCriticalSection(
    _Inout_ LPCRITICAL_SECTION lpCriticalSection
);
```

### 退出关键代码段

```cpp
VOID
WINAPI
LeaveCriticalSection(
    _Inout_ LPCRITICAL_SECTION lpCriticalSection
);
```

### 删除临界区

```cpp
WINBASEAPI
VOID
WINAPI
DeleteCriticalSection(
    _Inout_ LPCRITICAL_SECTION lpCriticalSection
);
```

---

## 与互斥量的区别

| 特性 | CRITICAL_SECTION (临界区对象) | [[3.4 互斥量\|Mutex (互斥量内核对象)]] |
|------|-------------------------------|----------------------------------------|
| **同步范围** | 仅限进程内的线程同步 | 跨进程和进程内线程同步 |
| **性能** | 高性能/快速，通常在用户模式下完成 | 涉及内核模式切换，相对较慢 |
| **安全性** | 无法检测线程终止，如果线程在临界区内异常终止，可能导致死锁（无放弃通知） | 支持放弃通知。如果拥有 Mutex 的线程异常终止，Mutex 会进入"已放弃"状态，可以被检测到 |
| **句柄** | 不需要句柄 (HANDLE) | 需要句柄 (HANDLE)，必须使用 CloseHandle 关闭 |

## 案例：售票系统

```cpp
#include <stdio.h>
#include <windows.h>
#include <process.h>

int iTickets = 5000;
CRITICAL_SECTION g_cs;

// A窗口      B窗口
DWORD WINAPI SellTicketA(void* lpParam)
{
    while (1)
    {
        EnterCriticalSection(&g_cs);  // 进入临界区
        if (iTickets > 0)
        {
            Sleep(1);
            iTickets--;
            printf("A remain %d\n", iTickets);
            LeaveCriticalSection(&g_cs);  // 离开临界区
        }
        else
        {
            LeaveCriticalSection(&g_cs);  // 离开临界区
            break;
        }
    }
    return 0;
}

DWORD WINAPI SellTicketB(void* lpParam)
{
    while (1)
    {
        EnterCriticalSection(&g_cs);  // 进入临界区
        if (iTickets > 0)
        {
            Sleep(1);
            iTickets--;
            printf("B remain %d\n", iTickets);
            LeaveCriticalSection(&g_cs);  // 离开临界区
        }
        else
        {
            LeaveCriticalSection(&g_cs);  // 离开临界区
            break;
        }
    }
    return 0;
}

int main()
{
    HANDLE hThreadA, hThreadB;
    hThreadA = CreateThread(NULL, 0, SellTicketA, NULL, 0, NULL);  // 2
    hThreadB = CreateThread(NULL, 0, SellTicketB, NULL, 0, NULL);  // 2
    CloseHandle(hThreadA);  // 1
    CloseHandle(hThreadB);  // 1

    InitializeCriticalSection(&g_cs);  // 初始化关键代码段

    Sleep(40000);

    DeleteCriticalSection(&g_cs);  // 删除临界区
    system("pause");
    return 0;
}
```

### 代码说明

1. `CRITICAL_SECTION g_cs` 是一个全局的临界区对象
2. `InitializeCriticalSection(&g_cs)` 初始化临界区，必须在使用前调用
3. `EnterCriticalSection(&g_cs)` 进入临界区，如果已被其他线程持有则等待
4. `LeaveCriticalSection(&g_cs)` 离开临界区，允许其他等待的线程进入
5. `DeleteCriticalSection(&g_cs)` 删除临界区，释放相关资源

> **注意**：在上面的代码中，`InitializeCriticalSection` 应该在创建线程之前调用，否则可能导致未定义行为。这是原始代码的一个问题。

## 使用建议

1. **优先选择 CRITICAL_SECTION**：在单进程多线程场景下，CRITICAL_SECTION 比 Mutex 性能更好
2. **确保配对使用**：每次 `EnterCriticalSection` 必须对应一次 `LeaveCriticalSection`
3. **避免长时间持有**：[[3.8 临界区|临界区]]内的代码应尽量简短，减少其他线程的等待时间
4. **注意异常处理**：如果临界区内可能抛出异常，确保在异常处理中也能正确离开临界区
5. **防止[[3.10 线程死锁|死锁]]**：避免在持有一个临界区时尝试获取另一个临界区（嵌套锁定）

---

## 关键要点

1. **CRITICAL_SECTION 是用户模式同步对象**，比内核对象更快
2. **仅限进程内同步**，不能用于跨进程
3. **四个核心函数**：Initialize、Enter、Leave、Delete
4. **无放弃通知**：线程异常终止可能导致死锁
5. **递归安全**：同一线程可以多次进入同一临界区
