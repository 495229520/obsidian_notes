# 3.11 线程同步机制总结

## 四种同步机制对比

| 比较项 | [[3.4 互斥量\|互斥量 (Mutex)]] | [[3.6 事件对象与线程同步\|事件对象 (Event)]] | [[3.7 信号量\|信号量 (Semaphore)]] | [[3.9 关键代码段\|关键代码段 (CriticalSection)]] |
|--------|------|------|------|------|
| **是否为内核对象** | 是 | 是 | 是 | **否** |
| **速度** | 较慢 | 较慢 | 慢 | **快** |
| **多个进程中的线程同步** | 支持 | 支持 | 支持 | **不支持** |
| **发生[[3.10 线程死锁\|死锁]]** | 否 | 否 | 否 | **是** |
| **组成** | 1. 一个线程ID，用来标识哪个线程拥有该互斥量<br>2. 一个计数器，用来指明该线程用于互斥对象的次数 | 1. 一个使用计数<br>2. 一个布尔值：用来标识该事件是自动重置还是人工重置<br>3. 一个布尔值：标识该事件处于有信号状态还是无信号状态 | 1. 一个使用计数<br>2. 最大资源数<br>3. 标识当前可用的资源数 | 一个小代码段；在代码能够执行前，必须占用对某些资源的访问权 |
| **相关函数** | `CreateMutex`<br>`WaitForSingleObject`<br>`ReleaseMutex` | `CreateEvent`<br>`ResetEvent`<br>`WaitForSingleObject`<br>`SetEvent` | `CreateSemaphore`<br>`WaitForSingleObject`<br>`ReleaseSemaphore` | `InitializeCriticalSection`<br>`EnterCriticalSection`<br>`LeaveCriticalSection`<br>`DeleteCriticalSection` |
| **注意事项** | 谁拥有互斥对象，谁释放；如果多次在同一个线程中请求同一个互斥对象，需要多次调用 `ReleaseMutex` | 为了实现线程间的同步，不应该使用人工重置，应该把第二个参数设置 false；设置为自动重置 | 它允许多个线程在同一时间访问同一个资源；但是需要限制访问此资源的最大线程数目 | 防止[[3.10 线程死锁\|死锁]]：使用多个关键代码段变量的时候 |
| **类比** | 一把钥匙 | 钥匙（自动/人工） | 停车场和保安 | 电话亭 |
| **句柄** | 不需要句柄 (HANDLE) | 需要句柄 (HANDLE)，必须使用 `CloseHandle` 关闭 | 需要句柄 (HANDLE)，必须使用 `CloseHandle` 关闭 | 不需要句柄 |

## Mutex（互斥量）详解

### 重点：独占性和所有权

- **用途**：当您有一段代码（[[3.8 临界区|临界区]]）需要访问或修改一个共享变量时，您使用 Mutex 来锁定这段代码

### 工作方式

```
线程 A 获取 Mutex ──→ Mutex 被锁定 ──→ 线程 B 尝试获取 Mutex 时被阻塞
       │
       ↓
线程 A 完成工作后释放 Mutex ──→ 线程 B 被唤醒并获取 Mutex
```

### 特点

- 具有**所有权**概念：只有持有者能释放
- 可**跨进程**使用（通过命名）
- 如果拥有 Mutex 的线程异常终止，Mutex 会进入"已放弃"状态，可以被检测到

## Event Object（事件对象）详解

### 重点：通知和状态

- **用途**：当您需要确保**线程 B 必须在线程 A 完成特定操作后才能开始执行**时，您使用 Event

### 工作方式

```
Event 初始设置为"无信号" ──→ 线程 B 在 WaitForSingleObject 处阻塞
       │
       ↓
线程 A 完成前置操作 ──→ 线程 A 调用 SetEvent ──→ Event 变为"有信号"
       │
       ↓
线程 B 被唤醒并继续执行
```

### 两种重置模式

| 模式 | 描述 |
|------|------|
| **自动重置** | 唤醒一个等待线程后自动变回无信号状态 |
| **人工重置** | 需要手动调用 `ResetEvent` 重置，可同时唤醒多个线程 |

## 选择指南

| 场景 | 推荐机制 | 理由 |
|------|----------|------|
| 同一进程内高性能同步 | [[3.9 关键代码段\|CriticalSection]] | 不涉及内核切换，速度最快 |
| 跨进程同步 | [[3.4 互斥量\|Mutex]] | 支持命名，可跨进程访问 |
| 线程间顺序控制/通知 | [[3.6 事件对象与线程同步\|Event]] | 适合生产者-消费者模式的通知场景 |
| 限制并发访问数量 | [[3.7 信号量\|Semaphore]] | 计数机制适合资源池管理 |

---

## 关键要点

1. **CriticalSection 最快**：不是内核对象，无需内核态切换
2. **CriticalSection 有死锁风险**：需要注意锁的获取顺序
3. **Mutex 有所有权**：谁获取谁释放，可追踪持有者
4. **Event 用于通知**：适合"等待某个条件满足"的场景
5. **Semaphore 用于计数**：适合限制并发访问数量
