# 3.8 临界区 (Critical Section)

## 简介

临界区是指程序中访问**共享资源 (Shared Resources)** 的那段代码。它之所以"临界"，是因为这段代码在任何时刻最多只能被一个线程执行。

---

## 核心定义

| 特性 | 描述 |
|------|------|
| **代码区域** | 临界区不是一个同步对象，而是一段代码 |
| **访问对象** | 这段代码操作一个或多个共享资源（例如全局变量、共享内存、文件、打印机等） |
| **互斥要求** | 临界区必须确保互斥访问 (Mutual Exclusion)。即：当一个线程在执行这段代码时，其他任何线程都必须被阻止进入这段代码 |

---

## 竞态条件 (Race Condition)

如果多个线程同时进入临界区并修改共享数据，就会发生**竞态条件 (Race Condition)**，导致不可预测和错误的结果。

### 经典案例：num++ 的非原子性

假设全局变量 `num = 10`，两个线程同时执行 `num++`：

| 线程 1 | 线程 2 | 共享资源 num |
|--------|--------|--------------|
| 1. 读取 num (10) | | 10 |
| | 1. 读取 num (10) | 10 |
| 2. 计算 10 + 1 = 11 | | 10 |
| | 2. 计算 10 + 1 = 11 | 10 |
| 3. 写入 num (11) | | 11 |
| | 3. 写入 num (11) | 11 (预期应为 12) |

> 如果没有同步机制保护，两次递增操作最终只生效了一次。临界区保护就是为了防止这种情况发生。

这个问题的根本原因是 `num++` 不是[[15.3 atomic|原子操作]]，它实际上包含三个步骤：读取、计算、写入。在多线程环境下，这三个步骤可能被其他线程打断。

---

## 实现临界区保护

```cpp
// ** 加锁操作 ** (Lock/Wait)
// 只有获取锁的线程才能通过
{
    // 这段代码就是【临界区】
    // 访问和修改共享资源（例如：num++）
}
// ** 解锁操作 ** (Unlock/Release)
```

## 常用同步工具比较

| 工具名称 | 操作系统/库 | 保护临界区的方式 |
|----------|-------------|------------------|
| [[3.4 互斥量\|互斥量 (Mutex)]] | Windows API (`CreateMutex`) / POSIX (`pthread_mutex_t`) | 最常用。通过独占所有权实现互斥 |
| [[3.9 关键代码段\|临界区对象 (Critical Section)]] | Windows API (`CRITICAL_SECTION`) | 一种比 Mutex 更轻量级的，仅用于进程内线程同步的机制 |
| [[3.7 信号量\|信号量 (Semaphore)]] | Windows API / POSIX (`sem_t`) | 如果临界区可以允许多个（N 个）线程同时访问，可以使用信号量限制访问数量 |

---

## 临界区 vs 原子操作

对于简单的计数器操作（如 `num++`），可以使用[[15.3 atomic]]来替代显式的临界区保护：

```cpp
#include <atomic>

std::atomic<int> counter{0};

// 多线程安全的自增
counter++;  // 原子操作，无需额外同步
```

但对于复杂的操作（涉及多个变量或多个步骤），仍然需要使用互斥量或其他同步机制来保护临界区。

---

## 关键要点

1. **临界区是代码段**，不是同步对象，是需要保护的共享资源访问区域
2. **竞态条件**是多线程同时访问临界区导致的错误
3. **必须使用同步机制**（互斥量、临界区对象、信号量等）来保护临界区
4. **简单操作可用原子类型**，复杂操作需要显式同步
5. **临界区应尽量短小**，减少锁持有时间以提高并发性能
