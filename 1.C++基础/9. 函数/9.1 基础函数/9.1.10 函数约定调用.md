# 9.1.10 函数调用约定 (Calling Conventions)

> 函数调用约定定义了函数调用时参数如何传递、返回值如何获取、以及谁负责清理栈空间。这些约定直接影响[[1.2 函数栈帧建立与销毁原理|函数栈帧]]的建立和销毁方式。

---

## 1. 调用约定的核心规则

| 规则 | 说明 |
|------|------|
| **参数传递顺序** | 从左向右传递还是从右向左传递 |
| **参数传递机制** | 指定参数是通过寄存器传递还是堆栈传递 |
| **堆栈清理责任** | 跳转到函数之后需要压栈分配空间，那么最后是由调用者还是被调用者对函数进行堆栈的清理呢 |
| **名称修饰** | 函数名称如何修改来支持函数重载 |

---

## 2. 常用调用约定

### 2.1 `__cdecl`（默认约定）

C 语言的默认约定，也是许多 x86 C++ 编译器的默认约定。

| 特性 | 说明 |
|------|------|
| **参数传递** | 参数从右到左压入堆栈 |
| **堆栈清理** | 调用者 (caller) 负责清理堆栈 |
| **返回值** | 通常通过 EAX/RAX 寄存器返回（整数/指针），浮点数通过浮点寄存器（x87 ST0 或 XMM0） |
| **优点** | 支持可变参数函数（如 `printf`），因为调用者知道传递了多少参数，可以正确清理 |
| **常见用途** | C 语言的默认约定，也是许多 x86 C++ 编译器的默认约定。在需要 C/C++ 互操作且需要支持可变参数时使用 |
| **名字修饰** | 通常在函数名前加下划线 `_`（例如 `_my_function`） |

### 2.2 `__stdcall`

| 特性 | 说明 |
|------|------|
| **参数传递** | 参数从右到左压入堆栈 |
| **堆栈清理** | 被调用者 (callee) 负责清理堆栈 |
| **返回值** | 同 `__cdecl` |
| **缺点** | 不支持可变参数函数，因为被调用者不知道实际传递了多少参数，无法正确清理 |
| **常见用途** | Win32 API 函数（Windows 核心 API）的标准约定 |
| **名字修饰** | 通常在函数名前加下划线 `_`，并在末尾附加 `@` 和参数总字节数（例如 `_MyFunction@8`） |

### 2.3 `__fastcall`

| 特性 | 说明 |
|------|------|
| **参数传递** | 尝试将前几个（通常是两个）参数通过寄存器传递（例如，在 x86 MSVC 上是 ECX 和 EDX），其余参数从右到左压入堆栈 |
| **堆栈清理** | 通常是被调用者 (callee) 负责清理堆栈上传递的参数 |
| **返回值** | 同 `__cdecl` |
| **优点** | 通过使用寄存器传递参数，理论上可以提高性能 |
| **常见用途** | 性能敏感的代码，但由于编译器实现差异，跨编译器使用可能存在问题 |
| **名字修饰** | 通常在函数名前加 `@`，并在末尾附加 `@` 和参数总字节数（例如 `@MyFunction@8`） |

### 2.4 `__thiscall`

| 特性 | 说明 |
|------|------|
| **参数传递** | `this` 指针通常通过特定寄存器传递（在 x86 MSVC 上是 ECX），其他参数通常从右到左压入堆栈（类似 `__stdcall`） |
| **堆栈清理** | 通常是被调用者 (callee) 负责清理堆栈（除 `this` 指针外的参数） |
| **注意** | `__thiscall` 不是一个可以显式使用的标准关键字，它是编译器内部处理非静态成员函数的一种方式，具体实现可能因编译器和平台而异 |
| **常见用途** | C++ 编译器内部用于非静态成员函数的调用 |

### 2.5 `__vectorcall`

| 特性 | 说明 |
|------|------|
| **参数传递** | `__fastcall` 的扩展，旨在更有效地传递 SIMD 矢量类型（如 `__m128`, `__m256`）参数，会使用更多的 SSE/AVX 寄存器 |
| **堆栈清理** | 被调用者 (callee) |
| **常见用途** | 需要传递 SIMD 类型参数并追求高性能的场景 |

---

## 3. 调用约定对比表

| 约定 | 参数顺序 | 堆栈清理 | 寄存器参数 | 可变参数 |
|------|---------|---------|-----------|---------|
| `__cdecl` | 右→左 | 调用者 | 无 | 支持 |
| `__stdcall` | 右→左 | 被调用者 | 无 | 不支持 |
| `__fastcall` | 右→左 | 被调用者 | 前2个 | 不支持 |
| `__thiscall` | 右→左 | 被调用者 | this 指针 | 不支持 |
| `__vectorcall` | 右→左 | 被调用者 | 更多 | 不支持 |

## 4. 汇编层面的差异

### `__cdecl` 示例

```cpp
int __cdecl add(int a, int b) {
    return a + b;
}

int main() {
    int result = add(3, 5);
}
```

```asm
; 调用 add(3, 5)
push 5          ; 参数 b 先入栈（右到左）
push 3          ; 参数 a 后入栈
call add
add esp, 8      ; 调用者清理栈（2个int = 8字节）
```

### `__stdcall` 示例

```cpp
int __stdcall add(int a, int b) {
    return a + b;
}
```

```asm
; 调用 add(3, 5)
push 5          ; 参数 b
push 3          ; 参数 a
call add
; 无需清理栈，被调用者已经清理

; add 函数内部
add:
    ; ... 函数体 ...
    ret 8       ; 返回并清理 8 字节的参数
```

---

## 5. 关键要点

1. **默认约定**：C/C++ 默认使用 `__cdecl`
2. **Win32 API**：使用 `__stdcall`
3. **可变参数**：只有 `__cdecl` 支持（如 `printf`）
4. **性能优化**：`__fastcall` 和 `__vectorcall` 使用寄存器传参
5. **C++ 成员函数**：编译器自动使用 `__thiscall` 处理 `this` 指针
