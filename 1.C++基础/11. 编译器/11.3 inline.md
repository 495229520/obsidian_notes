# 11.3 inline 内联

## 1. 简介

`inline` 是一个关键字，用于建议编译器将函数调用**内联展开**（Inline Expansion），即用函数体直接替换函数调用，从而消除函数调用的开销。

> **重要**：`inline` 只是对编译器的**建议**，编译器可以忽略这个建议。

---

## 2. 内联展开的原理

### 普通函数调用

```cpp
int add(int a, int b) {
    return a + b;
}

int main() {
    int result = add(3, 5);  // 函数调用开销：压栈、跳转、返回
}
```

### 内联展开后

```cpp
inline int add(int a, int b) {
    return a + b;
}

int main() {
    int result = 3 + 5;  // 直接替换，无函数调用开销
}
```

---

## 3. 语法

### 3.1 内联函数

```cpp
// 方式一：在定义处使用 inline
inline int square(int x) {
    return x * x;
}

// 方式二：类内定义的成员函数隐式 inline
class Counter {
public:
    int getValue() const { return value; }  // 隐式 inline
private:
    int value;
};
```

### 3.2 inline 变量（C++17）

```cpp
// C++17: 可以在头文件中定义 inline 变量
inline int global_counter = 0;

// 类的静态成员
class Config {
public:
    inline static int max_size = 100;  // C++17
};
```

## 4. inline 与 ODR

`inline` 的重要作用是允许在多个[[11.2 编译器基本概念#翻译单元|翻译单元]]中定义同一个函数/变量，而不违反 [[11.1 ODR唯一定义规则|ODR]]。

### 问题：头文件中定义普通函数

```cpp
// my_utils.h
#ifndef MY_UTILS_H
#define MY_UTILS_H

int add(int a, int b) {  // 非 inline
    return a + b;
}

#endif

// file1.cpp
#include "my_utils.h"  // 定义了 add

// file2.cpp
#include "my_utils.h"  // 又定义了 add
// 链接错误：multiple definition of 'add'
```

### 解决方案：使用 inline

```cpp
// my_utils.h
#ifndef MY_UTILS_H
#define MY_UTILS_H

inline int add(int a, int b) {  // inline 函数
    return a + b;
}

#endif

// file1.cpp 和 file2.cpp 都可以包含此头文件
// 链接器会合并重复的定义
```

## 5. 类成员函数的 inline

### 5.1 类内定义（隐式 inline）

```cpp
class Counter {
private:
    int count = 0;

public:
    // 类内定义的成员函数隐式为 inline
    int getCount() const { return count; }
    void increment() { ++count; }
};
```

### 5.2 类外定义（需显式 inline）

```cpp
// counter.h
class Counter {
private:
    int count = 0;

public:
    int getCount() const;  // 声明
};

// 方式一：在头文件中定义（需要 inline）
inline int Counter::getCount() const {
    return count;
}

// 方式二：在 .cpp 文件中定义（不需要 inline）
// counter.cpp
int Counter::getCount() const {
    return count;
}
```

## 6. 内联的优缺点

### 优点

| 优点 | 说明 |
|------|------|
| **消除调用开销** | 无需压栈、跳转、返回 |
| **启用更多优化** | 内联后编译器可进行常量传播、死代码消除等 |
| **减少间接跳转** | 对 CPU 分支预测更友好 |

### 缺点

| 缺点 | 说明 |
|------|------|
| **代码膨胀** | 函数体在每个调用点复制，增加可执行文件大小 |
| **编译时间增加** | 头文件修改导致更多重新编译 |
| **可能降低缓存命中率** | 代码体积增大可能影响指令缓存 |

---

## 7. 编译器的内联决策

编译器会根据以下因素决定是否内联：

### 7.1 倾向于内联的情况

- 函数体很小（几行代码）
- 函数被频繁调用
- 编译优化级别较高（`-O2`、`-O3`）

### 7.2 不会内联的情况

- 函数体很大
- 递归函数（除非尾递归优化）
- 包含复杂控制流（如多层循环）
- 通过函数指针调用
- 虚函数（除非编译器能确定具体类型）

### 7.3 强制内联提示

```cpp
// GCC/Clang
__attribute__((always_inline)) inline void must_inline() { }

// MSVC
__forceinline void must_inline() { }
```

> **注意**：即使使用强制内联，编译器在某些情况下仍可能拒绝内联。

## 8. inline vs [[11.4 define|宏]]

| 特性 | `inline` 函数 | `#define` 宏 |
|------|---------------|--------------|
| **类型检查** | 有 | 无 |
| **调试支持** | 可调试 | 难以调试 |
| **作用域** | 遵循作用域规则 | 全局替换 |
| **副作用** | 参数只求值一次 | 可能多次求值 |
| **代码安全** | 安全 | 容易出错 |

### 宏的副作用问题

```cpp
// 宏定义 - 有问题
#define SQUARE(x) ((x) * (x))
int a = 5;
int result = SQUARE(a++);  // 展开为 ((a++) * (a++))，a 被递增两次！

// inline 函数 - 安全
inline int square(int x) { return x * x; }
int result2 = square(a++);  // a 只递增一次
```

## 9. constexpr 与 inline

C++11 起，`constexpr` 函数隐式为 `inline`：

```cpp
constexpr int factorial(int n) {
    return (n <= 1) ? 1 : n * factorial(n - 1);
}
// factorial 隐式为 inline，可以在头文件中定义
```

C++17 起，`constexpr` 静态数据成员隐式为 `inline`：

```cpp
struct Config {
    static constexpr int MAX_SIZE = 100;  // C++17: 隐式 inline
};
```

## 10. 最佳实践

### 适合使用 inline 的场景

1. **简单的 getter/setter 函数**
   ```cpp
   inline int getValue() const { return value; }
   ```

2. **短小的工具函数**
   ```cpp
   inline int max(int a, int b) { return (a > b) ? a : b; }
   ```

3. **头文件中需要定义的函数**
   ```cpp
   // header-only 库
   inline void utility_function() { /* ... */ }
   ```

### 不适合使用 inline 的场景

1. 函数体很大
2. 包含循环或复杂逻辑
3. 很少被调用的函数
4. 需要取函数地址的场景

---

## 11. 关键要点

1. **建议性质**：`inline` 是建议，编译器可以忽略
2. **ODR 豁免**：`inline` 允许在多个翻译单元中定义同一函数
3. **隐式 inline**：类内定义的成员函数、`constexpr` 函数
4. **C++17 新特性**：`inline` 变量可以在头文件中定义
5. **优于宏**：`inline` 函数比宏更安全、可调试
6. **权衡利弊**：消除调用开销 vs 代码膨胀
