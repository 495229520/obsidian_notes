# 进程的虚拟地址空间 (Process Virtual Address Space)

## 1. 简介

> **任何的编程语言** => 都会产生两种东西：**指令**和**数据**
>
> C++ 在[[1.1 程序编译与链接原理|编译和链接]]之后 => `xxx.exe` => 存放在磁盘上 => 加载在内存中

---

## 2. 什么是虚拟地址空间？

一个形象的比喻：

| 描述 | 类型 |
|------|------|
| 它存在，你能看得见 | 它是**物理**的 |
| 它存在，你看不见 | 它是**透明**的 |
| 它不存在，你却看得见 | 它是**虚拟**的 |
| 它不存在，你也看不见 | 它被**删除**了！|

虚拟地址空间就是第三种情况：**它实际上不存在于物理内存中，但进程却能"看到"并使用它**。操作系统通过内存管理单元（MMU）将虚拟地址映射到实际的物理地址。

---

## 3. X86 32位系统的地址空间布局

> **关键点**：X86 是 32 位系统！
>
> - Linux 系统会给当前进程分配一个 **2³² 字节大小**的虚拟地址空间，容量为 **4GB**
> - 每一个进程的**用户空间是私有的**，但**内核空间是公有的**！
> - 因此[[11.2 进程间通信IPC|进程之间的通讯]]需要通过内核空间的内存进行读写操作（如[[11.3 管道通讯|匿名管道通讯]]）

### 3.2 各段详细说明

| **段名**      | **地址范围**                    | **内容**                                  | **特性**            |
| ----------- | --------------------------- | --------------------------------------- | ----------------- |
| **不可访问**    | `0x00000000` 起始             | 保留区域                                    | 访问会导致系统崩溃（段错误）    |
| **.text**   | `0x08048000` 起始             | 存放汇编指令（机器码）                             | 只读、可执行            |
| **.rodata** | 紧随 .text                    | 只读数据（常量字符串等）                            | 只读，修改会报错          |
| **.data**   | 紧随 .rodata                  | 已初始化且初始值**不为零**的全局/静态变量                 | 可读写               |
| **.bss**    | 紧随 .data                    | 未初始化或初始化**为零**的全局/静态变量                  | 可读写，不占磁盘空间        |
| **.heap**   | 向上生长                        | 动态分配的内存（`malloc`/`new`）                 | 从低地址向高地址增长        |
| **共享库**     | 中间区域                        | `*.dll`（Windows）/ `*.so`（Linux）         | 动态链接库映射           |
| **.stack**  | 向下生长                        | [[1.2 函数栈帧建立与销毁原理\|函数栈帧建立与销毁原理函数]]、局部变量 | 从高地址向低地址增长，每个线程独有 |
| **内核空间**    | `0xC0000000` - `0xFFFFFFFF` | 操作系统内核代码和数据                             | 所有进程共享，用户态无法直接访问  |

### 3.3 内核空间分区

| **分区** | **用途** |
|----------|----------|
| ZONE_DMA | 用于 DMA（直接内存访问）操作，通常是低端物理内存 |
| ZONE_NORMAL | 内核可以直接映射的正常内存区域 |
| ZONE_HIGHMEM | 高端内存，内核需要特殊方式才能访问 |

## 4. 变量存储位置详解

### 4.1 存储规则总结

| **变量类型** | **存储位置** | **说明** |
|-------------|-------------|----------|
| 全局变量（已初始化且非零） | `.data` | 产生符号，链接时可见 |
| 全局变量（未初始化或为零） | `.bss` | 产生符号，链接时可见 |
| `static` 全局变量 | `.data` / `.bss` | 产生符号，但仅本文件可见 |
| `static` 局部变量 | `.data` / `.bss` | 产生符号，生命周期为整个程序 |
| 普通局部变量 | `.stack`（栈） | **不产生符号**，直接生成汇编指令 |
| 常量字符串 | `.rodata` | 只读，修改会导致段错误 |

### 4.2 代码示例

```cpp
#include <iostream>
using namespace std;

// ============ 全局变量 ============
int gdata1 = 10;         // 数据存储在 .data（已初始化且非零）
int gdata2 = 0;          // 数据存储在 .bss （初始化为零）
int gdata3;              // 数据存储在 .bss （未初始化）

// ============ 静态全局变量 ============
static int gdata4 = 10;  // 数据存储在 .data
static int gdata5 = 0;   // 数据存储在 .bss
static int gdata6;       // 数据存储在 .bss

int main()               // 汇编指令存储在 .text 中
{
    // ============ 普通局部变量 ============
    int a = 12;          // 汇编指令存储在 .text 中（栈上分配）
    int b = 0;           // 汇编指令存储在 .text 中
    int c;               // 汇编指令存储在 .text 中

    // ============ 静态局部变量 ============
    static int d = 13;   // 数据存储在 .data
    static int e = 0;    // 数据存储在 .bss
    static int f;        // 数据存储在 .bss

    // 汇编指令存储在 .text 中
    std::cout << a << std::endl;
}
```

### 4.3 关键理解

1. **静态/非静态全局变量**：产生符号，存放在 `.data` 或 `.bss` 段
2. **非静态局部变量**：**不产生符号**，直接产生汇编指令操作栈空间
3. **静态局部变量**：虽然作用域在函数内，但存储在 `.data` / `.bss`，生命周期为整个程序

## 5. 常见问题

### 5.1 为什么访问 `0x00000000` 附近会崩溃？

这是操作系统故意保留的**空指针陷阱区**。当程序错误地解引用空指针（`NULL`）时，会访问到这个区域，操作系统会立即终止程序并报告段错误（Segmentation Fault）。

### 5.2 为什么 `.rodata` 不能修改？

```cpp
char *p = "hello world";  // 字符串存储在 .rodata
*p = 'a';                 // 错误！会导致段错误
```

`.rodata` 段被操作系统标记为只读，任何写入尝试都会触发硬件异常。

### 5.3 堆和栈为什么相向生长？

- **栈**从高地址向低地址生长
- **堆**从低地址向高地址生长

这种设计最大化了两者的可用空间：它们可以在中间"相遇"，灵活分配内存，避免固定边界带来的浪费。

## 6. 补充：64位系统的变化

在 64 位系统（x86-64）中：
- 虚拟地址空间理论上为 **2⁶⁴ 字节**（16 EB）
- 实际上当前 CPU 只使用 **48 位**地址（256 TB）
- 用户空间和内核空间各占一半，中间有巨大的"空洞"
- 起始地址不再是 `0x08048000`，而是更高的地址（如 `0x400000`）
