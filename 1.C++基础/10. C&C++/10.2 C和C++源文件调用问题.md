# 10.2 C 和 C++ 源文件调用问题

## 1. 简介

在 C 和 C++ 混合编程中，由于两种语言的编译和链接机制存在差异，会遇到各种调用问题。本节详细分析这些问题及其解决方案。

---

## 2. 常见链接错误

### 2.1 未定义引用 (Undefined Reference)

**问题描述**：C++ 代码调用 C 函数时，链接器找不到符号。

```
error: undefined reference to `print_message(char const*)'
```

**原因**：C++ 编译器对函数名进行了[[11.2 编译器基本概念#名称修饰|名称修饰]]，生成的符号名与 C 编译器生成的不匹配。

**解决方案**：使用 [[10.1 C和C++联调#`extern "C"` 关键字|extern "C"]]

```cpp
// 错误方式
void print_message(const char* msg);  // C++ 会修饰为 _Z13print_messagePKc

// 正确方式
extern "C" void print_message(const char* msg);  // 保持为 print_message
```

### 2.2 符号重复定义 (Multiple Definition)

**问题描述**：多个源文件包含同一个头文件时，出现重复定义错误。

```
error: multiple definition of `global_var'
```

**原因**：在头文件中定义了变量或函数（而非声明）。

**解决方案**：

```cpp
// header.h - 错误方式
int global_var = 10;  // 定义，会导致重复定义

// header.h - 正确方式
extern int global_var;  // 声明

// source.cpp - 在一个源文件中定义
int global_var = 10;
```

## 3. 头文件包含顺序问题

### 问题示例

```cpp
// 可能导致问题的包含顺序
#include <windows.h>    // 可能定义了某些宏
#include "my_header.h"  // 可能与系统宏冲突
```

### 最佳实践

```cpp
// 推荐的包含顺序
#include "my_header.h"      // 1. 本项目头文件
#include <iostream>         // 2. C++ 标准库
#include <cstdio>           // 3. C 标准库（C++ 风格）
#include <stdio.h>          // 4. C 标准库（C 风格）
#include <windows.h>        // 5. 系统/平台头文件
#include <third_party.h>    // 6. 第三方库
```

## 4. 数据类型兼容性问题

### 4.1 bool 类型

```cpp
// C 语言（C99 之前没有 bool）
#include <stdbool.h>  // C99 引入
typedef int BOOL;     // 传统 C 方式

// C++ 语言
bool flag = true;     // 原生支持
```

### 4.2 结构体差异

```c
// C 语言 - 需要 struct 关键字
struct Point {
    int x, y;
};
struct Point p1;  // C 必须加 struct

// C++ 语言
Point p2;         // C++ 可以省略 struct
```

### 4.3 空指针

```c
// C 语言
#define NULL ((void*)0)
void* ptr = NULL;

// C++ 语言
nullptr  // C++11 引入，类型安全
```

## 5. 调用约定问题

不同编译器和平台可能使用不同的[[9.1.10 函数约定调用|调用约定]]：

```cpp
// Windows 平台特定调用约定
extern "C" {
    __cdecl void cdecl_func();      // C 默认调用约定
    __stdcall void stdcall_func();  // Windows API 常用
}
```

## 6. 异常处理边界

C 语言不支持 C++ 异常，在边界处必须处理：

```cpp
// C++ 实现
extern "C" int safe_divide(int a, int b) {
    try {
        if (b == 0) {
            throw std::runtime_error("Division by zero");
        }
        return a / b;
    } catch (...) {
        // 不能让异常传播到 C 代码
        return -1;  // 返回错误码
    }
}
```

## 7. 内存管理问题

### 7.1 不要混用分配函数

```cpp
// 错误示例
char* str = (char*)malloc(100);
delete[] str;  // 错误！malloc 不能用 delete

// 正确示例
// 方式1：C 风格
char* str1 = (char*)malloc(100);
free(str1);

// 方式2：C++ 风格
char* str2 = new char[100];
delete[] str2;
```

### 7.2 跨边界传递内存

```cpp
// 头文件声明分配和释放函数
extern "C" {
    void* lib_alloc(size_t size);
    void lib_free(void* ptr);
}

// 使用时配对调用
void* data = lib_alloc(1024);
// ... 使用 data ...
lib_free(data);  // 使用配套的释放函数
```

## 8. 静态初始化顺序问题

### 问题

C++ 全局/静态对象的初始化顺序在不同[[11.2 编译器基本概念#翻译单元|翻译单元]]之间是未定义的。

```cpp
// file1.cpp
std::string global_str = "Hello";

// file2.cpp
extern std::string global_str;
int len = global_str.length();  // 危险！global_str 可能还未初始化
```

### 解决方案

使用函数内静态变量（Construct On First Use）：

```cpp
// 安全的方式
std::string& get_global_str() {
    static std::string str = "Hello";
    return str;
}
```

## 9. 诊断与调试

### 查看符号名

```bash
# Linux/macOS
nm -C program.o          # -C 选项解码修饰后的名称
c++filt _Z13print_messagePKc  # 解码单个符号

# Windows (MSVC)
dumpbin /symbols program.obj
undname ?print_message@@YAXPEBD@Z
```

### 常见符号前缀

| 前缀 | 含义 |
|------|------|
| `_Z` | GCC/Clang C++ 符号 |
| `?` | MSVC C++ 符号 |
| `_` | C 符号（某些平台） |

## 10. LNK4042 链接器警告

此链接器警告通常表示链接器发现一个或多个具有相同符号名称的目标文件，但这些目标文件是使用不同的[[11.2 编译器基本概念#翻译单元|编译单元]]（例如，不同的包含文件集或不同的编译器设置）进行编译的。

### 10.1 产生原因

混合 C 和 C++ 源文件可能导致 LNK4042 的场景：

**场景 1：C++ 调用 C，未使用 `extern "C"`**

```c
// my_c_file.c
int my_function(int x) {
    return x * 2;
}
```

```cpp
// my_cpp_file.cpp
#include <iostream>

int my_function(int x);  // 未使用 extern "C" 声明

int main() {
    int result = my_function(5);
    std::cout << "Result: " << result << std::endl;
    return 0;
}
```

问题分析：
- C++ 编译器会修饰名称 `my_function`（如 `?my_function@@YAHH@Z`）
- C 编译器不会修饰该名称
- 链接器在目标文件中查找修饰后的名称，但在 `my_c_file.obj` 中找不到
- 导致链接器错误（如 `LNK2001: 无法解析的外部符号`）

**场景 2：重复目标文件**

如果存在同名的 C 和 C++ 源文件（如 `example.c` 和 `example.cpp`），构建系统可能生成同名的目标文件（`example.obj`），链接时会检测到重复文件，导致 LNK4042 警告。

### 10.2 解决方案

1. **使用 `extern "C"`**：在 C++ 中调用 C 函数时，正确使用 [[10.1 C和C++联调#`extern "C"` 关键字|extern "C"]] 声明
2. **避免同名源文件**：确保 C 和 C++ 源文件使用不同的文件名
3. **分离构建输出**：配置构建系统将 C 和 C++ 的目标文件输出到不同目录

---

## 11. 关键要点

1. **链接错误**：通常由名称修饰不匹配导致，使用 `extern "C"` 解决
2. **头文件保护**：使用 `#ifdef __cplusplus` 创建兼容头文件
3. **类型兼容**：注意 `bool`、`struct`、`nullptr` 的差异
4. **异常边界**：C++ 异常不能传播到 C 代码
5. **内存管理**：`malloc/free` 和 `new/delete` 不要混用
6. **静态初始化**：跨翻译单元的初始化顺序未定义
7. **LNK4042 警告**：注意同名源文件和缺少 `extern "C"` 导致的链接问题
